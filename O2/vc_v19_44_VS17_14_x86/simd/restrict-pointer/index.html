    restrict-pointer¶

    Details

    Source: simd/restrict-pointer.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    restrict-pointer.c
    /*
 * Restrict pointers enable better optimization.
 *
 * The 'restrict' keyword promises that pointers don't alias,
 * allowing the compiler to optimize more aggressively.
 */

/* Without restrict - compiler must be conservative */
void copy_no_restrict(int *dst, const int *src, int n)
{
    /* dst and src MIGHT overlap, so compiler must handle that */
    for (int i = 0; i < n; i++) {
        dst[i] = src[i];
    }
}

/* With restrict - compiler knows no aliasing */
void copy_restrict(int *restrict dst, const int *restrict src, int n)
{
    /* dst and src guaranteed not to overlap
     * Compiler can use faster copy or vectorize freely
     */
    for (int i = 0; i < n; i++) {
        dst[i] = src[i];
    }
}

/* Multiplication example - dramatic difference */
void scale_no_restrict(float *result, const float *input,
                       const float *scale, int n)
{
    /* Compiler doesn't know if result aliases input or scale */
    for (int i = 0; i < n; i++) {
        result[i] = input[i] * (*scale);
    }
    /* Must reload *scale each iteration! */
}

void scale_restrict(float *restrict result,
                    const float *restrict input,
                    const float *restrict scale, int n)
{
    /* Compiler knows result doesn't alias scale */
    for (int i = 0; i < n; i++) {
        result[i] = input[i] * (*scale);
    }
    /* Can load *scale once and reuse */
}

    Explanation¶

    the key insight here is how the restrict keyword dramatically changes the compiler's optimization strategy.

    The Core Difference: What restrict Enables¶

    The restrict keyword is a promise to the compiler: "these pointers will never point to overlapping memory." Without this promise, the compiler must assume the worst casethat pointers might overlapand generate conservative code. With restrict, the compiler can generate much faster code.

    copy_restrict vs copy_no_restrict¶

    The most striking difference appears in the copy_restrict function:

    rep movsd

    This is a single instruction that copies multiple 32-bit integers (movsd = "move scalar double-word") in a loop. The rep prefix repeats this operation, making it extremely efficient for bulk memory copying.

    In contrast, copy_no_restrict uses a manual loop with explicit bounds checking and conditional jumps. Why? Because without restrict, the compiler must handle the possibility that dst and src overlap. If they do overlap, copying forward (from start to end) could overwrite data before it's read. The compiler generates defensive code to handle this case, which is much slower.

    Key insight: rep movsd is a specialized CPU instruction optimized for exactly this task. It's faster than any loop the compiler could generate because it's a single operation that the CPU can execute very efficiently.

    scale_restrict vs scale_no_restrict¶

    The multiplication example shows an even more dramatic difference:

    Without restrict: The compiler must reload *scale (the value pointed to by the scale pointer) on every loop iteration. Why? Because without restrict, the compiler can't be sure that result[i] = ... doesn't modify the memory location that scale points to. So it must re-read *scale each time to be safe.

    With restrict: The compiler loads *scale once before the loop and reuses that value throughout. This is much faster because: - Memory reads are expensive (they might require waiting for data from RAM) - The CPU can keep the value in a register, which is nearly instant to access - Eliminating redundant memory reads is one of the biggest performance wins in optimization

    Why This Matters¶

    The restrict keyword lets the compiler make aggressive optimizations that would be unsafe without it. The compiler isn't being lazyit's being correct. Without restrict, generating code that assumes no aliasing would be wrong if the pointers actually did overlap.

    This is compiled with -O2 (moderate optimization), so the compiler is already quite aggressive. The restrict keyword gives it permission to be even more aggressive by eliminating the need for defensive checks.
