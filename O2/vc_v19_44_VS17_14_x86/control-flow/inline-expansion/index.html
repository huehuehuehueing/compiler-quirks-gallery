    inline-expansion¶

    Details

    Source: control-flow/inline-expansion.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    inline-expansion.c
    /*
 * Function inlining: small functions are expanded at call site.
 *
 * Eliminates call overhead and enables further optimizations
 * by exposing the function body to the caller's context.
 */

/* Tiny function - almost certainly inlined */
static inline int square(int x)
{
    return x * x;
}

/* Using static helps inlining (internal linkage) */
static int cube(int x)
{
    return x * x * x;
}

/* Wrapper that uses inlined functions */
int compute(int x)
{
    /* After inlining, this becomes:
     * return (x * x) + (x * x * x);
     * Which may further simplify to:
     * return x * x * (1 + x);
     */
    return square(x) + cube(x);
}

/* Recursive functions are NOT inlined (infinite expansion) */
int fibonacci(int n)
{
    if (n <= 1) return n;
    return fibonacci(n-1) + fibonacci(n-2);
}

/* Large functions may not be inlined (code size cost) */
int large_function(int x)
{
    int result = 0;
    for (int i = 0; i < 100; i++) {
        result += x * i;
        result ^= (result >> 3);
        result += (result << 5);
    }
    return result;
}

    Explanation¶

    the key insight is that the compiler has inlined the small functions (square and cube) into compute, but kept the larger functions (fibonacci and large_function) as separate function calls. This demonstrates the compiler's inlining strategy.

    Function Inlining Strategy¶

    The compiler uses -O2 optimization, which enables function inlining â€” a technique where the compiler copies a function's body directly into the calling code instead of making a function call. This eliminates the overhead of jumping to another location and back.

    square and cube Functions¶

    These remain as separate functions in the assembly (they're still defined), but notice something important: they're never actually called. The compiler inlined them into compute.

    square function: 

    mov     eax, DWORD PTR _x$[esp-4]    ; Load parameter x into eax
imul    eax, eax                      ; Multiply eax by itself (x * x)
ret     0

    cube function: 

    mov     ecx, DWORD PTR _x$[esp-4]    ; Load parameter x into ecx
mov     eax, ecx                      ; Copy x to eax
imul    eax, ecx                      ; eax = x * x
imul    eax, ecx                      ; eax = (x * x) * x = xÂ³
ret     0

    The compute Function â€” Where Inlining Happens¶

    This is where the optimization becomes visible:

    mov     ecx, DWORD PTR _x$[esp-4]    ; Load x into ecx
lea     eax, DWORD PTR [ecx+1]       ; eax = x + 1 (using lea for address calculation)
imul    eax, ecx                      ; eax = (x + 1) * x
imul    eax, ecx                      ; eax = ((x + 1) * x) * x = xÂ² * (x + 1)
ret     0

    The key optimization here: Instead of computing (x * x) + (x * x * x) separately, the compiler recognized that both expressions share x * x, so it factored the math to x * x * (1 + x). This reduces the number of multiplications from three to two.

    The lea (Load Effective Address) instruction is used to compute x + 1 without a separate addition instruction â€” lea calculates addresses but can also perform simple arithmetic like addition.

    Functions NOT Inlined¶

    fibonacci Function¶

    This function cannot be inlined because it's recursive â€” it calls itself. Inlining a recursive function would create infinite expansion (the compiler would need to inline the call within the inlined code, which calls itself again, forever). So the compiler keeps it as a separate function:

    push    edi                           ; Save edi (caller-saved register)
mov     edi, DWORD PTR _n$[esp]      ; Load parameter n
cmp     edi, 1                        ; Compare n with 1
jg      SHORT $LN2@fibonacci          ; Jump if n > 1
mov     eax, edi                      ; Return n (base case)
pop     edi
ret     0

$LN2@fibonacci:                       ; Recursive case
lea     eax, DWORD PTR [edi-2]       ; Calculate n-2
push    esi                           ; Save esi
push    eax                           ; Push n-2 as argument
call    _fibonacci                    ; Call fibonacci(n-2)
lea     ecx, DWORD PTR [edi-1]       ; Calculate n-1
mov     esi, eax                      ; Save result in esi
push    ecx                           ; Push n-1 as argument
call    _fibonacci                    ; Call fibonacci(n-1)
add     esp, 8                        ; Clean up stack (2 arguments Ã— 4 bytes)
add     eax, esi                      ; Add the two results
pop     esi
pop     edi
ret     0

    The function uses edi and esi registers to preserve values across the two recursive calls.
