    static-memcpy¶

    Details

    Source: string-literals/static-memcpy.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    static-memcpy.c
    #include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[5];

    memcpy(buf, "AAAA", 4);
    buf[sizeof(buf) - 1] = '\0';

    /* make sure the above is not treated as dead code */
    puts(buf);
}

    Explanation¶

    the key insight here is that this code demonstrates stack buffer security protection - the compiler is adding extra checks to prevent buffer overflow attacks.

    The Core Pattern: Security Cookie Protection¶

    The most important thing to understand is that MSVC (Microsoft Visual C++) is protecting the local buffer buf from being overwritten by malicious code. It does this by:

    1. Creating a "security cookie" - a random value stored on the stack
    2. Checking it before returning - verifying the cookie wasn't corrupted
    3. Aborting if corrupted - calling @__security_check_cookie@4 to validate

    This is called stack buffer overflow protection and is a security feature enabled by default in MSVC.

    Step-by-Step Breakdown¶

    Stack Setup (lines 1-3): 

    sub     esp, 12                    ; Make room for local variables
mov     eax, DWORD PTR ___security_cookie  ; Load the security cookie
xor     eax, esp                   ; Mix it with current stack position
mov     DWORD PTR __$ArrayPad$[esp+12], eax ; Store it on stack

    The compiler allocates 12 bytes on the stack: 5 bytes for buf, 4 bytes for the security cookie (__$ArrayPad$), and 3 bytes of padding. The security cookie is XORed with the stack pointer (esp) to make it harder to predict.

    Buffer Initialization (lines 4-5): 

    lea     eax, DWORD PTR _buf$[esp+12]  ; Calculate address of buf
mov     DWORD PTR _buf$[esp+12], 1094795585  ; Write "AAAA" (0x41414141)

    The lea instruction (Load Effective Address) calculates where buf is located without actually reading from memory - it's just arithmetic. Then the compiler writes the 4-byte string "AAAA" directly as the hex value 0x41414141 (since 'A' = 0x41 in ASCII).

    Null Terminator and Function Call (lines 6-8): 

    push    eax                        ; Push buf's address (parameter for puts)
mov     BYTE PTR _buf$[esp+20], 0  ; Write null terminator at buf[4]
call    _puts                      ; Call puts() to print the string

    The buffer address is pushed onto the stack as a parameter for puts(), and the null terminator is written at position 4 (the last byte of the 5-byte buffer).

    Security Check Before Return (lines 9-12): 

    mov     ecx, DWORD PTR __$ArrayPad$[esp+16]  ; Retrieve stored cookie
add     esp, 4                     ; Adjust stack (from the push)
xor     ecx, esp                   ; XOR with current esp again
call    @__security_check_cookie@4 ; Verify cookie is unchanged

    Before returning, the compiler retrieves the security cookie and XORs it with the current stack pointer. If the cookie was corrupted (meaning a buffer overflow occurred), this check will fail and the program terminates.

    Cleanup and Return (lines 13-14): 

    add     esp, 12                    ; Deallocate local variables
ret     0                          ; Return to caller

    The stack is cleaned up and the function returns.

    Key Observations¶

    • Optimization applied: The compiler optimized the memcpy() call into a single mov instruction writing 0x41414141 directly - this is a common optimization when copying small, constant data.
    • Security overhead: The security cookie mechanism adds several instructions (load, XOR, store, retrieve, verify) that wouldn't be present in unprotected code. This is the trade-off for buffer overflow protection.
    • Calling convention: Parameters are passed via the stack (you see push eax before calling puts), which is typical for x86 calling conventions.
