    unrollme-2¶

    Details

    Source: loops/unrollme-2.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    unrollme-2.c
    #include <stdio.h>

void foo(void)
{
    int i;
    char buf[8];

    for (i = 0; i < sizeof(buf); i++)
    {
        buf[i] = 'A';
    }

    /*
     * by dereferencing buf this should force the compile to NOT
     * eliminate the above as dead code. loop will likely still
     * be unrolled
     */
    printf("%s", buf);
}

    Explanation¶

    the key insight is that the compiler has completely unrolled the loop and eliminated it entirely, replacing it with direct memory writes. Let me walk through what's happening.

    Loop Unrolling and Elimination¶

    The original C code has a loop that fills an 8-byte buffer with the character 'A' (ASCII value 65, or 0x41). Instead of generating actual loop code, the MSVC compiler with -O2 optimization has:

    1. Eliminated the loop entirely - there's no cmp, jne, or loop counter logic
    2. Unrolled it into direct writes - the buffer is filled with two mov instructions that write 0x41414141 (four 'A' characters packed into a 32-bit value)

    This is a classic optimization called loop unrolling: when the compiler knows the loop runs a fixed number of times (8 iterations), it can replace the loop with the equivalent sequence of operations written out explicitly.

    Stack Frame Setup and Security¶

    The function starts with security cookie protection (a Microsoft-specific feature):

    sub     esp, 12              ; allocate 12 bytes on stack
mov     eax, DWORD PTR ___security_cookie
xor     eax, esp
mov     DWORD PTR __$ArrayPad$[esp+12], eax

    This allocates space for the 8-byte buf array plus 4 bytes for a security cookie. The cookie is XORed with the stack pointer and stored to detect stack buffer overflows. This is why you see the security check at the end (call @__security_check_cookie@4).

    Buffer Initialization¶

    The actual buffer filling happens with two instructions:

    mov     DWORD PTR _buf$[esp+12], 1094795585       ; 41414141H
mov     DWORD PTR _buf$[esp+24], 1094795585       ; 41414141H

    The value 0x41414141 is eight ASCII 'A' characters packed into two 32-bit words (little-endian, so each byte is 0x41). This fills all 8 bytes of the buffer in just two writes instead of eight individual byte writes.

    Function Call Setup¶

    Before calling printf, the code:

    lea     eax, DWORD PTR _buf$[esp+12]    ; load address of buf
push    eax                              ; push buf pointer (2nd arg)
push    OFFSET `string'                  ; push format string (1st arg)
call    _printf

    The lea (Load Effective Address) instruction calculates the memory address of the buffer without actually reading from memoryit's just arithmetic to compute where buf is located. This address is then pushed as the second argument to printf, and the format string "%s" is pushed as the first argument. By convention in x86 calling, arguments are pushed right-to-left.

    Key Observations¶

    • Loop completely eliminated: The compiler recognized the loop runs exactly 8 times and unrolled it into direct writes
    • Packed writes: Instead of 8 byte-sized writes, the compiler uses 2 word-sized writes with the value 0x41414141
    • Security overhead: The security cookie adds a few extra instructions for buffer overflow detection
    • No actual loop instructions: There's no counter, no comparison, no jumpjust straight-line code that fills the buffer and calls printf

    This demonstrates how -O2 optimization can transform simple loops into much more efficient code by eliminating the loop overhead entirely.
