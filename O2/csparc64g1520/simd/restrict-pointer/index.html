    restrict-pointer¶

    Details

    Source: simd/restrict-pointer.c Compiler: csparc64g1520 (SPARC64) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    restrict-pointer.c
    /*
 * Restrict pointers enable better optimization.
 *
 * The 'restrict' keyword promises that pointers don't alias,
 * allowing the compiler to optimize more aggressively.
 */

/* Without restrict - compiler must be conservative */
void copy_no_restrict(int *dst, const int *src, int n)
{
    /* dst and src MIGHT overlap, so compiler must handle that */
    for (int i = 0; i < n; i++) {
        dst[i] = src[i];
    }
}

/* With restrict - compiler knows no aliasing */
void copy_restrict(int *restrict dst, const int *restrict src, int n)
{
    /* dst and src guaranteed not to overlap
     * Compiler can use faster copy or vectorize freely
     */
    for (int i = 0; i < n; i++) {
        dst[i] = src[i];
    }
}

/* Multiplication example - dramatic difference */
void scale_no_restrict(float *result, const float *input,
                       const float *scale, int n)
{
    /* Compiler doesn't know if result aliases input or scale */
    for (int i = 0; i < n; i++) {
        result[i] = input[i] * (*scale);
    }
    /* Must reload *scale each iteration! */
}

void scale_restrict(float *restrict result,
                    const float *restrict input,
                    const float *restrict scale, int n)
{
    /* Compiler knows result doesn't alias scale */
    for (int i = 0; i < n; i++) {
        result[i] = input[i] * (*scale);
    }
    /* Can load *scale once and reuse */
}

    Explanation¶

    the key insight is how the restrict keyword dramatically changes the compiler's optimization strategy.

    The Core Difference: What restrict Enables¶

    The restrict keyword is a promise to the compiler: "these pointers don't overlap in memory." Without this promise, the compiler must be conservative and reload values from memory repeatedly. With it, the compiler can make aggressive optimizations—in one case, even replacing the entire loop with a memcpy call.

    copy_no_restrict vs copy_restrict¶

    Without restrict (lines 1-14): The compiler generates a manual loop that copies integers one at a time: - lduw [%o1+%g1], %g2 — load from source - st %g2, [%o0+%g1] — store to destination - Loop back and repeat

    This is straightforward but requires the compiler to be careful about potential overlaps.

    With restrict (lines 16-24): The compiler recognizes that since pointers can't alias, it can simply call memcpy—a highly optimized library function that copies memory in bulk. This is much faster than the manual loop because memcpy uses specialized techniques (like copying multiple bytes at once) that a simple loop cannot.

    The key instructions: - sllx %o2, 2, %o2 — converts the count to byte length (multiply by 4, since each int is 4 bytes) - call memcpy, 0 — delegates to the optimized library function

    scale_no_restrict vs scale_restrict¶

    This example shows an even more dramatic difference. Both functions multiply each element of input by a constant scale value.

    Without restrict (lines 26-40): Inside the loop (.L15), notice this instruction appears every iteration: 

    ld [%o2], %f9

    This reloads the scale value from memory each time through the loop. Why? Because without restrict, the compiler must assume that result might alias scale—meaning writing to result[i] could change the value at scale. So it conservatively reloads *scale each iteration to be safe.

    With restrict (lines 42-56): The scale value is loaded once, before the loop: 

    ld [%o2], %f9

    Then %f9 is reused in every iteration without reloading. The compiler knows that writing to result cannot affect scale, so it safely keeps the value in a register.

    Register and Calling Convention Basics¶

    In SPARC, function parameters arrive in specific registers: - %o0, %o1, %o2, %o3 — the first four integer arguments - %f8, %f9, etc. — floating-point registers (used for float operations)

    For example, in scale_no_restrict: - %o0 = result pointer - %o1 = input pointer
    - %o2 = scale pointer - %o3 = n (the count)

    The return address is in %o7, and the final jmp %o7+8 returns from the function.

    Loop Structure Pattern¶

    All loops follow this pattern: 1. Check if n ≤ 0: cmp followed by ble,pn (branch if less-or-equal, predicted not taken) 2. Initialize: Set %g1 = 0 as the loop counter 3. Convert count to bytes: sllx %o2, 2, %o2 multiplies by 4 4. Loop body (labeled .L6, .L15, .L21): Do the work 5. Increment and test: add %g1, 4, %g1 then cmp and bne,a,pt (branch if not equal, with annul bit and predicted taken)

    The bne,a,pt instruction includes the next instruction in its delay slot (the lduw or ld that preloads the next value), which is a SPARC optimization technique.
