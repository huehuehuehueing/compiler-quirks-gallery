    fast-math¶

    Details

    Source: arithmetic/fast-math.c Compiler: csparc64g1520 (SPARC64) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    fast-math.c
    /*
 * Fast-math optimizations: -ffast-math / -Ofast effects.
 *
 * These flags allow the compiler to reorder floating-point
 * operations, assuming associativity (which floats don't have).
 * Can change results or break special value handling.
 */

/* With -ffast-math, this may be reordered and give different results */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++)
        sum += arr[i];
    return sum;
}

/* NaN check may be optimized away with -ffast-math */
int is_nan(float x)
{
    /* With -ffast-math, compiler assumes no NaN values exist,
     * so this comparison may be optimized to always return 0.
     */
    return x != x;
}

/* Division by variable may become multiplication by reciprocal */
float divide_by_y(float x, float y)
{
    /* With -ffast-math: may become x * (1.0f / y)
     * Less accurate but faster on some architectures.
     */
    return x / y;
}

/* Algebraic simplification (inexact with floats!) */
float algebraic(float a, float b, float c)
{
    /* With -ffast-math: (a + b) + c may be reordered to a + (b + c)
     * These can give different results with floating point!
     */
    return (a + b) + c;
}

    Explanation¶

    the key insight is that this code demonstrates how the -O2 optimization level handles floating-point operations, with some interesting patterns in how the compiler manages the code.

    Key Pattern: Floating-Point Operations and Optimization¶

    The most important thing to understand is that SPARC uses separate floating-point registers (the %f registers) for floating-point math, distinct from integer registers (the %o and %g registers). This separation is fundamental to how the code works.

    Function-by-Function Breakdown¶

    sum_array - Loop with Floating-Point Accumulation¶

    This function demonstrates a classic loop pattern:

    cmp     %o1, 0              // Compare n (in %o1) with 0
ble,pn  %icc, .L4           // If n <= 0, branch to .L4 (return 0.0)
 srl    %o1, 0, %o1         // Delay slot: zero-extend %o1 (prepare for 64-bit math)

    The cmp instruction compares the loop count. The ble,pn (branch if less-or-equal, predict not taken) jumps to .L4 if the array is empty. The ,pn suffix is a branch prediction hint telling the CPU "we don't expect this branch to be taken" — the compiler predicts arrays usually have elements.

    fzeros  %f0                 // Initialize sum to 0.0 in floating-point register %f0
sllx    %o1, 2, %o1         // Multiply n by 4 (shift left 2 bits) to get byte offset
add     %o0, %o1, %g1       // Calculate end address: arr + (n * 4)

    Here, fzeros initializes the accumulator. The sllx and add instructions calculate where the loop should stop — since each float is 4 bytes, we multiply the count by 4 and add it to the starting address.

    .L3:                        // Loop label
ld      [%o0], %f8          // Load one float from memory into %f8
add     %o0, 4, %o0         // Advance pointer by 4 bytes
cmp     %o0, %g1            // Compare current position with end address
bne,pt  %xcc, .L3           // If not equal, branch back (predict taken)
 fadds  %f0, %f8, %f0       // Delay slot: add loaded value to running sum

    The loop loads a float, advances the pointer, and accumulates. Notice the fadds (floating-point add single-precision) is in the delay slot — the instruction after the branch that executes before the branch takes effect. This is a SPARC architectural feature that allows the CPU to do useful work while deciding whether to branch.

    .L4:                        // Empty array case
fzeros %f0                  // Return 0.0

    is_nan - NaN Detection¶

    fcmps   %fcc0, %f1, %f1     // Compare %f1 (the input) with itself
mov     0, %o0              // Set return value to 0
jmp     %o7+8               // Return
 movne  %fcc0, 1, %o0       // Delay slot: if not equal, set return to 1

    This is clever: a NaN is the only value that doesn't equal itself. The fcmps compares the input with itself and sets condition flags. The movne (move if not equal) conditionally sets the return value to 1 only if the comparison showed inequality. By convention, %o0 holds the return value for integer results.

    divide_by_y - Simple Division¶

    jmp     %o7+8               // Return immediately
 fdivs  %f1, %f3, %f0       // Delay slot: divide %f1 by %f3, store in %f0

    This is minimal: the division happens in the delay slot of the return instruction. By SPARC convention, %f1 and %f3 hold the first and second floating-point arguments, and %f0 holds the return value.
