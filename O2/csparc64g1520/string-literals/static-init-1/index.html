    static-init-1¶

    Details

    Source: string-literals/static-init-1.c Compiler: csparc64g1520 (SPARC64) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    static-init-1.c
    /*
 * Static string initialization with concatenation.
 *
 * Adjacent string literals are concatenated at compile time.
 * The initializer may become a single memcpy or inline stores.
 */
#include <string.h>
#include <stdio.h>

void foo(void)
{
    /* Adjacent strings concatenated at compile time: "AAAABBBB" */
    char buf[100] = { "AAAA" "BBBB" };

    /* Make observable to prevent dead code elimination */
    puts(buf);
}

    Explanation¶

    the key insight here is how the compiler efficiently initializes a 100-byte buffer with a string using optimized memory operations.

    Key Pattern: Efficient String Initialization¶

    The most important thing to understand is that the compiler doesn't copy the string byte-by-byte. Instead, it constructs the string value in registers and writes it to memory in large chunks using 64-bit stores (stx instructions). This is much faster than individual byte operations.

    Stack Frame Setup¶

    save    %sp, -288, %sp

    This instruction creates a new stack frame for the function. The -288 means the compiler allocates 288 bytes of local storage on the stack. This space holds the 100-byte buf array plus some padding for alignment. In SPARC, the save instruction also shifts the register window, which is part of SPARC's unique register management system—think of it as automatically saving the caller's registers and giving the function a fresh set to work with.

    Building the String in Registers¶

    The next several instructions construct the string "AAAABBBB" in registers %g1 and %g2:

    sethi   %hi(1094795264), %g1      // Load upper 32 bits
or      %g1, 321, %g1             // OR in lower 32 bits
sllx    %g1, 32, %g1              // Shift left 32 bits (move to upper half of 64-bit register)

sethi   %hi(1111638016), %g2      // Load upper 32 bits of second value
or      %g2, 578, %g2             // OR in lower 32 bits

add     %g1, %g2, %g1             // Combine both halves

    These instructions are building the ASCII representation of "AAAABBBB" as a 64-bit value. The sethi instruction loads the upper 22 bits, and or fills in the lower 10 bits. By shifting %g1 left 32 bits and adding %g2, the compiler creates an 8-byte chunk containing the full string.

    Clearing the Rest of the Buffer¶

    stx     %g0, [%fp+1951]
stx     %g0, [%fp+1959]
stx     %g0, [%fp+1967]
... (more stx instructions)

    The stx instruction stores a 64-bit value. %g0 is the zero register (always contains 0 in SPARC). These instructions clear the remaining 92 bytes of the buffer by writing zeros in 8-byte chunks. The offsets (1951, 1959, 1967, etc.) are spaced 8 bytes apart, filling the buffer after the initial string.

    Preparing the Function Call¶

    add     %fp, 1943, %o0
call    puts, 0
 st     %g0, [%fp+2039]

    The add instruction calculates the address of the buffer (frame pointer + 1943) and stores it in %o0. By convention in SPARC, the first argument to a function goes in %o0. The call instruction invokes puts with the buffer address. The st instruction in the delay slot (the line after call) clears the final 4 bytes of the buffer—this executes before the function call completes due to SPARC's branch delay slot.

    Function Return¶

    return  %i7+8
 nop

    The return instruction jumps back to the caller (the address stored in %i7 plus 8). The nop (no operation) in the delay slot is a placeholder—it does nothing but is required because SPARC always executes the instruction after a branch before the branch takes effect.

    Why This Approach?¶

    The compiler uses 64-bit stores instead of byte-by-byte initialization because modern processors can move data much faster in larger chunks. This is an optimization enabled by the -O2 flag—an unoptimized compiler would likely use a loop or individual byte stores, which would be significantly slower.
