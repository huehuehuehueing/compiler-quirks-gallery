    memset-removed¶

    Details

    Source: security/memset-removed.c Compiler: csparc64g1520 (SPARC64) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    memset-removed.c
    /*
 * Security issue: memset() to clear sensitive data is removed
 * as dead code when the buffer is not used afterward.
 *
 * This is a real security vulnerability pattern - passwords
 * and keys may remain in memory after "clearing".
 */
#include <string.h>
#include <stdio.h>

void process_password(const char *password)
{
    char local_copy[64];

    /* Copy password for processing */
    strncpy(local_copy, password, sizeof(local_copy) - 1);
    local_copy[sizeof(local_copy) - 1] = '\0';

    /* Do something observable with the password (simulate auth check) */
    if (local_copy[0] == 's' && local_copy[1] == 'e' &&
        local_copy[2] == 'c' && local_copy[3] == 'r' &&
        local_copy[4] == 'e' && local_copy[5] == 't') {
        puts("Access granted");
    } else {
        puts("Access denied");
    }

    /* Security: clear the password from stack
     * BUG: This memset is often REMOVED by optimizer
     * because local_copy is not used afterward!
     */
    memset(local_copy, 0, sizeof(local_copy));
}

void test_password(void)
{
    process_password("secret");
}

    Explanation¶

    the key insight here is that the memset() call to clear the password has been completely removed by the optimizer, which is the security vulnerability the code comments warn about.

    The Critical Security Issue¶

    Look at the source code: after checking the password, there's a memset(local_copy, 0, sizeof(local_copy)) call intended to erase the password from memory. This call is completely absent from the assembly output. The compiler's -O2 optimization level recognized that local_copy is never used after the memset() call, so it eliminated the entire operation as "dead code." This leaves the password sitting in the stack memory where it could potentially be recovered.

    How the Function Works¶

    Stack Setup: 

    save    %sp, -240, %sp
    This creates a new stack frame with 240 bytes of space. The 64-byte local_copy buffer lives somewhere in this allocated space (at offset 1983 from the frame pointer, as you'll see in the code).

    Calling strncpy(): 

    mov     63, %o2              # Third parameter: 63 (copy up to 63 chars)
mov     %i0, %o1             # Second parameter: password pointer (from caller)
add     %fp, 1983, %o0       # First parameter: address of local_copy buffer
call    strncpy, 0
    In SPARC's calling convention, function parameters go in registers %o0, %o1, %o2, etc. The code sets up the three strncpy() parameters: destination buffer, source password, and length.

    Password Validation: The code then checks each character of the password one by one: 

    ldsb    [%fp+1983], %g1      # Load first byte ('s' = 115)
cmp     %g1, 115
bne,pn  %icc, .L2            # Branch if not equal (pessimistic prediction)

    This pattern repeats for each character: load a byte, compare it to the expected ASCII value ('e'=101, 'c'=99, 'r'=114, 't'=116), and branch to the "Access denied" path if any character doesn't match.

    Calling puts(): If all characters match, the code loads the address of the "Access granted" string and calls puts(): 

    sethi   %hi(.LC0), %i0       # Load high 22 bits of string address
restore %i0, %lo(.LC0), %o0  # Restore register and add low 10 bits
call    puts, 0

    The restore instruction is interestingit both returns from the current function frame and prepares the parameter for puts() in a single instruction.

    What's Missing: The memset() Call¶

    The source code includes: 

    memset(local_copy, 0, sizeof(local_copy));

    This entire call is gone from the assembly. The optimizer saw that: 1. local_copy is only used for the password check 2. After the check, local_copy is never read again 3. The function returns immediately after

    Therefore, the compiler concluded the memset() has no observable effect and removed it. This is technically correct from a pure C semantics perspective, but it's a serious security problem because the password data remains in stack memory.

    Key Observations¶

    • Optimization level matters for security: With -O2, the compiler aggressively removes code it considers "dead." Higher optimization levels can inadvertently eliminate security-critical operations.
    • SPARC calling conventions: Parameters are passed in %o0-%o5 registers; the save/restore instructions manage the register window (a SPARC-specific feature for efficient function calls).
    • Branch prediction hints: The bne,pn (branch if not equal, pessimistic) and bne,pt (branch if not equal, pessimistic) annotations tell the CPU which branch is more likely, helping with pipeline efficiency.
