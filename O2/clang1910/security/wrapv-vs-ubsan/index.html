    wrapv-vs-ubsan¶

    Details

    Source: security/wrapv-vs-ubsan.c Compiler: clang1910 (x86/x64) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    wrapv-vs-ubsan.c
    /* @gallery-hints
 *   extra-flags: -fwrapv
 */

/*
 * -fwrapv: well-defined signed integer overflow
 *
 * By default, signed integer overflow is undefined behaviour in C,
 * and compilers exploit this for optimisation (e.g. assuming i+1 > i).
 * With -fwrapv, signed overflow wraps around in two's complement,
 * which is well-defined.
 *
 * Compare the assembly with and without -fwrapv: the optimizer may
 * remove overflow checks or fold comparisons differently when
 * overflow is undefined.
 */

int will_overflow(int x)
{
    /* Without -fwrapv the compiler may assume this never wraps,
     * optimising the comparison to always-true. */
    if (x + 1 > x)
        return 1;
    return 0;
}

int abs_safe(int x)
{
    /* With well-defined wrapping, the compiler must handle INT_MIN. */
    if (x < 0)
        x = -x;
    return x;
}

    Explanation¶

    the key insight is that -fwrapv makes signed integer overflow well-defined, which changes how the compiler optimizes these functions.

    will_overflow(int) Function¶

    The most important thing to understand here: the compiler recognizes that x + 1 > x can only be false when x is at its maximum value (INT_MAX = 2147483647), because with -fwrapv, overflow wraps around in two's complement arithmetic.

    Here's what happens step-by-step:

    1. xor eax, eax — Clear the eax register (set it to 0). This is the return value register by convention.

    2. cmp edi, 2147483647 — Compare the input parameter (in edi, the first integer parameter register) against INT_MAX. This checks: "Is the input equal to the maximum 32-bit signed integer?"

    3. setne al — Set the low byte of eax to 1 if the comparison was "not equal" (the ne suffix means "not equal"). If edi ≠ 2147483647, this sets al to 1; otherwise it stays 0.

    4. ret — Return the value in eax.

    Why this works: When x < INT_MAX, adding 1 to it produces a valid result that's greater than x. When x = INT_MAX, adding 1 wraps to INT_MIN (the most negative value), which is not greater than INT_MAX. So the function returns 1 (true) for all inputs except INT_MAX, where it returns 0 (false).

    abs_safe(int) Function¶

    This function computes the absolute value while handling the edge case of INT_MIN (the most negative value), which has no positive counterpart in two's complement.

    1. mov eax, edi — Copy the input parameter into eax (the return value register).

    2. neg eax — Negate eax (compute -eax). This is the arithmetic negation instruction.

    3. cmovs eax, edi — Conditional move if signed (the s suffix means "if sign flag is set"). This instruction checks if the previous neg operation set the sign flag, which happens when the result is negative. If so, it moves edi (the original input) back into eax; otherwise, eax keeps the negated value.

    4. ret — Return the value in eax.

    Why this works: For normal negative numbers, neg produces a positive result. But for INT_MIN, negating it wraps back to INT_MIN itself (because there's no positive value large enough to represent the opposite). The cmovs detects this by checking if the result is still negative after negation—if so, it means we had INT_MIN, so we return the original input unchanged.

    Key Observations¶

    • Optimization enabled: The -O2 flag combined with -fwrapv allows the compiler to make strong assumptions about overflow behavior. Without -fwrapv, the first function might be optimized to always return 1 (since undefined overflow could be assumed impossible).

    • Conditional move pattern: The cmovs instruction in abs_safe is a branchless optimization—it avoids jumping, which is faster on modern processors because it doesn't disrupt instruction pipeline flow.

    • Register conventions: Both functions use edi for the input (first integer parameter) and eax for the return value, following the AMD64 calling convention.
