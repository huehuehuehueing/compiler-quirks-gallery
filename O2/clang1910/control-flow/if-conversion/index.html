    if-conversion¶

    Details

    Source: control-flow/if-conversion.c Compiler: clang1910 (x86/x64) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    if-conversion.c
    /*
 * If-Conversion (branch to conditional move)
 *
 * Short if-else chains that select between two values can be
 * replaced with conditional-move instructions (cmov on x86).
 * This eliminates a branch, avoiding potential misprediction
 * penalties on modern super-scalar CPUs.
 *
 * Look for: cmovl / cmovge / cmovne instead of jl / jge / jne
 * with a branch over a mov.
 */

int min(int a, int b)
{
    if (a < b)
        return a;
    return b;
}

int clamp(int x, int lo, int hi)
{
    if (x < lo)
        x = lo;
    if (x > hi)
        x = hi;
    return x;
}

int abs_val(int x)
{
    return x < 0 ? -x : x;
}

    Explanation¶

    the key insight is that the compiler has replaced traditional branching (jumps) with conditional move instructions (cmov*), which is a powerful optimization technique.

    The Core Pattern: If-Conversion¶

    Instead of using branches that could cause CPU pipeline delays, the compiler uses conditional moves. Here's why this matters: when a CPU encounters a branch, it must guess which direction the code will go. If it guesses wrong, it wastes time flushing its instruction pipeline. Conditional moves avoid this by executing both paths' instructions but only keeping the result of the correct one.

    min(int, int) - The Simplest Case¶

    mov     eax, esi          // eax = b (second parameter)
cmp     edi, esi          // compare a (first param) with b
cmovl   eax, edi          // if a < b, move a into eax
ret

    What's happening: 1. mov eax, esi - Start by assuming the result is b (the second parameter, passed in esi by convention) 2. cmp edi, esi - Compare a (in edi) with b (in esi). This sets CPU flags based on the comparison 3. cmovl eax, edi - Conditional move less: if the comparison showed a < b, overwrite eax with a (from edi). Otherwise, keep b 4. ret - Return the result in eax

    The beauty here: no branch instruction, no pipeline stall. The CPU executes both the mov and the cmovl, but only the correct result ends up in eax.

    clamp(int, int, int) - Two Conditions Chained¶

    mov     eax, esi          // eax = lo (second parameter)
cmp     edi, esi          // compare x (first param) with lo
cmovg   eax, edi          // if x > lo, move x into eax
cmp     eax, edx          // compare result with hi (third param)
cmovge  eax, edx          // if result >= hi, move hi into eax
ret

    What's happening: This function clamps x between lo and hi. The compiler chains two conditional moves:

    1. First block: Start with lo, then conditionally replace it with x if x > lo
    2. Second block: Take that result, then conditionally replace it with hi if the result is >= hi

    Again, no branches—just sequential conditional moves. Each cmov* instruction only updates eax if its condition is true.

    abs_val(int) - A Clever Trick¶

    mov     eax, edi          // eax = x (first parameter)
neg     eax               // eax = -x (negate it)
cmovs   eax, edi          // if sign flag set, move x back into eax
ret

    What's happening: This is elegant: the compiler computes both possibilities (x and -x) and uses a conditional move to pick the right one:

    1. mov eax, edi - Put x in eax
    2. neg eax - Negate it to get -x
    3. cmovs eax, edi - Conditional move if sign: if the negation produced a negative result (sign flag set), that means the original x was positive, so move the original x back. Otherwise, keep -x

    The neg instruction sets the sign flag based on its result, which the cmovs then uses to decide.

    Why This Matters¶

    With -O2 optimization, the compiler recognizes these simple if-else patterns and converts them to branchless code. This is faster on modern CPUs because: - No branch misprediction penalties - Instructions execute in parallel without pipeline stalls - The CPU doesn't have to guess which way the code goes

    This is a textbook example of if-conversion, one of the most impactful optimizations for short conditional sequences.
