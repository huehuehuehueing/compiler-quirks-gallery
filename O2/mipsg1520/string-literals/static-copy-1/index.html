    static-copy-1¶

    Details

    Source: string-literals/static-copy-1.c Compiler: mipsg1520 (MIPS) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    static-copy-1.c
    /*
 * String copy with strcpy - compiler may inline as immediate stores.
 *
 * For small, known-length strings, strcpy becomes direct
 * memory stores with the string data embedded in instructions.
 */
#include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[100];
    strcpy(buf, "ABCDXXXXCCCC");

    /* Make observable to prevent dead code elimination */
    puts(buf);
}

    Explanation¶

    the key insight is that the compiler has inlined the string copy operation - instead of calling strcpy, it directly copies the string data as immediate values into the buffer.

    The Core Strategy: Inlining String Copies¶

    When the compiler sees strcpy(buf, "ABCDXXXXCCCC") with a known string literal, it recognizes this as an optimization opportunity. Rather than calling the strcpy function (which would loop through each character), the compiler loads the string data directly from memory and stores it into the buffer in one shot using multi-byte operations.

    Breaking Down the Assembly¶

    Setting up the string address: 

    lui     $3,%hi($LC0)        # Load upper 16 bits of string address into $3
addiu   $2,$3,%lo($LC0)     # Add lower 16 bits to form complete address in $2
    These instructions construct the full memory address of the string literal "ABCDXXXXCCCC" stored in the .data section (labeled $LC0). MIPS uses a two-step process because 16-bit immediates can't hold full addresses.

    Loading the string data in chunks: 

    lw      $6,%lo($LC0)($3)    # Load first 4 bytes (word) of string
lw      $5,4($2)            # Load next 4 bytes
lw      $3,8($2)            # Load next 4 bytes
lbu     $2,12($2)           # Load final byte (null terminator)
    The compiler loads the 13-byte string (12 characters + null terminator) in chunks: three 4-byte words (lw = load word) and one final byte (lbu = load byte unsigned). This is much faster than a byte-by-byte loop.

    Stack frame setup: 

    addiu   $sp,$sp,-136        # Allocate 136 bytes on stack for local variables
addiu   $4,$sp,24           # Calculate address of buf (at offset 24 from stack pointer)
sw      $31,132($sp)        # Save return address (required before calling another function)
    The stack grows downward (subtracting from $sp), and the buffer buf[100] is placed at offset 24 from the new stack pointer.

    Copying the loaded data into the buffer: 

    sw      $6,24($sp)          # Store first 4 bytes at buf[0]
sw      $5,28($sp)          # Store next 4 bytes at buf[4]
sw      $3,32($sp)          # Store next 4 bytes at buf[8]
sb      $2,36($sp)          # Store final byte at buf[12]
    The loaded string chunks are written directly into the buffer's stack location.

    Function call and cleanup: 

    jal     puts                # Jump and link to puts() function (call puts with buf address in $4)
lw      $31,132($sp)        # Restore return address
jr      $31                 # Return to caller
addiu   $sp,$sp,136         # Deallocate stack space
    The jal instruction calls puts() to print the buffer. By convention, the first argument (buf's address) is passed in register $4. The return address is restored before jumping back.

    Why This Matters¶

    This is an example of compiler optimization at work with -O2. The compiler recognized that: 1. The string length is known at compile time (12 characters) 2. The string is small enough to copy efficiently with a few multi-byte loads/stores 3. Calling the full strcpy function would be slower than inlining the copy

    The result is faster code that avoids function call overhead entirely.
