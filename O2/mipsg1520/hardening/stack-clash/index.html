    stack-clash¶

    Details

    Source: hardening/stack-clash.c Compiler: mipsg1520 (MIPS) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    stack-clash.c
    /* @gallery-hints
 *   extra-flags: -fstack-clash-protection
 *   compiler-exclude: vc_v19_44_VS17_14_x64, vc_v19_44_VS17_14_x86, avrg1520
 */

/*
 * Stack Clash Protection
 *
 * -fstack-clash-protection inserts probes into large stack
 * allocations so that every page of the stack is touched in order.
 * This prevents "stack clash" attacks where a large allocation
 * jumps the guard page and overlaps the heap.
 *
 * Look for: loop-based or sequential page-sized probes (usually
 * stores to rsp at 4096-byte intervals) when the function has a
 * large local frame.
 */

void large_frame(int n)
{
    volatile char buf[16384];  /* larger than one page */
    /* Touch a few spots so the compiler keeps the buffer. */
    buf[0] = 'A';
    buf[4096] = 'B';
    buf[8192] = 'C';
    buf[16383] = 'D';
    (void)n;
}

    Explanation¶

    the key insight here is that the compiler is implementing stack clash protection - a security feature that prevents attackers from exploiting large stack allocations.

    The Core Pattern: Probing the Stack¶

    The first five instructions are the critical security mechanism:

    sw      $0,-4096($sp)
sw      $0,-8192($sp)
sw      $0,-12288($sp)
sw      $0,-16384($sp)
sw      $0,-16392($sp)

    These sw (store word) instructions write zeros to memory at regular intervals before the stack pointer is actually moved. Think of this like testing each step of a staircase before committing your full weight to it. Each sw instruction touches a different page of memory (pages are typically 4096 bytes). By writing to these addresses first, the OS is forced to allocate physical memory for each page, preventing an attacker from jumping over the guard page that normally protects the stack.

    Why This Matters¶

    Without stack clash protection, an attacker could allocate a huge buffer that skips over the guard page and collides with the heap, corrupting data. These probe instructions ensure every page is touched in sequence, making such attacks impossible.

    The Actual Stack Allocation¶

    After the probes, the function allocates the actual stack space:

    addiu   $sp,$sp,-16392

    This decrements the stack pointer by 16,392 bytes (16 KB), creating space for the 16,384-byte buffer plus 8 bytes of padding. The stack grows downward in MIPS, so negative offsets move to lower addresses.

    Writing the Buffer Values¶

    The next four instructions write the actual test values into the buffer:

    li      $2,65                 # Load 65 ('A') into $2
sb      $2,8($sp)             # Store byte at buf[0]

li      $2,66                 # Load 66 ('B') into $2
sb      $2,4104($sp)          # Store byte at buf[4096]

li      $2,67                 # Load 67 ('C') into $2
sb      $2,8200($sp)          # Store byte at buf[8192]

li      $2,68                 # Load 68 ('D') into $2
sb      $2,16391($sp)         # Store byte at buf[16383]

    The li (load immediate) instruction loads ASCII values into register $2. The sb (store byte) instructions write these single bytes to the buffer at the specified offsets. Notice the offsets: 8, 4104, 8200, and 16391 - these correspond to the source code's buf[0], buf[4096], buf[8192], and buf[16383] after accounting for the 8-byte padding.

    Function Return¶

    jr      $31
addiu   $sp,$sp,16392

    The jr $31 jumps back to the caller (register $31 holds the return address by convention). The addiu restores the stack pointer by adding back 16,392 bytes. This instruction executes in the delay slot (a MIPS quirk where the instruction after a branch executes before the branch takes effect).

    Key Observations¶

    • Compilation with -O2 optimizes the code well - the compiler eliminates the buffer variable itself since it's only used for these writes, but keeps the writes to prevent dead code elimination
    • Stack clash protection adds overhead - those five probe instructions wouldn't exist without -fstack-clash-protection, representing the security cost
    • The probes happen before stack allocation - this is the critical ordering that makes the protection work
