    volatile-access¶

    Details

    Source: memory/volatile-access.c Compiler: csparcg1520 (SPARC) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    volatile-access.c
    /*
 * Volatile prevents optimization of memory accesses.
 *
 * Without volatile, repeated reads/writes may be cached in
 * registers or eliminated. Volatile forces actual memory access.
 */

/* Hardware register simulation */
volatile int *const HARDWARE_REG = (volatile int *)0x40000000;

void write_hardware(int value)
{
    /* Each write MUST go to memory - cannot be cached or reordered */
    *HARDWARE_REG = value;
    *HARDWARE_REG = value;  /* Not eliminated - both writes happen */
}

int read_hardware(void)
{
    /* Each read MUST come from memory - cannot use cached value */
    int a = *HARDWARE_REG;
    int b = *HARDWARE_REG;
    return a + b;  /* Two actual memory reads */
}

/* Compare: non-volatile version */
int *non_volatile_ptr;

int read_non_volatile(void)
{
    /* Optimizer may read once and reuse the value */
    int a = *non_volatile_ptr;
    int b = *non_volatile_ptr;  /* May be optimized to: b = a */
    return a + b;
}

    Explanation¶

    the key insight is how the volatile keyword forces the compiler to perform actual memory operations, while non-volatile pointers allow optimization.

    The Core Difference: Volatile vs. Non-Volatile¶

    The most important pattern here is how volatile prevents compiler optimizations. When you mark a pointer as volatile, you're telling the compiler "I know this points to hardware or shared memory—don't try to be clever and cache values in registers." Without volatile, the compiler assumes it can optimize repeated reads to the same location.

    write_hardware() - Two Writes to Hardware¶

    write_hardware:
    sethi   %hi(1073741824), %g1      # Load address 0x40000000 into %g1
    st      %o0, [%g1]                # Store parameter to memory
    st      %o0, [%g1]                # Store parameter again (NOT eliminated!)
    jmp     %o7+8                     # Return to caller
     nop                              # Delay slot (no operation)

    What's happening: The function receives a value in %o0 (the first parameter register by SPARC calling convention). It then:

    1. sethi %hi(1073741824), %g1 - Loads the high 22 bits of the address 0x40000000 into register %g1. This is the hardware register address.
    2. Two identical st (store) instructions - Both write the same value to the same memory address. Normally, a compiler would eliminate the second write as redundant, but because HARDWARE_REG is volatile, both writes must happen. This is critical for hardware—the second write might trigger a different action than the first.
    3. jmp %o7+8 - Returns to the caller. The %o7 register holds the return address (by SPARC convention).

    read_hardware() - Two Reads from Hardware¶

    read_hardware:
    sethi   %hi(1073741824), %g1      # Load address 0x40000000 into %g1
    ld      [%g1], %o0                # Read first value into %o0
    ld      [%g1], %g1                # Read second value into %g1
    jmp     %o7+8                     # Return to caller
     add    %o0, %g1, %o0             # Add them (in delay slot)

    What's happening: This function reads from the hardware register twice and returns their sum:

    1. First ld (load) - Reads from address in %g1, stores result in %o0
    2. Second ld - Reads from the same address again, stores in %g1. Without volatile, the compiler would skip this second read and just reuse the first value. But volatile forces both reads.
    3. add in the delay slot - The add instruction executes after the jmp returns (this is how SPARC's branch delay slot works). It adds the two values and puts the result in %o0, which is the return value register.

    read_non_volatile() - Compiler Optimization in Action¶

    read_non_volatile:
    sethi   %hi(non_volatile_ptr), %g1        # Load address of pointer variable
    ld      [%g1+%lo(non_volatile_ptr)], %g1  # Load the pointer itself
    ld      [%g1], %o0                        # Read value once
    jmp     %o7+8                             # Return to caller
     sll    %o0, 1, %o0                       # Multiply by 2 (shift left 1 bit)

    What's happening: This is the crucial difference. The source code reads twice, but the assembly only reads once:

    1. First two instructions - Load the pointer variable from memory into %g1
    2. Single ld - Read the value once into %o0
    3. sll %o0, 1, %o0 - Shift left by 1 bit, which multiplies by 2. This is equivalent to a + a (or a + b when b = a
