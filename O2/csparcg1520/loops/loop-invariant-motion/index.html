    loop-invariant-motion¶

    Details

    Source: loops/loop-invariant-motion.c Compiler: csparcg1520 (SPARC) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    loop-invariant-motion.c
    /*
 * Loop-Invariant Code Motion (LICM)
 *
 * The optimizer recognises computations inside a loop whose result
 * does not change across iterations and moves ("hoists") them before
 * the loop.  This avoids redundant work on every iteration.
 *
 * In the example below, `base * scale` is invariant with respect to
 * the loop index i, so the compiler should compute it once and reuse
 * the result inside the loop body.
 */

void fill_scaled(int *arr, int n, int base, int scale)
{
    for (int i = 0; i < n; i++)
    {
        /* base * scale is loop-invariant: it does not depend on i. */
        arr[i] = base * scale + i;
    }
}

    Explanation¶

    the key insight here is that the compiler has successfully applied Loop-Invariant Code Motion (LICM) — it moved the multiplication base * scale outside the loop so it's computed only once instead of on every iteration.

    The Core Optimization¶

    Look at line 3: smul %o2, %o3, %o2 — this multiplication happens before the loop starts (at .L3). The compiler recognized that base * scale doesn't change across loop iterations, so it computed it once and reused the result. This is the optimization the code comment describes.

    Register Usage (SPARC Calling Convention)¶

    SPARC uses %o0 through %o7 for parameter passing and temporary values: - %o0: pointer to the array (arr) - %o1: loop count (n) - %o2: base parameter (later reused for the invariant result) - %o3: scale parameter

    Step-by-Step Execution¶

    Before the loop (lines 1-4): 1. cmp %o1, 0 — compare n with 0 2. ble,pn %icc, .L6 — if n ≤ 0, branch to .L6 (exit) with prediction that this is unlikely (pn = predict not taken) 3. smul %o2, %o3, %o2 — multiply base * scale, store result in %o2 (this is the hoisted computation) 4. add %o1, %o2, %o1 — compute the loop exit condition: %o1 = n + (base*scale). This is used as the stopping point for the loop counter

    The loop body (lines 5-9, labeled .L3): 1. st %o2, [%o0] — store the value in %o2 to the array at address %o0 2. add %o2, 1, %o2 — increment the value (i++ in the source) 3. cmp %o2, %o1 — compare current value with the exit condition 4. bne,pt %icc, .L3 — branch back to .L3 if not equal (predict taken, pt) 5. add %o0, 4, %o0 — increment array pointer by 4 bytes (one int) in the delay slot

    Exit (lines 10-11): - jmp %o7+8 — return to caller - nop — no operation (delay slot filler)

    Why This Matters¶

    Without LICM, the compiler would recalculate base * scale on every loop iteration — wasteful. With -O2 optimization enabled, the compiler recognized this pattern and moved it outside, reducing the work inside the loop from 2 multiplications per iteration to just 1 addition and 1 store. This is a significant performance improvement for loops with many iterations.
