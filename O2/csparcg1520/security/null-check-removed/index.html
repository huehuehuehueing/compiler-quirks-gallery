    null-check-removed¶

    Details

    Source: security/null-check-removed.c Compiler: csparcg1520 (SPARC) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    null-check-removed.c
    /*
 * Security issue: null pointer check eliminated after dereference.
 *
 * Compiler assumes dereferencing null is UB, so if you dereference
 * first, any subsequent null check is "impossible" and removed.
 */
#include <stddef.h>

struct data {
    int value;
    int flags;
};

int process(struct data *p)
{
    int val;

    /* Dereference first */
    val = p->value;

    /* Check for null afterward - THIS MAY BE REMOVED!
     * Compiler reasons: "p was dereferenced, so p cannot be null,
     * therefore this check is always false"
     */
    if (p == NULL)
        return -1;

    return val + p->flags;
}

    Explanation¶

    the most important insight here is that the null pointer check has been completely eliminated by the compiler.

    The Key Issue: Undefined Behavior Optimization¶

    When you dereference a pointer (access memory through it) before checking if it's null, you're creating undefined behavior in C. The compiler interprets this as: "if the code reached this point, the pointer must be valid, because dereferencing null is impossible." Therefore, any null check that comes after a dereference is logically unreachable and gets removed entirely.

    What the Assembly Does¶

    The function loads two values from the structure and adds them together:

    ld      [%o0+4], %g2      // Load flags (at offset +4) into g2
ld      [%o0], %o0        // Load value (at offset 0) into o0
jmp     %o7+8             // Return to caller
 add    %g2, %o0, %o0     // Add them together, result goes to o0

    Register conventions in SPARC: - %o0 is the first parameter (the pointer p) and also the return value register - %g2 is a temporary register used here for intermediate storage - %o7 holds the return address

    Step-by-Step Execution¶

    1. ld [%o0+4], %g2 — Load the flags field (4 bytes into the structure) into %g2
    2. ld [%o0], %o0 — Load the value field (at the start of the structure) into %o0, overwriting the pointer
    3. jmp %o7+8 — Jump back to the caller (the +8 accounts for SPARC's delay slot)
    4. add %g2, %o0, %o0 — Add flags and value together, storing the result in %o0 (the return value)

    Note that instruction 4 executes before the jump due to SPARC's delay slot—a quirk of this architecture where the instruction after a branch executes before the branch takes effect.

    What's Missing¶

    The null check (if (p == NULL) return -1;) has vanished entirely. There's no comparison instruction, no conditional branch—nothing. The compiler optimized it away because it reasoned: "we already dereferenced p on line 1, so it cannot be null." This is technically correct according to C's rules, but it means if null is passed, the program will crash with undefined behavior rather than returning -1 as the source code intended.

    This is a common security pitfall: the source code looks defensive, but the compiler's optimization removes the defense.
