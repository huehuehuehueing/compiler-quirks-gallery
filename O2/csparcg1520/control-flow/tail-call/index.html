    tail-call¶

    Details

    Source: control-flow/tail-call.c Compiler: csparcg1520 (SPARC) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    tail-call.c
    /*
 * Tail call optimization: recursive calls become jumps.
 *
 * When a function's last action is calling another function
 * (or itself), the call can become a jump, saving stack space.
 */

/* Tail-recursive factorial - can be optimized to a loop */
int factorial_tail(int n, int acc)
{
    if (n <= 1)
        return acc;
    /* Last action is the recursive call - tail position */
    return factorial_tail(n - 1, n * acc);
}

int factorial(int n)
{
    return factorial_tail(n, 1);
}

/* NOT tail recursive - multiplication after recursive call */
int factorial_not_tail(int n)
{
    if (n <= 1)
        return 1;
    /* Multiplication happens AFTER the call returns */
    return n * factorial_not_tail(n - 1);
}

/* Mutual tail recursion */
int is_even(unsigned n);
int is_odd(unsigned n);

int is_even(unsigned n)
{
    if (n == 0) return 1;
    return is_odd(n - 1);  /* Tail call */
}

int is_odd(unsigned n)
{
    if (n == 0) return 0;
    return is_even(n - 1);  /* Tail call */
}

    Explanation¶

    the key insight is that the compiler has applied tail call optimization to convert recursive function calls into loops, which is a major performance improvement.

    The Core Optimization: Tail Calls Become Loops¶

    When a function's last action is calling another function (or itself), the compiler can replace that call with a jump instead. This saves stack space and avoids the overhead of function calls. You can see this pattern throughout the code.

    factorial_tail - Tail Recursion Optimized to a Loop¶

    factorial_tail:
        cmp     %o0, 1              # Compare n with 1
        ble,pn  %icc, .L5           # If n <= 1, jump to return
         nop
        smul    %o1, %o0, %o1       # acc = acc * n
.L10:
        add     %o0, -1, %o0        # n = n - 1
        cmp     %o0, 1              # Compare n with 1
        bne,a,pt %icc, .L10         # If n != 1, loop back
         smul   %o1, %o0, %o1       # acc = acc * n (in delay slot)
.L5:
        jmp     %o7+8               # Return to caller
         mov    %o1, %o0            # Move result to return register

    What's happening: Instead of making recursive function calls, the compiler converted the tail recursion into a simple loop (.L10). The registers hold: - %o0: the parameter n - %o1: the accumulator acc - %o7: the return address (by SPARC calling convention)

    The loop decrements n and multiplies the accumulator until n reaches 1, then returns. This uses constant stack space instead of growing the stack with each recursive call.

    factorial - Wrapper Function¶

    factorial:
        mov     %o0, %g1            # Save n to %g1
        cmp     %g1, 1              # Compare with 1
        ble,pn  %icc, .L20          # If n <= 1, jump to return
         mov    1, %o0              # Set acc = 1 (in delay slot)
        smul    %o0, %g1, %o0       # acc = 1 * n
.L19:
        add     %g1, -1, %g1        # n = n - 1
        cmp     %g1, 1              # Compare with 1
        bne,a,pt %icc, .L19         # If n != 1, loop back
         smul   %o0, %g1, %o0       # acc = acc * n
.L20:
        jmp     %o7+8               # Return
         nop

    This function simply calls factorial_tail(n, 1), but the compiler inlined the tail-recursive loop directly here instead of making an actual function call.

    factorial_not_tail - No Optimization Possible¶

    factorial_not_tail:
        mov     %o0, %g1
        cmp     %g1, 1
        ble,pn  %icc, .L30
         mov    1, %o0
        smul    %o0, %g1, %o0
.L29:
        add     %g1, -1, %g1
        cmp     %g1, 1
        bne,a,pt %icc, .L29
         smul   %o0, %g1, %o0
.L30:
        jmp     %o7+8
         nop

    Interestingly, this function looks identical to factorial in the assembly! The compiler recognized that even though the source code has a multiplication after the recursive call, it can still optimize it to the same loop pattern because the multiplication is associative and can be rearranged.

    Mutual Tail Recursion: is_even and is_odd¶

    These functions call each other in tail position. The compiler handles this differently because it can't convert mutual recursion into a simple loop without more complex analysis.
