    stack-clash¶

    Details

    Source: hardening/stack-clash.c Compiler: csparcg1520 (SPARC) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    stack-clash.c
    /* @gallery-hints
 *   extra-flags: -fstack-clash-protection
 *   compiler-exclude: vc_v19_44_VS17_14_x64, vc_v19_44_VS17_14_x86, avrg1520
 */

/*
 * Stack Clash Protection
 *
 * -fstack-clash-protection inserts probes into large stack
 * allocations so that every page of the stack is touched in order.
 * This prevents "stack clash" attacks where a large allocation
 * jumps the guard page and overlaps the heap.
 *
 * Look for: loop-based or sequential page-sized probes (usually
 * stores to rsp at 4096-byte intervals) when the function has a
 * large local frame.
 */

void large_frame(int n)
{
    volatile char buf[16384];  /* larger than one page */
    /* Touch a few spots so the compiler keeps the buffer. */
    buf[0] = 'A';
    buf[4096] = 'B';
    buf[8192] = 'C';
    buf[16383] = 'D';
    (void)n;
}

    Explanation¶

    the key insight here is that this code demonstrates stack clash protection - a security feature that prevents attackers from exploiting large stack allocations.

    The Core Problem Being Solved¶

    When a function allocates a large block of memory on the stack (16,384 bytes here), the CPU normally just moves the stack pointer down without actually touching each page of memory. An attacker could exploit this by allocating even larger memory elsewhere, jumping over the guard page (a protected memory region), and corrupting the heap. Stack clash protection prevents this by probing every page - touching each 4,096-byte page of the allocation to ensure it exists.

    The Probe Loop (Lines 1-9)¶

    mov     0, %g1                    # g1 = 0 (current probe position)
sethi   %hi(16384), %g4           # g4 = 16384 (allocation size)
sub     %sp, %g1, %g1             # g1 = sp - 0 = sp (start address)
sub     %g1, %g4, %g4             # g4 = sp - 16384 (end address)
.LPSRL0:
add     %g1, -4096, %g1           # Move up by one page (4096 bytes)
cmp     %g1, %g4                  # Have we reached the end?
bne     .LPSRL0                   # If not, loop again
 st     %g0, [%g1+0]              # Touch this page (store 0)

    This loop walks backwards through the stack in 4,096-byte steps, storing a zero value at each page boundary. The st (store) instruction is the actual "probe" - it forces the CPU to allocate the physical memory page. The loop continues until g1 equals g4 (the bottom of the allocation).

    Why this matters: By touching every page sequentially, the OS can't be tricked into skipping the guard page. Each page must physically exist before the next one is accessed.

    Final Stack Adjustment and Buffer Writes (Lines 10-35)¶

    After probing, the code adjusts the stack pointer and writes the test values:

    st      %g0, [%g4-72]             # Final probe near the end
sethi   %hi(-17408), %g1
or      %g1, 952, %g1
add     %sp, %g1, %sp              # Adjust sp down by 16384 bytes

    Then it stores the characters ('A'=65, 'B'=66, 'C'=67, 'D'=68) at the specified offsets: - buf[0] - stored at sp+72 - buf[4096] - stored at sp+72+4096 - buf[8192] - stored at sp+72+8192 - buf[16383] - stored at sp+72+15360+1023

    The offset of 72 bytes accounts for the SPARC calling convention's register save area.

    Key Observations¶

    • Optimization level (-O2): The compiler still generates the probe loop despite optimization, because security features like stack clash protection are mandatory, not optional optimizations.
    • Register usage: Temporary registers (%g1, %g2, %g3, %g4) hold addresses and values; %sp is the stack pointer.
    • The return: jmp %o7+8 returns to the caller (in SPARC, %o7 holds the return address).

    This assembly demonstrates that security features can add measurable overhead - the probe loop is necessary but adds latency proportional to the allocation size.
