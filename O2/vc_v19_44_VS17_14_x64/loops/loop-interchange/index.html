    loop-interchange¶

    Details

    Source: loops/loop-interchange.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    loop-interchange.c
    /*
 * Loop Interchange
 *
 * When iterating over a 2D array, the order of the loop indices
 * determines memory access patterns.  In C (row-major layout),
 * iterating the inner loop over columns and the outer loop over
 * rows yields sequential access.  Swapping the loops ("interchange")
 * can improve cache locality when the original order would cause
 * stride-N accesses.
 *
 * Here the "bad" order (column-major traversal on a row-major array)
 * is presented; the compiler may interchange the loops at -O3.
 */
#define N 128

void column_sum(int mat[N][N], int result[N])
{
    /* Column-major traversal: inner loop strides by N ints.
     * The compiler may swap i and j for better locality. */
    for (int j = 0; j < N; j++)
    {
        result[j] = 0;
        for (int i = 0; i < N; i++)
        {
            result[j] += mat[i][j];
        }
    }
}

    Key Insight: Loop Unrolling for Performance¶

    The compiler has unrolled the inner loop by a factor of 16, processing 16 array elements per iteration instead of 1. This is a classic -O2 optimization that reduces loop overhead and improves CPU pipeline efficiency.

    Setup and Parameter Handling¶

    The function receives two parameters via the x64 calling convention: - rcx = pointer to the 2D matrix (mat) - rdx = pointer to the result array (result)

    The first few instructions prepare for the main computation:

    mov     rcx, rdx              ; rcx = rdx (parameter swap)
sub     rcx, rdx              ; rcx = mat - result (offset calculation)
mov     r11d, 128             ; r11d = 128 (outer loop counter)
xor     edi, edi              ; edi = 0 (will hold accumulator)
lea     rbx, QWORD PTR [rcx+1024]  ; rbx = mat + 1024 bytes

    The lea (Load Effective Address) instruction calculates mat + 1024 without accessing memoryit's pure address arithmetic. The value 1024 = 128 8 bytes, which is one full row of the matrix (since each int is 4 bytes, and 128 ints 4 = 512 bytes per row... actually 1024 bytes suggests the matrix uses 256-byte rows or there's pointer arithmetic optimization happening).

    Outer Loop: Processing Each Column¶

    The label $LL4@column_sum marks the outer loop that iterates 128 times (once per column):

    mov     ecx, edi              ; ecx = 0 (reset accumulator)
mov     DWORD PTR [rdx], edi  ; result[j] = 0 (initialize column sum)
lea     r9, QWORD PTR [rbx+rdx]  ; r9 = starting address for this column
mov     r10d, 8               ; r10d = 8 (inner loop counter)

    The r10d counter of 8 means the inner loop runs 8 times, and since each iteration processes 16 elements (unrolled), that's 8 16 = 128 total elements per columnexactly what we need.

    Inner Loop: Unrolled Accumulation¶

    The label $LL7@column_sum marks the unrolled inner loop. Here's the pattern repeated 16 times:

    add     ecx, DWORD PTR [r9-1024]  ; Load and add element
lea     r9, QWORD PTR [r9+8192]   ; Advance pointer by 8192 bytes (one full matrix row)
mov     DWORD PTR [rdx], ecx      ; Store running sum

    Each iteration: 1. Loads one element from the matrix at offset r9-1024 (or similar negative offsets like -8704, -8192, etc.) 2. Adds it to the running sum in ecx 3. Stores the updated sum back to result[j] 4. Advances r9 by 8192 bytes (the stride to the next row in the same column)

    The negative offsets (-1024, -8704, -8192, etc.) are relative to r9, which the compiler pre-increments. This is a pointer arithmetic optimizationinstead of decrementing r9, the compiler increments it and uses negative offsets, which can be more efficient on some CPU architectures.

    Loop Control and Cleanup¶

    sub     r10, 1                ; Decrement inner loop counter
jne     $LL7@column_sum       ; Jump if not equal (inner loop continues)

add     rdx, 4                ; Move to next result element (4 bytes = one int)
sub     r11, 1                ; Decrement outer loop counter
jne     $LL4@column_sum       ; Jump if not equal (outer loop continues)

    After the inner loop completes, rdx advances by 4 bytes to point to the next column's result, and the outer loop counter decrements.
