    tail-call¶

    Details

    Source: control-flow/tail-call.c Compiler: avrg1520 (AVR) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    tail-call.c
    /*
 * Tail call optimization: recursive calls become jumps.
 *
 * When a function's last action is calling another function
 * (or itself), the call can become a jump, saving stack space.
 */

/* Tail-recursive factorial - can be optimized to a loop */
int factorial_tail(int n, int acc)
{
    if (n <= 1)
        return acc;
    /* Last action is the recursive call - tail position */
    return factorial_tail(n - 1, n * acc);
}

int factorial(int n)
{
    return factorial_tail(n, 1);
}

/* NOT tail recursive - multiplication after recursive call */
int factorial_not_tail(int n)
{
    if (n <= 1)
        return 1;
    /* Multiplication happens AFTER the call returns */
    return n * factorial_not_tail(n - 1);
}

/* Mutual tail recursion */
int is_even(unsigned n);
int is_odd(unsigned n);

int is_even(unsigned n)
{
    if (n == 0) return 1;
    return is_odd(n - 1);  /* Tail call */
}

int is_odd(unsigned n)
{
    if (n == 0) return 0;
    return is_even(n - 1);  /* Tail call */
}

    Explanation¶

    the key insight is that the compiler has successfully converted tail-recursive functions into loops, eliminating function call overhead.

    The Core Optimization: Tail Call Elimination¶

    The most important pattern here is that factorial_tail and factorial don't use rcall (recursive call) instructions in their main loops. Instead, they use rjmp (relative jump) or loop back with brne (branch if not equal). This is tail call optimization — when a function's last action is calling itself, the compiler converts it into a loop instead of a recursive call. This saves stack space and avoids the overhead of function calls.

    Understanding AVR Registers and Calling Conventions¶

    Before diving into the code, you need to know how AVR passes parameters and returns values: - r24:r25 (a pair of 8-bit registers forming a 16-bit value) hold the first parameter - r22:r23 hold the second parameter - Return values come back in r24:r25 - r18:r19 are caller-saved temporary registers (the function can use them freely) - __zero_reg__ (r1) is always zero, used for comparisons

    factorial_tail(int n, int acc) — The Optimized Loop¶

    mov r18,r24          // Copy n into r18:r19 (working copy)
mov r19,r25
cpi r18,2            // Compare n with 2
cpc r19,__zero_reg__ // Compare with carry (for 16-bit comparison)
brlt .L5             // If n < 2, jump to return
.L2:                 // Loop starts here
    mov r24,r18      // Set up multiplication: r24:r25 = n
    mov r25,r19
    rcall __mulhi3   // Multiply: acc * n (acc is in r22:r23)
    mov r22,r24      // Store result back in acc (r22:r23)
    mov r23,r25
    subi r18,1       // Decrement n
    sbc r19,__zero_reg__
    cpi r18,1        // Check if n > 1
    cpc r19,__zero_reg__
    brne .L2         // Loop back if n > 1
.L5:
    mov r24,r22      // Return acc
    mov r25,r23
ret

    Key insight: This is a loop, not recursion. The brne .L2 branches back to the loop start instead of calling the function again. The accumulator pattern (acc) is maintained in r22:r23 across loop iterations.

    factorial(int n) — Wrapper Function¶

    This function simply initializes the accumulator to 1 and jumps into the same loop logic:

    mov r18,r24          // Copy n
mov r19,r25
ldi r22,lo8(1)       // Initialize acc = 1
ldi r23,0
cpi r18,2            // Same loop as factorial_tail
cpc r19,__zero_reg__
brlt .L10
.L11:                // Same loop structure
    // ... (identical loop body)
    brne .L11

    The compiler inlined the tail-recursive call and merged it into a single loop.

    factorial_not_tail(int n) — Why It Can't Be Optimized¶

    This function looks similar but has a critical difference:

    .L18:
    mov r24,r18      // Copy n
    mov r25,r19
    subi r18,1       // Decrement n BEFORE the call
    sbc r19,__zero_reg__
    rcall __mulhi3   // Multiply happens AFTER recursion would return
    mov r22,r24      // Store result
    mov r23,r25
    cpi r18,1
    cpc r19,__zero_reg__
    brne .L18

    Notice the multiplication (__mulhi3) happens inside the loop after the decrement. In the original source, the multiplication happens after the recursive call returns, so it can't be converted to a simple tail call.
