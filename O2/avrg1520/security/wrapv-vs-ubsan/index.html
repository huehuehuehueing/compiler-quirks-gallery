    wrapv-vs-ubsan¶

    Details

    Source: security/wrapv-vs-ubsan.c Compiler: avrg1520 (AVR) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    wrapv-vs-ubsan.c
    /* @gallery-hints
 *   extra-flags: -fwrapv
 */

/*
 * -fwrapv: well-defined signed integer overflow
 *
 * By default, signed integer overflow is undefined behaviour in C,
 * and compilers exploit this for optimisation (e.g. assuming i+1 > i).
 * With -fwrapv, signed overflow wraps around in two's complement,
 * which is well-defined.
 *
 * Compare the assembly with and without -fwrapv: the optimizer may
 * remove overflow checks or fold comparisons differently when
 * overflow is undefined.
 */

int will_overflow(int x)
{
    /* Without -fwrapv the compiler may assume this never wraps,
     * optimising the comparison to always-true. */
    if (x + 1 > x)
        return 1;
    return 0;
}

int abs_safe(int x)
{
    /* With well-defined wrapping, the compiler must handle INT_MIN. */
    if (x < 0)
        x = -x;
    return x;
}

    Explanation¶

    the key insight is how the -fwrapv flag affects overflow handling, and how the compiler optimizes these two functions differently.

    Key Insight: Overflow Handling with -fwrapv¶

    The -fwrapv flag tells the compiler that signed integer overflow is well-defined (wraps around in two's complement), rather than undefined behavior. This changes how the compiler optimizes these functions. Notice that both functions avoid the naive approach of actually computing x + 1 or -x — instead, they use clever bit-level tricks to check conditions without risking overflow.

    will_overflow(int) Function¶

    This function checks if x + 1 > x would overflow.

    Parameter passing: On AVR, a 32-bit int is passed in registers r24:r25 (lower 16 bits in r24:r25, upper 16 bits in r26:r27, but here we see r24:r25 used). The function receives x in r24:r25.

    mov r18,r24          // Copy x's low byte to r18
mov r19,r25          // Copy x's high byte to r19
    The compiler copies the input to r18:r19 for safe manipulation.

    ldi r24,lo8(1)       // Load 1 into r24 (return value, low byte)
ldi r25,0            // Load 0 into r25 (return value, high byte)
    Pre-load the return value 1 into r24:r25 (the return register pair).

    cpi r18,-1           // Compare r18 with -1 (0xFF)
sbci r19,127         // Subtract with carry: r19 - 127 - carry_flag
breq .L4             // Branch if equal (zero flag set)
ret                  // Return 1 (already in r24:r25)

    Here's the clever part: instead of computing x + 1, the compiler checks if x == INT_MIN (which is 0x80000000 on a 32-bit system). Why? Because INT_MIN + 1 is the only case where adding 1 doesn't make the value larger — it wraps around to a negative number. The comparison cpi r18,-1 checks if the low byte equals 0xFF, and sbci r19,127 checks if the high byte equals 0x80 (127 is the high byte of INT_MIN). If both match, the breq branches to .L4 and returns 0; otherwise, it returns 1.

    .L4:
ldi r24,0            // Load 0 into r24
ret                  // Return 0

    abs_safe(int) Function¶

    This function computes the absolute value of x, handling the special case where x == INT_MIN.

    sbrs r25,7           // Skip next instruction if bit 7 of r25 is set
ret                  // Return immediately if x >= 0

    sbrs (Skip if Bit in Register is Set) checks bit 7 of r25 (the sign bit of the high byte). If the sign bit is not set, x is non-negative, so the function returns immediately with x unchanged.

    neg r25              // Negate r25 (two's complement: flip bits and add 1)
neg r24              // Negate r24
sbc r25,__zero_reg__ // Subtract with carry: r25 - 0 - carry_flag
ret                  // Return the negated value

    If x is negative, the compiler negates it. The neg instruction performs two's complement negation on each byte. The sbc (Subtract with Carry) adjusts the high byte to account for any borrow from the low byte negation. This correctly handles the 32-bit negation across the two register pairs.

    Why this works with -fwrapv: Even if x == INT_MIN, negating it wraps around to INT_MIN itself (since -INT_MIN overflows). With -fwrapv, this wrapping is well-defined, so the compiler doesn't need special handling — it just negates
