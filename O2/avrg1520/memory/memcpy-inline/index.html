    memcpy-inline¶

    Details

    Source: memory/memcpy-inline.c Compiler: avrg1520 (AVR) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    memcpy-inline.c
    /*
 * memcpy inlining: small copies become inline loads/stores.
 *
 * For small, known-size copies, the compiler generates
 * inline code instead of calling memcpy.
 */
#include <string.h>

struct small {
    int a, b;
};

void copy_8_bytes(void *dst, const void *src)
{
    /* Small copy - likely inlined as single 64-bit load/store
     * on 64-bit architectures, or two 32-bit ops on 32-bit.
     */
    memcpy(dst, src, 8);
}

void copy_struct(struct small *dst, const struct small *src)
{
    /* Structure copy - same as memcpy but type-aware.
     * Compiler knows alignment and size.
     */
    *dst = *src;
}

void copy_varying(void *dst, const void *src, int use_large)
{
    /* Compile-time unknown size - must call memcpy or
     * generate size-checking code.
     */
    if (use_large)
        memcpy(dst, src, 1024);
    else
        memcpy(dst, src, 8);
}

    Explanation¶

    the key insight is that the compiler is inlining memory copies as explicit load/store loops instead of calling the memcpy function. This is a common optimization for small, known-size copies.

    AVR Architecture Basics¶

    Before diving in, here's what you need to know about AVR: - Registers: AVR has 32 general-purpose registers (r0-r31). By convention, r24-r25 hold the first parameter, r22-r23 the second, r20-r21 the third. - Pointer registers: X (r26:r27), Y (r28:r29), and Z (r30:r31) are 16-bit register pairs used for memory addressing. - Post-increment addressing: ld r0,Z+ loads from the address in Z, then increments Z automatically—useful for loops.

    Function 1: copy_8_bytes — Simple Byte Loop¶

    mov r26,r24          // X = destination pointer
mov r27,r25
ldi r25,lo8(8)       // r25 = 8 (loop counter)
mov r30,r22          // Z = source pointer
mov r31,r23
0:
ld r0,Z+             // Load byte from source, increment Z
st X+,r0             // Store byte to destination, increment X
dec r25              // Decrement counter
brne 0b              // Branch if not equal (loop back)
ret

    What's happening: This is a byte-by-byte copy loop. The compiler inlines the 8-byte copy as: 1. Set up X (destination) and Z (source) as 16-bit pointers 2. Load counter into r25 (8 bytes to copy) 3. Loop: load one byte from source, store to destination, decrement counter, repeat until done

    This is much faster than calling memcpy because there's no function call overhead—just 8 iterations of a 4-instruction loop.

    Function 2: copy_struct — Direct Register Copy¶

    mov r30,r22          // Z = source pointer
mov r31,r23
ld r20,Z             // Load 4 bytes from source into r20-r23
ldd r21,Z+1
ldd r22,Z+2
ldd r23,Z+3
mov r30,r24          // Z = destination pointer
mov r31,r25
st Z,r20             // Store 4 bytes to destination
std Z+1,r21
std Z+2,r22
std Z+3,r23
ret

    What's happening: Since the compiler knows the struct is exactly 8 bytes (two 32-bit integers), it: 1. Loads all 4 bytes from source into registers r20-r23 2. Stores all 4 bytes to destination

    This is even faster than the loop—no branching, no counter, just 8 load/store instructions. The ldd (load with displacement) and std (store with displacement) instructions access memory at offsets from the pointer (e.g., Z+1 means address Z plus 1).

    Function 3: copy_varying — Conditional Size Handling¶

    This function takes a runtime parameter (use_large) to decide between copying 8 bytes or 1024 bytes:

    or r20,r21           // Test if use_large is zero
breq .L5             // If zero, branch to 8-byte copy
