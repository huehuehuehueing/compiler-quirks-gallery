    stack-reuse¶

    Details

    Source: memory/stack-reuse.c Compiler: avrg1520 (AVR) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    stack-reuse.c
    /*
 * Stack slot reuse: non-overlapping locals share stack space.
 *
 * The compiler analyzes variable lifetimes and reuses
 * stack slots for variables that are never live at the same time.
 */
#include <stdio.h>

void example(int flag)
{
    if (flag) {
        /* This buffer is only live in this branch */
        char buffer_a[100];
        snprintf(buffer_a, sizeof(buffer_a), "Path A: %d", flag);
        printf("%s\n", buffer_a);
    } else {
        /* This buffer may reuse the same stack slot as buffer_a
         * since they're never both alive at the same time.
         */
        char buffer_b[100];
        snprintf(buffer_b, sizeof(buffer_b), "Path B: %d", flag);
        printf("%s\n", buffer_b);
    }

    /* Total stack usage may be ~100 bytes, not 200 */
}

    Explanation¶

    the key insight here is stack slot reuse - the compiler allocates only 100 bytes of stack space total, not 200, because the two buffers never exist at the same time.

    Stack Frame Setup¶

    The function starts by establishing a stack frame:

    push r28
push r29
in r28,__SP_L__
in r29,__SP_H__
subi r28,100
sbc r29,__zero_reg__

    This saves the frame pointer registers (r28 and r29) and allocates exactly 100 bytes of stack space. The subi (subtract immediate) and sbc (subtract with carry) instructions decrement the stack pointer by 100. Notice it's 100 bytes, not 200 - this is the optimization in action.

    The subsequent in/out instructions with __SREG__ and cli are interrupt-safety guards: they disable interrupts while modifying the stack pointer, then restore the interrupt state. This prevents corruption if an interrupt fires mid-operation.

    The Branch Logic¶

    sbiw r24,0
breq .L2

    The flag parameter arrives in r24:r25 (a 16-bit pair). sbiw (subtract immediate from word) checks if it's zero. If zero, breq (branch if equal) jumps to .L2 (the else branch). Otherwise, execution continues to the if branch.

    Both Branches Use the Same Stack Space¶

    If branch (flag != 0): 

    push r25
push r24          // Save flag value
ldi r24,lo8(.LC0)
ldi r25,hi8(.LC0) // Load address of "Path A: %d"
.L4:
push r25
push r24          // Push format string address
push __zero_reg__ // Padding byte
ldi r24,lo8(100)
push r24          // Push buffer size (100)
mov r24,r28
mov r25,r29
adiw r24,1        // Calculate buffer address (frame pointer + 1)
push r25
push r24          // Push buffer address
rcall snprintf    // Call snprintf

    Else branch (flag == 0): 

    push __zero_reg__
push __zero_reg__ // Padding (replaces the saved flag)
ldi r24,lo8(.LC1)
ldi r25,hi8(.LC1) // Load address of "Path B: %d"
rjmp .L4          // Jump to shared code

    Both branches jump to .L4, which calls snprintf with the buffer at the same stack location. The key is: buffer_a and buffer_b occupy the same 100 bytes because only one branch executes.

    Stack Cleanup and Return¶

    After the function body completes, the stack pointer is restored:

    subi r28,-100     // Add 100 back to stack pointer
sbci r29,-1       // Add carry
// ... interrupt-safety guards ...
pop r29
pop r28
ret

    The subi r28,-100 is equivalent to addi r28,100 - it adds 100 back, deallocating the local variables. Then the saved frame pointers are restored and the function returns.

    Why This Matters¶

    The comment in the source code says "Total stack usage may be ~100 bytes, not 200". The assembly confirms this: the compiler analyzed that buffer_a and buffer_b have non-overlapping lifetimes (one exists only in the if branch, the other only in the else branch), so it reused the same stack slot for both. This is a form of register allocation applied to stack memory - the compiler's optimizer recognized the opportunity and eliminated the waste.
