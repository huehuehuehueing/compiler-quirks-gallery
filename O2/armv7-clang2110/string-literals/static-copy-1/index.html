    static-copy-1¶

    Details

    Source: string-literals/static-copy-1.c Compiler: armv7-clang2110 (ARM) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    static-copy-1.c
    /*
 * String copy with strcpy - compiler may inline as immediate stores.
 *
 * For small, known-length strings, strcpy becomes direct
 * memory stores with the string data embedded in instructions.
 */
#include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[100];
    strcpy(buf, "ABCDXXXXCCCC");

    /* Make observable to prevent dead code elimination */
    puts(buf);
}

    Explanation¶

    the key insight here is that the compiler has replaced the strcpy call with direct memory writes of the string data, avoiding the overhead of a string copy function.

    Function Setup and Stack Frame¶

    The function starts by preparing space for the local buffer:

    push    {r11, lr}           // Save frame pointer and return address
sub     sp, sp, #104        // Allocate 104 bytes on stack (100 for buf + padding)

    In ARM32, the stack pointer (sp) grows downward in memory. The push instruction saves two registers that the function needs to restore later. The sub instruction makes room for the 100-byte buf array plus some extra space for alignment.

    The Optimization: Inlining String Data as Constants¶

    Instead of calling strcpy to copy the string "ABCDXXXXCCCC" byte-by-byte, the compiler has embedded the string data directly as constants and writes them to the buffer in bulk:

    ldr     r0, .LCPI0_0        // Load first 4 bytes of string (ABCD)
ldr     r1, .LCPI0_1        // Load next 4 bytes (XXXX)
ldr     r2, .LCPI0_2        // Load final 4 bytes (CCCC)
mov     r3, #0              // r3 = 0 (null terminator)
stm     sp, {r0, r1, r2}    // Store all three 4-byte chunks to buffer
strb    r3, [sp, #12]       // Store null terminator at position 12

    The three .long values at the end contain the ASCII bytes of the string packed into 32-bit words. The ldr (Load Register) instructions fetch these pre-computed values, and stm (Store Multiple) writes all three registers to the buffer in one efficient operation. Finally, strb (Store Byte) places the null terminator (\0) at the end.

    This is much faster than calling strcpy, which would loop through the string checking for the null terminator.

    Preparing the Function Call¶

    mov     r0, sp              // r0 = address of buffer (first parameter)
bl      puts                // Call puts() with buffer address

    By ARM32 calling convention, the first function parameter goes in r0. The mov instruction sets r0 to point to the buffer on the stack, then bl (Branch with Link) calls puts() to print it.

    Function Cleanup and Return¶

    add     sp, sp, #104        // Deallocate stack space
pop     {r11, pc}           // Restore frame pointer and return to caller

    The stack is restored to its original size, and pop {r11, pc} returns control to the caller by loading the return address into the program counter (pc).

    Why This Optimization Works¶

    The compiler can apply this optimization because: - The string is a compile-time constant with a known, small length (12 bytes) - It can pre-compute the string data and embed it as constants - Bulk writes with stm are more efficient than a function call with per-byte copying

    This is a form of inlining and constant folding—the compiler replaces a function call with direct operations on constant data.
