    inline-expansion¶

    Details

    Source: control-flow/inline-expansion.c Compiler: armv7-clang2110 (ARM) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    inline-expansion.c
    /*
 * Function inlining: small functions are expanded at call site.
 *
 * Eliminates call overhead and enables further optimizations
 * by exposing the function body to the caller's context.
 */

/* Tiny function - almost certainly inlined */
static inline int square(int x)
{
    return x * x;
}

/* Using static helps inlining (internal linkage) */
static int cube(int x)
{
    return x * x * x;
}

/* Wrapper that uses inlined functions */
int compute(int x)
{
    /* After inlining, this becomes:
     * return (x * x) + (x * x * x);
     * Which may further simplify to:
     * return x * x * (1 + x);
     */
    return square(x) + cube(x);
}

/* Recursive functions are NOT inlined (infinite expansion) */
int fibonacci(int n)
{
    if (n <= 1) return n;
    return fibonacci(n-1) + fibonacci(n-2);
}

/* Large functions may not be inlined (code size cost) */
int large_function(int x)
{
    int result = 0;
    for (int i = 0; i < 100; i++) {
        result += x * i;
        result ^= (result >> 3);
        result += (result << 5);
    }
    return result;
}

    Explanation¶

    the key insight is that the compiler has inlined the small functions but kept the large and recursive ones as separate function calls.

    compute(int) - Inlining in Action¶

    mul     r1, r0, r0      // r1 = x * x (square)
add     r0, r0, #1      // r0 = x + 1
mul     r0, r1, r0      // r0 = (x*x) * (x+1) = x(x+1) = x + x
bx      lr              // return

    This is the most important pattern: both square() and cube() have been completely inlined and algebraically simplified. Instead of computing x*x + x*x*x separately, the compiler recognized that this equals x*x * (x+1) and generates just three instructions. The function call overhead is eliminated entirely.

    By convention in ARM32, r0 holds the input parameter and the return value. The bx lr instruction returns to the caller (the lr register holds the return address).

    fibonacci(int) - Recursive Functions Stay as Calls¶

    push    {r4, r5, r11, lr}    // Save registers and return address
mov     r4, r0               // r4 = n (save input)
mov     r5, #0               // r5 = 0 (accumulator)
cmp     r0, #2               // Compare n with 2
bge     .LBB1_2              // If n >= 2, jump to recursive case

    Recursive functions cannot be inlined because inlining would create infinite code expansion. Instead, the compiler generates a proper function with: - Stack frame setup (push saves registers that need to be preserved across the recursive calls) - Base case check (if n < 2, return n) - Recursive calls using bl fibonacci(int) (branch with link - a function call)

    The recursive calls happen at .LBB1_2, where the function calls itself twice per iteration.

    large_function(int) - Loop Unrolling Avoided¶

    mov     r2, #100        // Loop counter = 100
mov     r3, #0          // i = 0
mov     r1, #0          // result = 0
.LBB2_1:                // Loop start
    add     r1, r1, r3      // result += i
    eor     r1, r1, r1, asr #3   // result ^= (result >> 3)
    add     r1, r1, r1, lsl #5   // result += (result << 5)
    add     r3, r3, r0      // i += x
    subs    r2, r2, #1      // Decrement counter, set flags
    bne     .LBB2_1         // Loop if not zero

    This function stays as a loop rather than being unrolled. The compiler chose not to expand the 100 iterations because: - The loop body contains complex operations (bitwise XOR, shifts) - Unrolling would create very large code - The performance gain wouldn't justify the code size increase

    The subs instruction (subtract and set flags) is used to decrement the counter and check if we should loop again with bne (branch if not equal to zero).

    Key takeaway: The -O2 optimization level aggressively inlines tiny functions like square() and cube(), even simplifying the math. But it respects the limits of inliningrecursive functions and large loops remain as function calls and loops to avoid code bloat.
