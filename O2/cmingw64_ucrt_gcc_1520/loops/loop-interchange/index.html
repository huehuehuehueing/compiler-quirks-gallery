    loop-interchange¶

    Details

    Source: loops/loop-interchange.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    loop-interchange.c
    /*
 * Loop Interchange
 *
 * When iterating over a 2D array, the order of the loop indices
 * determines memory access patterns.  In C (row-major layout),
 * iterating the inner loop over columns and the outer loop over
 * rows yields sequential access.  Swapping the loops ("interchange")
 * can improve cache locality when the original order would cause
 * stride-N accesses.
 *
 * Here the "bad" order (column-major traversal on a row-major array)
 * is presented; the compiler may interchange the loops at -O3.
 */
#define N 128

void column_sum(int mat[N][N], int result[N])
{
    /* Column-major traversal: inner loop strides by N ints.
     * The compiler may swap i and j for better locality. */
    for (int j = 0; j < N; j++)
    {
        result[j] = 0;
        for (int i = 0; i < N; i++)
        {
            result[j] += mat[i][j];
        }
    }
}

    Explanation¶

    the key insight is that the compiler has interchanged the loops to improve memory access patterns, which is exactly what the comment predicted at -O2 optimization.

    The Core Pattern: Loop Interchange¶

    The original C code iterates column-by-column (outer loop over columns j, inner loop over rows i). This causes stride-N memory access — each access jumps 512 bytes (128 integers × 4 bytes) through memory, which is terrible for CPU cache performance.

    The compiler has rearranged the loops so the inner loop now iterates sequentially through memory. This is a classic optimization called loop interchange.

    Register Setup (Lines 1-3)¶

    lea     r8, 65536[rcx]      ; r8 = end of matrix (rcx + 65536)
lea     r9, 66048[rcx]      ; r9 = end of result array (rcx + 66048)
mov     rcx, rdx            ; rcx = result array pointer (from rdx parameter)

    These lea (Load Effective Address) instructions calculate boundary pointers without accessing memory — they just compute addresses. The compiler pre-calculates where the data ends so it can use simple comparisons in the loop instead of recalculating bounds repeatedly.

    • 65536 = 128 × 128 × 4 bytes (the entire matrix)
    • 66048 = 65536 + 512 (the entire result array, which has 128 elements × 4 bytes)

    Outer Loop: Iterating Over Columns (.L3)¶

    .L3:
    mov     DWORD PTR [rcx], 0      ; result[j] = 0
    lea     rax, -65536[r8]         ; rax = start of current column
    xor     edx, edx                ; edx = 0 (accumulator)

    For each column j: 1. Initialize result[j] to 0 2. Set rax to point to the start of the current column in the matrix 3. Clear edx (the accumulator register that will sum the column)

    Inner Loop: Summing a Column (.L2)¶

    .L2:
    add     edx, DWORD PTR [rax]    ; accumulator += mat[i][j]
    add     rax, 512                ; rax += 512 (move to next row, same column)
    mov     DWORD PTR [rcx], edx    ; result[j] = accumulator
    cmp     rax, r8                 ; have we reached the end?
    jne     .L2                     ; if not, loop

    This inner loop walks sequentially down a single column: - add edx, DWORD PTR [rax] loads one integer from the current position and adds it to the running sum - add rax, 512 moves to the next row in the same column (512 = 128 integers × 4 bytes per row) - The cmp and jne check if we've processed all 128 rows

    Notice the memory access pattern: each iteration accesses memory at addresses that differ by exactly 512 bytes — this is sequential column traversal, which is cache-friendly.

    Outer Loop Continuation (Lines 15-17)¶

        lea     r8, 4[rax]              ; r8 = next column's starting address
    add     rcx, 4                  ; rcx = next result[j] position
    cmp     r8, r9                  ; have we processed all columns?
    jne     .L3                     ; if not, loop

    After finishing one column, the compiler: 1. Advances r8 to the next column (by adding 4 bytes, moving one integer to the right) 2. Advances rcx to the next result array element 3. Checks if we've finished all 128 columns

    The original code would access memory like this: [0,0] → [1,0] → [2,0] → ... → [127,0] (jumping 512 bytes each time). The interchanged version accesses: `[0,0] → [0,1] → [0,2] → ... →
