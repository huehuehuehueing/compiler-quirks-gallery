    stack-reuse¶

    Details

    Source: memory/stack-reuse.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    stack-reuse.c
    /*
 * Stack slot reuse: non-overlapping locals share stack space.
 *
 * The compiler analyzes variable lifetimes and reuses
 * stack slots for variables that are never live at the same time.
 */
#include <stdio.h>

void example(int flag)
{
    if (flag) {
        /* This buffer is only live in this branch */
        char buffer_a[100];
        snprintf(buffer_a, sizeof(buffer_a), "Path A: %d", flag);
        printf("%s\n", buffer_a);
    } else {
        /* This buffer may reuse the same stack slot as buffer_a
         * since they're never both alive at the same time.
         */
        char buffer_b[100];
        snprintf(buffer_b, sizeof(buffer_b), "Path B: %d", flag);
        printf("%s\n", buffer_b);
    }

    /* Total stack usage may be ~100 bytes, not 200 */
}

    Explanation¶

    the key insight is that the compiler is reusing the same stack space for both buffers since they never exist at the same time.

    The Core Optimization: Stack Slot Reuse¶

    The most important pattern here is that both buffer_a and buffer_b share the same 100-byte stack location. Notice that the function allocates only 168 bytes total (sub rsp, 168), not 200+ bytes. This is stack slot reuse in action—the compiler analyzed that the two buffers have non-overlapping lifetimes (one exists in the if branch, the other in the else branch), so it reuses the same memory location for both.

    Stack Frame Setup¶

    sub     rsp, 168

    This instruction moves the stack pointer down by 168 bytes to reserve space for local variables. The stack grows downward in amd64 (toward lower addresses), so subtracting from rsp allocates space. The 168 bytes includes: - 100 bytes for the buffer (shared between both branches) - Additional space for alignment and temporary storage during function calls

    The if Branch (Path A)¶

    lea     r10, 48[rsp]           // Calculate buffer address: rsp + 48
mov     r9d, ecx               // r9d = flag (4th parameter)
mov     edx, 100               // edx = 100 (3rd parameter: buffer size)
mov     rcx, r10               // rcx = buffer address (1st parameter)
lea     r8, .LC0[rip]          // r8 = address of "Path A: %d" string
mov     QWORD PTR 40[rsp], r10 // Save buffer address on stack
call    snprintf               // Call snprintf(buffer, 100, "Path A: %d", flag)

    The lea (Load Effective Address) instruction calculates the buffer's address without accessing memory—it computes rsp + 48 and stores the result in r10. This is where the 100-byte buffer lives. The buffer address is then saved at 40[rsp] (a temporary storage location) before calling snprintf.

    By convention in the amd64 calling convention: - rcx = 1st parameter (buffer pointer) - rdx = 2nd parameter (buffer size) - r8 = 3rd parameter (format string) - r9 = 4th parameter (flag value)

    After snprintf returns, the buffer address is reloaded and passed to puts:

    mov     rcx, QWORD PTR 40[rsp]
call    puts

    The else Branch (Path B)¶

    lea     rcx, 48[rsp]           // Calculate buffer address: rsp + 48 (SAME location!)
xor     r9d, r9d               // r9d = 0 (flag is 0 in else branch)
mov     edx, 100               // edx = 100
lea     r8, .LC1[rip]          // r8 = address of "Path B: %d" string
mov     QWORD PTR 40[rsp], rcx // Save buffer address on stack
call    snprintf

    Notice the critical detail: both branches use 48[rsp] as the buffer location. The compiler reused the same stack slot because buffer_a is dead (no longer needed) when we enter the else branch, and buffer_b is only created here.

    The xor r9d, r9d instruction sets r9d to 0 (a common idiom for zeroing a register—it's faster than mov r9d, 0). This represents the flag parameter in the else branch, which is 0 since the condition was false.

    Stack Cleanup and Return¶

    add     rsp, 168
ret

    The add instruction deallocates the 168 bytes by moving the stack pointer back up, restoring the stack to its original state. Then ret returns to the caller.
