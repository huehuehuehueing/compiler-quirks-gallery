    inline-expansion¶

    Details

    Source: control-flow/inline-expansion.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    inline-expansion.c
    /*
 * Function inlining: small functions are expanded at call site.
 *
 * Eliminates call overhead and enables further optimizations
 * by exposing the function body to the caller's context.
 */

/* Tiny function - almost certainly inlined */
static inline int square(int x)
{
    return x * x;
}

/* Using static helps inlining (internal linkage) */
static int cube(int x)
{
    return x * x * x;
}

/* Wrapper that uses inlined functions */
int compute(int x)
{
    /* After inlining, this becomes:
     * return (x * x) + (x * x * x);
     * Which may further simplify to:
     * return x * x * (1 + x);
     */
    return square(x) + cube(x);
}

/* Recursive functions are NOT inlined (infinite expansion) */
int fibonacci(int n)
{
    if (n <= 1) return n;
    return fibonacci(n-1) + fibonacci(n-2);
}

/* Large functions may not be inlined (code size cost) */
int large_function(int x)
{
    int result = 0;
    for (int i = 0; i < 100; i++) {
        result += x * i;
        result ^= (result >> 3);
        result += (result << 5);
    }
    return result;
}

    Explanation¶

    the key insight here is function inlining in action you can see how the compiler transforms function calls into direct code expansion.

    The compute Function: Inlining in Action¶

    compute:
    mov     eax, ecx
    imul    eax, ecx
    imul    ecx, eax
    add     eax, ecx
    ret

    This is the most important pattern to understand. The source code calls two functions (square(x) and cube(x)), but there are no function call instructions here. Instead, the compiler has inlined both functions meaning it replaced the function calls with the actual computation code.

    Let's trace through what happens:

    1. mov eax, ecx Copy the input parameter x (passed in ecx by calling convention) into eax. This is the first step of computing x * x.

    2. imul eax, ecx Multiply eax by ecx. Since eax contains x and ecx contains x, this computes x * x and stores the result in eax. This is the square(x) function inlined.

    3. imul ecx, eax Multiply ecx by eax. Now ecx contains x (still), and eax contains x * x, so this computes x * (x * x) = x * x * x and stores it in ecx. This is the cube(x) function inlined.

    4. add eax, ecx Add the two results together: (x * x) + (x * x * x). The final result goes into eax, which is where return values are placed by convention.

    5. ret Return to the caller.

    Why is this important? Without inlining, the code would have two call instructions (expensive jumps to other functions) and two ret instructions. By inlining, the compiler eliminates this overhead and can optimize the combined computation more effectively.

    The fibonacci Function: Why Recursion Isn't Inlined¶

    The fibonacci function shows the opposite pattern it's not inlined because it's recursive (calls itself). The compiler can't inline a recursive function because that would create infinite expansion. Instead, you see:

    fibonacci:
    push    r15
    ... (285 lines omitted) ...
    pop     r15
    ret

    The function saves registers on the stack (push instructions) and has a large body (285 lines omitted), indicating it contains the actual recursive logic with multiple call instructions to itself.

    The large_function: Loop-Based Code¶

    The visible snippet shows loop operations:

        mov     eax, edx
    sal     eax, 5          ; shift left by 5 (multiply by 32)
    add     eax, edx
    sub     r8d, 1          ; decrement loop counter
    jne     .L56            ; jump if not equal (loop back)
    ret

    This function isn't inlined either it's too large. The loop structure (with sub and jne for loop control) would expand the caller's code too much if inlined, so the compiler keeps it as a separate function.

    Key Observations¶

    • Inlining eliminates call overhead: Small functions like square and cube are expanded inline, removing expensive function call instructions.
    • The compiler is smart about trade-offs: It inlines tiny functions but keeps large or recursive functions separate to avoid code bloat.
    • Register usage follows conventions: Input parameters arrive in ecx (first integer parameter), and the return value goes in eax.
