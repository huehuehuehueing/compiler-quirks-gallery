    if-conversion¶

    Details

    Source: control-flow/if-conversion.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    if-conversion.c
    /*
 * If-Conversion (branch to conditional move)
 *
 * Short if-else chains that select between two values can be
 * replaced with conditional-move instructions (cmov on x86).
 * This eliminates a branch, avoiding potential misprediction
 * penalties on modern super-scalar CPUs.
 *
 * Look for: cmovl / cmovge / cmovne instead of jl / jge / jne
 * with a branch over a mov.
 */

int min(int a, int b)
{
    if (a < b)
        return a;
    return b;
}

int clamp(int x, int lo, int hi)
{
    if (x < lo)
        x = lo;
    if (x > hi)
        x = hi;
    return x;
}

int abs_val(int x)
{
    return x < 0 ? -x : x;
}

    Explanation¶

    the key insight here is that the compiler has replaced traditional branching (jumps) with conditional move instructions (cmov), which is a powerful optimization technique.

    The Core Optimization: If-Conversion¶

    Instead of using branches that could cause CPU pipeline delays, the compiler uses cmov instructions. These instructions conditionally move a value into a register based on CPU flags, eliminating the need for jumps. This is faster on modern CPUs because branches can cause pipeline stalls when the CPU guesses wrong about which direction the code will take.

    Function-by-Function Breakdown¶

    min function¶

    cmp     ecx, edx        # Compare first parameter (a) with second (b)
mov     eax, edx        # Move b into return register (eax)
cmovle  eax, ecx        # If a <= b, move a into eax instead
ret

    What's happening: - cmp ecx, edx compares the two input values and sets CPU flags - mov eax, edx assumes we'll return b (the second parameter) - cmovle eax, ecx says: "if the comparison showed less-than-or-equal, use a instead" - The result is whichever value is smaller, returned in eax (the standard return register for integers)

    Why this is clever: There's no jump instruction. The CPU doesn't have to guess which path to take—it executes both the mov and the cmov, but only one actually affects the result.

    clamp function¶

    cmp     ecx, edx        # Compare x with lo
mov     eax, r8d        # Move hi into return register
cmovl   ecx, edx        # If x < lo, set x to lo
cmp     ecx, r8d        # Compare (possibly adjusted) x with hi
cmovle  eax, ecx        # If x <= hi, use x; otherwise use hi
ret

    What's happening: - First cmp and cmovl handle the lower bound: if x < lo, replace it with lo - Second cmp and cmovle handle the upper bound: if the adjusted value exceeds hi, cap it at hi - The final result in eax is the clamped value

    Key detail: Notice ecx is reused—after the first conditional move, it holds the adjusted value, which is then compared against the upper limit.

    abs_val function¶

    mov     eax, ecx        # Move input into return register
neg     eax             # Negate it (compute -x)
cmovs   eax, ecx        # If sign flag is set, use original x instead
ret

    What's happening: - neg eax computes the negative of the input - cmovs checks the sign flag (set when a negative result occurs) - If the original value was negative, we keep the negated version; otherwise, we use the original

    The trick: By negating first and then conditionally moving, the compiler avoids branching entirely. The cmovs instruction reads the sign flag that neg just set.

    Why This Matters¶

    Traditional approach (without optimization): 

    cmp ecx, edx
jl  use_a        # Jump if less
mov eax, edx     # Use b
jmp done
use_a:
mov eax, ecx     # Use a
done:
ret

    Optimized approach (what you see): 

    cmp ecx, edx
mov eax, edx
cmovle eax, ecx
ret

    The optimized version has fewer instructions and no jumps, so the CPU's instruction pipeline stays full and predictable. Modern CPUs are very fast at executing instructions in sequence, but jumps can cause the pipeline to stall while the CPU figures out where to go next.
