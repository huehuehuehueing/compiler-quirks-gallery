    static-init-1¶

    Details

    Source: string-literals/static-init-1.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: O2 - Standard Optimization (-O2)

    Source Code¶

    static-init-1.c
    /*
 * Static string initialization with concatenation.
 *
 * Adjacent string literals are concatenated at compile time.
 * The initializer may become a single memcpy or inline stores.
 */
#include <string.h>
#include <stdio.h>

void foo(void)
{
    /* Adjacent strings concatenated at compile time: "AAAABBBB" */
    char buf[100] = { "AAAA" "BBBB" };

    /* Make observable to prevent dead code elimination */
    puts(buf);
}

    Explanation¶

    the key insight is that the compiler has pre-computed the concatenated string at compile time and is initializing the buffer with direct memory writes.

    Stack Frame Setup¶

    The function starts by allocating space on the stack: 

    sub     rsp, 152

    This reserves 152 bytes of stack memory. Since the source code declares char buf[100], you might wonder why 152 bytes are needed. The extra space accounts for stack alignment requirements on amd64 (the stack must be 16-byte aligned for function calls) and provides working space for the initialization.

    String Initialization Strategy¶

    Rather than copying the string byte-by-byte, the compiler uses a clever optimization: it pre-computes the concatenated string "AAAABBBB" and stores it as a single 8-byte value.

    movabs  rax, 4774451407296217409

    This instruction loads a large constant into rax. This number is actually the concatenated string "AAAABBBB" encoded as a 64-bit integer (in little-endian format, the bytes spell out the ASCII characters). The movabs instruction is needed because the constant is too large for a regular mov instruction.

    lea     rcx, 32[rsp]

    This calculates the address where the buffer starts (32 bytes into the stack frame) and stores it in rcx. The lea (Load Effective Address) instruction performs address arithmetic without accessing memory—it's just computing rsp + 32.

    mov     QWORD PTR 32[rsp], rax

    This writes the 8-byte string value to the buffer's starting location. "QWORD PTR" means we're writing a 64-bit (8-byte) value.

    Zeroing the Rest of the Buffer¶

    The remaining 92 bytes of the 100-byte buffer are cleared to zero:

    xor     edx, edx          // edx = 0
mov     QWORD PTR 40[rsp], rdx
mov     QWORD PTR 48[rsp], 0
mov     QWORD PTR 56[rsp], 0
// ... (continues for remaining 8-byte chunks)
mov     DWORD PTR 128[rsp], 0  // final 4 bytes

    The xor edx, edx instruction sets edx to zero (XORing a register with itself always produces zero—this is a common idiom). Then the compiler writes zeros in 8-byte chunks to clear the rest of the buffer. The final mov DWORD PTR 128[rsp], 0 writes the last 4 bytes as a 32-bit value.

    Function Call and Cleanup¶

    call    puts
nop
add     rsp, 152
ret

    The call puts invokes the standard library function to print the string. The nop (no operation) is padding, likely for alignment. Finally, add rsp, 152 deallocates the stack space we reserved at the beginning, and ret returns to the caller.

    Key Observations¶

    String concatenation happens at compile time: The compiler merged "AAAA" and "BBBB" into a single 8-byte constant before generating any code—this is a compile-time optimization, not a runtime operation.

    Efficient initialization: Rather than initializing the buffer byte-by-byte, the compiler uses 8-byte writes (and one 4-byte write), which is much faster on modern processors.

    The rcx register is computed but unused: The lea rcx, 32[rsp] instruction calculates the buffer address, but rcx is never actually used afterward. This is likely a minor inefficiency in the code generation—the address calculation was unnecessary since puts doesn't need it (the buffer address is already on the stack where puts expects it).
