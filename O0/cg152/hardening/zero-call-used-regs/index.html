    zero-call-used-regs¶

    Details

    Source: hardening/zero-call-used-regs.c Compiler: cg152 (x86/x64) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    zero-call-used-regs.c
    /* @gallery-hints
 *   extra-flags: -fzero-call-used-regs=used
 *   compiler-only: cg152, clang1910
 */

/*
 * Zero Call-Used Registers
 *
 * -fzero-call-used-regs=used clears every register that the function
 * actually used before returning. This mitigates information leakage
 * through registers and makes ROP gadgets less useful because
 * attackers cannot rely on register contents at a ret instruction.
 *
 * Look for: xor or zero instructions on registers (eax, ecx, edx,
 * xmm0, etc.) immediately before the ret instruction.
 */

int compute(int a, int b, int c)
{
    int result = a * b + c;
    return result;
}

long long widen(int x)
{
    return (long long)x * x;
}

    Explanation¶

    the key insight here is that this code demonstrates a security feature that clears registers before returning from functions.

    The Security Feature: Register Zeroing¶

    The most important pattern to notice is the xor instructions immediately before each ret instruction. These instructions zero out registers that the function used during execution. This is the -fzero-call-used-regs=used flag in action—it's a security hardening technique that prevents sensitive data from leaking through registers after the function returns.

    compute Function Breakdown¶

    Stack Setup: 

    push    rbp              // Save old frame pointer
mov     rbp, rsp         // Create new stack frame
    This establishes a stack frame—think of it as creating a local workspace on the stack where the function stores its variables.

    Parameter Storage: 

    mov     DWORD PTR [rbp-20], edi    // Store parameter 'a' (passed in edi)
mov     DWORD PTR [rbp-24], esi    // Store parameter 'b' (passed in esi)
mov     DWORD PTR [rbp-28], edx    // Store parameter 'c' (passed in edx)
    By convention, the first three integer parameters are passed in registers edi, esi, and edx. This code immediately saves them to the stack—this is typical of unoptimized code, which doesn't keep values in registers.

    Computation: 

    mov     eax, DWORD PTR [rbp-20]    // Load 'a' into eax
imul    eax, DWORD PTR [rbp-24]    // Multiply by 'b': eax = a * b
mov     edx, eax                    // Save result to edx
mov     eax, DWORD PTR [rbp-28]    // Load 'c' into eax
add     eax, edx                    // Add: eax = c + (a*b)
mov     DWORD PTR [rbp-4], eax     // Store result on stack
    The computation follows the source code: multiply a * b, then add c. The result is stored on the stack at [rbp-4].

    Return and Register Clearing: 

    mov     eax, DWORD PTR [rbp-4]    // Load result into eax (return value)
pop     rbp                         // Restore frame pointer
xor     edx, edx                    // Zero edx (used for parameter 'c')
xor     esi, esi                    // Zero esi (used for parameter 'b')
xor     edi, edi                    // Zero edi (used for parameter 'a')
ret
    The xor instructions clear the three registers that held input parameters. xor reg, reg is a common idiom to zero a register—it's faster than moving zero directly.

    widen Function Breakdown¶

    Parameter Storage: 

    push    rbp
mov     rbp, rsp
mov     DWORD PTR [rbp-4], edi     // Store parameter 'x' on stack

    Type Conversion and Multiplication: 

    mov     eax, DWORD PTR [rbp-4]     // Load x into eax (32-bit)
movsx   rdx, eax                    // Sign-extend eax to 64-bit rdx
mov     eax, DWORD PTR [rbp-4]     // Load x again into eax
cdqe                                // Sign-extend eax to 64-bit rax
imul    rax, rdx                    // Multiply: rax = x * x (64-bit result)

    The key here is handling the type conversion from int (32-bit) to long long (64-bit). The function loads x twice and sign-extends it two different ways: - movsx rdx, eax extends the 32-bit value in eax to a full 64-bit value in rdx - cdqe (Convert Doubleword to Quadword Extended) extends eax to rax

    Then imul rax, rdx multiplies these 64-bit values, producing the final result in rax.
