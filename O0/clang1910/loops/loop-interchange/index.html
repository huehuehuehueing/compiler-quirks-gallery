    loop-interchange¶

    Details

    Source: loops/loop-interchange.c Compiler: clang1910 (x86/x64) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    loop-interchange.c
    /*
 * Loop Interchange
 *
 * When iterating over a 2D array, the order of the loop indices
 * determines memory access patterns.  In C (row-major layout),
 * iterating the inner loop over columns and the outer loop over
 * rows yields sequential access.  Swapping the loops ("interchange")
 * can improve cache locality when the original order would cause
 * stride-N accesses.
 *
 * Here the "bad" order (column-major traversal on a row-major array)
 * is presented; the compiler may interchange the loops at -O3.
 */
#define N 128

void column_sum(int mat[N][N], int result[N])
{
    /* Column-major traversal: inner loop strides by N ints.
     * The compiler may swap i and j for better locality. */
    for (int j = 0; j < N; j++)
    {
        result[j] = 0;
        for (int i = 0; i < N; i++)
        {
            result[j] += mat[i][j];
        }
    }
}

    Explanation¶

    the key insight is that this is unoptimized code (-O0) that implements a column-major traversal of a 2D array with poor memory access patterns. The compiler makes no attempt to optimize the loop structure, resulting in inefficient stride-N memory accesses.

    Stack Frame Setup¶

    The function begins with standard stack frame management: - push rbp saves the old base pointer - mov rbp, rsp establishes a new stack frame - The next four mov instructions store the function parameters on the stack: - [rbp - 8]: the matrix pointer (mat) - [rbp - 16]: the result array pointer (result) - [rbp - 20]: the outer loop variable j - [rbp - 24]: the inner loop variable i

    This is typical unoptimized behaviorparameters and local variables are stored on the stack rather than kept in registers, requiring repeated memory loads.

    Outer Loop Structure (.LBB0_1)¶

    The outer loop iterates j from 0 to 128: - cmp dword ptr [rbp - 20], 128 compares j against 128 - jge .LBB0_8 jumps to the end when j >= 128

    Before the inner loop, result[j] is initialized to 0: - mov rax, qword ptr [rbp - 16] loads the result pointer - movsxd rcx, dword ptr [rbp - 20] sign-extends j to 64-bit (necessary for 64-bit address arithmetic) - mov dword ptr [rax + 4*rcx], 0 stores 0 at result[j] (the 4*rcx scales the index by 4 bytes since integers are 4 bytes)

    Inner Loop Structure (.LBB0_3)¶

    The inner loop iterates i from 0 to 128, accumulating mat[i][j] into result[j]:

    Computing the matrix element address: - mov rax, qword ptr [rbp - 8] loads the matrix pointer - movsxd rcx, dword ptr [rbp - 24] sign-extends i to 64-bit - shl rcx, 9 shifts i left by 9 bits (multiplying by 512, which is 128 integers 4 bytes per integer)this moves to row i - add rax, rcx adds this offset to the matrix base address - movsxd rcx, dword ptr [rbp - 20] sign-extends j to 64-bit - mov edx, dword ptr [rax + 4*rcx] loads mat[i][j] (the 4*rcx adds the column offset)

    Accumulating the sum: - mov rax, qword ptr [rbp - 16] reloads the result pointer - movsxd rcx, dword ptr [rbp - 20] sign-extends j again - add edx, dword ptr [rax + 4*rcx] adds result[j] to the loaded matrix element - mov dword ptr [rax + 4*rcx], edx stores the updated sum back to result[j]

    Loop increment: - mov eax, dword ptr [rbp - 24] loads i - add eax, 1 increments it - mov dword ptr [rbp - 24], eax stores it back - jmp .LBB0_3 jumps back to the inner loop condition

    The Performance Problem¶

    This code exhibits poor cache locality because it accesses the matrix in column-major order (jumping by 512 bytesone full rowbetween consecutive accesses). Modern CPUs fetch data in cache lines, so these large strides cause frequent cache misses. At -O3, the compiler would interchange the loops to iterate rows in the inner loop, achieving sequential memory access.

    Additionally, this is unoptimized code with significant redundancy: variables are repeatedly loaded from and stored to the stack, and the compiler makes no attempt to keep loop variables
