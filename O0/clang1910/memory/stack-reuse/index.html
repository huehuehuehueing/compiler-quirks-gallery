    stack-reuse¶

    Details

    Source: memory/stack-reuse.c Compiler: clang1910 (x86/x64) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    stack-reuse.c
    /*
 * Stack slot reuse: non-overlapping locals share stack space.
 *
 * The compiler analyzes variable lifetimes and reuses
 * stack slots for variables that are never live at the same time.
 */
#include <stdio.h>

void example(int flag)
{
    if (flag) {
        /* This buffer is only live in this branch */
        char buffer_a[100];
        snprintf(buffer_a, sizeof(buffer_a), "Path A: %d", flag);
        printf("%s\n", buffer_a);
    } else {
        /* This buffer may reuse the same stack slot as buffer_a
         * since they're never both alive at the same time.
         */
        char buffer_b[100];
        snprintf(buffer_b, sizeof(buffer_b), "Path B: %d", flag);
        printf("%s\n", buffer_b);
    }

    /* Total stack usage may be ~100 bytes, not 200 */
}

    Explanation¶

    the key insight is that the compiler allocates 224 bytes of stack space upfront, but reuses it for both buffers since they never exist at the same time.

    Stack Frame Setup¶

    push    rbp
mov     rbp, rsp
sub     rsp, 224

    These three instructions establish the function's stack frame. Think of the stack as a region of memory that grows downward (toward lower addresses). The rbp register acts as a "frame pointer"â€”a fixed reference point for accessing local variables. The sub rsp, 224 instruction reserves 224 bytes of stack space for all local variables in this function.

    Why 224 bytes when each buffer is only 100 bytes? The compiler allocates enough space to hold the largest possible set of variables that could be live simultaneously. Since the two buffers are in mutually exclusive branches (one in the if, one in the else), only one exists at a time, so 224 bytes covers both with some padding for alignment.

    Parameter Storage¶

    mov     dword ptr [rbp - 4], edi

    The flag parameter arrives in the edi register (by the x86-64 calling convention, the first integer argument goes in rdi). This instruction stores it at [rbp - 4]â€”4 bytes below the frame pointer. This is unoptimized code; an optimized compiler would keep the parameter in the register instead of storing and reloading it.

    Conditional Branch¶

    cmp     dword ptr [rbp - 4], 0
je      .LBB0_2

    The cmp instruction compares the stored flag value to 0. The je (jump if equal) instruction branches to .LBB0_2 if the comparison is trueâ€”that's the else branch. If the comparison is false, execution continues to the if branch code.

    The if Branch (Path A)¶

    lea     rdi, [rbp - 112]
mov     ecx, dword ptr [rbp - 4]
mov     esi, 100
lea     rdx, [rip + .L.str]
mov     al, 0
call    snprintf@PLT

    This prepares arguments for snprintf(). The lea (Load Effective Address) instruction calculates an address without accessing memoryâ€”it's just arithmetic. [rbp - 112] is where buffer_a lives on the stack.

    The arguments are set up following the x86-64 calling convention: - rdi = pointer to buffer_a (the destination buffer) - esi = 100 (the buffer size) - rdx = pointer to the format string "Path A: %d" - ecx = the flag value - al = 0 (used for variadic functions to indicate no vector registers contain arguments)

    Then snprintf is called. After it returns, printf is called similarly to print the buffer.

    The else Branch (Path B)¶

    lea     rdi, [rbp - 224]
mov     ecx, dword ptr [rbp - 4]
mov     esi, 100
lea     rdx, [rip + .L.str.2]
mov     al, 0
call    snprintf@PLT

    This is nearly identical to the if branch, but notice the critical difference: buffer_b is placed at [rbp - 224] instead of [rbp - 112]. This is the stack slot reuse in action.

    The compiler analyzed the code and determined: - buffer_a (at [rbp - 112]) is only used in the if branch - buffer_b (at [rbp - 224]) is only used in the else branch - They're never both alive, so they can occupy different regions of the same 224-byte allocation

    If the compiler couldn't reuse stack slots, it would need 200 bytes (100 + 100). Instead, it cleverly uses the same space for both, just at different offsets.

    add     rsp, 224
pop     rbp
ret
