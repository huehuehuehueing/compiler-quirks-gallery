    zero-call-used-regs¶

    Details

    Source: hardening/zero-call-used-regs.c Compiler: clang1910 (x86/x64) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    zero-call-used-regs.c
    /* @gallery-hints
 *   extra-flags: -fzero-call-used-regs=used
 *   compiler-only: cg152, clang1910
 */

/*
 * Zero Call-Used Registers
 *
 * -fzero-call-used-regs=used clears every register that the function
 * actually used before returning. This mitigates information leakage
 * through registers and makes ROP gadgets less useful because
 * attackers cannot rely on register contents at a ret instruction.
 *
 * Look for: xor or zero instructions on registers (eax, ecx, edx,
 * xmm0, etc.) immediately before the ret instruction.
 */

int compute(int a, int b, int c)
{
    int result = a * b + c;
    return result;
}

long long widen(int x)
{
    return (long long)x * x;
}

    Explanation¶

    the key insight here is that the -fzero-call-used-regs=used flag is causing the compiler to deliberately clear registers before returning, which is a security feature.

    The Core Pattern: Register Zeroing for Security¶

    The most important thing to notice is the xor instructions immediately before each ret instruction. These are not part of the actual computation—they're security measures that clear registers to prevent information leakage.

    xor     edi, edi      // Clear rdi register
xor     edx, edx      // Clear rdx register
xor     esi, esi      // Clear rsi register
ret

    This pattern appears in both functions. The xor instruction with the same register as both operands (like xor edi, edi) is a standard way to set a register to zero. This happens because the -fzero-call-used-regs=used flag tells the compiler: "Before returning, zero out every register this function actually used."

    Why This Matters¶

    Normally, when a function returns, the registers it used still contain data. An attacker could potentially read this leftover data or use it in a "return-oriented programming" (ROP) attack. By explicitly zeroing these registers, the compiler eliminates this attack surface.

    Function 1: compute(int, int, int)¶

    This function computes a * b + c. Here's the flow:

    Parameter storage (unoptimized pattern): 

    mov     dword ptr [rbp - 4], edi    // Store parameter a on stack
mov     dword ptr [rbp - 8], esi    // Store parameter b on stack
mov     dword ptr [rbp - 12], edx   // Store parameter c on stack

    By convention, the first three integer parameters arrive in registers rdi, rsi, and rdx. This code immediately spills them to the stack—this is typical unoptimized behavior (-O0). An optimized compiler would keep them in registers.

    Computation: 

    mov     eax, dword ptr [rbp - 4]    // Load a into eax
imul    eax, dword ptr [rbp - 8]    // Multiply by b: eax = a * b
add     eax, dword ptr [rbp - 12]   // Add c: eax = (a * b) + c
mov     dword ptr [rbp - 16], eax   // Store result on stack
mov     eax, dword ptr [rbp - 16]   // Load result back into eax

    The result is computed in eax (the 32-bit portion of rax), which is the standard register for returning integer values. Notice the inefficiency: the result is stored to the stack, then immediately loaded back—this is unoptimized code.

    Security cleanup: 

    xor     edi, edi      // Zero rdi (held parameter a)
xor     edx, edx      // Zero rdx (held parameter c)
xor     esi, esi      // Zero rsi (held parameter b)
ret

    The three parameter registers are zeroed because the function used them.

    Function 2: widen(int)¶

    This function converts an int to long long and squares it: (long long)x * x.

    Parameter storage: 

    mov     dword ptr [rbp - 4], edi    // Store parameter x on stack

    Type conversion and multiplication: 

    movsxd  rax, dword ptr [rbp - 4]    // Sign-extend x to 64-bit in rax
movsxd  rcx, dword ptr [rbp - 4]    // Sign-extend x to 64-bit in rcx
imul    rax, rcx                     // Multiply: rax = x * x (as 64-bit)

    The movsxd instruction (move with sign extension, 64-bit destination) converts the 32-bit int to a 64-bit long long. This is necessary because multiplying two 32-bit values could overflow, so the compiler promotes to 64-bit before multiplying. The result stays in rax, which is the standard register for returning 64-bit values.
