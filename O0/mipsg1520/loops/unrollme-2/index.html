    unrollme-2¶

    Details

    Source: loops/unrollme-2.c Compiler: mipsg1520 (MIPS) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    unrollme-2.c
    #include <stdio.h>

void foo(void)
{
    int i;
    char buf[8];

    for (i = 0; i < sizeof(buf); i++)
    {
        buf[i] = 'A';
    }

    /*
     * by dereferencing buf this should force the compile to NOT
     * eliminate the above as dead code. loop will likely still
     * be unrolled
     */
    printf("%s", buf);
}

    Explanation¶

    This is unoptimized code (compiled with -O0), which means the compiler prioritizes correctness and simplicity over performance. You'll see several inefficiencies that an optimized compiler would eliminate.

    Key Insight: Stack Frame Setup and Loop Structure¶

    The most important pattern here is how the function manages its stack frame (the memory region where local variables are stored). The function allocates 48 bytes on the stack to hold an 8-byte character array and other data, then executes a simple loop that fills the array with the character 'A' (ASCII value 65).

    Stack Frame Setup¶

    addiu   $sp,$sp,-48          # Allocate 48 bytes on stack
sw      $31,44($sp)          # Save return address
sw      $fp,40($sp)          # Save frame pointer
move    $fp,$sp              # Set up new frame pointer

    The function starts by making room on the stack (the memory region used for local variables). The $sp register is the stack pointer (points to the top of the stack), and $fp is the frame pointer (marks the base of this function's stack frame). By convention in MIPS, $31 holds the return address—where execution should jump back to after this function completes. The function saves both the return address and the old frame pointer so they can be restored later.

    Loop Initialization and Condition Check¶

    sw      $0,24($fp)           # Initialize i = 0 (store 0 at offset 24)
b       $L2                  # Jump to loop condition check

    The variable i is stored at offset 24 from the frame pointer. The code jumps directly to the condition check ($L2) rather than executing the loop body first—this is a standard pattern for for loops.

    Loop Condition ($L2)¶

    $L2:
    lw      $2,24($fp)       # Load i into register $2
    nop                      # Delay slot (explained below)
    sltu    $2,$2,8          # Set $2 = 1 if i < 8, else 0
    bne     $2,$0,$L3        # Branch to loop body if condition is true
    nop

    This checks if i < 8 (the size of the buffer). The sltu instruction performs an unsigned less-than comparison. If true, it branches to $L3 (the loop body); if false, the loop exits.

    Note on nop instructions: These are "no operation" instructions that fill delay slots—a MIPS quirk where the instruction after a branch or load executes before the branch/load takes effect. The compiler inserts nop to avoid unexpected behavior.

    Loop Body ($L3)¶

    $L3:
    lw      $2,24($fp)       # Load i
    addiu   $3,$fp,24        # Calculate address of buf (frame pointer + 24)
    addu    $2,$3,$2         # Add i to buf's address: &buf[i]
    li      $3,65            # Load 65 (ASCII 'A') into $3
    sb      $3,4($2)         # Store byte 'A' at buf[i]

    lw      $2,24($fp)       # Load i again
    nop
    addiu   $2,$2,1          # Increment i
    sw      $2,24($fp)       # Store updated i back
    b       $L2              # Jump back to condition check

    This is where the inefficiency of unoptimized code becomes obvious. To execute buf[i] = 'A': 1. Load i from memory 2. Calculate the address of buf[i] by adding i to the buffer's base address 3. Store the character

    Then to increment i: 4. Load i from memory again 5. Increment it 6. Store it back

    An optimized compiler would keep i in a register throughout the loop, avoiding these repeated memory operations.
