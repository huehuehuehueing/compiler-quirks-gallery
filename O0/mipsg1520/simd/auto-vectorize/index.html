    auto-vectorize¶

    Details

    Source: simd/auto-vectorize.c Compiler: mipsg1520 (MIPS) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    auto-vectorize.c
    /*
 * Auto-vectorization: loops converted to SIMD operations.
 *
 * Compiler transforms scalar loops into vector operations
 * (SSE, AVX, NEON, etc.) processing multiple elements at once.
 */

/* Simple loop - prime candidate for vectorization */
void add_arrays(float *dst, const float *a, const float *b, int n)
{
    for (int i = 0; i < n; i++) {
        dst[i] = a[i] + b[i];
    }
    /* With -O3: becomes SIMD adds processing 4-8 floats at once */
}

/* Reduction - trickier to vectorize but often done */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++) {
        sum += arr[i];
    }
    return sum;
    /* May use horizontal add instructions or partial sums */
}

/* Loop with dependency - CANNOT be vectorized */
void fibonacci_array(int *arr, int n)
{
    arr[0] = 0;
    arr[1] = 1;
    for (int i = 2; i < n; i++) {
        /* Each iteration depends on previous two - serial */
        arr[i] = arr[i-1] + arr[i-2];
    }
}

/* Conditional in loop - may use masked operations */
void clamp_array(float *arr, int n, float min, float max)
{
    for (int i = 0; i < n; i++) {
        if (arr[i] < min) arr[i] = min;
        else if (arr[i] > max) arr[i] = max;
    }
}

    Explanation¶

    This is unoptimized code (compiled with -O0), which means the compiler prioritizes correctness and simplicity over performance. You'll see lots of redundant memory operations and function calls that optimized code would eliminate.

    Key Pattern: No Vectorization Despite Loop Opportunities¶

    The most important insight is that none of these loops use SIMD (vectorization) even though they're prime candidates for it. The source code comments mention vectorization, but with -O0, the compiler generates scalar code—processing one element at a time instead of multiple elements in parallel.

    Understanding MIPS Basics First¶

    Before diving into each function, here's what you need to know about MIPS:

    • Registers: $4-$7 hold the first four function parameters (by calling convention)
    • Stack frame: Functions use $sp (stack pointer) and $fp (frame pointer) to manage local variables
    • Memory access: lw (load word) reads from memory, sw (store word) writes to memory
    • Floating-point calls: MIPS doesn't have built-in floating-point arithmetic in this code, so it calls library functions like __addsf3 (add single-precision floats)

    Function 1: add_arrays — Simple Loop, No Vectorization¶

    This function should add two arrays element-by-element. Here's what happens:

    Loop body (label $L3): 1. Load i from stack, multiply by 4 (shift left 2 bits) to convert array index to byte offset 2. Load a[i] and b[i] from memory into registers $4 and $5 3. Call __addsf3 to add them (because MIPS needs a library function for floating-point math) 4. Store result to dst[i] 5. Increment i and loop back

    Why this is inefficient: - Each iteration calls __addsf3, which is expensive - With vectorization (-O3), the compiler would use SIMD instructions to add 4-8 floats in a single operation - The code loads and stores one float at a time instead of processing multiple floats together

    What optimized code would look like: With -O3, you'd see SIMD instructions (like addps on x86 or equivalent on MIPS) processing multiple array elements per iteration.

    Function 2: sum_array — Reduction Loop¶

    This accumulates all array elements into a single sum:

    Loop body (label $L6): 1. Load arr[i] into a register 2. Call __addsf3 to add it to the running sum 3. Store updated sum back to stack 4. Increment i and loop

    Why this is inefficient: - Every iteration stores sum to memory, then loads it back next iteration (store-then-load pattern) - With optimization, the compiler would keep sum in a register across iterations - Vectorized code could use horizontal add instructions to sum multiple elements in parallel

    Function 3: fibonacci_array — Why This Can't Be Vectorized¶

    This function demonstrates a data dependency that prevents vectorization:

    arr[i] = arr[i-1] + arr[i-2]

    Each element depends on the previous two elements. The assembly shows: - Complex address calculations using li (load immediate) with large constants to compute i-1 and i-2 offsets - The loop must execute serially because iteration i can't start until iteration i-2 completes

    Key insight: No amount of SIMD can help here—the CPU must compute elements sequentially. This is why the source code comments say "CANNOT be vectorized."
