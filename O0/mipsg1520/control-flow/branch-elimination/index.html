    branch-elimination¶

    Details

    Source: control-flow/branch-elimination.c Compiler: mipsg1520 (MIPS) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    branch-elimination.c
    /*
 * Branch elimination: compile-time constant conditions.
 *
 * When a branch condition is known at compile time,
 * the dead branch is eliminated entirely.
 */
#include <limits.h>

/* Debug flag - in release builds, debug code is eliminated */
#define DEBUG 0

void maybe_log(const char *msg)
{
    if (DEBUG) {
        /* This entire block is removed when DEBUG is 0 */
        extern void log_message(const char *);
        log_message(msg);
    }
}

/* Platform-specific code */
int get_pointer_size(void)
{
    if (sizeof(void *) == 8) {
        /* Only this branch exists on 64-bit platforms */
        return 64;
    } else {
        /* Only this branch exists on 32-bit platforms */
        return 32;
    }
}

/* Impossible conditions eliminated */
int check_range(int x)
{
    /* If x is already validated as positive elsewhere,
     * and optimizer knows it, this check may be eliminated.
     */
    if (x < 0)
        return -1;

    /* This is always false for signed int */
    if (x > INT_MAX)
        return -2;

    return x;
}

    Explanation¶

    the key insight is that the compiler has eliminated dead code at compile time — branches that can never execute are completely removed from the final assembly.

    The Core Pattern: Compile-Time Branch Elimination¶

    When the compiler can determine that a condition is always true or always false at compile time, it removes the unreachable branch entirely. This is called dead code elimination. You won't see any conditional branch instructions (beq, bne, etc.) for conditions the compiler already knows the answer to.

    Function-by-Function Breakdown¶

    maybe_log() — Complete Function Elimination¶

    maybe_log(char const*):
    addiu   $sp,$sp,-8          # Set up stack frame
    sw      $fp,4($sp)
    move    $fp,$sp
    sw      $4,8($fp)           # Save parameter (msg pointer)
    nop
    move    $sp,$fp             # Tear down stack frame
    lw      $fp,4($sp)
    addiu   $sp,$sp,8
    jr      $31                 # Return
    nop

    What happened: Since DEBUG is defined as 0, the compiler knows the if (DEBUG) condition is always false. The entire function body (the call to log_message()) is eliminated. The function does nothing except set up and tear down its stack frame — it's essentially a no-op.

    Key registers: - $sp (stack pointer): manages the stack frame - $fp (frame pointer): marks the current function's stack location - $4: holds the first parameter by MIPS calling convention (the msg pointer), but it's never used

    get_pointer_size() — One Branch Survives¶

    get_pointer_size():
    addiu   $sp,$sp,-8
    sw      $fp,4($sp)
    move    $fp,$sp
    li      $2,32               # Load immediate: put 32 into $2
    move    $sp,$fp
    lw      $fp,4($sp)
    addiu   $sp,$sp,8
    jr      $31
    nop

    What happened: The compiler evaluated sizeof(void *) at compile time. On this 32-bit MIPS platform, a pointer is 4 bytes, so sizeof(void *) == 8 is false. The compiler kept only the else branch and eliminated the if branch entirely.

    The function simply loads 32 into $2 (the return value register by MIPS convention) and returns. No conditional branch instruction exists — the compiler made the decision before generating code.

    check_range() — Partial Elimination¶

    check_range(int):
    addiu   $sp,$sp,-8
    sw      $fp,4($sp)
    move    $fp,$sp
    sw      $4,8($fp)           # Save parameter x
    lw      $2,8($fp)           # Load x into $2
    nop
    bgez    $2,$L5              # Branch if x >= 0 (skip negative return)
    nop

    li      $2,-1               # If x < 0: load -1 into $2
    b       $L6                 # Jump to end
    nop

$L5:
    lw      $2,8($fp)           # If x >= 0: load x into $2
$L6:
    move    $sp,$fp             # Tear down and return
    lw      $fp,4($sp)
    addiu   $sp,$sp,8
    jr      $31
    nop

    What happened: The first check (if (x < 0)) is kept because x is a runtime value — the compiler can't know at compile time whether it's negative.

    However, the second check (if (x > INT_MAX)) is completely eliminated. For a signed 32-bit integer, the maximum value is INT_MAX (2,147,483,647). A signed int can never exceed this value by definition — it's mathematically impossible. The compiler recognized this impossible condition and removed the entire branch.
