    stack-reuse¶

    Details

    Source: memory/stack-reuse.c Compiler: armv8-clang2110 (ARM) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    stack-reuse.c
    /*
 * Stack slot reuse: non-overlapping locals share stack space.
 *
 * The compiler analyzes variable lifetimes and reuses
 * stack slots for variables that are never live at the same time.
 */
#include <stdio.h>

void example(int flag)
{
    if (flag) {
        /* This buffer is only live in this branch */
        char buffer_a[100];
        snprintf(buffer_a, sizeof(buffer_a), "Path A: %d", flag);
        printf("%s\n", buffer_a);
    } else {
        /* This buffer may reuse the same stack slot as buffer_a
         * since they're never both alive at the same time.
         */
        char buffer_b[100];
        snprintf(buffer_b, sizeof(buffer_b), "Path B: %d", flag);
        printf("%s\n", buffer_b);
    }

    /* Total stack usage may be ~100 bytes, not 200 */
}

    Explanation¶

    the key insight is that both buffers share the same stack space because they're never alive at the same time, demonstrating stack slot reuse.

    Function Prologue: Setting Up the Stack Frame¶

    sub     sp, sp, #240
stp     x29, x30, [sp, #224]
add     x29, sp, #224

    The function allocates 240 bytes of stack space (even though each buffer is only 100 bytes). Here's why: - sub sp, sp, #240 moves the stack pointer down by 240 bytes to make room - stp x29, x30, [sp, #224] saves two registers (x29 and x30) at the top of the stack. By convention, x29 is the frame pointer (marks the current function's stack frame) and x30 is the return address (where to jump back after this function ends) - add x29, sp, #224 sets up the frame pointer to point to where we saved those registers

    The 240-byte allocation includes space for both buffers plus some extra working space.

    Parameter Storage and Branch Decision¶

    stur    w0, [x29, #-4]
ldur    w8, [x29, #-4]
cbz     w8, .LBB0_2
b       .LBB0_1

    This is unoptimized code. The flag parameter arrives in w0 (the lower 32 bits of x0, following ARM calling conventions). The compiler: 1. Stores it to the stack at offset -4 from the frame pointer 2. Immediately loads it back into w8 3. Uses cbz w8 (compare and branch if zero) to jump to .LBB0_2 if the flag is zero, otherwise falls through to .LBB0_1

    The store-then-load pattern is typical of unoptimized codean optimized compiler would just use w0 directly.

    Path A: The "if" Branch (buffer_a)¶

    .LBB0_1:
        ldur    w3, [x29, #-4]
        sub     x0, x29, #104
        str     x0, [sp, #8]
        mov     x1, #100
        adrp    x2, .L.str
        add     x2, x2, :lo12:.L.str
        bl      snprintf
        ldr     x1, [sp, #8]
        adrp    x0, .L.str.1
        add     x0, x0, :lo12:.L.str.1
        bl      printf

    Here's the stack slot reuse in action: - sub x0, x29, #104 calculates the address of buffer_a by going 104 bytes below the frame pointer. This is where the 100-byte buffer lives, plus 4 bytes of padding. - str x0, [sp, #8] stores this buffer address on the stack (at offset 8 from the current stack pointer) as a temporary - mov x1, #100 sets the buffer size to 100 - The adrp/add pair loads the address of the format string "Path A: %d" into x2 - bl snprintf calls snprintf to write formatted text into the buffer - The buffer address is reloaded from [sp, #8] and passed to printf
