    static-memcpy¶

    Details

    Source: string-literals/static-memcpy.c Compiler: armv8-clang2110 (ARM) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    static-memcpy.c
    #include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[5];

    memcpy(buf, "AAAA", 4);
    buf[sizeof(buf) - 1] = '\0';

    /* make sure the above is not treated as dead code */
    puts(buf);
}

    Explanation¶

    the key insight is that this code demonstrates how the compiler handles a local buffer and copies data into it, using the stack for storage.

    Stack Frame Setup¶

    The function starts by allocating space on the stack:

    sub     sp, sp, #32
stp     x29, x30, [sp, #16]
add     x29, sp, #16

    This is unoptimized code. The compiler allocates 32 bytes on the stack (even though only 5 bytes are needed for buf). The stp instruction saves two registers (x29 and x30) — these are the frame pointer and return address, which the function needs to restore before returning. By convention in ARM64, x29 holds the frame pointer (marking the current function's stack frame) and x30 holds the return address (where to jump back to after the function completes).

    Loading the String Constant¶

    adrp    x8, .L.str
add     x8, x8, :lo12:.L.str
ldr     w8, [x8]

    These three instructions load the string "AAAA" into register x8. Here's why three instructions are needed: - adrp loads the high 12 bits of the address (the "page" containing the string) - add adds the low 12 bits to complete the full address - ldr w8, [x8] loads 4 bytes (a 32-bit word, indicated by the w prefix) from that address into x8

    This two-step address calculation is necessary because ARM64 can't load a full 64-bit address in a single instruction.

    Copying Data to the Buffer¶

    add     x0, sp, #8
str     w8, [sp, #8]

    The first instruction calculates where the buffer lives: sp + 8 (8 bytes into the stack frame). This address is stored in x0 — by convention, x0 is the first argument register, so this prepares the buffer address for the upcoming puts() call.

    The second instruction copies the 4 bytes from x8 (the string "AAAA") into the buffer at [sp, #8].

    Null Termination¶

    strb    wzr, [sp, #12]

    This stores a single byte of zero (wzr is the zero register) at offset sp + 12. Since the buffer starts at sp + 8 and is 5 bytes long, offset 12 is the last byte — this null-terminates the string as required by the source code.

    Function Call and Cleanup¶

    bl      puts
ldp     x29, x30, [sp, #16]
add     sp, sp, #32
ret

    The bl instruction calls puts() with the buffer address already in x0. After puts() returns, the function restores the saved registers and deallocates the stack space before returning to the caller with ret.

    Key Observations¶

    This is unoptimized code — the compiler allocates far more stack space (32 bytes) than necessary (5 bytes), and uses multiple instructions to load a simple 4-byte constant. An optimized version would use fewer stack bytes and potentially inline the string copy more efficiently. The inefficiency is typical of -O0 compilation, where the compiler prioritizes simplicity and debuggability over performance.
