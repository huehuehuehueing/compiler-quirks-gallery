    tail-call¶

    Details

    Source: control-flow/tail-call.c Compiler: armv8-clang2110 (ARM) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    tail-call.c
    /*
 * Tail call optimization: recursive calls become jumps.
 *
 * When a function's last action is calling another function
 * (or itself), the call can become a jump, saving stack space.
 */

/* Tail-recursive factorial - can be optimized to a loop */
int factorial_tail(int n, int acc)
{
    if (n <= 1)
        return acc;
    /* Last action is the recursive call - tail position */
    return factorial_tail(n - 1, n * acc);
}

int factorial(int n)
{
    return factorial_tail(n, 1);
}

/* NOT tail recursive - multiplication after recursive call */
int factorial_not_tail(int n)
{
    if (n <= 1)
        return 1;
    /* Multiplication happens AFTER the call returns */
    return n * factorial_not_tail(n - 1);
}

/* Mutual tail recursion */
int is_even(unsigned n);
int is_odd(unsigned n);

int is_even(unsigned n)
{
    if (n == 0) return 1;
    return is_odd(n - 1);  /* Tail call */
}

int is_odd(unsigned n)
{
    if (n == 0) return 0;
    return is_even(n - 1);  /* Tail call */
}

    Explanation¶

    the key insight is that this is unoptimized code (-O0 flag), so tail call optimization is not applied even though the source code is written to enable it. Let me walk through what's happening.

    The Core Issue: No Tail Call Optimization¶

    At -O0, the compiler generates straightforward, unoptimized code. Even though factorial_tail is written as a tail-recursive function (where the recursive call is the last action), the compiler still uses full function call overhead (bl instructions) instead of converting the tail call into a jump. With optimization enabled (-O2 or -O3), the compiler would convert these into loops or tail jumps, saving stack space and improving performance.

    Stack Frame Setup Pattern¶

    Every function follows the same pattern:

    sub     sp, sp, #32              // Allocate 32 bytes on stack
stp     x29, x30, [sp, #16]      // Save frame pointer and return address
add     x29, sp, #16             // Set up new frame pointer

    This creates a stack frame (a reserved area of memory for the function's local variables). Think of it like a workspace on a tableeach function gets its own space. The x29 register (frame pointer) marks where this function's workspace begins, and x30 (link register) stores the return address so the function knows where to jump back to when it finishes.

    factorial_tail(int, int) - The Tail-Recursive Function¶

    The function receives two parameters: - w0 = n (the countdown value) - w1 = acc (the accumulator)

    Parameter storage: 

    str     w0, [sp, #8]             // Store n at sp+8
str     w1, [sp, #4]             // Store acc at sp+4

    The w prefix means we're working with 32-bit integers (the lower 32 bits of 64-bit registers).

    Base case check: 

    ldr     w8, [sp, #8]             // Load n into w8
subs    w8, w8, #1               // Subtract 1 and set condition flags
b.gt    .LBB0_2                  // Branch if greater than (n > 1)
b       .LBB0_1                  // Otherwise, go to base case

    The subs instruction subtracts and updates the condition flags (like a comparison). If n > 1, we jump to .LBB0_2 (recursive case); otherwise, we return the accumulator.

    Base case (.LBB0_1): 

    ldr     w8, [sp, #4]             // Load acc
stur    w8, [x29, #-4]           // Store as return value

    Recursive case (.LBB0_2): 

    ldr     w8, [sp, #8]             // Load n
subs    w0, w8, #1               // Calculate n-1, store in w0 (first parameter)
ldr     w8, [sp, #8]             // Load n again
ldr     w9, [sp, #4]             // Load acc
mul     w1, w8, w9               // Calculate n * acc, store in w1 (second parameter)
bl      factorial_tail(int, int) // Call the function

    Notice the inefficiency: the compiler loads n twice (once for the subtraction, once for the multiplication). This is typical of unoptimized codeno dead code elimination or register reuse optimization.

    The bl instruction is a branch with linkit calls the function and saves the return address in x30. This is a full function call, not a tail jump. With optimization, this would become a b (unconditional branch) instead, reusing the current stack frame.

    factorial(int) - The Wrapper¶

    This function simply calls factorial_tail with the accumulator initialized to 1: 

    ldur    w0, [x29, #-4]           // Load parameter n
mov     w1, #1                   // Set acc = 1
bl      factorial_tail(int, int) // Call tail-recursive version
