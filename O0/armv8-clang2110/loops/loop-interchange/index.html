    loop-interchange¶

    Details

    Source: loops/loop-interchange.c Compiler: armv8-clang2110 (ARM) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    loop-interchange.c
    /*
 * Loop Interchange
 *
 * When iterating over a 2D array, the order of the loop indices
 * determines memory access patterns.  In C (row-major layout),
 * iterating the inner loop over columns and the outer loop over
 * rows yields sequential access.  Swapping the loops ("interchange")
 * can improve cache locality when the original order would cause
 * stride-N accesses.
 *
 * Here the "bad" order (column-major traversal on a row-major array)
 * is presented; the compiler may interchange the loops at -O3.
 */
#define N 128

void column_sum(int mat[N][N], int result[N])
{
    /* Column-major traversal: inner loop strides by N ints.
     * The compiler may swap i and j for better locality. */
    for (int j = 0; j < N; j++)
    {
        result[j] = 0;
        for (int i = 0; i < N; i++)
        {
            result[j] += mat[i][j];
        }
    }
}

    Explanation¶

    the key insight is that this is unoptimized code (-O0) that uses the stack heavily to store loop variables instead of keeping them in registers. This creates a lot of unnecessary memory traffic that optimized code would eliminate.

    Function Setup and Parameter Storage¶

    The function starts by allocating 32 bytes of stack space: 

    sub     sp, sp, #32
str     x0, [sp, #24]
str     x1, [sp, #16]

    In ARM64 calling conventions, function parameters are passed in registers: x0 contains the matrix pointer and x1 contains the result array pointer. Rather than using these registers directly, the unoptimized compiler immediately stores them on the stack. This is typical of -O0 compilationthe compiler treats the stack as temporary storage for everything.

    Loop Variable Storage Pattern¶

    The code allocates stack locations for loop variables: - [sp, #12] stores the outer loop variable j (column index) - [sp, #8] stores the inner loop variable i (row index)

    This is the core inefficiency: every time the code needs to read or modify j or i, it must load from or store to the stack. An optimized compiler would keep these in registers.

    Outer Loop Structure (.LBB0_1)¶

    ldr     w8, [sp, #12]      // Load j from stack
subs    w8, w8, #128       // Compare j with 128
b.ge    .LBB0_8            // Exit if j >= 128

    The outer loop loads j, subtracts 128 to check if we've reached the limit, and branches to the exit if done. The subs instruction performs the subtraction and sets condition flags, which b.ge (branch if greater-or-equal) uses.

    Result Array Initialization (.LBB0_2)¶

    ldr     x9, [sp, #16]      // Load result array pointer
ldrsw   x10, [sp, #12]     // Load j (sign-extended to 64-bit)
mov     w8, wzr            // w8 = 0
str     w8, [x9, x10, lsl #2]  // result[j] = 0

    Before summing, result[j] is initialized to zero. The ldrsw instruction loads the 32-bit j value and sign-extends it to 64 bits (necessary for array indexing). The lsl #2 shifts left by 2 bits, which multiplies by 4 (converting array index to byte offset, since integers are 4 bytes).

    Inner Loop: The Memory Access Problem (.LBB0_4)¶

    This is where the inefficiency becomes severe:

    ldr     x8, [sp, #24]      // Load matrix pointer
ldrsw   x9, [sp, #8]       // Load i (row index)
add     x8, x8, x9, lsl #9 // x8 = matrix + i*512
ldrsw   x9, [sp, #12]      // Load j (column index)
ldr     w11, [x8, x9, lsl #2]  // Load mat[i][j]

    To access mat[i][j], the code: 1. Loads the matrix pointer from the stack 2. Loads i from the stack and shifts left by 9 bits (multiplies by 512, which is 128 integers 4 bytes per row) 3. Adds this to the matrix pointer to get the row address 4. Loads j from the stack again and uses it as a column offset

    The lsl #9 is the key calculation: since each row has 128 integers and each integer is 4 bytes, one row is 512 bytes (2^9).
