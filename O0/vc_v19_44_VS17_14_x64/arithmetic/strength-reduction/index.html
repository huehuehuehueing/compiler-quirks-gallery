    strength-reduction¶

    Details

    Source: arithmetic/strength-reduction.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    strength-reduction.c
    /*
 * Strength reduction: expensive operations replaced with cheaper ones.
 *
 * Multiplication by powers of 2 becomes shifts.
 * Division by constants becomes multiplication + shift.
 */

int multiply_by_8(int x)
{
    /* Becomes: x << 3 */
    return x * 8;
}

int multiply_by_15(int x)
{
    /* May become: (x << 4) - x */
    return x * 15;
}

unsigned divide_by_3(unsigned x)
{
    /* Division is expensive. Compiler uses "magic number" multiplication:
     * Becomes something like: (x * 0xAAAAAAAB) >> 33
     */
    return x / 3;
}

unsigned modulo_power_of_2(unsigned x)
{
    /* Becomes: x & 7 */
    return x % 8;
}

    Explanation¶

    the key insight is that this is unoptimized code (-O0), so the compiler is not applying the strength reduction optimizations mentioned in the comments. Let me walk through what's happening in each function.

    Overall Pattern: Unoptimized Code¶

    This is unoptimized code. Even though the source comments describe optimizations like "multiply by 8 becomes a shift," the compiler is not actually applying these transformations. You'll see expensive operations like div (division) and imul (multiplication) used directly instead of cheaper alternatives.

    Function-by-Function Breakdown¶

    multiply_by_8: Shift Optimization Applied¶

    mov     DWORD PTR [rsp+8], ecx      # Store parameter on stack
mov     eax, DWORD PTR x$[rsp]      # Load it back into eax
shl     eax, 3                      # Shift left by 3 (multiply by 2³ = 8)
ret     0

    What's happening: The compiler does optimize this one—it replaces x * 8 with shl eax, 3 (shift left by 3 bits). This is because multiplying by powers of 2 is so obviously optimizable that even unoptimized compilers do it. Shifting is much faster than multiplication.

    The unnecessary store-then-load pattern (mov [rsp+8], ecx followed by mov eax, [rsp+8]) is typical of unoptimized code—the compiler doesn't bother keeping values in registers.

    multiply_by_15: Direct Multiplication (Not Optimized)¶

    mov     DWORD PTR [rsp+8], ecx      # Store parameter on stack
imul    eax, DWORD PTR x$[rsp], 15  # Multiply by 15 directly
ret     0

    What's happening: The compiler uses imul (signed multiply) with the constant 15. The source comment suggests this could become (x << 4) - x (shift left 4 to get 16x, then subtract x to get 15x), which would be cheaper. But at -O0, the compiler doesn't bother—it just does the multiplication directly.

    divide_by_3: Expensive Division (Not Optimized)¶

    mov     DWORD PTR [rsp+8], ecx      # Store parameter on stack
xor     edx, edx                    # Clear edx (set to 0)
mov     eax, DWORD PTR x$[rsp]      # Load x into eax
mov     ecx, 3                      # Load divisor 3 into ecx
div     ecx                         # Divide eax by ecx; quotient → eax, remainder → edx
ret     0

    What's happening: The div instruction is one of the slowest operations on modern CPUs. The source comment mentions that optimized code would use a "magic number" multiplication trick: (x * 0xAAAAAAAB) >> 33. This is a clever technique where division by a constant is replaced with multiplication by a carefully chosen value followed by a shift—much faster.

    But at -O0, the compiler just uses div directly. Notice the setup: edx must be cleared before division (it holds the remainder), and the divisor goes in ecx.
