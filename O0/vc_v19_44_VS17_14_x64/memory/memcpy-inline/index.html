    memcpy-inline¶

    Details

    Source: memory/memcpy-inline.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    memcpy-inline.c
    /*
 * memcpy inlining: small copies become inline loads/stores.
 *
 * For small, known-size copies, the compiler generates
 * inline code instead of calling memcpy.
 */
#include <string.h>

struct small {
    int a, b;
};

void copy_8_bytes(void *dst, const void *src)
{
    /* Small copy - likely inlined as single 64-bit load/store
     * on 64-bit architectures, or two 32-bit ops on 32-bit.
     */
    memcpy(dst, src, 8);
}

void copy_struct(struct small *dst, const struct small *src)
{
    /* Structure copy - same as memcpy but type-aware.
     * Compiler knows alignment and size.
     */
    *dst = *src;
}

void copy_varying(void *dst, const void *src, int use_large)
{
    /* Compile-time unknown size - must call memcpy or
     * generate size-checking code.
     */
    if (use_large)
        memcpy(dst, src, 1024);
    else
        memcpy(dst, src, 8);
}

    Explanation¶

    the key insight is that the compiler handles different copy scenarios very differently, and understanding why reveals important compiler behavior.

    The Core Pattern: Three Different Copy Strategies¶

    The code shows three functions that copy memory, but the compiler generates completely different assembly for each:

    1. copy_8_bytes: Calls memcpy (the library function)
    2. copy_struct: Inlines the copy as direct load/store instructions
    3. copy_varying: Calls memcpy with runtime branching

    Let me walk through each:

    Function 1: copy_8_bytes - Calling memcpy¶

    mov     QWORD PTR [rsp+16], rdx    ; Save src parameter
mov     QWORD PTR [rsp+8], rcx     ; Save dst parameter
sub     rsp, 40                    ; Allocate stack space
mov     r8d, 8                     ; Set size = 8
mov     rdx, QWORD PTR src$[rsp]   ; Load src into rdx
mov     rcx, QWORD PTR dst$[rsp]   ; Load dst into rcx
call    memcpy                     ; Call the library function
add     rsp, 40                    ; Deallocate stack
ret     0

    This is unoptimized code. The compiler saves the incoming parameters to the stack, then immediately reloads them. Here's what's happening:

    • Parameter passing: On x64 calling conventions, the first two parameters arrive in rcx (dst) and rdx (src). The function immediately saves these to the stack at offsets 8 and 16.
    • Stack frame setup: sub rsp, 40 allocates 40 bytes of stack space (this is unoptimizedit's more than needed, but provides alignment).
    • Preparing for memcpy call: The function reloads the parameters from the stack back into rcx and rdx, and puts the size (8) into r8d. This is the calling convention for memcpy(dst, src, size).
    • The actual copy: call memcpy delegates the work to the library function.

    Why not inline? Even though 8 bytes is small, the compiler chose to call memcpy rather than generate inline load/store instructions. This is a compiler choicesome optimizers would inline this, but at -O0 (no optimization), the compiler takes the straightforward path of calling the library function.

    Function 2: copy_struct - Inlined Copy¶

    mov     rax, QWORD PTR src$[rsp]   ; Load src pointer
mov     rax, QWORD PTR [rax]       ; Load 8 bytes from *src
mov     rcx, QWORD PTR dst$[rsp]   ; Load dst pointer
mov     QWORD PTR [rcx], rax       ; Store 8 bytes to *dst
ret     0

    This is the key difference. The compiler recognizes that *dst = *src is a structure assignment of known size (8 bytes), so it inlines the copy directly:

    • Load from source: mov rax, QWORD PTR [rax] loads 8 bytes (a QWORD) from the source address into rax.
    • Store to destination: mov QWORD PTR [rcx], rax writes those 8 bytes to the destination address.

    Why inline here? The compiler understands the structure size at compile time and can generate a simple load/store pair. This is more efficient than calling memcpy because there's no function call overhead.

    Notice: No stack frame setup (sub rsp) is needed herethe function just uses the parameters directly from rcx and rdx.
