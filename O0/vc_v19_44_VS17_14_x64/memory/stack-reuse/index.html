    stack-reuse¶

    Details

    Source: memory/stack-reuse.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    stack-reuse.c
    /*
 * Stack slot reuse: non-overlapping locals share stack space.
 *
 * The compiler analyzes variable lifetimes and reuses
 * stack slots for variables that are never live at the same time.
 */
#include <stdio.h>

void example(int flag)
{
    if (flag) {
        /* This buffer is only live in this branch */
        char buffer_a[100];
        snprintf(buffer_a, sizeof(buffer_a), "Path A: %d", flag);
        printf("%s\n", buffer_a);
    } else {
        /* This buffer may reuse the same stack slot as buffer_a
         * since they're never both alive at the same time.
         */
        char buffer_b[100];
        snprintf(buffer_b, sizeof(buffer_b), "Path B: %d", flag);
        printf("%s\n", buffer_b);
    }

    /* Total stack usage may be ~100 bytes, not 200 */
}

    Explanation¶

    the key insight is how the compiler reuses stack space for variables that never exist at the same time.

    The Core Concept: Stack Slot Reuse¶

    The most important thing to understand here is that buffer_a and buffer_b don't each get their own 100-byte stack spacethey share the same location. Look at these lines:

    buffer_a$1 = 32
buffer_b$2 = 144

    These offsets are different (32 vs 144), which initially seems wasteful. However, the total stack allocation is only 280 bytes, not 400 bytes (which would be needed for two separate 100-byte buffers). This is the compiler's optimization at work.

    Stack Frame Setup¶

    The function starts by allocating space:

    sub     rsp, 280                      ; 00000118H

    The rsp register is the stack pointer (think of it as pointing to the top of a stack of memory). The sub instruction subtracts from it, which grows the stack downward and reserves 280 bytes of local space. This single allocation serves both buffers because they're never both "alive" (in use) simultaneously.

    Security Cookie Protection¶

    Before doing anything useful, the compiler adds security checks:

    mov     rax, QWORD PTR __security_cookie
xor     rax, rsp
mov     QWORD PTR __$ArrayPad$[rsp], rax

    This is a stack canarya random value stored on the stack to detect buffer overflows. The xor instruction combines the security cookie with the stack pointer address. At the end of the function, this value is verified to ensure no corruption occurred.

    The Conditional Branch¶

    cmp     DWORD PTR flag$[rsp], 0
je      SHORT $LN2@example

    The cmp instruction compares the flag parameter to zero. The je (jump if equal) instruction branches to $LN2@example if the flag is zerothis is the else branch.

    Path A: When flag is non-zero¶

    lea     rcx, QWORD PTR buffer_a$1[rsp]
call    snprintf

    The lea instruction (Load Effective Address) calculates the address of buffer_a by taking the stack pointer and adding offset 32. This address is placed in rcx, which by calling convention is the first parameter register. Then snprintf is called to format the string into the buffer.

    The other parameters are set up in registers: - edx = 100 (buffer size) - r8 = address of format string "Path A: %d" - r9d = the flag value

    Path B: When flag is zero¶

    lea     rcx, QWORD PTR buffer_b$2[rsp]
call    snprintf

    This is nearly identical, except buffer_b is at offset 144 instead of 32. The format string is different ("Path B: %d"), but the mechanism is the same.

    Why Different Offsets?¶

    You might wonder: if they share space, why are the offsets different? The answer is that they don't actually share the exact same bytesbuffer_b starts at offset 144, which is 112 bytes after buffer_a starts at offset 32. This gives each buffer its own 100-byte region, but because the if/else branches are mutually exclusive, only one buffer is ever used in a single execution. The compiler allocates enough total space (280 bytes) to accommodate whichever buffer is needed, rather than allocating 200 bytes upfront.

    Cleanup and Return¶

    mov     rcx, QWORD PTR __$ArrayPad$[rsp]
xor     rcx, rsp
call    __security_check_cookie
add     rsp, 280
ret     0

    Before returning, the security cookie is verified by XORing it again with the current stack pointer. If the value matches what was stored, the function is safe. Then the stack is deallocated (add rsp, 280 reverses the earlier sub), and the function returns.

    The -O0 flag means no optimizations are applied. You'll notice inefficiencies like storing and reloading
