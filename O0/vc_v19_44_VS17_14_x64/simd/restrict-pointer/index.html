    restrict-pointer¶

    Details

    Source: simd/restrict-pointer.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    restrict-pointer.c
    /*
 * Restrict pointers enable better optimization.
 *
 * The 'restrict' keyword promises that pointers don't alias,
 * allowing the compiler to optimize more aggressively.
 */

/* Without restrict - compiler must be conservative */
void copy_no_restrict(int *dst, const int *src, int n)
{
    /* dst and src MIGHT overlap, so compiler must handle that */
    for (int i = 0; i < n; i++) {
        dst[i] = src[i];
    }
}

/* With restrict - compiler knows no aliasing */
void copy_restrict(int *restrict dst, const int *restrict src, int n)
{
    /* dst and src guaranteed not to overlap
     * Compiler can use faster copy or vectorize freely
     */
    for (int i = 0; i < n; i++) {
        dst[i] = src[i];
    }
}

/* Multiplication example - dramatic difference */
void scale_no_restrict(float *result, const float *input,
                       const float *scale, int n)
{
    /* Compiler doesn't know if result aliases input or scale */
    for (int i = 0; i < n; i++) {
        result[i] = input[i] * (*scale);
    }
    /* Must reload *scale each iteration! */
}

void scale_restrict(float *restrict result,
                    const float *restrict input,
                    const float *restrict scale, int n)
{
    /* Compiler knows result doesn't alias scale */
    for (int i = 0; i < n; i++) {
        result[i] = input[i] * (*scale);
    }
    /* Can load *scale once and reuse */
}

    Explanation¶

    the key insight is that at -O0 (unoptimized), the restrict keyword makes no difference in the generated code. Both versions produce identical assembly. Let me walk you through what's happening.

    The Big Picture¶

    This is unoptimized code. The compiler is not applying any optimizations, so it doesn't matter whether you promise no aliasing with restrict or notthe assembly is the same. At higher optimization levels like -O2 or -O3, restrict would enable significant improvements, but we're not seeing that here.

    Understanding the Functions¶

    Let me explain the two simpler functions first (copy_no_restrict and copy_restrict), then the more interesting scaling functions.

    Function Setup (All Functions)¶

    Every function starts the same way: 

    mov     DWORD PTR [rsp+24], r8d    ; Save 4th parameter (n)
mov     QWORD PTR [rsp+16], rdx    ; Save 3rd parameter (src/scale)
mov     QWORD PTR [rsp+8], rcx     ; Save 2nd parameter (dst/input)
sub     rsp, 24                    ; Allocate stack space

    This follows the x64 calling convention: parameters come in registers rcx, rdx, r8, r9 (for the first four parameters). The function immediately saves these to the stack because it needs to use those registers for other work. The sub rsp, 24 allocates 24 bytes of local stack space for the loop variable i.

    The Copy Loop (Both Versions Identical)¶

    mov     DWORD PTR i$1[rsp], 0      ; i = 0
jmp     SHORT $LN4@copy_no_re      ; Jump to loop condition

    The loop structure is: 1. Increment (at $LN2): Load i, increment it, store it back 2. Condition check (at $LN4): Load n, compare with i, jump out if i >= n 3. Loop body: Load from src[i], store to dst[i], jump back to increment

    The loop body does this: 

    movsxd  rax, DWORD PTR i$1[rsp]    ; Convert i to 64-bit (sign-extend)
movsxd  rcx, DWORD PTR i$1[rsp]    ; Convert i to 64-bit again
mov     rdx, QWORD PTR dst$[rsp]   ; Load dst pointer
mov     r8, QWORD PTR src$[rsp]    ; Load src pointer
mov     eax, DWORD PTR [r8+rax*4]  ; Load src[i] (rax*4 because ints are 4 bytes)
mov     DWORD PTR [rdx+rcx*4], eax ; Store to dst[i]

    Notice the redundancy: i is converted to 64-bit twice (in rax and rcx), and the pointers are reloaded from the stack every iteration. This is typical unoptimized codethe compiler doesn't eliminate these redundant operations.

    Key observation: Both copy_no_restrict and copy_restrict produce identical assembly. The restrict keyword provides no benefit at -O0.

    The Scaling Functions (Also Identical)¶

    The scaling functions use floating-point instructions: 

    movss   xmm0, DWORD PTR [rcx+rax*4]  ; Load input[i] into xmm0 (SSE register)
mulss   xmm0, DWORD PTR [rdx]        ; Multiply by *scale (the dereferenced pointer)
movss   DWORD PTR [rcx+rax*4], xmm0  ; Store result[i]

    The movss instruction moves a single-precision (32-bit) floating-point value. The xmm0 register is part of the SSE (Streaming SIMD Extensions) instruction set, used for floating-point operations.
