    tail-call¶

    Details

    Source: control-flow/tail-call.c Compiler: csparcg1520 (SPARC) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    tail-call.c
    /*
 * Tail call optimization: recursive calls become jumps.
 *
 * When a function's last action is calling another function
 * (or itself), the call can become a jump, saving stack space.
 */

/* Tail-recursive factorial - can be optimized to a loop */
int factorial_tail(int n, int acc)
{
    if (n <= 1)
        return acc;
    /* Last action is the recursive call - tail position */
    return factorial_tail(n - 1, n * acc);
}

int factorial(int n)
{
    return factorial_tail(n, 1);
}

/* NOT tail recursive - multiplication after recursive call */
int factorial_not_tail(int n)
{
    if (n <= 1)
        return 1;
    /* Multiplication happens AFTER the call returns */
    return n * factorial_not_tail(n - 1);
}

/* Mutual tail recursion */
int is_even(unsigned n);
int is_odd(unsigned n);

int is_even(unsigned n)
{
    if (n == 0) return 1;
    return is_odd(n - 1);  /* Tail call */
}

int is_odd(unsigned n)
{
    if (n == 0) return 0;
    return is_even(n - 1);  /* Tail call */
}

    Explanation¶

    the key insight is that this is unoptimized code (-O0), so tail call optimization is NOT being applied even though the source code is written to enable it. Let me walk through what's happening.

    The Core Issue: No Tail Call Optimization¶

    In the source code, factorial_tail is written as a tail-recursive functionits last action is calling itself. In optimized code, this would become a simple loop that reuses the same stack frame. However, with -O0 (no optimization), the compiler generates a full function call for every recursive step, creating a new stack frame each time.

    How SPARC Functions Work (Brief Background)¶

    SPARC uses a register window calling convention: - %i0 and %i1 are input registers (parameters come in here) - %o0 and %o1 are output registers (used to pass parameters to called functions) - %g1, %g2, %g3 are temporary registers - %fp is the frame pointer (points to the current stack frame) - %i7 is the return address

    Walking Through factorial_tail¶

    save    %sp, -96, %sp          // Create new stack frame (96 bytes)
st      %i0, [%fp+68]          // Store parameter n on stack
st      %i1, [%fp+72]          // Store parameter acc on stack

    The function immediately saves both parameters to the stack. This is typical unoptimized behavioreven though the values are already in registers, the compiler stores them to memory.

    ld      [%fp+68], %g1          // Load n from stack
cmp     %g1, 1                 // Compare n with 1
bg      %icc, .L2              // Branch if greater (n > 1)

    This is the base case check: if n <= 1, return acc. The bg (branch if greater) instruction jumps to .L2 if n > 1.

    ld      [%fp+72], %g1          // Load acc from stack
ba,pt   %xcc, .L3              // Jump to return

    If the base case is true, load acc and jump to the return sequence.

    .L2:                           // Recursive case
ld      [%fp+68], %g1          // Load n
add     %g1, -1, %g3           // Calculate n-1
ld      [%fp+68], %g2          // Load n again
ld      [%fp+72], %g1          // Load acc
smul    %g2, %g1, %g1          // Multiply: n * acc
mov     %g1, %o1               // Move result to output register o1 (2nd param)
mov     %g3, %o0               // Move n-1 to output register o0 (1st param)
call    factorial_tail, 0      // Call factorial_tail
 nop                           // Delay slot (executed before call)
mov     %o0, %g1               // Move return value to g1

    Here's the critical inefficiency: the compiler generates a full call instruction instead of converting this to a loop. In optimized code, this would be a jmp (jump) that reuses the current stack frame. Instead, each recursive call creates a new 96-byte stack frame, consuming stack space proportional to the recursion depth.

    Comparing factorial_not_tail (Non-Tail-Recursive)¶

    .L7:
ld      [%fp+68], %g1
add     %g1, -1, %g1
mov     %g1, %o0
call    factorial_not_tail, 0  // Recursive call
 nop
mov     %o0, %g2               // Save return value
ld      [%fp+68], %g1          // Load n again
smul    %g2, %g1, %g1          // Multiply AFTER the call returns

    Notice the key difference: the multiplication happens after the call returns. This is why it's not tail-recursivethere's work to do after the recursive call completes, so the compiler cannot optimize it away. The function must preserve its stack frame to perform the multiplication.
