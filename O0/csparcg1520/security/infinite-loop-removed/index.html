    infinite-loop-removed¶

    Details

    Source: security/infinite-loop-removed.c Compiler: csparcg1520 (SPARC) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    infinite-loop-removed.c
    /*
 * Infinite loop removed as undefined behavior.
 *
 * A loop with no side effects that doesn't terminate is UB.
 * Compiler may remove it entirely or assume it terminates.
 */

/* This function may not loop forever as expected! */
void spin_forever(void)
{
    /* No volatile, no side effects - UB if it never terminates.
     * Compiler may remove this entirely.
     */
    while (1)
        ;
}

/* Correct way to spin (e.g., for embedded busy-wait) */
void spin_forever_safe(void)
{
    /* volatile prevents optimization */
    while (1)
    {
        __asm__ volatile("" ::: "memory");
    }
}

    Explanation¶

    the key insight here is that both functions compile to identical infinite loops, despite the source code's comment claiming one is "unsafe". This reveals important compiler behavior around undefined behavior.

    The Core Pattern: Function Setup and Infinite Loop¶

    Both functions follow the same structure:

    save    %sp, -96, %sp    // Set up stack frame
.L2:                      // Loop label
ba,pt   %xcc, .L2        // Branch always back to loop
 nop                      // Delay slot (execute before branch)

    Let me break down what's happening:

    Stack Frame Setup: save %sp, -96, %sp¶

    The save instruction is SPARC's way of creating a new stack frame for the function. Think of it like marking off a workspace on the stack: - %sp is the stack pointer (the current top of the stack) - -96 is the frame size (96 bytes of space reserved) - The result goes back into %sp, moving it down by 96 bytes

    This is standard function prologue on SPARC, even though these functions don't actually use any local variables. The compiler includes it because it's following the calling convention rules.

    The Infinite Loop: ba,pt %xcc, .L2 + nop¶

    This is where the actual looping happens: - ba means "branch always" — unconditionally jump - %xcc is the condition code register (though "always" means we ignore it) - .L2 is the label pointing back to the start of the loop - nop is a "no operation" instruction that does nothing

    The nop appears in the delay slot — a SPARC quirk where the instruction after a branch executes before the branch takes effect. The compiler puts nop there because there's nothing useful to execute.

    Why Both Functions Are Identical¶

    This is the surprising part: the compiler treats both functions the same way. Even though spin_forever() has no volatile keyword and technically contains undefined behavior (an infinite loop with no side effects), the compiler still generates an infinite loop for it.

    The reason: the compiler assumes the loop will eventually terminate or be interrupted (by a signal, exception, or external event). Since it can't prove the loop is truly infinite, it generates the loop anyway. The volatile keyword in spin_forever_safe() doesn't change the generated code — it just makes the intent explicit to the compiler and prevents certain optimizations.

    Key Observations¶

    • This is unoptimized code (-O0 flag), so the compiler includes the full save instruction even though it's not strictly necessary for these trivial functions
    • Both functions generate identical machine code, showing that the compiler doesn't distinguish between the "unsafe" and "safe" versions at this optimization level
    • The infinite loop is implemented as a simple backward branch with a delay slot nop
    • No actual return instruction appears — these functions never return to their caller
