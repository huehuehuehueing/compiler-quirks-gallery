    static-copy-1¶

    Details

    Source: string-literals/static-copy-1.c Compiler: avrg1520 (AVR) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    static-copy-1.c
    /*
 * String copy with strcpy - compiler may inline as immediate stores.
 *
 * For small, known-length strings, strcpy becomes direct
 * memory stores with the string data embedded in instructions.
 */
#include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[100];
    strcpy(buf, "ABCDXXXXCCCC");

    /* Make observable to prevent dead code elimination */
    puts(buf);
}

    Explanation¶

    the key insight is that this code manually implements a string copy loop rather than calling a library function, and it uses AVR's special addressing modes to efficiently copy data.

    Stack Frame Setup¶

    The function starts by creating space for the 100-byte buf array on the stack:

    push r28
push r29
in r28,__SP_L__
in r29,__SP_H__
subi r28,100
sbc r29,__zero_reg__

    On AVR, r28 and r29 form the frame pointer (Y register), which points to the current stack frame. The code: 1. Saves the old frame pointer by pushing r28 and r29 2. Reads the current stack pointer into the frame pointer 3. Subtracts 100 from it to allocate space for the buffer

    The sbc (subtract with carry) instruction handles the high byte of the 16-bit subtraction. The cli/out sequence that follows temporarily disables interrupts while updating the stack pointer—this is necessary because the stack pointer is a 16-bit value that requires two separate writes on AVR, and an interrupt during this operation could corrupt the stack.

    String Copy Loop¶

    The actual copying happens in this loop:

    ldi r18,lo8(13)           # Load 13 (string length + null terminator)
ldi r30,lo8(.LC0)         # Load source address (low byte)
ldi r31,hi8(.LC0)         # Load source address (high byte)
mov r24,r28               # Copy frame pointer to r24
mov r25,r29               # Copy frame pointer to r25
adiw r24,1                # Add 1 to get buffer address
mov r26,r24               # Copy to X register (destination pointer)
mov r27,r25
0:
ld r0,Z+                  # Load byte from source, auto-increment
st X+,r0                  # Store byte to destination, auto-increment
dec r18                   # Decrement counter
brne 0b                   # Branch if not equal (loop back)

    This is a byte-by-byte copy loop. The key registers are: - Z register (r30:r31): Points to the source string in program memory (.LC0) - X register (r26:r27): Points to the destination buffer on the stack - r18: Loop counter (13 bytes to copy) - r0: Temporary register holding each byte during transfer

    The ld r0,Z+ and st X+,r0 instructions use post-increment addressing—the + means the pointer automatically increments after each access. This is more efficient than manually incrementing pointers.

    Function Call and Cleanup¶

    After the copy completes, the code calls puts() to print the buffer:

    mov r24,r28               # Set up first argument (buffer address)
mov r25,r29
adiw r24,1
rcall puts                # Call puts function

    By AVR calling convention, the first argument goes in r24:r25 (a 16-bit pointer). The code recalculates the buffer address and passes it to puts().

    Finally, the stack frame is deallocated and the function returns:

    subi r28,-100             # Add 100 back to stack pointer
sbci r29,-1
### ... (interrupt-safe stack pointer update)
pop r29
pop r28
ret

    Why This Approach?¶

    This is unoptimized code (-O0 flag). An optimized compiler would likely inline the string data directly as immediate stores rather than using a loop, since the string is small and known at compile time. The loop-based approach here is correct but inefficient—it trades code size for execution speed, which is typical of unoptimized compilation.
