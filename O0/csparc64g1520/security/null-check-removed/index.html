    null-check-removed¶

    Details

    Source: security/null-check-removed.c Compiler: csparc64g1520 (SPARC64) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    null-check-removed.c
    /*
 * Security issue: null pointer check eliminated after dereference.
 *
 * Compiler assumes dereferencing null is UB, so if you dereference
 * first, any subsequent null check is "impossible" and removed.
 */
#include <stddef.h>

struct data {
    int value;
    int flags;
};

int process(struct data *p)
{
    int val;

    /* Dereference first */
    val = p->value;

    /* Check for null afterward - THIS MAY BE REMOVED!
     * Compiler reasons: "p was dereferenced, so p cannot be null,
     * therefore this check is always false"
     */
    if (p == NULL)
        return -1;

    return val + p->flags;
}

    Explanation¶

    the key insight here is that the compiler generates a null pointer check even though it's unreachable code — this demonstrates undefined behavior in the source.

    The Core Issue¶

    The source code dereferences pointer p first (val = p->value), then checks if p == NULL afterward. In C, dereferencing a null pointer is undefined behavior. The compiler is technically allowed to assume this never happens and optimize accordingly. However, this unoptimized code (-O0) still generates the null check instructions, even though they can never execute.

    Stack Frame Setup¶

    save    %sp, -192, %sp

    This instruction creates a new stack frame for the function. %sp is the stack pointer register. The -192 allocates 192 bytes of local storage on the stack. This is standard SPARC calling convention — each function gets its own stack frame to store local variables and temporary values.

    Storing the Parameter¶

    stx     %i0, [%fp+2175]

    The parameter p arrives in %i0 (input register 0, by SPARC calling convention). This instruction stores it to the stack at offset 2175 from the frame pointer (%fp). In unoptimized code, parameters are often immediately spilled to memory rather than kept in registers — this is inefficient but makes debugging easier.

    Dereferencing the Pointer (First Access)¶

    ldx     [%fp+2175], %g1      // Load p from stack into %g1
ld      [%g1], %g1           // Dereference: load p->value into %g1
st      %g1, [%fp+2043]      // Store the value to stack

    These three instructions: 1. Reload p from the stack into %g1 2. Dereference it with ld [%g1] — this loads the 32-bit integer at the address stored in %g1 (the value field) 3. Store that result back to the stack at offset 2043

    This is the critical dereference that makes the null check undefined behavior.

    The Null Check (Unreachable Code)¶

    ldx     [%fp+2175], %g1      // Load p from stack again
brnz    %g1, .L2             // Branch if non-zero (not null)
 nop                         // Delay slot (executes before branch)

    brnz means "branch if register non-zero." If %g1 (containing p) is non-zero, jump to label .L2. If it's zero (null), fall through to the next instruction.

    mov     -1, %g1              // Load -1 into %g1 (return value)
ba,pt   %xcc, .L3            // Unconditional branch to .L3
 nop

    If the pointer were null, this would return -1. However, this code is unreachable because we already dereferenced p earlier without checking.

    Computing the Result (When Not Null)¶

    .L2:
ldx     [%fp+2175], %g1      // Load p from stack
ld      [%g1+4], %g2         // Load p->flags (offset +4 bytes)
ld      [%fp+2043], %g1      // Load the saved value
add     %g2, %g1, %g1        // Add: flags + value

    At label .L2, the code: 1. Reloads p from the stack 2. Loads the flags field (at offset +4 from the struct start) 3. Reloads the previously saved value 4. Adds them together with add

    Return Value Setup¶

    .L3:
sra     %g1, 0, %g1          // Sign-extend %g1 (shift right arithmetic by 0)
mov     %g1, %i0             // Move result to %i0 (return register)
return  %i7+8                // Return to caller
 nop

    The sra instruction performs a sign-extending shift (shift right arithmetic by 0 bits, which just sign-extends). The result is moved to %i0, which is the return value register by SPAR
