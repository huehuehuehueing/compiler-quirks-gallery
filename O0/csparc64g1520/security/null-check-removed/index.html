    null-check-removed¶

    Details

    Source: security/null-check-removed.c Compiler: csparc64g1520 (SPARC64) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    null-check-removed.c
    /*
 * Security issue: null pointer check eliminated after dereference.
 *
 * Compiler assumes dereferencing null is UB, so if you dereference
 * first, any subsequent null check is "impossible" and removed.
 */
#include <stddef.h>

struct data {
    int value;
    int flags;
};

int process(struct data *p)
{
    int val;

    /* Dereference first */
    val = p->value;

    /* Check for null afterward - THIS MAY BE REMOVED!
     * Compiler reasons: "p was dereferenced, so p cannot be null,
     * therefore this check is always false"
     */
    if (p == NULL)
        return -1;

    return val + p->flags;
}

    Explanation¶

    the key insight here is that this code demonstrates undefined behavior elimination - the compiler removes the null pointer check because dereferencing the pointer first makes the check logically impossible.

    The Critical Pattern: Undefined Behavior Elimination¶

    The most important thing to understand is what happens when you dereference a pointer before checking if it's null. In C, dereferencing a null pointer is undefined behavior - the language doesn't define what should happen. The compiler assumes this will never occur, so if you dereference first, any subsequent null check becomes "impossible" and gets optimized away.

    Stack Setup and Parameter Storage¶

    save    %sp, -192, %sp
stx     %i0, [%fp+2175]

    The function starts by creating a new stack frame (the save instruction). The parameter p arrives in register %i0 (by SPARC calling convention, the first integer argument goes here). The code immediately stores it to the stack at offset 2175 from the frame pointer. This is unoptimized code - an optimized compiler would keep p in a register instead of storing and reloading it repeatedly.

    The Dereference (First Operation)¶

    ldx     [%fp+2175], %g1      ; Load p from stack into %g1
ld      [%g1], %g1           ; Dereference: load p->value into %g1
st      %g1, [%fp+2043]      ; Store val to stack

    Here's where the undefined behavior occurs. The code loads p from the stack, then immediately dereferences it with ld [%g1] (load from the address in %g1). If p were null, this would crash. The compiler now "knows" that p cannot be null, because if it were, the program would have already crashed.

    The Null Check (That Gets Kept But Is Dead Code)¶

    ldx     [%fp+2175], %g1      ; Load p again
brnz    %g1, .L2             ; Branch if non-zero (not null)
 nop
mov     -1, %g1              ; Return -1 if null
ba,pt   %xcc, .L3            ; Jump to end
 nop

    Interestingly, the compiler does generate the null check instruction (brnz - "branch if register non-zero"). However, this is dead code - it will never actually take the null path because the compiler has already proven that p cannot be null after the dereference. The branch will always go to .L2.

    The Normal Path (When Not Null)¶

    .L2:
ldx     [%fp+2175], %g1      ; Load p again
ld      [%g1+4], %g2         ; Load p->flags (offset 4 bytes)
ld      [%fp+2043], %g1      ; Load val from stack
add     %g2, %g1, %g1        ; Add: flags + val

    This path loads the flags field (at offset 4 in the struct), retrieves the previously stored val, and adds them together. The result ends up in %g1.

    Return Value Setup¶

    .L3:
sra     %g1, 0, %g1          ; Sign-extend (arithmetic right shift by 0)
mov     %g1, %i0             ; Move result to %i0 (return register)
return  %i7+8                ; Return to caller
 nop

    The result is prepared for return by moving it to %i0 (the return value register by SPARC convention). The sra instruction performs a sign-extending right shift by 0 bits - this is a no-op that just ensures the value is properly sign-extended for a 32-bit return value.

    Why This Code Is Unoptimized¶

    This is unoptimized code (-O0 flag). You can see several inefficiencies: - The pointer p is stored to the stack and reloaded multiple times instead of staying in a register - The value val is stored to the stack and reloaded instead of staying in a register
    - The null check code is generated even though it's unreachable - The sra %g1, 0, %g1 instruction is unnecessary
