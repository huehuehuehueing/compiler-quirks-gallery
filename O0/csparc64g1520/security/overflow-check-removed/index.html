    overflow-check-removed¶

    Details

    Source: security/overflow-check-removed.c Compiler: csparc64g1520 (SPARC64) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    overflow-check-removed.c
    /*
 * Security issue: integer overflow check removed.
 *
 * Signed integer overflow is undefined behavior in C.
 * Compiler may assume it never happens and remove checks.
 */
#include <limits.h>

int safe_add(int a, int b)
{
    /* Attempt to check for overflow before it happens.
     * With optimization, this check may be REMOVED because
     * the compiler assumes signed overflow is impossible.
     */
    if (a > 0 && b > 0 && a > INT_MAX - b)
        return -1;  /* overflow would occur */

    if (a < 0 && b < 0 && a < INT_MIN - b)
        return -1;  /* underflow would occur */

    return a + b;
}

    Explanation¶

    the key insight is that this is unoptimized code (-O0 flag), and it demonstrates how the compiler handles the overflow-checking logic through repeated memory operations and conditional branches.

    Function Prologue and Parameter Setup¶

    save    %sp, -176, %sp
mov     %i0, %g1
mov     %i1, %g2
st      %g1, [%fp+2175]
mov     %g2, %g1
st      %g1, [%fp+2183]

    The function starts by creating a stack frame (the save instruction allocates 176 bytes). The two input parameters arrive in %i0 and %i1 (by SPARC calling convention, input registers hold function arguments). The compiler then immediately stores these values to the stack at offsets 2175 and 2183 from the frame pointer. This is typical unoptimized behaviorthe compiler doesn't keep values in registers; instead, it spills them to memory and reloads them repeatedly.

    First Overflow Check: Positive Numbers¶

    ld      [%fp+2175], %g1      // load a
cmp     %g1, 0
ble     %icc, .L2            // if a <= 0, skip to .L2
ld      [%fp+2183], %g1      // load b
cmp     %g1, 0
ble     %icc, .L2            // if b <= 0, skip to .L2

    This checks the condition a > 0 && b > 0. The compiler loads a from the stack, compares it to zero, and branches to .L2 if it's not positive. If that passes, it loads b and checks if it's positive. Both conditions must be true to continue.

    sethi   %hi(2147482624), %g1
or      %g1, 1023, %g2       // construct INT_MAX (2147483647)
ld      [%fp+2183], %g1      // load b
sub     %g2, %g1, %g1        // compute INT_MAX - b
ld      [%fp+2175], %g2      // load a
cmp     %g2, %g1             // compare a with (INT_MAX - b)
ble     %icc, .L2            // if a <= INT_MAX - b, no overflow
mov     -1, %g1              // else set return value to -1
ba,pt   %xcc, .L3            // branch to exit

    If both a and b are positive, the code constructs INT_MAX using sethi (set high 22 bits) and or (set low 10 bits). It then computes INT_MAX - b and checks if a > INT_MAX - b. If the overflow condition is detected, it sets %g1 to -1 and jumps to the exit label .L3.

    Second Underflow Check: Negative Numbers¶

    .L2:
ld      [%fp+2175], %g1      // load a
cmp     %g1, 0
bge     %icc, .L4            // if a >= 0, skip to .L4
ld      [%fp+2183], %g1      // load b
cmp     %g1, 0
bge     %icc, .L4            // if b >= 0, skip to .L4

    This checks a < 0 && b < 0. The logic mirrors the first check but uses bge (branch if greater-or-equal) to skip when conditions aren't met.
