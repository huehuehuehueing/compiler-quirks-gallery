    if-conversion¶

    Details

    Source: control-flow/if-conversion.c Compiler: csparc64g1520 (SPARC64) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    if-conversion.c
    /*
 * If-Conversion (branch to conditional move)
 *
 * Short if-else chains that select between two values can be
 * replaced with conditional-move instructions (cmov on x86).
 * This eliminates a branch, avoiding potential misprediction
 * penalties on modern super-scalar CPUs.
 *
 * Look for: cmovl / cmovge / cmovne instead of jl / jge / jne
 * with a branch over a mov.
 */

int min(int a, int b)
{
    if (a < b)
        return a;
    return b;
}

int clamp(int x, int lo, int hi)
{
    if (x < lo)
        x = lo;
    if (x > hi)
        x = hi;
    return x;
}

int abs_val(int x)
{
    return x < 0 ? -x : x;
}

    Explanation¶

    the key insight is that this is unoptimized code (-O0 flag), and it demonstrates why the comment about "if-conversion" (replacing branches with conditional moves) is importantthe compiler is NOT doing this optimization.

    The Core Problem: Unnecessary Branches Instead of Conditional Moves¶

    The comment in your source code mentions that modern CPUs prefer conditional moves over branches. A conditional move is a single instruction that says "move this value only if a condition is true," avoiding the need to jump around code. However, this unoptimized code uses actual branches (bge, ble, ba) instead, which is less efficient on modern processors.

    Understanding SPARC Calling Conventions¶

    Before diving into each function, here's the calling convention being used: - Input registers (%i0, %i1, %i2): Parameters passed to the function - Output register (%i0): Used to return the result - %sp: Stack pointer (points to top of stack) - %fp: Frame pointer (points to current function's stack frame) - %i7: Return address register

    The min Function Walkthrough¶

    save    %sp, -176, %sp
    This creates a new stack frame by allocating 176 bytes of local storage. The save instruction is SPARC's way of setting up a function's workspace.

    mov     %i0, %g1        // Copy parameter a into %g1
mov     %i1, %g2        // Copy parameter b into %g2
st      %g1, [%fp+2175] // Store a on the stack
st      %g1, [%fp+2183] // Store b on the stack

    Here's the first inefficiency: the parameters are already in registers, but the compiler stores them to the stack and then loads them back. This is typical unoptimized behaviorthe compiler doesn't optimize away these redundant memory operations.

    ld      [%fp+2175], %g2 // Load a from stack
ld      [%fp+2183], %g1 // Load b from stack
cmp     %g2, %g1        // Compare: is a < b?
bge     %icc, .L2       // If a >= b, jump to .L2 (branch if greater-or-equal)
 nop                    // Delay slot (explained below)

    The cmp instruction sets condition flags. The bge (branch if greater-or-equal) jumps to label .L2 if the condition is true. The nop (no operation) after the branch is a delay slotSPARC requires one instruction after a branch before the jump takes effect, so nop fills this slot.

    ld      [%fp+2175], %g1 // Load a from stack (the "true" case)
ba,pt   %xcc, .L3       // Unconditional branch to .L3
 nop
.L2:
ld      [%fp+2183], %g1 // Load b from stack (the "false" case)
.L3:

    This is the branch structure: if a < b, load a; otherwise, load b. Then both paths converge at .L3.

    sra     %g1, 0, %g1     // Sign-extend (shift right arithmetic by 0)
mov     %g1, %i0        // Move result to return register
return  %i7+8           // Return to caller
 nop

    The sra instruction with a shift of 0 is a sign-extension operation (ensuring the value is properly formatted as a 32-bit signed integer). Then the result is moved to %i0 for return.

    The clamp Function¶

    This function has the same pattern repeated twiceonce to clamp the lower bound, once for the upper bound:

    if (x < lo)
    x = lo;
if (x > hi)
    x = hi;

    The assembly mirrors this with two separate branch sequences (.L5 and .L6 labels), each following the same store-to-stack, load-from-stack, compare, and branch pattern.
