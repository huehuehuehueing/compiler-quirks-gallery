    unrollme-2¶

    Details

    Source: loops/unrollme-2.c Compiler: csparc64g1520 (SPARC64) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    unrollme-2.c
    #include <stdio.h>

void foo(void)
{
    int i;
    char buf[8];

    for (i = 0; i < sizeof(buf); i++)
    {
        buf[i] = 'A';
    }

    /*
     * by dereferencing buf this should force the compile to NOT
     * eliminate the above as dead code. loop will likely still
     * be unrolled
     */
    printf("%s", buf);
}

    Explanation¶

    the key insight is that this is unoptimized code that uses stack memory inefficiently and repeatedly loads/stores the loop variable instead of keeping it in a register.

    Function Prologue and Stack Setup¶

    save    %sp, -192, %sp

    This instruction creates a new stack frame for the function. In SPARC, save is the standard way to enter a function—it allocates 192 bytes of local space on the stack and sets up the frame pointer (%fp). This large allocation is typical of unoptimized code; an optimized compiler would use much less space since we only need 8 bytes for the buf array and 4 bytes for the loop variable i.

    Loop Variable Initialization¶

    st      %g0, [%fp+2043]

    This stores zero (from %g0, which is always zero in SPARC) into memory at offset 2043 from the frame pointer. This initializes the loop variable i to 0. Notice it's stored in memory rather than kept in a register—this is a hallmark of unoptimized code.

    Loop Structure¶

    The loop uses a branch-to-label pattern:

    ba,pt   %xcc, .L2        // Branch always to loop condition check
 nop                      // Delay slot (executed before branch)

    This jumps to .L2, which is the loop condition check. The ,pt suffix is a branch prediction hint meaning "predict taken."

    Loop Condition Check (.L2)¶

    ld      [%fp+2043], %g1  // Load i from memory
cmp     %g1, 7           // Compare i with 7 (sizeof(buf) - 1)
bleu    %icc, .L3        // Branch if less-than-or-equal to loop body
 nop

    This loads i from memory, compares it to 7, and branches to .L3 (the loop body) if the condition is true. The bleu instruction means "branch if less-than-or-equal, unsigned." This happens every iteration—even though i is a simple loop counter, the compiler loads it from memory each time.

    Loop Body (.L3)¶

    ld      [%fp+2043], %g1  // Load i again
sra     %g1, 0, %g1      // Sign-extend (arithmetic right shift by 0)
add     %g1, 2047, %g1   // Add 2047 to i
add     %g1, %fp, %g1    // Add frame pointer to get buf address
mov     65, %g2          // Move 65 ('A') into g2
stb     %g2, [%g1-16]    // Store byte 'A' at buf[i]

    This sequence calculates the address of buf[i] and stores the character 'A' (ASCII 65). The calculation is convoluted: it loads i, adds 2047 (an offset), adds the frame pointer, then subtracts 16 when storing. This is how the compiler maps the local array address to the stack frame.

    Then the loop variable is incremented:

    ld      [%fp+2043], %g1  // Load i from memory
add     %g1, 1, %g1      // Increment
st      %g1, [%fp+2043]  // Store back to memory

    Again, this is inefficient—the variable is loaded, modified, and stored back to memory every iteration.

    Preparing the printf Call¶

    add     %fp, 2031, %g1   // Calculate address of buf
mov     %g1, %o1         // Move to %o1 (second parameter)
sethi   %hi(.LC0), %g1   // Load high bits of format string address
or      %g1, %lo(.LC0), %o0  // Load low bits and place in %o0 (first parameter)
call    printf, 0        // Call printf
 nop                      // Delay slot

    This sets up the two arguments for printf: the format string "%s" in %o0 and the buffer address in %o1. In SPARC's calling convention, the first two arguments go in %o0 and %o1.
