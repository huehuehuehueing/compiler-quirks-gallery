    memcpy-inline¶

    Details

    Source: memory/memcpy-inline.c Compiler: csparc64g1520 (SPARC64) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    memcpy-inline.c
    /*
 * memcpy inlining: small copies become inline loads/stores.
 *
 * For small, known-size copies, the compiler generates
 * inline code instead of calling memcpy.
 */
#include <string.h>

struct small {
    int a, b;
};

void copy_8_bytes(void *dst, const void *src)
{
    /* Small copy - likely inlined as single 64-bit load/store
     * on 64-bit architectures, or two 32-bit ops on 32-bit.
     */
    memcpy(dst, src, 8);
}

void copy_struct(struct small *dst, const struct small *src)
{
    /* Structure copy - same as memcpy but type-aware.
     * Compiler knows alignment and size.
     */
    *dst = *src;
}

void copy_varying(void *dst, const void *src, int use_large)
{
    /* Compile-time unknown size - must call memcpy or
     * generate size-checking code.
     */
    if (use_large)
        memcpy(dst, src, 1024);
    else
        memcpy(dst, src, 8);
}

    Explanation¶

    the key insight is that this is unoptimized code (compiled with -O0), and it shows three different approaches to copying memory, with dramatically different efficiency levels.

    The Most Important Pattern: Unoptimized Stack Overhead¶

    This is unoptimized code. Notice how every function starts with save %sp, -176, %sp — this creates a 176-byte stack frame even for simple operations. Then parameters are immediately stored to the stack and reloaded. This is characteristic of -O0 compilation: the compiler doesn't optimize away unnecessary memory operations.

    Function 1: copy_8_bytes — Calling memcpy¶

    save    %sp, -176, %sp          # Create stack frame
stx     %i0, [%fp+2175]         # Store dst pointer to stack
stx     %i1, [%fp+2183]         # Store src pointer to stack
mov     8, %o2                  # Set size = 8 bytes
ldx     [%fp+2183], %o1         # Reload src from stack into %o1
ldx     [%fp+2175], %o0         # Reload dst from stack into %o0
call    memcpy, 0               # Call memcpy function

    What's happening: The compiler receives parameters in %i0 (dst) and %i1 (src) — these are the "input" registers in SPARC's calling convention. Instead of using them directly, the unoptimized compiler: 1. Stores them to the stack 2. Reloads them into %o0, %o1, %o2 (the "output" registers used for function arguments) 3. Calls memcpy

    This is inefficient but correct. An optimized compiler would skip the stack operations and move parameters directly to the output registers.

    Function 2: copy_struct — Inline Structure Copy¶

    ldx     [%fp+2175], %g1         # Load dst pointer
ldx     [%fp+2183], %g2         # Load src pointer
ld      [%g2], %g3              # Load first 4 bytes from src
st      %g3, [%g1]              # Store first 4 bytes to dst
ld      [%g2+4], %g2            # Load second 4 bytes from src
st      %g2, [%g1+4]            # Store second 4 bytes to dst

    This is the key difference: For structure assignment (*dst = *src), the compiler inlines the copy as two 32-bit load/store pairs instead of calling memcpy. This is much faster because: - No function call overhead - Direct memory operations - The compiler knows the exact size (8 bytes) at compile time

    The compiler uses %g1 and %g2 (general-purpose registers) as temporary storage for pointers and data.

    Function 3: copy_varying — Runtime Size Decision¶

    mov     %i2, %g1                # Move use_large parameter to %g1
st      %g1, [%fp+2191]         # Store to stack
ld      [%fp+2191], %g1         # Reload from stack
cmp     %g1, 0                  # Compare with 0
be      %icc, .L4               # Branch if equal (use_large is false)

    The conditional logic: The function checks the use_large parameter: - If true (non-zero): calls memcpy(dst, src, 1024) - If false (zero): calls memcpy(dst, src, 8)

    Both branches follow the same inefficient pattern as copy_8_bytes — storing parameters to stack, reloading them, then calling memcpy. The compiler cannot inline this copy because the size is unknown at compile time.

    Why This Code Is Inefficient¶

    The -O0 compilation shows several wasteful patterns:

    1. Unnecessary stack operations: Parameters are stored then immediately reloaded. Optimized code would use registers directly.
