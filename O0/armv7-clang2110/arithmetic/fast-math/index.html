    fast-math¶

    Details

    Source: arithmetic/fast-math.c Compiler: armv7-clang2110 (ARM) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    fast-math.c
    /*
 * Fast-math optimizations: -ffast-math / -Ofast effects.
 *
 * These flags allow the compiler to reorder floating-point
 * operations, assuming associativity (which floats don't have).
 * Can change results or break special value handling.
 */

/* With -ffast-math, this may be reordered and give different results */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++)
        sum += arr[i];
    return sum;
}

/* NaN check may be optimized away with -ffast-math */
int is_nan(float x)
{
    /* With -ffast-math, compiler assumes no NaN values exist,
     * so this comparison may be optimized to always return 0.
     */
    return x != x;
}

/* Division by variable may become multiplication by reciprocal */
float divide_by_y(float x, float y)
{
    /* With -ffast-math: may become x * (1.0f / y)
     * Less accurate but faster on some architectures.
     */
    return x / y;
}

/* Algebraic simplification (inexact with floats!) */
float algebraic(float a, float b, float c)
{
    /* With -ffast-math: (a + b) + c may be reordered to a + (b + c)
     * These can give different results with floating point!
     */
    return (a + b) + c;
}

    Explanation¶

    the key insight is that this is unoptimized code (-O0), so the compiler prioritizes correctness and simplicity over efficiency. You'll see lots of unnecessary memory operations (storing values then immediately loading them back) that an optimized compiler would eliminate.

    Overall Structure¶

    The code contains four functions. Each follows the ARM32 calling convention where: - Floating-point parameters come in s0, s1, s2 (32-bit float registers) - Integer parameters come in r0, r1, r2 - The return value goes in s0 (for floats) or r0 (for integers)

    sum_array() - Loop with Accumulation¶

    Key pattern: This function demonstrates a loop that accumulates floating-point values. The inefficiency here is dramaticthe compiler stores and reloads the loop counter repeatedly.

    Setup phase (lines 1-9): 

    push    {r11, lr}           // Save frame pointer and return address
mov     r11, sp             // Set up new frame pointer
sub     sp, sp, #16         // Allocate 16 bytes of stack space
str     r0, [r11, #-4]      // Store array pointer
str     r1, [sp, #8]        // Store n (loop limit)
vldr    s0, .LCPI0_0        // Load 0.0 into s0 (floating-point register)
vstr    s0, [sp, #4]        // Store sum on stack at offset 4
ldr     r0, .LCPI0_0        // Load 0 into r0 (integer register)
str     r0, [sp]            // Store loop counter i on stack at offset 0

    Notice the redundancy: the compiler loads the constant 0.0 into both a floating-point register (s0) and an integer register (r0), then stores both. This is typical unoptimized behaviorno attempt to reuse values.

    Loop body (lines 10-24): 

    .LBB0_1:                     // Loop start label
ldr     r0, [sp]            // Load i from stack
ldr     r1, [sp, #8]        // Load n from stack
cmp     r0, r1              // Compare i with n
bge     .LBB0_4             // If i >= n, jump to exit

    The loop condition check reloads i and n from memory every iterationthese values never change during the loop body, so an optimized compiler would keep them in registers.

    ldr     r0, [r11, #-4]      // Load array pointer
ldr     r1, [sp]            // Load i (again!)
add     r0, r0, r1, lsl #2  // Calculate address: arr + (i * 4)
vldr    s2, [r0]            // Load arr[i] into s2
vldr    s0, [sp, #4]        // Load sum from stack
vadd.f32 s0, s0, s2         // Add: sum = sum + arr[i]
vstr    s0, [sp, #4]        // Store sum back to stack

    The add r0, r0, r1, lsl #2 instruction calculates the array element address: it shifts i left by 2 bits (multiplying by 4, since floats are 4 bytes) and adds it to the base pointer.

    ldr     r0, [sp]            // Load i (yet again!)
add     r0, r0, #1          // Increment i
str     r0, [sp]            // Store i back to stack
b       .LBB0_1             // Jump back to loop start

    The increment operation loads i, adds 1, and stores it backall from/to stack memory.

    Exit and return (lines 25-27): 

    .LBB0_4:
vldr    s0, [sp, #4]        // Load sum into s0 (return value)
mov     sp, r11             // Restore stack pointer
pop     {r11, pc}           // Restore frame pointer and return
