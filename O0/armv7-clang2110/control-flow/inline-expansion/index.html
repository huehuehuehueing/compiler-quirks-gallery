    inline-expansion¶

    Details

    Source: control-flow/inline-expansion.c Compiler: armv7-clang2110 (ARM) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    inline-expansion.c
    /*
 * Function inlining: small functions are expanded at call site.
 *
 * Eliminates call overhead and enables further optimizations
 * by exposing the function body to the caller's context.
 */

/* Tiny function - almost certainly inlined */
static inline int square(int x)
{
    return x * x;
}

/* Using static helps inlining (internal linkage) */
static int cube(int x)
{
    return x * x * x;
}

/* Wrapper that uses inlined functions */
int compute(int x)
{
    /* After inlining, this becomes:
     * return (x * x) + (x * x * x);
     * Which may further simplify to:
     * return x * x * (1 + x);
     */
    return square(x) + cube(x);
}

/* Recursive functions are NOT inlined (infinite expansion) */
int fibonacci(int n)
{
    if (n <= 1) return n;
    return fibonacci(n-1) + fibonacci(n-2);
}

/* Large functions may not be inlined (code size cost) */
int large_function(int x)
{
    int result = 0;
    for (int i = 0; i < 100; i++) {
        result += x * i;
        result ^= (result >> 3);
        result += (result << 5);
    }
    return result;
}

    Explanation¶

    the key insight is that this is unoptimized code (-O0), so functions are NOT being inlined despite the source code's comments suggesting they would be. Instead, each function is compiled separately with full function call overhead.

    Stack Frame Setup Pattern¶

    Every function follows the same ARM32 calling convention pattern:

    push    {r11, lr}      // Save frame pointer and return address
mov     r11, sp        // Set up new frame pointer
sub     sp, sp, #N     // Allocate local variables on stack

    In ARM32, r11 is the frame pointer (tracks the current function's stack frame), and lr (link register) holds the return address. The push instruction saves both, and sub sp, sp, #N makes room for local variables. This setup happens in every function because the compiler isn't optimizing it away.

    compute(int) - Function Calls Without Inlining¶

    Despite square() being marked inline in the source, the compiler generates actual function calls:

    str     r0, [sp, #4]        // Store parameter x on stack
ldr     r0, [sp, #4]        // Load x back into r0
bl      square(int)         // Call square (bl = branch with link)
str     r0, [sp]            // Store result on stack
ldr     r0, [sp, #4]        // Load x again
bl      cube(int)           // Call cube
mov     r1, r0              // Move cube result to r1
ldr     r0, [sp]            // Load square result back
add     r0, r0, r1          // Add them together

    Notice the inefficiency: the parameter x is stored to stack, then loaded back just to pass it to the function. The results are also stored and reloaded. At -O0, the compiler doesn't optimize away these redundant memory operations.

    square(int) and cube(int) - Redundant Load-Store Patterns¶

    square() computes x * x:

    str     r0, [sp]        // Store x on stack
ldr     r0, [sp]        // Load x back
ldr     r1, [sp]        // Load x again (same value!)
mul     r0, r0, r1      // Multiply: r0 = r0 * r1

    This is unoptimized: the value is stored then loaded twice when it could stay in a register. cube() does the same thing twice (for x * x * x):

    ldr     r0, [sp]        // Load x
ldr     r1, [sp]        // Load x again
mul     r0, r0, r1      // r0 = x * x
ldr     r1, [sp]        // Load x a third time
mul     r0, r0, r1      // r0 = (x*x) * x

    fibonacci(int) - Recursive Function with Branching¶

    This function cannot be inlined (recursion would cause infinite expansion). The control flow uses conditional branching:

    cmp     r0, #1          // Compare n with 1
bgt     .LBB3_2         // Branch if greater than 1

    If n <= 1, it returns n directly. Otherwise, it makes two recursive calls:

    sub     r0, r0, #1
bl      fibonacci(int)  // Call fibonacci(n-1)
str     r0, [sp, #4]    // Save result

sub     r0, r0, #2      // (Note: r0 was modified, so this is n-1-2 = n-3, not n-2!)
bl      fibonacci(int)  // Call fibonacci(n-2)

    There's a subtle issue here: after the first recursive call, r0 contains the result, not the original n. The second sub operates on the wrong value. This is unoptimized code behaviorthe compiler doesn't optimize the register usage.
