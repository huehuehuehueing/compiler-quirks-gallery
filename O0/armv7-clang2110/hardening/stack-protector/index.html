    stack-protector¶

    Details

    Source: hardening/stack-protector.c Compiler: armv7-clang2110 (ARM) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    stack-protector.c
    /* @gallery-hints
 *   extra-flags: -fstack-protector-all
 *   compiler-exclude: vc_v19_44_VS17_14_x64, vc_v19_44_VS17_14_x86
 */

/*
 * Stack Protector (Stack Canary)
 *
 * With -fstack-protector-all the compiler inserts a canary value
 * between local variables and the saved return address. Before the
 * function returns, the canary is checked; if it was overwritten
 * (e.g. by a buffer overflow), the program aborts.
 *
 * Look for: canary load from %fs:0x28 (or %gs:0x14 on 32-bit),
 * comparison before the function epilogue, and a call to
 * __stack_chk_fail on mismatch.
 */
#include <string.h>

void copy_input(const char *input)
{
    char buf[64];
    strcpy(buf, input);   /* intentionally unsafe for demonstration */
    (void)buf;
}

    Explanation¶

    the key insight here is that the compiler is protecting against buffer overflow attacks by inserting a "canary" value — a secret number that guards the return address.

    Stack Frame Setup¶

    The function starts by establishing a stack frame:

    push    {r11, lr}        // Save frame pointer (r11) and return address (lr)
mov     r11, sp          // Set up new frame pointer
sub     sp, sp, #72      // Allocate 72 bytes on stack (64 for buffer + 8 extra)

    In ARM32, r11 is the frame pointer (by convention, it tracks the current function's stack frame), and lr is the link register (holds the return address). The sub sp, sp, #72 allocates space for the 64-byte buffer plus extra space for the canary and other data.

    Loading the Canary Value¶

    The next section loads a secret "canary" value from a protected location:

    ldr     r1, .LCPI0_0     // Load address offset to __stack_chk_guard
add     r1, pc, r1       // Calculate actual address using PC-relative addressing
ldr     r1, [r1]         // Load pointer from GOT (Global Offset Table)
ldr     r1, [r1]         // Load the actual canary value
str     r1, [r11, #-4]   // Store canary at [r11 - 4] (just before saved r11)

    This is a position-independent code (PIC) pattern. The compiler uses pc-relative addressing to find the canary without knowing where the code will be loaded in memory. The canary is stored at [r11, #-4], which places it right before the saved frame pointer — directly above where a buffer overflow would overwrite the return address.

    Function Body¶

    str     r0, [sp]         // Store input pointer on stack
add     r0, sp, #4       // Calculate buffer address (sp + 4)
ldr     r1, [sp]         // Load input pointer back into r1
bl      strcpy           // Call strcpy(buf, input)

    By convention in ARM32, r0 holds the first parameter (the input pointer). The code stores it on the stack, then passes the buffer address in r0 and the input pointer in r1 to strcpy. This is unoptimized — a better compiler would keep values in registers rather than storing and reloading them.

    Canary Verification (The Protection)¶

    Before returning, the compiler checks if the canary was corrupted:

    ldr     r0, .LCPI0_2     // Load address offset to __stack_chk_guard
add     r0, pc, r0       // Calculate actual address
ldr     r0, [r0]         // Load pointer from GOT
ldr     r0, [r0]         // Load current canary value
ldr     r1, [r11, #-4]   // Load stored canary from stack
cmp     r0, r1           // Compare them
bne     .LBB0_2          // If not equal, jump to error handler

    The cmp instruction compares the original canary (now in r0) with the stored canary (in r1). If they don't match, a buffer overflow has occurred, and the code branches to .LBB0_2.

    Normal Return vs. Error Path¶

    If the canary is intact: 

    mov     sp, r11          // Restore stack pointer
pop     {r11, pc}        // Restore frame pointer and return

    If the canary was corrupted: 

    .LBB0_2:
bl      __stack_chk_fail // Call abort function

    The __stack_chk_fail function terminates the program, preventing the attacker from continuing execution.

    Key Observations¶

    • Three canary loads: The code loads the canary value three times (once to store it, twice to verify it). This is unoptimized — -O2 or -O3 would likely cache the value in a register.
    • PC-relative addressing: All three canary accesses use the same pattern (ldr + add + ldr + ldr), which is necessary for position-independent code but adds overhead.
