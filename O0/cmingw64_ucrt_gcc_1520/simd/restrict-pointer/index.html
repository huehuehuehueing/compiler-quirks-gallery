    restrict-pointer¶

    Details

    Source: simd/restrict-pointer.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    restrict-pointer.c
    /*
 * Restrict pointers enable better optimization.
 *
 * The 'restrict' keyword promises that pointers don't alias,
 * allowing the compiler to optimize more aggressively.
 */

/* Without restrict - compiler must be conservative */
void copy_no_restrict(int *dst, const int *src, int n)
{
    /* dst and src MIGHT overlap, so compiler must handle that */
    for (int i = 0; i < n; i++) {
        dst[i] = src[i];
    }
}

/* With restrict - compiler knows no aliasing */
void copy_restrict(int *restrict dst, const int *restrict src, int n)
{
    /* dst and src guaranteed not to overlap
     * Compiler can use faster copy or vectorize freely
     */
    for (int i = 0; i < n; i++) {
        dst[i] = src[i];
    }
}

/* Multiplication example - dramatic difference */
void scale_no_restrict(float *result, const float *input,
                       const float *scale, int n)
{
    /* Compiler doesn't know if result aliases input or scale */
    for (int i = 0; i < n; i++) {
        result[i] = input[i] * (*scale);
    }
    /* Must reload *scale each iteration! */
}

void scale_restrict(float *restrict result,
                    const float *restrict input,
                    const float *restrict scale, int n)
{
    /* Compiler knows result doesn't alias scale */
    for (int i = 0; i < n; i++) {
        result[i] = input[i] * (*scale);
    }
    /* Can load *scale once and reuse */
}

    Explanation¶

    the key insight is that at -O0 (unoptimized), the restrict keyword makes no difference in the generated code. Both versions produce identical assembly. This is unoptimized code, so the compiler isn't applying any optimizations that would benefit from the restrict promise.

    Understanding the Key Pattern¶

    This is unoptimized code. The compiler stores parameters on the stack immediately and reloads them repeatedly from memory rather than keeping them in registers. The restrict keyword is an optimization hint, but without optimization enabled (-O0), the compiler ignores it entirely.

    Stack Frame Setup (All Functions)¶

    Each function follows the same pattern: 

    push    rbp              # Save old frame pointer
mov     rbp, rsp         # Set up new frame pointer
sub     rsp, 16          # Allocate 16 bytes of local stack space

    This creates a stack frame. The parameters (passed in registers by the calling convention) are immediately stored to the stack: - rcx 16[rbp] (first pointer parameter) - rdx 24[rbp] (second pointer parameter)
    - r8/r8d 32[rbp] (third parameter) - r9d 40[rbp] (fourth parameter, in scale_* functions)

    The loop counter i lives at -4[rbp] on the stack.

    The Copy Functions (copy_no_restrict and copy_restrict)¶

    Both functions are identical in assembly, demonstrating that -O0 ignores restrict:

    Loop body (simplified flow): 1. Load loop counter i from stack 2. Convert to 64-bit with cdqe (sign-extend 32-bit to 64-bit) 3. Calculate src[i] address: lea rdx, 0[0+rax*4] computes i*4 (byte offset for 4-byte integers), then add rax, rdx adds it to the src base pointer 4. Load src[i] into eax 5. Repeat address calculation for dst[i] 6. Store the value to dst[i] 7. Increment loop counter and jump back

    The redundancy here is striking: the loop counter is loaded from stack, converted, and used multiple times per iteration. An optimized version would keep i in a register and reuse it.

    The Scale Functions (scale_no_restrict and scale_restrict)¶

    Again, both are identical in assembly. The key difference from the copy functions is the use of floating-point registers:

    Loop body: 1. Load input[i] into xmm1 (a 128-bit floating-point register, using only the lower 32 bits for a single float) 2. Load *scale (dereference the scale pointer) into xmm0 3. Calculate result[i] address 4. Multiply: mulss xmm0, xmm1 (scalar single-precision multiply) 5. Store result to result[i]

    The instructions movss (move scalar single-precision) and mulss (multiply scalar single-precision) operate on 32-bit floating-point values within the XMM registers.

    Why restrict Doesn't Help Here¶

    At -O0, the compiler makes no attempt to: - Keep values in registers across iterations - Hoist invariant loads (like *scale) outside the loop - Eliminate redundant address calculations - Vectorize operations

    With -O2 or -O3, the restrict keyword would allow the compiler to: - Load *scale once before the loop and reuse it (in scale_restrict) - Keep *scale in memory and reload it each iteration (in scale_no_restrict, being conservative) - Potentially vectorize the copy operations

    The unoptimized code treats both versions identically because optimization is disabled entirely.
