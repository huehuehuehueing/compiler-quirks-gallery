    static-memcpy¶

    Details

    Source: string-literals/static-memcpy.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    static-memcpy.c
    #include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[5];

    memcpy(buf, "AAAA", 4);
    buf[sizeof(buf) - 1] = '\0';

    /* make sure the above is not treated as dead code */
    puts(buf);
}

    Explanation¶

    the key insight is that this is unoptimized code that sets up a stack frame, calls two library functions, and cleans up afterward.

    Stack Frame Setup¶

    The function starts by establishing a stack frame — a region of memory on the stack that belongs to this function:

    push    rbp              # Save the old base pointer
mov     rbp, rsp         # Set up new base pointer for this function
sub     rsp, 48          # Allocate 48 bytes of local storage

    The rbp register (base pointer) marks the start of the function's local memory. By subtracting 48 from rsp (stack pointer), the function reserves space for the 5-byte buf array plus extra padding. This extra space (48 bytes instead of just 5) is typical of unoptimized code — the compiler doesn't optimize stack allocation.

    Preparing the memcpy Call¶

    The next section sets up arguments for the memcpy function call. On amd64, the calling convention passes the first four integer arguments in specific registers: rcx, rdx, r8, and r9.

    lea     rdx, .LC0[rip]   # Load address of "AAAA" string into rdx (2nd argument)
lea     rax, -5[rbp]     # Calculate address of buf: rbp - 5
mov     r8d, 4           # Load 4 (byte count) into r8 (3rd argument)
mov     rcx, rax         # Move buf address into rcx (1st argument)
call    memcpy           # Call memcpy(buf, "AAAA", 4)

    The lea (Load Effective Address) instruction calculates addresses without accessing memory — it's just arithmetic. The address -5[rbp] means "5 bytes below rbp", which is where the 5-byte buffer lives on the stack.

    Null-Terminating the Buffer¶

    mov     BYTE PTR -1[rbp], 0

    This writes a single byte (0, which is the null terminator '\0') to the location at rbp - 1. Since the buffer starts at rbp - 5, this writes to the last byte of the buffer (buf[4]).

    Calling puts¶

    lea     rax, -5[rbp]     # Calculate buf address again
mov     rcx, rax         # Move buf address into rcx (1st argument)
call    puts             # Call puts(buf)

    The buffer address is calculated again and passed to puts, which prints the string.

    Cleanup and Return¶

    nop                      # No operation (padding, does nothing)
add     rsp, 48          # Deallocate the 48 bytes we reserved
pop     rbp              # Restore the old base pointer
ret                      # Return to caller

    The add rsp, 48 reverses the stack allocation, and pop rbp restores the previous function's base pointer. The nop instruction is padding — unoptimized code often includes these.

    Key Observations¶

    This is unoptimized code (-O0 flag). Notice the inefficiencies: - The buffer address is calculated twice (once for memcpy, once for puts) instead of reusing it - 48 bytes are allocated when only 5 are needed - The nop instruction serves no purpose - Unnecessary register-to-register moves (like mov rcx, rax) could be eliminated

    With optimization enabled (-O2 or -O3), the compiler would eliminate these redundancies and likely inline or optimize the function further.
