    fast-math¶

    Details

    Source: arithmetic/fast-math.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    fast-math.c
    /*
 * Fast-math optimizations: -ffast-math / -Ofast effects.
 *
 * These flags allow the compiler to reorder floating-point
 * operations, assuming associativity (which floats don't have).
 * Can change results or break special value handling.
 */

/* With -ffast-math, this may be reordered and give different results */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++)
        sum += arr[i];
    return sum;
}

/* NaN check may be optimized away with -ffast-math */
int is_nan(float x)
{
    /* With -ffast-math, compiler assumes no NaN values exist,
     * so this comparison may be optimized to always return 0.
     */
    return x != x;
}

/* Division by variable may become multiplication by reciprocal */
float divide_by_y(float x, float y)
{
    /* With -ffast-math: may become x * (1.0f / y)
     * Less accurate but faster on some architectures.
     */
    return x / y;
}

/* Algebraic simplification (inexact with floats!) */
float algebraic(float a, float b, float c)
{
    /* With -ffast-math: (a + b) + c may be reordered to a + (b + c)
     * These can give different results with floating point!
     */
    return (a + b) + c;
}

    Explanation¶

    the key insight is that this is unoptimized code (-O0), which means the compiler prioritizes correctness and simplicity over speed. You'll see inefficient patterns like storing values to the stack and immediately loading them back.

    Function Calling Convention¶

    All four functions use the x64 calling convention (Microsoft x64 ABI on Windows). This means: - Floating-point parameters arrive in xmm0, xmm1, xmm2 (SSE registers) - Integer parameters arrive in rcx, rdx, r8, r9 - Return values go back in xmm0 (for floats) or rax (for integers)

    sum_array Function¶

    This function accumulates array elements into a running sum.

    Stack setup: 

    push    rbp              // Save old frame pointer
mov     rbp, rsp         // Create new frame
sub     rsp, 16          // Allocate 16 bytes of local storage

    Parameter storage (unoptimized pattern): 

    mov     QWORD PTR 16[rbp], rcx    // Store array pointer
mov     DWORD PTR 24[rbp], edx    // Store array length

    Even though these parameters are already in registers, the compiler stores them to the stack. This is typical unoptimized behaviorthe compiler doesn't optimize away unnecessary memory operations.

    Loop initialization: 

    pxor    xmm0, xmm0                    // Zero out xmm0 (0.0f)
movss   DWORD PTR -4[rbp], xmm0       // Store sum to stack at [rbp-4]
mov     DWORD PTR -8[rbp], 0          // Store loop counter i to stack at [rbp-8]
jmp     .L2                           // Jump to loop condition

    The pxor instruction XORs a register with itselfa common way to zero it. movss (move scalar single-precision) stores the 32-bit float value to the stack.

    Loop body (.L3): 

    mov     eax, DWORD PTR -8[rbp]        // Load i from stack into eax
cdqe                                  // Sign-extend eax to rax (convert int to long)
lea     rdx, 0[0+rax*4]               // Calculate offset: i * 4 (each float is 4 bytes)
mov     rax, QWORD PTR 16[rbp]        // Load array pointer
add     rax, rdx                      // Calculate &arr[i]
movss   xmm0, DWORD PTR [rax]         // Load arr[i] into xmm0
movss   xmm1, DWORD PTR -4[rbp]       // Load sum from stack into xmm1
addss   xmm0, xmm1                    // Add: xmm0 = arr[i] + sum
movss   DWORD PTR -4[rbp], xmm0       // Store result back to stack
add     DWORD PTR -8[rbp], 1          // Increment i

    Notice the store-then-load pattern: the sum is stored to the stack, then immediately loaded back. This is inefficientan optimized compiler would keep the sum in a register throughout the loop.

    Loop condition (.L2): 

    mov     eax, DWORD PTR -8[rbp]        // Load i
cmp     eax, DWORD PTR 24[rbp]        // Compare i with n
jl      .L3                           // Jump back if i < n

    Return: 

    movss   xmm0, DWORD PTR -4[rbp]       // Load sum into xmm0 (return register)
add     rsp, 16                       // Deallocate stack space
pop     rbp                           // Restore frame pointer
ret
