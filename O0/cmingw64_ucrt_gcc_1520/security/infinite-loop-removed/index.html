    infinite-loop-removed¶

    Details

    Source: security/infinite-loop-removed.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    infinite-loop-removed.c
    /*
 * Infinite loop removed as undefined behavior.
 *
 * A loop with no side effects that doesn't terminate is UB.
 * Compiler may remove it entirely or assume it terminates.
 */

/* This function may not loop forever as expected! */
void spin_forever(void)
{
    /* No volatile, no side effects - UB if it never terminates.
     * Compiler may remove this entirely.
     */
    while (1)
        ;
}

/* Correct way to spin (e.g., for embedded busy-wait) */
void spin_forever_safe(void)
{
    /* volatile prevents optimization */
    while (1)
    {
        __asm__ volatile("" ::: "memory");
    }
}

    Explanation¶

    the key insight here is that both functions produce identical infinite loops, but they represent fundamentally different compiler behaviors due to undefined behavior.

    The Core Issue: Undefined Behavior¶

    The first function spin_forever() contains undefined behavior (UB) — a loop with no side effects that never terminates. The C standard doesn't define what should happen in this case, so the compiler is free to do anything. The second function spin_forever_safe() uses volatile inline assembly to prevent the compiler from optimizing away the loop.

    Assembly Breakdown¶

    spin_forever() — The Problematic Version¶

    spin_forever:
    push    rbp              # Save old base pointer
    mov     rbp, rsp         # Set up new stack frame
.L2:
    jmp     .L2              # Jump back to .L2 (infinite loop)

    The compiler generates an infinite loop here, but this is one possible implementation of undefined behavior. The compiler could have: - Removed the loop entirely (since it has no observable effect) - Assumed the loop terminates and optimized accordingly - Generated the infinite loop as shown

    At -O0 (unoptimized), the compiler takes the loop literally and generates the jump instruction.

    spin_forever_safe() — The Correct Version¶

    spin_forever_safe:
    push    rbp              # Save old base pointer
    mov     rbp, rsp         # Set up new stack frame
.L4:
    jmp     .L4              # Jump back to .L4 (infinite loop)

    This produces the same infinite loop, but for a valid reason: the __asm__ volatile("" ::: "memory") inline assembly acts as a compiler barrier. The volatile keyword tells the compiler "this has side effects I can't see," and the "memory" clobber list says "this might affect memory in unpredictable ways." This prevents the compiler from optimizing away the loop.

    Stack Frame Setup¶

    Both functions perform identical stack frame setup: - push rbp saves the previous base pointer (following the System V AMD64 calling convention) - mov rbp, rsp establishes a new stack frame by making rbp point to the current stack position

    This is typical unoptimized code — even though these functions don't use local variables, the compiler still sets up a full stack frame.

    Why They Look Identical¶

    At -O0, the compiler hasn't applied optimizations that would reveal the difference between these functions. With higher optimization levels like -O2 or -O3, the first function might be optimized away entirely (since its loop has no observable effect), while the second would remain as an infinite loop (due to the volatile barrier preventing removal).
