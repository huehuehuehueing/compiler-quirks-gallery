    static-copy-1¶

    Details

    Source: string-literals/static-copy-1.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    static-copy-1.c
    /*
 * String copy with strcpy - compiler may inline as immediate stores.
 *
 * For small, known-length strings, strcpy becomes direct
 * memory stores with the string data embedded in instructions.
 */
#include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[100];
    strcpy(buf, "ABCDXXXXCCCC");

    /* Make observable to prevent dead code elimination */
    puts(buf);
}

    Explanation¶

    the key insight here is that this is unoptimized code (-O0), and it demonstrates how the compiler handles a local buffer and function calls using the x86 calling convention with explicit stack management.

    Stack Frame Setup¶

    The function begins by establishing a stack frame — a dedicated region of memory for local variables:

    push    ebp                          ; Save old frame pointer
mov     ebp, esp                     ; Set up new frame pointer
sub     esp, 104                     ; Allocate 104 bytes for locals

    Think of the stack frame like a workspace on a table: ebp (base pointer) marks the start of your workspace, and esp (stack pointer) marks the current top. The sub esp, 104 instruction reserves 104 bytes — enough for the 100-byte buf array plus 4 bytes for security padding.

    Security Cookie (Stack Canary)¶

    The next section implements a stack canary — a security feature that detects buffer overflows:

    mov     eax, DWORD PTR ___security_cookie    ; Load a random value
xor     eax, ebp                             ; Mix it with frame pointer
mov     DWORD PTR __$ArrayPad$[ebp], eax    ; Store at end of locals

    This stores a special value at the end of the local variables. If a buffer overflow corrupts this value, the security check at the end of the function will detect it.

    The Core Operations: strcpy and puts¶

    The actual work happens here:

    push    OFFSET $SG10544              ; Push address of "ABCDXXXXCCCC" string
lea     eax, DWORD PTR _buf$[ebp]   ; Load address of buf into eax
push    eax                          ; Push buf's address
call    _strcpy                      ; Call strcpy(buf, "ABCDXXXXCCCC")
add     esp, 8                       ; Clean up 2 arguments from stack

    By convention in x86 calling, arguments are passed on the stack (pushed right-to-left). The lea instruction calculates the address of buf without accessing memory — it's just arithmetic to compute where buf lives relative to ebp.

    Then strcpy is called to copy the string into the buffer. After the call, add esp, 8 removes the two 4-byte arguments from the stack.

    The same pattern repeats for puts:

    lea     ecx, DWORD PTR _buf$[ebp]   ; Load buf's address into ecx
push    ecx                          ; Push it as argument
call    _puts                        ; Call puts(buf)
add     esp, 4                       ; Clean up 1 argument

    Security Check and Cleanup¶

    Before returning, the security cookie is verified:

    mov     ecx, DWORD PTR __$ArrayPad$[ebp]    ; Retrieve stored cookie
xor     ecx, ebp                            ; XOR with frame pointer again
call    @__security_check_cookie@4         ; Verify it matches original

    If the cookie was corrupted, this call terminates the program. Finally:

    mov     esp, ebp                    ; Restore stack pointer
pop     ebp                         ; Restore old frame pointer
ret     0                           ; Return to caller

    Key Observations¶

    • This is unoptimized code: The compiler doesn't inline strcpy or eliminate the stack frame, even though it could. An optimized version (-O2) would likely inline the string copy as direct memory writes.
    • Stack-based calling convention: Arguments are passed via the stack, not registers, which is typical for x86 (32-bit) calling conventions.
    • Security overhead: The stack canary adds 4 bytes of storage and two XOR operations, representing the security-first approach of this compiler.
    • Explicit stack cleanup: The add esp instructions manually remove arguments, which is necessary because the caller is responsible for cleanup in this calling convention.
