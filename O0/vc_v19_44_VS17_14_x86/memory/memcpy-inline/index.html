    memcpy-inline¶

    Details

    Source: memory/memcpy-inline.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: O0 - No Optimization (-O0)

    Source Code¶

    memcpy-inline.c
    /*
 * memcpy inlining: small copies become inline loads/stores.
 *
 * For small, known-size copies, the compiler generates
 * inline code instead of calling memcpy.
 */
#include <string.h>

struct small {
    int a, b;
};

void copy_8_bytes(void *dst, const void *src)
{
    /* Small copy - likely inlined as single 64-bit load/store
     * on 64-bit architectures, or two 32-bit ops on 32-bit.
     */
    memcpy(dst, src, 8);
}

void copy_struct(struct small *dst, const struct small *src)
{
    /* Structure copy - same as memcpy but type-aware.
     * Compiler knows alignment and size.
     */
    *dst = *src;
}

void copy_varying(void *dst, const void *src, int use_large)
{
    /* Compile-time unknown size - must call memcpy or
     * generate size-checking code.
     */
    if (use_large)
        memcpy(dst, src, 1024);
    else
        memcpy(dst, src, 8);
}

    Explanation¶

    the key insight is that this is unoptimized code (-O0), and the compiler makes very different choices for each function based on whether the copy size is known at compile time.

    Function 1: copy_8_bytes - Known Size, Still Calls memcpy¶

    push    ebp
mov     ebp, esp
push    8                    ; size argument
mov     eax, DWORD PTR _src$[ebp]
push    eax                  ; src argument
mov     ecx, DWORD PTR _dst$[ebp]
push    ecx                  ; dst argument
call    _memcpy
add     esp, 12
pop     ebp
ret     0

    Even though the compiler knows this copies exactly 8 bytes, it still calls the memcpy function rather than inlining the copy. This is typical unoptimized behaviorthe compiler prioritizes simplicity over performance. The function:

    1. Sets up a stack frame (push ebp, mov ebp, esp)
    2. Pushes three arguments onto the stack in reverse order (this is the cdecl calling convention, where arguments go on the stack right-to-left):
    3. Size: 8
    4. Source pointer: loaded into eax, then pushed
    5. Destination pointer: loaded into ecx, then pushed
    6. Calls memcpy
    7. Cleans up the stack by adding 12 to esp (removing 3 4-byte arguments)

    Why not inline? At -O0, the compiler doesn't perform the optimization of replacing small memcpy calls with inline load/store instructions. With -O2 or -O3, this would become a single 64-bit load and store.

    Function 2: copy_struct - Structure Assignment Gets Inlined¶

    mov     eax, DWORD PTR _src$[ebp]
mov     ecx, DWORD PTR [eax]        ; load first 4 bytes
mov     edx, DWORD PTR [eax+4]      ; load next 4 bytes
mov     eax, DWORD PTR _dst$[ebp]
mov     DWORD PTR [eax], ecx        ; store first 4 bytes
mov     DWORD PTR [eax+4], edx      ; store next 4 bytes

    This is the key difference: structure assignment (*dst = *src) is inlined as direct memory operations, not a memcpy call. The compiler:

    1. Loads the source pointer into eax
    2. Reads two 32-bit values from the source (at offsets 0 and 4 bytes)
    3. Loads the destination pointer into eax
    4. Writes those two values to the destination

    This works because the compiler knows the exact size and layout of the struct (8 bytes: two int fields). The struct assignment is treated as a language-level operation that the compiler can directly translate to memory operations, whereas memcpy is a library function call.

    Function 3: copy_varying - Runtime Size Requires Branching¶

    cmp     DWORD PTR _use_large$[ebp], 0
je      SHORT $LN2@copy_varyi       ; jump if equal (zero)

    Since the size is not known at compile time (it depends on the use_large parameter), the compiler must:

    1. Test the condition with cmp (compare)
    2. Branch with je (jump if equal) to one of two paths
