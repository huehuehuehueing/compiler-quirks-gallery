    stack-clash¶

    Details

    Source: hardening/stack-clash.c Compiler: armv8-clang2110 (ARM) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    stack-clash.c
    /* @gallery-hints
 *   extra-flags: -fstack-clash-protection
 *   compiler-exclude: vc_v19_44_VS17_14_x64, vc_v19_44_VS17_14_x86, avrg1520
 */

/*
 * Stack Clash Protection
 *
 * -fstack-clash-protection inserts probes into large stack
 * allocations so that every page of the stack is touched in order.
 * This prevents "stack clash" attacks where a large allocation
 * jumps the guard page and overlaps the heap.
 *
 * Look for: loop-based or sequential page-sized probes (usually
 * stores to rsp at 4096-byte intervals) when the function has a
 * large local frame.
 */

void large_frame(int n)
{
    volatile char buf[16384];  /* larger than one page */
    /* Touch a few spots so the compiler keeps the buffer. */
    buf[0] = 'A';
    buf[4096] = 'B';
    buf[8192] = 'C';
    buf[16383] = 'D';
    (void)n;
}

    Explanation¶

    the key insight here is that the compiler has completely eliminated the large 16KB buffer allocation despite the -fstack-clash-protection flag being enabled.

    What's happening¶

    The source code declares a 16,384-byte local array (volatile char buf[16384]), which is larger than a typical memory page (4096 bytes). Normally, -fstack-clash-protection would insert special "probe" instructions to safely allocate such large stack space. However, the compiler has optimized this away entirely.

    Here's why: The buffer is only used to store four individual characters at specific locations. The compiler recognized that it doesn't need to allocate the full 16KB array—it can just store those four values directly on the stack using a minimal 16-byte frame.

    Breaking down the assembly¶

    Stack setup: 

    sub     sp, sp, #16
    This subtracts 16 bytes from the stack pointer (sp), creating a small 16-byte working area. This is far smaller than the original 16KB request.

    Loading the character values: 

    mov     w8, #65      ; Load 'A' (ASCII 65) into w8
mov     w9, #66      ; Load 'B' (ASCII 66) into w9
mov     w8, #67      ; Load 'C' (ASCII 67) into w8
mov     w8, #68      ; Load 'D' (ASCII 68) into w8
    The compiler loads each character value (65='A', 66='B', 67='C', 68='D') into 32-bit registers. Note that w8 and w9 are the lower 32 bits of the 64-bit registers x8 and x9.

    Storing the values: 

    strb    w8, [sp, #12]    ; Store 'A' at sp+12
strb    w9, [sp, #8]     ; Store 'B' at sp+8
strb    w8, [sp, #4]     ; Store 'C' at sp+4
strb    w8, [sp], #16    ; Store 'D' at sp, then restore sp
    The strb instruction stores a single byte (the 'b' suffix means "byte"). The addresses are calculated relative to sp: - [sp, #12] means the address at sp + 12 bytes - [sp], #16 is a special form: it stores at the current sp, then adds 16 to sp afterward (restoring the original stack pointer)

    Return: 

    ret
    Returns to the caller.

    Why no stack clash protection?¶

    The -fstack-clash-protection flag is designed to prevent "stack clash" attacks by probing large allocations. However, since the compiler optimized away the large allocation entirely (replacing it with a tiny 16-byte frame), there's no large allocation to protect against. The protection mechanism simply isn't needed here.

    This is an example of dead code elimination and constant folding—the compiler recognized that the buffer's contents don't matter (it's only written to, never read), so it replaced the expensive allocation with minimal code that satisfies the volatile semantics.
