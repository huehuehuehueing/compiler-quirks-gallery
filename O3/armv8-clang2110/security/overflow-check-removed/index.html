    overflow-check-removed¶

    Details

    Source: security/overflow-check-removed.c Compiler: armv8-clang2110 (ARM) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    overflow-check-removed.c
    /*
 * Security issue: integer overflow check removed.
 *
 * Signed integer overflow is undefined behavior in C.
 * Compiler may assume it never happens and remove checks.
 */
#include <limits.h>

int safe_add(int a, int b)
{
    /* Attempt to check for overflow before it happens.
     * With optimization, this check may be REMOVED because
     * the compiler assumes signed overflow is impossible.
     */
    if (a > 0 && b > 0 && a > INT_MAX - b)
        return -1;  /* overflow would occur */

    if (a < 0 && b < 0 && a < INT_MIN - b)
        return -1;  /* underflow would occur */

    return a + b;
}

    Explanation¶

    the key insight is that the compiler has aggressively optimized this overflow-checking code in ways that may defeat the safety checks, which is a critical security issue.

    The Core Problem¶

    This code attempts to detect integer overflow before it happens, but with -O3 optimization, the compiler makes assumptions that can eliminate or transform these checks in unexpected ways. Let's trace through what's actually happening:

    Register and Calling Convention Setup¶

    In ARM64 (aarch64), the calling convention passes the first two integer arguments in w0 and w1 (the 32-bit versions of x0 and x1). The function receives: - w0 = parameter a - w1 = parameter b

    The return value goes back in w0.

    First Check: Positive Overflow (a > 0 && b > 0 && a > INT_MAX - b)¶

    cmp     w0, #1              // Compare a with 1
b.lt    .LBB0_4             // If a < 1, skip to .LBB0_4
cmp     w1, #1              // Compare b with 1
b.lt    .LBB0_4             // If b < 1, skip to .LBB0_4

    These instructions check if both a and b are positive (≥ 1). If either is less than 1, the code jumps to .LBB0_4 to handle the negative/underflow case.

    eor     w8, w1, #0x7fffffff // XOR b with 0x7fffffff (INT_MAX)
cmp     w0, w8              // Compare a with (b XOR INT_MAX)
b.ls    .LBB0_4             // If a <= result, skip overflow return

    This is where the compiler gets clever. Instead of computing INT_MAX - b directly, it uses eor (exclusive OR) with 0x7fffffff (which is INT_MAX). This is a bit manipulation trick, but the comparison logic here is transformed from the original source.

    mov     w0, #-1             // Return -1 (overflow detected)
ret

    If the overflow condition is true, return -1.

    Second Check: Negative Underflow (a < 0 && b < 0 && a < INT_MIN - b)¶

    .LBB0_4:
mov     w8, #-2147483648    // Load INT_MIN into w8
and     w9, w1, w0          // AND a and b together, store in w9
sub     w8, w8, w1          // Compute INT_MIN - b
cmp     w0, w8              // Compare a with (INT_MIN - b)

    The and instruction checks if both a and b have their sign bits set (both negative). The sub computes INT_MIN - b.

    add     w8, w1, w0          // Compute a + b (the actual sum)
ccmp    w9, #0, #0, lt      // Conditional compare: if a < INT_MIN - b, 
                            // compare w9 with 0
csinv   w0, w8, wzr, ge     // Conditional select: if condition is true (ge),
                            // return w8 (the sum); else return -1 (inverted 0)
ret

    The ccmp (conditional compare) and csinv (conditional select inverted) instructions combine multiple conditions efficiently. If the underflow condition is met, return -1; otherwise return the sum.

    The Critical Issue¶

    The compiler has transformed the overflow checks using bit manipulation and conditional instructions. While this is technically correct for the specific cases it handles, the compiler may have made assumptions about undefined behavior that could cause it to skip or misoptimize these checks entirely in certain scenarios. The use of -O3 means the compiler is permitted to assume signed integer overflow never occurs—which directly contradicts what this function is trying to prevent.
