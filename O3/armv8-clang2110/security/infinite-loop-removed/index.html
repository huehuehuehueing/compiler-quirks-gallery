    infinite-loop-removed¶

    Details

    Source: security/infinite-loop-removed.c Compiler: armv8-clang2110 (ARM) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    infinite-loop-removed.c
    /*
 * Infinite loop removed as undefined behavior.
 *
 * A loop with no side effects that doesn't terminate is UB.
 * Compiler may remove it entirely or assume it terminates.
 */

/* This function may not loop forever as expected! */
void spin_forever(void)
{
    /* No volatile, no side effects - UB if it never terminates.
     * Compiler may remove this entirely.
     */
    while (1)
        ;
}

/* Correct way to spin (e.g., for embedded busy-wait) */
void spin_forever_safe(void)
{
    /* volatile prevents optimization */
    while (1)
    {
        __asm__ volatile("" ::: "memory");
    }
}

    Explanation¶

    the key insight here is that both functions compile to identical infinite loops, but they represent very different levels of undefined behavior.

    The Assembly Code¶

    Both functions compile to the same pattern: 

    .LBB0_1:
    b       .LBB0_1

    This is a branch instruction (b) that jumps to its own label, creating an infinite loop. The label .LBB0_1 points to the b instruction itself, so the instruction unconditionally branches back to itself forever.

    Why They Look Identical (But Aren't)¶

    spin_forever() - The problematic version: - The C code has a while(1) loop with no side effects - According to C language rules, a loop that never terminates and does nothing is undefined behavior - The compiler is allowed to assume this code is unreachable or will terminate - In this case, the compiler chose to keep the infinite loop, but it could have removed it entirely or done something else

    spin_forever_safe() - The correct version: - Uses __asm__ volatile("") - an inline assembly statement with the volatile keyword - The volatile keyword tells the compiler "this has side effects I can't see, so don't optimize it away" - The empty assembly string with "memory" clobber tells the compiler "this instruction might affect memory in unpredictable ways" - This prevents the compiler from removing or optimizing the loop, making the infinite loop well-defined behavior

    Why Both Produce the Same Assembly¶

    With -O3 optimization enabled, the compiler: 1. For spin_forever(): Recognizes the infinite loop and generates the branch instruction (though it could have removed it) 2. For spin_forever_safe(): Must respect the volatile constraint and generate code that actually loops

    The assembly looks identical because both end up as infinite loops, but the safety guarantee is completely different. The first one relies on compiler behavior that isn't guaranteed by the C standard, while the second one is guaranteed to work as intended.
