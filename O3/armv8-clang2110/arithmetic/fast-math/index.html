    fast-math¶

    Details

    Source: arithmetic/fast-math.c Compiler: armv8-clang2110 (ARM) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    fast-math.c
    /*
 * Fast-math optimizations: -ffast-math / -Ofast effects.
 *
 * These flags allow the compiler to reorder floating-point
 * operations, assuming associativity (which floats don't have).
 * Can change results or break special value handling.
 */

/* With -ffast-math, this may be reordered and give different results */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++)
        sum += arr[i];
    return sum;
}

/* NaN check may be optimized away with -ffast-math */
int is_nan(float x)
{
    /* With -ffast-math, compiler assumes no NaN values exist,
     * so this comparison may be optimized to always return 0.
     */
    return x != x;
}

/* Division by variable may become multiplication by reciprocal */
float divide_by_y(float x, float y)
{
    /* With -ffast-math: may become x * (1.0f / y)
     * Less accurate but faster on some architectures.
     */
    return x / y;
}

/* Algebraic simplification (inexact with floats!) */
float algebraic(float a, float b, float c)
{
    /* With -ffast-math: (a + b) + c may be reordered to a + (b + c)
     * These can give different results with floating point!
     */
    return (a + b) + c;
}

    Explanation¶

    the key insight is that this code demonstrates loop vectorization and fast-math optimizations at -O3.

    sum_array - The Main Story: Vectorized Loop¶

    The most important pattern here is that the compiler has unrolled and vectorized the loop to process 8 floats at once using SIMD (Single Instruction Multiple Data) operations. Here's why this matters:

    Initial checks (lines 1-8): - cmp w1, #1 and b.lt .LBB0_3: If n < 1, skip to the end and return 0.0 - cmp w1, #8 and b.hs .LBB0_4: If n >= 8, jump to the vectorized path

    The vectorized loop (.LBB0_4 and .LBB0_5): - and x9, x8, #0x7ffffff8: Calculate how many complete groups of 8 elements to process (rounds down) - ldp q1, q2, [x10, #-16]: Load two 128-bit SIMD registers (q1 and q2), each holding 4 floats. This loads 8 floats total per iteration. - The next 12 lines extract individual floats from these SIMD registers using mov s3, v1.s[1] (extract element 1 from register v1) and add them one-by-one to s0 using fadd instructions - subs x11, x11, #8 and b.ne .LBB0_5: Loop while there are more groups of 8

    Why extract and add individually? The compiler loads 8 floats efficiently but then adds them sequentially to s0. This is a trade-off: SIMD loads are fast, but the actual additions must maintain order (floating-point addition isn't truly associative due to rounding).

    Cleanup loop (.LBB0_7 and .LBB0_8): - Handles remaining elements (0-7) that didn't fit into complete groups of 8 - ldr s1, [x10], #4: Load one float at a time - fadd s0, s0, s1: Add to running sum

    is_nan - NaN Detection Still Works¶

    fcmp    s0, s0      // Compare s0 with itself
cset    w0, vs      // Set w0 to 1 if "overflow/underflow" flag is set
ret

    Key insight: A NaN value has the special property that it's not equal to itself. When you compare NaN with NaN, the floating-point status flags set the "overflow/underflow" (vs) condition. The cset instruction sets w0 to 1 if this condition is true, effectively detecting NaN. Despite -ffast-math, this optimization is preserved because it's mathematically sound.

    divide_by_y - Direct Division¶

    fdiv    s0, s0, s1  // Divide s0 by s1
ret

    This is straightforward: s0 (first parameter, x) is divided by s1 (second parameter, y). Even with -ffast-math, the compiler chose not to convert this to multiplication by reciprocal, likely because the performance benefit is minimal on modern ARM processors.

    algebraic - Reordered Addition¶

    fadd    s0, s0, s1  // s0 = s0 + s1 (a + b)
fadd    s0, s0, s2  // s0 = s0 + s2 ((a + b) + c)
ret

    This computes (a + b) + c exactly as written in the source. With -ffast-math, the compiler is allowed to reorder floating-point operations, but it chose not to here. The result is the same as the source code order.
