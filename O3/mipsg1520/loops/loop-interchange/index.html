    loop-interchange¶

    Details

    Source: loops/loop-interchange.c Compiler: mipsg1520 (MIPS) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    loop-interchange.c
    /*
 * Loop Interchange
 *
 * When iterating over a 2D array, the order of the loop indices
 * determines memory access patterns.  In C (row-major layout),
 * iterating the inner loop over columns and the outer loop over
 * rows yields sequential access.  Swapping the loops ("interchange")
 * can improve cache locality when the original order would cause
 * stride-N accesses.
 *
 * Here the "bad" order (column-major traversal on a row-major array)
 * is presented; the compiler may interchange the loops at -O3.
 */
#define N 128

void column_sum(int mat[N][N], int result[N])
{
    /* Column-major traversal: inner loop strides by N ints.
     * The compiler may swap i and j for better locality. */
    for (int j = 0; j < N; j++)
    {
        result[j] = 0;
        for (int i = 0; i < N; i++)
        {
            result[j] += mat[i][j];
        }
    }
}

    Explanation¶

    the key insight is that this code implements a 2D array column sum using loop interchange optimization - the compiler has rearranged the loops to access memory more efficiently.

    The Core Pattern: Loop Interchange¶

    The original C code iterates column-by-column (outer loop over columns j, inner loop over rows i). This creates stride-N memory access - each element is 512 bytes apart (N=128 integers × 4 bytes). The compiler at -O3 has interchanged the loops to iterate row-by-row instead, creating sequential memory access that's much faster for CPU caches.

    Setup Phase (Lines 1-6)¶

    li      $2,65536                    # $2 = 0x10000 (65536)
addiu   $3,$2,512                   # $3 = 65536 + 512 = 66048
move    $6,$5                       # $6 = result array pointer
addu    $7,$4,$3                    # $7 = matrix + 66048 (end boundary)
addu    $5,$4,$2                    # $5 = matrix + 65536 (current column start)
li      $8,-65536                   # $8 = -65536 (for subtracting)

    These instructions set up loop boundaries and pointers: - $2 and $8 represent 65536 bytes (128 × 512), which is the size of one full matrix row in bytes - $7 marks the end of the matrix (matrix + 66048 bytes) - $5 tracks the current column's starting position - $6 points to the result array

    Outer Loop: Processing Each Column ($L3)¶

    $L3:
    sw      $0,0($6)                # result[j] = 0 (initialize)
    addu    $2,$5,$8                # $2 = current_column - 65536 (start of column)
    move    $3,$0                   # $3 = 0 (accumulator for sum)

    For each column, the code initializes the result to zero and sets up the inner loop starting position.

    Inner Loop: Summing Down a Column ($L2)¶

    $L2:
    lw      $4,0($2)                # Load one element from matrix
    addiu   $2,$2,512               # Move to next row (512 bytes = 128 ints × 4)
    addu    $3,$3,$4                # Add to accumulator
    bne     $2,$5,$L2               # Loop while not at column end

    This inner loop: 1. Loads an integer from the current position 2. Advances by 512 bytes (one full row width) 3. Accumulates the value into $3 4. Branches back if we haven't reached the end of the column

    The key optimization: by incrementing by 512 bytes each iteration, we're now accessing sequential rows within the same column, which is much more cache-friendly than the original stride-N pattern.

    Storing Result and Advancing to Next Column¶

        sw      $3,0($6)                # Store accumulated sum
    addiu   $5,$2,4                 # Move to next column
    bne     $5,$7,$L3               # Loop while columns remain
    addiu   $6,$6,4                 # Advance result pointer

    After the inner loop completes, the code stores the sum and moves to the next column by incrementing the starting pointer by 4 bytes (one integer).

    Return¶

        jr      $31                     # Jump to return address
    nop                             # Delay slot (no operation)

    The nop (no operation) fills the branch delay slot - a MIPS architectural feature where the instruction after a branch executes before the branch takes effect.
