    static-memcpy¶

    Details

    Source: string-literals/static-memcpy.c Compiler: mipsg1520 (MIPS) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    static-memcpy.c
    #include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[5];

    memcpy(buf, "AAAA", 4);
    buf[sizeof(buf) - 1] = '\0';

    /* make sure the above is not treated as dead code */
    puts(buf);
}

    Explanation¶

    the key insight is that the compiler has eliminated the memcpy call entirely and directly embedded the string data into the stack, then calls puts to print it.

    Stack Frame Setup¶

    The function starts by allocating 40 bytes of stack space: 

    addiu   $sp,$sp,-40

    This creates a local buffer area. In MIPS, $sp is the stack pointer register (by convention, it points to the top of the stack). The addiu instruction (add immediate unsigned) subtracts 40 from $sp, growing the stack downward (the standard direction on most architectures).

    Embedding the String Data¶

    Rather than calling memcpy to copy "AAAA" into the buffer, the compiler has optimized this away:

    li      $2,1094778880                 # 0x41410000
addiu   $2,$2,16705
sw      $2,24($sp)

    Here's what's happening: - li (load immediate) loads the value 0x41410000 into $2 (a temporary register). In hexadecimal, 0x41 is the ASCII code for 'A' - addiu adds 16705 (which is 0x4141 in hex) to $2, resulting in 0x41414141 — this is four 'A' characters packed into a single 32-bit word - sw (store word) writes this 4-byte value directly to the stack at offset 24 from $sp

    This is a constant folding optimization: instead of calling memcpy at runtime, the compiler computed the exact bytes needed and embedded them directly.

    Null Termination¶

    sb      $0,28($sp)

    The sb instruction (store byte) writes a zero byte ($0 is always zero in MIPS) to offset 28 on the stack. This is the null terminator for the string, placed at buf[4] (since the buffer starts at offset 24 and is 5 bytes long).

    Notice this instruction executes in the delay slot of the jal instruction (the instruction immediately before the branch). This is a MIPS quirk: the instruction after a branch executes before the branch takes effect.

    Function Call Setup and Return¶

    addiu   $4,$sp,24
jal     puts
sb      $0,28($sp)
    • addiu $4,$sp,24 calculates the address of the buffer (offset 24 from $sp) and stores it in $4. By MIPS calling convention, $4 is the first argument register, so this passes the buffer address to puts
    • jal puts (jump and link) calls the puts function, storing the return address in $31 (the link register)

    After puts returns: 

    lw      $31,36($sp)
jr      $31
addiu   $sp,$sp,40

    • lw (load word) restores the return address from the stack
    • jr (jump register) returns to the caller by jumping to the address in $31
    • addiu $sp,$sp,40 deallocates the stack frame (the delay slot instruction, executing before the return)

    Key Observations¶

    Optimization applied: The compiler recognized that memcpy with a constant source and length could be replaced with direct constant embedding. This eliminates a function call and memory copy operation entirely.

    Stack frame: Even though the buffer is only 5 bytes, 40 bytes are allocated. This includes space for the buffer (5 bytes), padding for alignment, and space to save the return address ($31).
