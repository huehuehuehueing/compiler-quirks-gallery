    fortify-source¶

    Details

    Source: hardening/fortify-source.c Compiler: clang1910 (x86/x64) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    fortify-source.c
    /* @gallery-hints
 *   extra-flags: -D_FORTIFY_SOURCE=2
 *   compiler-only: cg152, clang1910
 *   scenario-exclude: O0
 */

/*
 * FORTIFY_SOURCE - compile-time and runtime buffer overflow checks.
 *
 * When _FORTIFY_SOURCE is defined (requires optimization >= -O1),
 * glibc replaces common functions like memcpy, strcpy, sprintf with
 * checked variants (__memcpy_chk, etc.) that verify the destination
 * buffer size at compile time or runtime.
 *
 * Look for: calls to __memcpy_chk or __strcpy_chk instead of plain
 * memcpy / strcpy.  If the compiler can prove the copy is safe at
 * compile time, the check is elided entirely.
 */
#include <string.h>

void safe_copy(char *dst, unsigned long dstsize, const char *src)
{
    /* With FORTIFY_SOURCE, this becomes __memcpy_chk which will
     * abort at runtime if srclen > dstsize. */
    size_t srclen = strlen(src) + 1;
    if (srclen <= dstsize)
        memcpy(dst, src, srclen);
}

void known_size_copy(void)
{
    char buf[32];
    /* Compiler knows buf is 32 bytes, so it can check at compile time. */
    strcpy(buf, "hello");
    (void)buf;
}

    Explanation¶

    the key insight here is how FORTIFY_SOURCE enables compile-time and runtime buffer overflow protection, and how the compiler optimizes based on what it can verify.

    safe_copy Function¶

    The core pattern: This function performs a runtime safety check before copying data.

    Setup (saving registers): 

    push    r15
push    r14
push    rbx
    The function saves three registers because it needs to preserve the function parameters across the strlen call. In the amd64 calling convention, the first three parameters arrive in rdi, rsi, and rdx, but these are caller-saved (the function can modify them). The code saves them to callee-saved registers (r14, r15, rbx) so they survive the function call.

    Parameter preservation: 

    mov     rbx, rdx        # src pointer → rbx
mov     r15, rsi        # dstsize → r15
mov     r14, rdi        # dst pointer → r14

    Finding the source length: 

    mov     rdi, rdx        # Move src to rdi (first parameter for strlen)
call    strlen@PLT      # Call strlen; result goes to rax
inc     rax             # Add 1 for null terminator (srclen = strlen + 1)

    The safety check: 

    cmp     rax, r15        # Compare srclen with dstsize
jbe     .LBB0_2         # Jump if srclen <= dstsize (safe to copy)
    If the source is larger than the destination buffer, the function returns early without copying—preventing a buffer overflow.

    Conditional memcpy: 

    .LBB0_2:
mov     rdi, r14        # dst → rdi (1st parameter)
mov     rsi, rbx        # src → rsi (2nd parameter)
mov     rdx, rax        # srclen → rdx (3rd parameter)
pop     rbx
pop     r14
pop     r15
jmp     memcpy@PLT      # Tail-call to memcpy
    If the check passes, the function sets up parameters for memcpy and jumps to it (a tail-call optimization—no return to safe_copy, just directly to memcpy's return).

    known_size_copy Function¶

    ret

    This is the dramatic optimization: The entire function body vanishes. Here's why:

    The source code calls strcpy(buf, "hello") where: - buf is a 32-byte stack buffer (known at compile time) - "hello" is a 6-byte string literal (known at compile time)

    The compiler can prove at compile time that the copy is safe (6 bytes ≤ 32 bytes), so: 1. The FORTIFY_SOURCE check is completely eliminated 2. The strcpy call itself is optimized away (likely inlined or dead-code eliminated since the result isn't used) 3. Only an empty function remains

    This demonstrates the power of FORTIFY_SOURCE with optimization: when the compiler has enough information, security checks become zero-cost.
