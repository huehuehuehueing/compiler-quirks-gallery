    tail-call¶

    Details

    Source: control-flow/tail-call.c Compiler: clang1910 (x86/x64) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    tail-call.c
    /*
 * Tail call optimization: recursive calls become jumps.
 *
 * When a function's last action is calling another function
 * (or itself), the call can become a jump, saving stack space.
 */

/* Tail-recursive factorial - can be optimized to a loop */
int factorial_tail(int n, int acc)
{
    if (n <= 1)
        return acc;
    /* Last action is the recursive call - tail position */
    return factorial_tail(n - 1, n * acc);
}

int factorial(int n)
{
    return factorial_tail(n, 1);
}

/* NOT tail recursive - multiplication after recursive call */
int factorial_not_tail(int n)
{
    if (n <= 1)
        return 1;
    /* Multiplication happens AFTER the call returns */
    return n * factorial_not_tail(n - 1);
}

/* Mutual tail recursion */
int is_even(unsigned n);
int is_odd(unsigned n);

int is_even(unsigned n)
{
    if (n == 0) return 1;
    return is_odd(n - 1);  /* Tail call */
}

int is_odd(unsigned n)
{
    if (n == 0) return 0;
    return is_even(n - 1);  /* Tail call */
}

    Explanation¶

    the key insight is that tail call optimization has converted recursive function calls into loops, eliminating the stack overhead that would normally accumulate with recursion.

    Tail Call Optimization: Recursion Becomes Loops¶

    The most important pattern here is that functions ending with a recursive call (tail calls) have been transformed into jmp instructions instead of call instructions. This means the function doesn't create a new stack frameit reuses the current one and jumps back to the beginning with updated parameters.

    How This Works¶

    In a normal recursive call, you'd see: 

    call function_name    ; Creates new stack frame

    But in tail-optimized code, you see: 

    jmp .LBB0_5          ; Jump back, reusing current frame

    The difference is crucial: a call instruction pushes a return address onto the stack and creates a new stack frame. A jmp just transfers control without stack overhead. This means factorial_tail(1000, 1) won't overflow the stackit runs like a loop.

    factorial_tail and factorial Functions¶

    Both functions use the same core strategy. Let me trace through factorial_tail:

    Initial setup: 

    mov     eax, esi          ; eax = acc (second parameter)
cmp     edi, 2            ; Compare n with 2
jl      .LBB0_6           ; If n < 2, jump to return

    The function checks if n < 2 (the base case). If true, it returns immediately with eax containing the accumulator.

    The optimization split:

    For small values (n < 9), the code uses a simple scalar loop: 

    .LBB0_5:
    imul    eax, edi      ; eax = eax * n
    lea     ecx, [rdi - 1] ; ecx = n - 1
    cmp     edi, 2        ; Check if n >= 2
    mov     edi, ecx      ; n = n - 1
    ja      .LBB0_5       ; Jump back if n > 1

    This is the tail-call-optimized loop: multiply the accumulator by n, decrement n, and jump back. No function call overhead.

    For larger values (n 9), the compiler uses SIMD vectorization (SSE instructions like pmuludq, paddd, pshufd). This processes multiple factorial values in parallel using 128-bit XMM registers, computing 4 values simultaneously. The vectorized section handles the bulk of the work, then falls through to the scalar loop for the remainder.

    factorial_not_tail - Why It Can't Be Optimized¶

    This function looks identical in assembly to factorial_tail and factorial. This is because the compiler is smart enough to recognize that even though the source code shows multiplication after the recursive call, the actual computation can be rearranged. The compiler transforms it into an accumulator-based loop anywaya form of strength reduction optimization.

    is_even and is_odd - Mutual Tail Recursion¶

    These functions demonstrate mutual recursion (each calls the other), and both have been optimized to loops:

    is_even: 

    test    edi, edi          ; Check if n == 0
je      .LBB3_4           ; If yes, return 1 (even)
.LBB3_2:
    cmp     edi, 1        ; Check if n == 1
    je      .LBB3_5       ; If yes, return 0 (odd)
    add     edi, -2       ; n = n - 2
    test    edi, edi      ; Check if n == 0
    jne     .LBB3_2       ; Loop back if n != 0

    Instead of calling is_odd(n-1), the code just decrements by 2 and loops. This works because: - is_even(n) calls is_odd(n-1) - is_odd(n) calls is_even(n-1) - So is_even(n) ultimately calls is_even(n-2)
