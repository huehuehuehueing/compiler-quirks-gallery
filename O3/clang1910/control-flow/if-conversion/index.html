    if-conversion¶

    Details

    Source: control-flow/if-conversion.c Compiler: clang1910 (x86/x64) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    if-conversion.c
    /*
 * If-Conversion (branch to conditional move)
 *
 * Short if-else chains that select between two values can be
 * replaced with conditional-move instructions (cmov on x86).
 * This eliminates a branch, avoiding potential misprediction
 * penalties on modern super-scalar CPUs.
 *
 * Look for: cmovl / cmovge / cmovne instead of jl / jge / jne
 * with a branch over a mov.
 */

int min(int a, int b)
{
    if (a < b)
        return a;
    return b;
}

int clamp(int x, int lo, int hi)
{
    if (x < lo)
        x = lo;
    if (x > hi)
        x = hi;
    return x;
}

int abs_val(int x)
{
    return x < 0 ? -x : x;
}

    Explanation¶

    the key insight here is that the compiler has replaced traditional branching (jumps) with conditional move instructions (cmov), which is a powerful optimization technique.

    The Core Optimization: If-Conversion¶

    Instead of using branches that could cause the CPU to guess wrong and waste time, the compiler uses cmov instructions. Think of cmov as a "conditional assignment"—it only moves a value into a register if a specific condition is true. This eliminates branches entirely, which is faster on modern CPUs because the processor doesn't have to pause and guess which direction the code will go.

    Function-by-Function Breakdown¶

    min(int, int) — Finding the Smaller Value¶

    mov     eax, esi          // eax = b (second parameter)
cmp     edi, esi          // compare a (first param) with b
cmovl   eax, edi          // if a < b, then eax = a
ret

    What's happening: 1. mov eax, esi — Start by assuming the result is b. By convention, esi holds the second parameter and edi holds the first parameter in the x86-64 calling convention. 2. cmp edi, esi — Compare a with b. This sets internal CPU flags based on the comparison. 3. cmovl eax, edi — The cmovl instruction means "conditional move if less." If a < b (the "less" condition is true), copy a into eax. Otherwise, eax already contains b, so nothing happens. 4. ret — Return the value in eax.

    Why this is clever: No jumps, no branch misprediction penalties. The CPU executes all instructions in sequence.

    clamp(int, int, int) — Constraining a Value to a Range¶

    mov     eax, esi          // eax = lo (lower bound)
cmp     edi, esi          // compare x with lo
cmovg   eax, edi          // if x > lo, then eax = x
cmp     eax, edx          // compare result with hi (upper bound)
cmovge  eax, edx          // if result >= hi, then eax = hi
ret

    What's happening: 1. mov eax, esi — Start with lo as the default result. 2. cmp edi, esi — Compare x with lo. 3. cmovg eax, edi — If x > lo (greater), use x instead. Otherwise keep lo. 4. cmp eax, edx — Now compare the current result with hi. 5. cmovge eax, edx — If the result is >= hi, cap it at hi. 6. ret — Return the clamped value.

    The pattern: Two sequential conditional moves handle the two if-statements from the source code. Each cmov is independent and doesn't require branching.

    abs_val(int) — Absolute Value¶

    mov     eax, edi          // eax = x
neg     eax               // eax = -x (negate it)
cmovs   eax, edi          // if sign flag set, eax = x (original)
ret

    What's happening: 1. mov eax, edi — Copy the input x into eax. 2. neg eax — Negate it (compute -x). This also sets the CPU's sign flag if the result is negative. 3. cmovs eax, edi — The cmovs instruction means "conditional move if sign flag is set." If the negation produced a negative number (meaning the original x was positive), restore the original x. Otherwise, keep the negation. 4. ret — Return the result.

    Why this works: For a negative input, -x is positive, so the sign flag is clear and we keep -x. For a positive input, -x is negative, so the sign flag is set and we restore the original positive x.
