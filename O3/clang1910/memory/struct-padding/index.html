    struct-padding¶

    Details

    Source: memory/struct-padding.c Compiler: clang1910 (x86/x64) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    struct-padding.c
    /*
 * Structure layout and padding optimizations.
 *
 * Compilers add padding for alignment but optimize access patterns.
 * Field reordering may be done with certain flags.
 */
#include <stddef.h>

/* Poor layout - lots of padding */
struct bad_layout {
    char a;       /* 1 byte + 3 padding */
    int b;        /* 4 bytes */
    char c;       /* 1 byte + 3 padding */
    int d;        /* 4 bytes */
};  /* Total: 16 bytes, 6 bytes wasted */

/* Good layout - minimal padding */
struct good_layout {
    int b;        /* 4 bytes */
    int d;        /* 4 bytes */
    char a;       /* 1 byte */
    char c;       /* 1 byte + 2 padding */
};  /* Total: 12 bytes, 2 bytes wasted */

/* Returns sizes - evaluate at compile time */
int get_bad_size(void)
{
    return sizeof(struct bad_layout);
}

int get_good_size(void)
{
    return sizeof(struct good_layout);
}

/* Offsetof shows where padding is */
int get_bad_c_offset(void)
{
    return offsetof(struct bad_layout, c);
}

    Explanation¶

    Key Insight: Compile-Time Constant Folding¶

    The most important thing to understand here is that all three functions are completely optimized away. The compiler evaluated the sizeof() and offsetof() operations at compile time (not at runtime), so each function simply returns a hardcoded constant value. This is possible because structure sizes and field offsets are known when the code is compiled.

    How It Works¶

    Register eax and Return Values

    By calling convention, the eax register (the lower 32 bits of rax) is used to return integer values from functions. Each function loads a constant directly into eax and returns:

    • get_bad_size(): Loads 16 into eax — the bad_layout struct is 16 bytes total
    • get_good_size(): Loads 12 into eax — the good_layout struct is 12 bytes total
    • get_bad_c_offset(): Loads 8 into eax — field c starts at byte offset 8 in bad_layout

    The ret Instruction

    The ret instruction simply returns control to the caller. Since there's no stack frame setup (no push or sub rsp), these are leaf functions that don't need to save any state.

    Why This Happens¶

    With -O3 optimization enabled, the compiler performs constant folding — it evaluates compile-time constants during compilation rather than generating code to compute them at runtime. Since sizeof() and offsetof() are compile-time operations in C, the compiler:

    1. Calculates the structure sizes and offsets based on the struct definitions
    2. Replaces the function bodies with direct constant returns
    3. Eliminates all unnecessary computation

    This is the most aggressive form of optimization for these particular functions — there's literally nothing left to optimize because the answer is known before the program even runs.
