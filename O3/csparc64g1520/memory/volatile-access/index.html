    volatile-access¶

    Details

    Source: memory/volatile-access.c Compiler: csparc64g1520 (SPARC64) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    volatile-access.c
    /*
 * Volatile prevents optimization of memory accesses.
 *
 * Without volatile, repeated reads/writes may be cached in
 * registers or eliminated. Volatile forces actual memory access.
 */

/* Hardware register simulation */
volatile int *const HARDWARE_REG = (volatile int *)0x40000000;

void write_hardware(int value)
{
    /* Each write MUST go to memory - cannot be cached or reordered */
    *HARDWARE_REG = value;
    *HARDWARE_REG = value;  /* Not eliminated - both writes happen */
}

int read_hardware(void)
{
    /* Each read MUST come from memory - cannot use cached value */
    int a = *HARDWARE_REG;
    int b = *HARDWARE_REG;
    return a + b;  /* Two actual memory reads */
}

/* Compare: non-volatile version */
int *non_volatile_ptr;

int read_non_volatile(void)
{
    /* Optimizer may read once and reuse the value */
    int a = *non_volatile_ptr;
    int b = *non_volatile_ptr;  /* May be optimized to: b = a */
    return a + b;
}

    Explanation¶

    the key insight is how the volatile keyword forces the compiler to perform actual memory operations instead of optimizing them away.

    The Core Concept: Volatile vs. Non-Volatile Memory Access¶

    The most important pattern here is the difference between volatile and non-volatile pointers. When you mark a pointer as volatile, you're telling the compiler "I know this points to something that changes outside your control (like hardware), so you must perform every memory access exactly as written—no caching, no reordering, no skipping."

    write_hardware() - Two Writes to Hardware¶

    sethi   %hi(1073741824), %g1      # Load upper bits of address 0x40000000 into %g1
st      %o0, [%g1]                # Store parameter (in %o0) to memory address in %g1
st      %o0, [%g1]                # Store the SAME value again to the SAME address
jmp     %o7+8                      # Return to caller
 nop                               # Delay slot (no operation)

    What's happening: The compiler loads the hardware register address (0x40000000) once, then performs two separate store instructions (st). Even though both writes are identical, the compiler cannot eliminate the second one because volatile forbids it. This is intentional—the hardware might require two writes to trigger an action, or the writes might have side effects.

    The sethi instruction loads the upper 22 bits of the address; since the lower bits are zero, no additional instruction is needed to complete the address.

    read_hardware() - Two Reads from Hardware¶

    sethi   %hi(1073741824), %g1      # Load upper bits of address 0x40000000 into %g1
lduw    [%g1], %o0                # Load unsigned word from hardware into %o0 (first read)
lduw    [%g1], %g1                # Load unsigned word from hardware into %g1 (second read)
add     %o0, %g1, %o0             # Add the two values: %o0 = %o0 + %g1
jmp     %o7+8                      # Return to caller
 sra    %o0, 0, %o0               # Sign-extend result (delay slot)

    What's happening: Two separate lduw (load unsigned word) instructions read from the same hardware address. The compiler cannot optimize this to a single read because volatile requires both accesses. The two values are then added together and returned.

    The sra %o0, 0, %o0 (shift right arithmetic by 0) is a sign-extension operation—it ensures the 32-bit result is properly extended to 64-bit format for the return value.

    read_non_volatile() - Optimization in Action¶

    sethi   %hi(non_volatile_ptr), %g1           # Load upper bits of pointer address
ldx     [%g1+%lo(non_volatile_ptr)], %g1    # Load the pointer value itself from memory
lduw    [%g1], %o0                           # Load value from the pointer (ONE read)
add     %o0, %o0, %o0                        # Add value to itself: %o0 = %o0 + %o0
jmp     %o7+8                                 # Return to caller
 sra    %o0, 0, %o0                          # Sign-extend result

    The key difference: Notice there's only one lduw instruction here, not two. The compiler optimized away the second read because non_volatile_ptr is not marked volatile. The compiler reasoned: "Since this pointer isn't volatile, the value at that address can't change between reads, so I can read once and reuse the value."

    The add %o0, %o0, %o0 instruction adds the value to itself (equivalent to multiplying by 2), which is faster than performing a second memory read.

    Why This Matters¶

    • Volatile hardware registers (like HARDWARE_REG): The compiler respects your intent and performs every access exactly as written, even if it seems redundant. This is essential for device drivers and embedded systems.
