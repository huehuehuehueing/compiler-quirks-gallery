    overflow-check-removed¶

    Details

    Source: security/overflow-check-removed.c Compiler: csparc64g1520 (SPARC64) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    overflow-check-removed.c
    /*
 * Security issue: integer overflow check removed.
 *
 * Signed integer overflow is undefined behavior in C.
 * Compiler may assume it never happens and remove checks.
 */
#include <limits.h>

int safe_add(int a, int b)
{
    /* Attempt to check for overflow before it happens.
     * With optimization, this check may be REMOVED because
     * the compiler assumes signed overflow is impossible.
     */
    if (a > 0 && b > 0 && a > INT_MAX - b)
        return -1;  /* overflow would occur */

    if (a < 0 && b < 0 && a < INT_MIN - b)
        return -1;  /* underflow would occur */

    return a + b;
}

    Explanation¶

    the key insight is that the compiler has kept the overflow checks intact despite using -O3 optimization, which is somewhat surprising given the comment about undefined behavior.

    Key Pattern: Conditional Overflow Detection¶

    The code uses a clever technique to detect overflow before it happens. Instead of checking a > INT_MAX - b directly (which could itself overflow), it rearranges the math: it computes INT_MAX - b first, then checks if a is greater than that result.

    Register Setup and Sign Detection¶

    mov     0, %g2
mov     0, %g1
movrgz  %o0, 1, %g2
movrgz  %o1, 1, %g1

    The code starts by initializing two temporary registers (%g2 and %g1) to 0. Then it uses movrgz (move if register greater than or equal to zero) to set them to 1 if the input parameters are non-negative. This is a way to detect the sign of each input: - %g2 becomes 1 if %o0 (parameter a) is ≥ 0 - %g1 becomes 1 if %o1 (parameter b) is ≥ 0

    By convention in SPARC, %o0 and %o1 hold the first two integer parameters passed to the function.

    Positive Overflow Check Path¶

    andcc   %g2, %g1, %g0
be,a,pt %icc, .L2
 andcc  %o0, %o1, %g0

    The andcc instruction performs a bitwise AND and sets condition codes. If both %g2 and %g1 are 1 (meaning both inputs are positive), the result is non-zero. The be,a,pt (branch if equal, annul, predict taken) jumps to .L2 if the result was zero—meaning at least one input is negative, so we skip the positive overflow check.

    If we don't branch, we continue to the positive overflow detection:

    sethi   %hi(2147482624), %g1
or      %g1, 1023, %g1
sub     %g1, %o1, %g1
cmp     %g1, %o0
bl,a,pn %icc, .L8
 mov    -1, %o0

    This constructs INT_MAX (2,147,483,647) in %g1 using sethi (set high 22 bits) and or (set low 10 bits). Then it computes INT_MAX - b and compares it with a. If a is greater (branch less), it jumps to .L8 and sets the return value to -1 (overflow detected).

    Negative Underflow Check Path¶

    .L2:
bge,pn  %icc, .L4
 sethi  %hi(-2147483648), %g1
sub     %g1, %o1, %g1
cmp     %g1, %o0
ble,a,pt %icc, .L8
 add    %o0, %o1, %o0

    At .L2, the code checks if both inputs are negative using bge (branch if greater or equal). If either is non-negative, it branches to .L4 to skip the underflow check. Otherwise, it constructs INT_MIN (-2,147,483,648) and performs a similar check: computing INT_MIN - b and comparing with a. If a is less than or equal to that value, underflow would occur, so it returns -1.

    Normal Addition Path¶

    .L4:
add     %o0, %o1, %o0

    If no overflow or underflow was detected, the function simply adds the two inputs and returns the result in %o0 (the return value register by convention).
