    static-memcpy¶

    Details

    Source: string-literals/static-memcpy.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    static-memcpy.c
    #include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[5];

    memcpy(buf, "AAAA", 4);
    buf[sizeof(buf) - 1] = '\0';

    /* make sure the above is not treated as dead code */
    puts(buf);
}

    Explanation¶

    the key insight here is that this code implements stack buffer overflow protection, which adds significant overhead to what would otherwise be a simple function.

    Stack Frame Setup and Security Cookie¶

    The function starts by establishing a stack frame and immediately sets up a security cookie — a random value used to detect buffer overflows:

    push    ebp
mov     ebp, esp
sub     esp, 16                    ; allocate 16 bytes on stack
mov     eax, DWORD PTR ___security_cookie
xor     eax, ebp
mov     DWORD PTR __$ArrayPad$[ebp], eax

    The compiler allocates 16 bytes of stack space for a 5-byte buffer (buf), plus padding and the security cookie. The cookie is XORed with the base pointer (ebp) and stored on the stack. This value will be checked before the function returns — if a buffer overflow corrupted it, the program will crash intentionally rather than continue with corrupted memory.

    The memcpy Call¶

    push    4
push    OFFSET $SG10545
lea     eax, DWORD PTR _buf$[ebp]
push    eax
call    _memcpy
add     esp, 12

    This uses the cdecl calling convention (common on x86), where arguments are pushed onto the stack in reverse order. The three arguments to memcpy are: 1. Destination: address of buf (calculated with lea — Load Effective Address) 2. Source: address of the string "AAAA" 3. Size: 4 bytes

    After the call, add esp, 12 cleans up the three 4-byte arguments from the stack.

    Range Check Before Array Access¶

    Here's where the security overhead becomes visible:

    mov     ecx, 1
shl     ecx, 2                     ; ecx = 1 << 2 = 4
mov     DWORD PTR $T1[ebp], ecx
cmp     DWORD PTR $T1[ebp], 5
jae     SHORT $LN3@foo
jmp     SHORT $LN4@foo
$LN3@foo:
call    ___report_rangecheckfailure

    The compiler calculates sizeof(buf) - 1 (which is 4) in a roundabout way: it loads 1 into ecx, shifts left by 2 bits (multiplying by 4), then stores it. It then compares this value against 5 (the buffer size) and calls a range-check failure handler if the index is out of bounds.

    This is defensive programming: the compiler is verifying that the array access buf[sizeof(buf) - 1] won't overflow before it happens.

    Null Terminator and Function Cleanup¶

    mov     edx, DWORD PTR $T1[ebp]
mov     BYTE PTR _buf$[ebp+edx], 0
lea     eax, DWORD PTR _buf$[ebp]
push    eax
call    _puts

    The verified index is loaded into edx, then a null byte is written to buf[4]. The buffer address is passed to puts.

    Finally, the security cookie is verified before returning:

    mov     ecx, DWORD PTR __$ArrayPad$[ebp]
xor     ecx, ebp
call    @__security_check_cookie@4
mov     esp, ebp
pop     ebp
ret     0

    The stored cookie is XORed with ebp again — if nothing corrupted the stack, this should produce the original ___security_cookie value. The @__security_check_cookie@4 function verifies this.

    Key Observations¶

    • Security overhead dominates: The range check, security cookie setup/verification, and extra stack allocation add substantial code compared to the actual work (copying 4 bytes and writing a null terminator).
    • Defensive compilation: Even though sizeof(buf) - 1 is a compile-time constant (4), the compiler still generates runtime checks. This is intentional — it catches programming errors at runtime.
    • Stack-based protection: The security cookie technique (also called "stack canary") is a common defense against buffer overflow exploits.
