    msvc-guard-cf¶

    Details

    Source: hardening/msvc-guard-cf.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    msvc-guard-cf.c
    /* @gallery-hints
 *   replace-flags: /O2 /guard:cf
 *   compiler-only: vc_v19_44_VS17_14_x64, vc_v19_44_VS17_14_x86
 */

/*
 * MSVC Control Flow Guard (/guard:cf)
 *
 * Control Flow Guard (CFG) is a Windows platform security feature
 * that validates every indirect call at runtime.  The compiler
 * replaces indirect CALL instructions with a dispatch through
 * __guard_dispatch_icall_fptr, which checks the target address
 * against a bitmap of valid call targets before transferring
 * control.
 *
 * Compare with cf-protection.c (GCC/Clang):
 *   - GCC/Clang use Intel CET hardware: ENDBR64 marks valid
 *     landing pads, and the CPU faults on violations.
 *   - MSVC CFG is a software bitmap check performed in user-mode
 *     before every indirect call; it works on all x86/x64 CPUs.
 *
 * Look for: rex_jmp __guard_dispatch_icall_fptr (or
 * __guard_check_icall_fptr) where a plain indirect call would
 * normally appear.
 */

typedef int (*op_fn)(int, int);

int add(int a, int b) { return a + b; }
int sub(int a, int b) { return a - b; }

int apply(op_fn f, int x, int y)
{
    return f(x, y);
}

int demo(void)
{
    return apply(add, 10, 3) + apply(sub, 10, 3);
}

    Explanation¶

    the key insight here is how Control Flow Guard (CFG) security protection changes indirect function calls.

    The Core Pattern: CFG Security Check¶

    The most important thing to understand is that when you call a function pointer (an indirect call), the /guard:cf flag forces the compiler to insert a security check before the actual call. Instead of directly jumping to the function pointer, the code calls __guard_check_icall_fptr, which validates that the target address is safe before execution continues.

    Walking Through the Code¶

    The add and sub functions are straightforward: - mov eax, DWORD PTR _a$[esp-4] — Load the first parameter into eax (the return value register by convention) - add eax, DWORD PTR _b$[esp-4] — Add the second parameter to it - ret 0 — Return

    These use the x86 calling convention where parameters are passed on the stack and the return value goes in eax.

    The apply function is where CFG becomes visible:

    push    DWORD PTR _y$[esp-4]      ; Push second argument (y)
mov     ecx, DWORD PTR _f$[esp]   ; Load function pointer into ecx
push    DWORD PTR _x$[esp]        ; Push first argument (x)
call    DWORD PTR ___guard_check_icall_fptr  ; Security check
call    DWORD PTR _f$[esp+4]      ; Actual indirect call
add     esp, 8                    ; Clean up stack
ret     0

    The critical lines are the two call instructions: 1. First call invokes the CFG security function with ecx containing the function pointer. This function checks if the address in ecx is in a whitelist of valid call targets. 2. Second call performs the actual indirect call to the function pointer.

    This two-step process is the security overhead: every indirect call is validated before execution.

    The demo function shows aggressive optimization:

    mov     eax, 20                   ; Load 20 directly
ret     0

    The compiler completely eliminated the function calls! It computed apply(add, 10, 3) + apply(sub, 10, 3) at compile time: (10+3) + (10-3) = 13 + 7 = 20. This is constant folding — a compiler optimization that evaluates constant expressions before runtime. With /O2 optimization enabled, the compiler inlined the function calls and computed the result statically.

    Key Observations¶

    • CFG overhead: The apply function shows the cost of /guard:cf — an extra function call (__guard_check_icall_fptr) before every indirect call. On systems without CFG support, this check would fail; on Windows, it validates against a bitmap of safe addresses.
    • Optimization wins: The demo function demonstrates that even with CFG enabled, the compiler still applies aggressive optimizations like inlining and constant folding when it can prove the calls are safe.
    • x86 calling convention: Parameters are passed on the stack (pushed before the call), and the return value is in eax.
