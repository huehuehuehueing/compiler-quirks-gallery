    restrict-pointer¶

    Details

    Source: simd/restrict-pointer.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    restrict-pointer.c
    /*
 * Restrict pointers enable better optimization.
 *
 * The 'restrict' keyword promises that pointers don't alias,
 * allowing the compiler to optimize more aggressively.
 */

/* Without restrict - compiler must be conservative */
void copy_no_restrict(int *dst, const int *src, int n)
{
    /* dst and src MIGHT overlap, so compiler must handle that */
    for (int i = 0; i < n; i++) {
        dst[i] = src[i];
    }
}

/* With restrict - compiler knows no aliasing */
void copy_restrict(int *restrict dst, const int *restrict src, int n)
{
    /* dst and src guaranteed not to overlap
     * Compiler can use faster copy or vectorize freely
     */
    for (int i = 0; i < n; i++) {
        dst[i] = src[i];
    }
}

/* Multiplication example - dramatic difference */
void scale_no_restrict(float *result, const float *input,
                       const float *scale, int n)
{
    /* Compiler doesn't know if result aliases input or scale */
    for (int i = 0; i < n; i++) {
        result[i] = input[i] * (*scale);
    }
    /* Must reload *scale each iteration! */
}

void scale_restrict(float *restrict result,
                    const float *restrict input,
                    const float *restrict scale, int n)
{
    /* Compiler knows result doesn't alias scale */
    for (int i = 0; i < n; i++) {
        result[i] = input[i] * (*scale);
    }
    /* Can load *scale once and reuse */
}

    Explanation¶

    the key insight is that the restrict keyword makes no difference in this MSVC output, which is surprising given the source code's claims. Let me walk through what's happening.

    The Surprising Finding: Identical Code¶

    Both copy_no_restrict and copy_restrict generate identical assembly. Similarly, scale_no_restrict and scale_restrict produce identical code. This means the compiler isn't taking advantage of the restrict promise at all in this case.

    Understanding the Function Structure¶

    All four functions follow the same pattern. Let me explain using copy_no_restrict as an example:

    Stack Frame Setup: 

    push    ebp              ; Save old base pointer
mov     ebp, esp         ; Set up new stack frame
push    ecx              ; Make room for local variable (i)
push    esi              ; Save esi (we'll use it in the loop)

    This creates a stack frame where local variable i is stored at [ebp-4]. The parameters are accessed via positive offsets: _dst$ at [ebp+8], _src$ at [ebp+12], _n$ at [ebp+16].

    The Loop Structure¶

    The loop uses a classic pattern with a forward jump:

    mov     DWORD PTR _i$1[ebp], 0    ; i = 0
jmp     SHORT $LN4@copy_no_re     ; Jump to loop condition

    This forward jump skips the increment code on the first iteration and goes straight to the condition checka common optimization pattern.

    Loop Condition Check: 

    mov     ecx, DWORD PTR _i$1[ebp]  ; Load i into ecx
cmp     ecx, DWORD PTR _n$[ebp]   ; Compare i with n
jge     SHORT $LN3@copy_no_re     ; Exit if i >= n

    Loop Body (the actual copy): 

    mov     edx, DWORD PTR _i$1[ebp]  ; Load i into edx
mov     eax, DWORD PTR _dst$[ebp] ; Load dst pointer
mov     ecx, DWORD PTR _i$1[ebp]  ; Load i again into ecx
mov     esi, DWORD PTR _src$[ebp] ; Load src pointer
mov     ecx, DWORD PTR [esi+ecx*4] ; Load src[i] (ecx = src[i])
mov     DWORD PTR [eax+edx*4], ecx ; Store to dst[i]

    Notice the address calculation: [esi+ecx*4] means "address of esi plus (ecx times 4)"this is how array indexing works in assembly. Since integers are 4 bytes, multiplying the index by 4 gives the byte offset.

    Loop Increment: 

    jmp     SHORT $LN2@copy_no_re     ; Jump back to increment
mov     eax, DWORD PTR _i$1[ebp]  ; Load i
add     eax, 1                     ; Increment i
mov     DWORD PTR _i$1[ebp], eax   ; Store i back

    The Floating-Point Functions¶

    The scale_no_restrict and scale_restrict functions are nearly identical to the copy functions, but they use SSE floating-point instructions instead of integer operations:

    movss   xmm0, DWORD PTR [eax+edx*4]  ; Load single-precision float from input[i] into xmm0
mulss   xmm0, DWORD PTR [ecx]        ; Multiply by *scale (the dereferenced pointer)
movss   DWORD PTR [eax+edx*4], xmm0  ; Store result back

    movss (move scalar single-precision) and mulss (multiply scalar single-precision) are SSE instructions that work with 32-bit floating-point values. The xmm0 register is a 128-bit SIMD register, but here it's only using the lower 32 bits for a single float.
