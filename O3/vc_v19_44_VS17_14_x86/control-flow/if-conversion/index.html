    if-conversion¶

    Details

    Source: control-flow/if-conversion.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    if-conversion.c
    /*
 * If-Conversion (branch to conditional move)
 *
 * Short if-else chains that select between two values can be
 * replaced with conditional-move instructions (cmov on x86).
 * This eliminates a branch, avoiding potential misprediction
 * penalties on modern super-scalar CPUs.
 *
 * Look for: cmovl / cmovge / cmovne instead of jl / jge / jne
 * with a branch over a mov.
 */

int min(int a, int b)
{
    if (a < b)
        return a;
    return b;
}

int clamp(int x, int lo, int hi)
{
    if (x < lo)
        x = lo;
    if (x > hi)
        x = hi;
    return x;
}

int abs_val(int x)
{
    return x < 0 ? -x : x;
}

    Explanation¶

    the key insight is that despite compiling with -O3 (maximum optimization), this code uses traditional branching instead of the conditional-move instructions mentioned in the comments. Let me walk through what's happening.

    Overall Pattern: Branch-Based Selection¶

    The compiler is using jmp (jump) and conditional branches (jge, jle) to select between values, rather than cmov (conditional move) instructions. This is surprising given the -O3 flag, but let's see why and what each function does.

    min Function: Simple Comparison with Branch¶

    mov     eax, DWORD PTR _a$[ebp]      ; Load parameter 'a' into eax
cmp     eax, DWORD PTR _b$[ebp]      ; Compare a with b
jge     SHORT $LN2@min               ; If a >= b, jump to $LN2@min
mov     eax, DWORD PTR _a$[ebp]      ; Otherwise, load a into eax
jmp     SHORT $LN1@min               ; Jump to end
$LN2@min:
mov     eax, DWORD PTR _b$[ebp]      ; Load b into eax
$LN1@min:
pop     ebp
ret     0

    What's happening: 1. The function loads a into eax (the return value register by convention) 2. It compares a with b using cmp 3. If a >= b, it jumps to $LN2@min and loads b instead 4. Otherwise, it falls through and keeps a in eax 5. Both paths converge at $LN1@min and return

    This is a branch-based approach: the CPU must predict which path to take, and a misprediction causes a pipeline stall.

    clamp Function: Two Sequential Checks¶

    mov     eax, DWORD PTR _x$[ebp]      ; Load x
cmp     eax, DWORD PTR _lo$[ebp]     ; Compare x with lo
jge     SHORT $LN2@clamp             ; If x >= lo, skip the first adjustment
mov     ecx, DWORD PTR _lo$[ebp]     ; Load lo
mov     DWORD PTR _x$[ebp], ecx      ; Store lo back to x (x = lo)
$LN2@clamp:
mov     edx, DWORD PTR _x$[ebp]      ; Load x again
cmp     edx, DWORD PTR _hi$[ebp]     ; Compare x with hi
jle     SHORT $LN3@clamp             ; If x <= hi, skip the second adjustment
mov     eax, DWORD PTR _hi$[ebp]     ; Load hi
mov     DWORD PTR _x$[ebp], eax      ; Store hi back to x (x = hi)
$LN3@clamp:
mov     eax, DWORD PTR _x$[ebp]      ; Load final x into eax for return
pop     ebp
ret     0

    What's happening: 1. First check: if x < lo, it stores lo back to the stack location of x 2. Second check: it reloads x from the stack, then if x > hi, it stores hi back to x 3. Finally, it loads the result from the stack into eax and returns

    Notice the store-then-load pattern: the function writes to the stack and immediately reads it back. This is inefficientan optimized version would keep values in registers.
