    loop-interchange¶

    Details

    Source: loops/loop-interchange.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    loop-interchange.c
    /*
 * Loop Interchange
 *
 * When iterating over a 2D array, the order of the loop indices
 * determines memory access patterns.  In C (row-major layout),
 * iterating the inner loop over columns and the outer loop over
 * rows yields sequential access.  Swapping the loops ("interchange")
 * can improve cache locality when the original order would cause
 * stride-N accesses.
 *
 * Here the "bad" order (column-major traversal on a row-major array)
 * is presented; the compiler may interchange the loops at -O3.
 */
#define N 128

void column_sum(int mat[N][N], int result[N])
{
    /* Column-major traversal: inner loop strides by N ints.
     * The compiler may swap i and j for better locality. */
    for (int j = 0; j < N; j++)
    {
        result[j] = 0;
        for (int i = 0; i < N; i++)
        {
            result[j] += mat[i][j];
        }
    }
}

    Explanation¶

    the key insight is that the compiler did NOT interchange the loops, despite the -O3 optimization flag. The assembly still performs column-major traversal (the inefficient access pattern), which means it's accessing memory with large strides.

    Stack Frame Setup¶

    The function starts by saving the two parameters and allocating stack space:

    mov     QWORD PTR [rsp+16], rdx    ; Save result pointer (2nd parameter)
mov     QWORD PTR [rsp+8], rcx     ; Save mat pointer (1st parameter)
sub     rsp, 24                     ; Allocate 24 bytes for local variables

    By convention in the x64 calling convention, rcx holds the first parameter (mat) and rdx holds the second parameter (result). The compiler saves these to the stack because it needs the registers for other work. The 24-byte allocation provides space for the loop variables j and i.

    Outer Loop Structure (Column Iteration)¶

    The outer loop iterates through columns (j = 0 to 127):

    mov     DWORD PTR j$1[rsp], 0      ; Initialize j = 0
jmp     SHORT $LN4@column_sum      ; Jump to loop condition

    The loop condition checks if j < 128:

    $LN4@column_sum:
cmp     DWORD PTR j$1[rsp], 128
jge     SHORT $LN3@column_sum      ; Exit if j >= 128

    At the end of each outer loop iteration, j is incremented:

    $LN2@column_sum:
mov     eax, DWORD PTR j$1[rsp]
inc     eax
mov     DWORD PTR j$1[rsp], eax

    Inner Loop: The Problematic Access Pattern¶

    Inside the outer loop, the code initializes result[j] = 0:

    movsxd  rax, DWORD PTR j$1[rsp]    ; Convert j to 64-bit
mov     rcx, QWORD PTR result$[rsp] ; Load result pointer
mov     DWORD PTR [rcx+rax*4], 0   ; result[j] = 0

    Then the inner loop iterates through rows (i = 0 to 127), accumulating result[j] += mat[i][j]:

    mov     DWORD PTR i$2[rsp], 0      ; Initialize i = 0
jmp     SHORT $LN7@column_sum      ; Jump to loop condition

    The critical part is how it accesses mat[i][j]. Since the matrix is stored in row-major order (rows are contiguous in memory), accessing mat[i][j] with i changing in the inner loop means jumping by 512 bytes each iteration (128 columns 4 bytes per int):

    movsxd  rcx, DWORD PTR i$2[rsp]    ; Convert i to 64-bit
imul    rcx, rcx, 512              ; Multiply by 512 (row stride)
mov     rdx, QWORD PTR mat$[rsp]   ; Load mat pointer
add     rdx, rcx                    ; Point to mat[i][0]
movsxd  rdx, DWORD PTR j$1[rsp]    ; Convert j to 64-bit
mov     eax, DWORD PTR [rcx+rdx*4] ; Load mat[i][j]

    This creates a stride-512 access pattern, which is terrible for CPU cache performance. Modern CPUs load data in cache lines (typically 64 bytes), so jumping 512 bytes means missing the cache almost every time.

    The Accumulation¶

    The loaded value is added to the current result:

    mov     eax, DWORD PTR [r8+rax*4]  ; Load result[j]
add     eax, DWORD PTR [rcx+rdx*4] ; Add mat[i][j]
mov     DWORD PTR [rdx+rcx*4], eax ; Store back to result[j]
