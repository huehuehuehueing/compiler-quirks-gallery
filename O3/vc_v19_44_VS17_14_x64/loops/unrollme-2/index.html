    unrollme-2¶

    Details

    Source: loops/unrollme-2.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    unrollme-2.c
    #include <stdio.h>

void foo(void)
{
    int i;
    char buf[8];

    for (i = 0; i < sizeof(buf); i++)
    {
        buf[i] = 'A';
    }

    /*
     * by dereferencing buf this should force the compile to NOT
     * eliminate the above as dead code. loop will likely still
     * be unrolled
     */
    printf("%s", buf);
}

    Explanation¶

    the key insight is that the compiler did NOT unroll the loop despite -O3 optimization, which is surprising given the comment in the source code. Let me walk through why and what's happening.

    The Loop Structure (The Main Event)¶

    The core of this function is a simple loop that fills an 8-byte buffer with the character 'A' (ASCII value 65 or 0x41). Here's how it works:

    Loop initialization: 

    mov     DWORD PTR i$[rsp], 0        ; i = 0
jmp     SHORT $LN4@foo              ; jump to loop condition

    The compiler initializes i to 0 on the stack and jumps directly to the condition check (this is a common pattern called a "do-while" style loop structure).

    Loop body (at $LN2@foo): 

    mov     eax, DWORD PTR i$[rsp]      ; load i into eax
inc     eax                          ; increment it
mov     DWORD PTR i$[rsp], eax      ; store it back

    Then it jumps back to the condition. This is inefficientit loads i, increments it, and stores it back to memory each iteration. A more optimized approach would keep i in a register.

    Loop condition (at $LN4@foo): 

    movsxd  rax, DWORD PTR i$[rsp]      ; load i and sign-extend to 64-bit
cmp     rax, 8                       ; compare with 8
jae     SHORT $LN3@foo              ; jump out if i >= 8

    The movsxd instruction sign-extends the 32-bit i to a 64-bit value in rax (necessary because array indexing uses 64-bit addresses). Then it compares with 8 and exits if the condition is false.

    Inside the loop body (the actual work): 

    movsxd  rax, DWORD PTR i$[rsp]      ; load i again and sign-extend
mov     BYTE PTR buf$[rsp+rax], 65  ; buf[i] = 'A' (65 is ASCII 'A')
jmp     SHORT $LN2@foo              ; loop back

    This stores the byte value 65 ('A') at position buf[i].

    Why Wasn't the Loop Unrolled?¶

    Despite -O3 optimization, the compiler kept this as an actual loop rather than unrolling it into 8 separate assignments. This is likely because:

    1. Stack buffer with security cookie: The compiler inserted a stack canary (security cookie) to detect buffer overflows. This adds complexity and may discourage aggressive loop unrolling.
    2. Small loop body: The loop is so simple that unrolling might not provide significant benefit relative to code size increase.

    Stack Frame Setup (Security Features)¶

    Notice the security-related code at the function start and end:

    sub     rsp, 72                     ; allocate 72 bytes on stack
mov     rax, QWORD PTR __security_cookie
xor     rax, rsp
mov     QWORD PTR __$ArrayPad$[rsp], rax

    This is stack buffer overflow protection. The compiler: 1. Allocates extra space on the stack 2. Loads a random security cookie value 3. XORs it with the stack pointer and stores it 4. Later verifies it hasn't been corrupted

    At the end: 

    mov     rcx, QWORD PTR __$ArrayPad$[rsp]
xor     rcx, rsp
call    __security_check_cookie

    This checks that the cookie is still valid before returning.
