    strict-aliasing¶

    Details

    Source: security/strict-aliasing.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    strict-aliasing.c
    /*
 * Strict aliasing violation: type punning through pointer casts.
 *
 * This code has undefined behavior under strict aliasing rules.
 * Results vary based on -fstrict-aliasing (default in O2+) vs
 * -fno-strict-aliasing.
 */
#include <stdint.h>

/* Convert float bits to int (WRONG WAY - UB with strict aliasing) */
uint32_t float_bits_bad(float f)
{
    /* Violates strict aliasing - float and uint32_t are
     * not compatible types, so compiler may assume they
     * don't alias and optimize incorrectly.
     */
    return *(uint32_t *)&f;
}

/* The safe way using union (well-defined in C99+) */
uint32_t float_bits_safe(float f)
{
    union {
        float f;
        uint32_t u;
    } conv;
    conv.f = f;
    return conv.u;
}

    Explanation¶

    the key insight here is how the compiler handles two different approaches to converting float bits to an integer, and how strict aliasing rules affect the generated code.

    The Core Difference¶

    The most important observation is that both functions produce nearly identical assembly despite their different source code approaches. This reveals something crucial about how modern compilers handle strict aliasing violations.

    float_bits_bad - The Undefined Behavior Version¶

    movss   DWORD PTR [rsp+8], xmm0    ; Store float parameter to stack
mov     eax, DWORD PTR f$[rsp]     ; Load it back as integer
ret     0

    This function: 1. Stores the float: The movss instruction (move scalar single-precision) takes the float parameter from xmm0 (the register used for floating-point parameters by convention) and stores it at [rsp+8] on the stack 2. Loads as integer: The mov instruction reads those same bytes back from the stack into eax (the return value register), but now interprets them as a 32-bit integer 3. Returns: The integer bits are now in eax and the function returns

    The code violates strict aliasing rules because it writes a float and reads a uint32_t from the same memory location—the compiler is technically allowed to assume these don't alias and optimize incorrectly.

    float_bits_safe - The Union Version¶

    movss   DWORD PTR [rsp+8], xmm0    ; Store float parameter to stack
sub     rsp, 24                     ; Allocate stack space for union
movss   xmm0, DWORD PTR f$[rsp]    ; Load float from parameter area
movss   DWORD PTR conv$[rsp], xmm0 ; Store to union's float member
mov     eax, DWORD PTR conv$[rsp]  ; Load from union's uint32_t member
add     rsp, 24                     ; Deallocate stack space
ret     0

    This function: 1. Stores the parameter: Same as before—the float goes to the stack 2. Allocates union space: sub rsp, 24 makes room for the union (though only 4 bytes are needed; the extra space is for alignment) 3. Copies through union: The float is loaded from the parameter area, stored to the union's f member, then loaded back as the u member 4. Returns the integer: eax contains the bit pattern 5. Cleans up: add rsp, 24 restores the stack pointer

    Why They're Nearly Identical¶

    Despite the union approach being "safe," the compiler generates extra stack operations because: - The union creates a separate memory location (conv$) on the stack - The compiler must respect the union's semantics by actually performing the store-then-load through it - This is why you see the extra movss instructions compared to the "bad" version

    The union version is well-defined behavior because unions explicitly allow accessing the same memory through different member types. The compiler cannot optimize away these operations.

    Key Observations¶

    • Parameter passing: Both functions receive the float in xmm0 (the standard register for floating-point parameters on x86-64)
    • Return value: Both return the integer result in eax (the standard register for 32-bit integer returns)
    • Stack usage: The "bad" version uses minimal stack space, while the "safe" version allocates extra space for the union, making it slightly less efficient
    • Undefined behavior doesn't always crash: The "bad" version happens to work correctly here because the compiler's assumptions about aliasing don't prevent the correct bit pattern from being retrieved—but this is not guaranteed and could fail with different optimization strategies
