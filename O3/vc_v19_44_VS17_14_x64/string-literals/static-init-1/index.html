    static-init-1¶

    Details

    Source: string-literals/static-init-1.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    static-init-1.c
    /*
 * Static string initialization with concatenation.
 *
 * Adjacent string literals are concatenated at compile time.
 * The initializer may become a single memcpy or inline stores.
 */
#include <string.h>
#include <stdio.h>

void foo(void)
{
    /* Adjacent strings concatenated at compile time: "AAAABBBB" */
    char buf[100] = { "AAAA" "BBBB" };

    /* Make observable to prevent dead code elimination */
    puts(buf);
}

    Explanation¶

    the key insight is that this code demonstrates stack buffer initialization with security protection, compiled with aggressive optimizations (-O3).

    The Big Picture¶

    The compiler has taken your simple string initialization and wrapped it with stack canary protection (a security feature that detects buffer overflows). Even though -O3 optimization is enabled, security features take precedence. The actual string copying is straightforward, but there's significant overhead for the security checks.

    Stack Setup and Security Cookie¶

    push    rsi
push    rdi
sub     rsp, 168
mov     rax, QWORD PTR __security_cookie
xor     rax, rsp
mov     QWORD PTR __$ArrayPad$[rsp], rax

    The function starts by: 1. Saving registers (rsi and rdi) - these are caller-saved registers that the function will use, so they must be preserved 2. Allocating stack space - sub rsp, 168 reserves 168 bytes on the stack (your 100-byte buffer plus padding and security data) 3. Installing a security cookie - this is a random value XORed with the stack pointer and stored on the stack. Before the function returns, it will verify this value hasn't been corrupted (a sign of a buffer overflow attack)

    String Initialization: Two-Phase Copy¶

    lea     rax, QWORD PTR buf$[rsp]        ; rax = address of buf
lea     rcx, OFFSET FLAT:$SG10443       ; rcx = address of "AAAABBBB"
mov     rdi, rax
mov     rsi, rcx
mov     ecx, 9
rep movsb

    The compiler uses rep movsb (repeat move string byte) to copy the concatenated string "AAAABBBB" (8 characters plus null terminator = 9 bytes) from the data section into your buffer. The registers are set up according to the x64 calling convention: - rdi = destination (your buffer) - rsi = source (the string in memory) - ecx = count (9 bytes)

    Clearing the Rest of the Buffer¶

    lea     rax, QWORD PTR buf$[rsp+9]      ; rax = buf + 9 (after the string)
mov     rdi, rax
xor     eax, eax                         ; eax = 0
mov     ecx, 91                          ; 91 remaining bytes
rep stosb

    After copying the string, the compiler clears the remaining 91 bytes of the buffer with zeros using rep stosb (repeat store byte). This is a defensive practiceâ€”it ensures uninitialized memory isn't accidentally used.

    Function Call and Return¶

    lea     rcx, QWORD PTR buf$[rsp]        ; rcx = address of buf (first parameter)
call    puts

    The buffer address is loaded into rcx (the first parameter register in x64 calling convention) and puts is called.

    mov     rcx, QWORD PTR __$ArrayPad$[rsp]
xor     rcx, rsp
call    __security_check_cookie
add     rsp, 168
pop     rdi
pop     rsi
ret     0

    Before returning, the security cookie is verified by XORing it with the current stack pointer againâ€”if the buffer was overflowed, this value would have changed. Finally, the stack is cleaned up and registers are restored.

    Key Observations¶

    • String concatenation is compile-time: "AAAA" "BBBB" becomes a single "AAAABBBB" string in the data sectionâ€”no runtime concatenation overhead
    • Security overhead dominates: Even with -O3, the stack canary protection adds significant code (the cookie setup and verification)
    • Defensive initialization: The compiler clears the entire buffer, not just the initialized portion, which is safe but adds overhead
    • Register usage follows conventions: rdi/rsi for string operations, rcx for the count and function parameters, following x64 calling conventions
