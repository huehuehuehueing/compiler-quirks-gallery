    fast-math¶

    Details

    Source: arithmetic/fast-math.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    fast-math.c
    /*
 * Fast-math optimizations: -ffast-math / -Ofast effects.
 *
 * These flags allow the compiler to reorder floating-point
 * operations, assuming associativity (which floats don't have).
 * Can change results or break special value handling.
 */

/* With -ffast-math, this may be reordered and give different results */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++)
        sum += arr[i];
    return sum;
}

/* NaN check may be optimized away with -ffast-math */
int is_nan(float x)
{
    /* With -ffast-math, compiler assumes no NaN values exist,
     * so this comparison may be optimized to always return 0.
     */
    return x != x;
}

/* Division by variable may become multiplication by reciprocal */
float divide_by_y(float x, float y)
{
    /* With -ffast-math: may become x * (1.0f / y)
     * Less accurate but faster on some architectures.
     */
    return x / y;
}

/* Algebraic simplification (inexact with floats!) */
float algebraic(float a, float b, float c)
{
    /* With -ffast-math: (a + b) + c may be reordered to a + (b + c)
     * These can give different results with floating point!
     */
    return (a + b) + c;
}

    Explanation¶

    the key insight is that this code demonstrates how the MSVC compiler handles floating-point operations with -O3 optimization, including some surprising behaviors around NaN checking and floating-point arithmetic.

    Key Pattern: Floating-Point Operations and Stack Management¶

    The most important thing to understand is that floating-point values in x86-64 are passed and returned using XMM registers (like xmm0, xmm1, xmm2), which are special 128-bit registers designed for floating-point math. This is different from integer parameters, which use rcx, rdx, etc.

    sum_array Function¶

    This function accumulates values from an array. Here's what happens:

    Setup and initialization: 

    mov     DWORD PTR [rsp+16], edx      # Save parameter n to stack
mov     QWORD PTR [rsp+8], rcx       # Save parameter arr pointer to stack
sub     rsp, 24                      # Allocate 24 bytes of stack space
xorps   xmm0, xmm0                   # Zero out xmm0 (sets sum to 0.0)
movss   DWORD PTR sum$[rsp], xmm0    # Store initial sum (0.0) on stack
mov     DWORD PTR i$1[rsp], 0        # Initialize loop counter i to 0

    The xorps instruction is a clever trick: XORing a register with itself always produces zero, and it's faster than loading a constant. The movss (move scalar single-precision) stores this zero as a 32-bit float on the stack.

    Loop structure: 

    $LN4@sum_array:                      # Loop condition check
    mov     eax, DWORD PTR n$[rsp]   # Load n into eax
    cmp     DWORD PTR i$1[rsp], eax  # Compare i with n
    jge     SHORT $LN3@sum_array     # Jump to exit if i >= n

    movsxd  rax, DWORD PTR i$1[rsp]  # Sign-extend i to 64-bit (for array indexing)
    mov     rcx, QWORD PTR arr$[rsp] # Load array pointer
    movss   xmm0, DWORD PTR sum$[rsp]# Load current sum
    addss   xmm0, DWORD PTR [rcx+rax*4]  # Add arr[i] to sum
    movss   DWORD PTR sum$[rsp], xmm0    # Store updated sum back
    jmp     SHORT $LN2@sum_array     # Jump to increment

    The key instruction is addss (add scalar single-precision), which adds one 32-bit float to another. The address [rcx+rax*4] calculates the array element: rcx is the base pointer, rax is the index, and *4 accounts for 4 bytes per float.

    is_nan Function¶

    This is the most interesting functionit demonstrates how the compiler still generates NaN-checking code even with -O3:

    movss   xmm0, DWORD PTR x$[rsp]      # Load x into xmm0
ucomiss xmm0, DWORD PTR x$[rsp]      # Compare x with itself
jp      SHORT $LN5@is_nan            # Jump if parity flag set (NaN detected)
je      SHORT $LN3@is_nan            # Jump if equal (not NaN)

    The ucomiss (unordered compare scalar single-precision) instruction compares x with itself. Normally, any value equals itself, so the je (jump if equal) would execute. However, NaN has a special property: NaN NaN. When comparing NaN with itself: - The jp (jump if parity) instruction triggers because NaN sets the parity flag - This jumps to code that returns 1 (true)

    If the value is not NaN, je executes and jumps to code that returns 0 (false).
