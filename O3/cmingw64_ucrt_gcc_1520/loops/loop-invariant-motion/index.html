    loop-invariant-motion¶

    Details

    Source: loops/loop-invariant-motion.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    loop-invariant-motion.c
    /*
 * Loop-Invariant Code Motion (LICM)
 *
 * The optimizer recognises computations inside a loop whose result
 * does not change across iterations and moves ("hoists") them before
 * the loop.  This avoids redundant work on every iteration.
 *
 * In the example below, `base * scale` is invariant with respect to
 * the loop index i, so the compiler should compute it once and reuse
 * the result inside the loop body.
 */

void fill_scaled(int *arr, int n, int base, int scale)
{
    for (int i = 0; i < n; i++)
    {
        /* base * scale is loop-invariant: it does not depend on i. */
        arr[i] = base * scale + i;
    }
}

    Explanation¶

    the key insight is that this code demonstrates Loop-Invariant Code Motion (LICM) - the compiler computed base * scale once before the loop rather than repeatedly inside it.

    The Core Optimization: Computing Once, Using Many Times¶

    The most important thing to understand is on line 5: 

    imul    r8d, r9d

    This single multiplication happens before the loop starts. The compiler recognized that base * scale doesn't change across loop iterations, so it computed this value once and stored it in r8d (the lower 32 bits of the r8 register). This is the LICM optimization in action - it eliminates redundant work.

    Parameter Setup and Register Conventions¶

    The function receives four parameters following the Windows x64 calling convention: - rcx = pointer to array (arr) saved to r10 - edx = loop count (n) saved to ecx - r8d = base value - r9d = scale value

    The first few instructions rearrange these into working registers: 

    mov     r10, rcx        # Save array pointer
mov     ecx, edx        # Save loop count

    Early Exit Check¶

    test    edx, edx
jle     .L1

    If n <= 0, the loop doesn't run, so jump to .L1 (the return statement). This prevents processing an empty or negative-sized array.

    The Loop-Invariant Computation¶

    imul    r8d, r9d        # r8d = base * scale (computed ONCE, before loop)

    This is the key optimization. Instead of computing base * scale inside the loop 4 times per iteration (in the vectorized version), it's computed once here. The result stays in r8d throughout the loop.

    Vectorized Loop (The Fast Path)¶

    The compiler uses SIMD instructions (Single Instruction, Multiple Data) to process 4 array elements at a time. This is an aggressive optimization at -O3:

    movd    xmm4, r8d       # Load base*scale into XMM register
movdxa  xmm0, .LC0[rip] # Load [0, 1, 2, 3] into XMM register

    The .LC0 constant contains [0, 1, 2, 3] - these are the offsets i for the first 4 iterations.

    Inside the loop (.L4): 

    pshufd  xmm1, xmm4, 0   # Broadcast base*scale to all 4 lanes
paddd   xmm1, xmm0      # Add [0,1,2,3] to get [base*scale+0, base*scale+1, ...]
paddd   xmm0, xmm2      # Increment offsets by 4 for next iteration
movups  XMMWORD PTR -16[rax], xmm1  # Store 4 values at once

    This processes 4 array elements per iteration instead of 1, making the loop 4 faster.

    Scalar Cleanup Loop (The Remainder)¶

    After the vectorized loop finishes, there may be 1-3 remaining elements (since the array size might not be divisible by 4). The code at .L3 handles these:

    lea     edx, [r8+rax]   # Compute base*scale + i
mov     DWORD PTR [r10+r9*4], edx  # Store one element

    This processes remaining elements one at a time using scalar (non-SIMD) instructions.

    Key Observations¶

    Loop-Invariant Code Motion: The multiplication base * scale is hoisted outside the loop entirely - it's computed once and reused. Without this optimization, the unoptimized version would multiply these values 4 times per iteration.

    Vectorization: The -O3 optimization level enables SIMD instructions to process 4 integers simultaneously, dramatically reducing the number of loop iterations needed.

    Two-Path Strategy: The compiler generates both a fast vectorized path (for most iterations) and a scalar cleanup path (for remaining elements), ensuring correctness regardless of array size.
