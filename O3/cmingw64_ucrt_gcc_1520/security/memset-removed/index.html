    memset-removed¶

    Details

    Source: security/memset-removed.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    memset-removed.c
    /*
 * Security issue: memset() to clear sensitive data is removed
 * as dead code when the buffer is not used afterward.
 *
 * This is a real security vulnerability pattern - passwords
 * and keys may remain in memory after "clearing".
 */
#include <string.h>
#include <stdio.h>

void process_password(const char *password)
{
    char local_copy[64];

    /* Copy password for processing */
    strncpy(local_copy, password, sizeof(local_copy) - 1);
    local_copy[sizeof(local_copy) - 1] = '\0';

    /* Do something observable with the password (simulate auth check) */
    if (local_copy[0] == 's' && local_copy[1] == 'e' &&
        local_copy[2] == 'c' && local_copy[3] == 'r' &&
        local_copy[4] == 'e' && local_copy[5] == 't') {
        puts("Access granted");
    } else {
        puts("Access denied");
    }

    /* Security: clear the password from stack
     * BUG: This memset is often REMOVED by optimizer
     * because local_copy is not used afterward!
     */
    memset(local_copy, 0, sizeof(local_copy));
}

void test_password(void)
{
    process_password("secret");
}

    Explanation¶

    the key insight, then the supporting details.

    The Critical Security Issue¶

    The most important thing to understand: the memset() call that was supposed to clear the password from memory is completely gone from the assembly code. This is the security vulnerability the source code comments warn about. The optimizer removed it because it determined the cleared buffer is never read again, treating it as "dead code." This is a real security problemthe password data remains on the stack even though the code intended to erase it.

    How the Code Works¶

    process_password Function¶

    The function receives a password string and checks if it matches "secret":

    1. Stack setup: sub rsp, 104 allocates 104 bytes of stack space for the local buffer and alignment.

    2. Copying the password:

    3. lea rcx, 32[rsp] calculates the address of the local buffer (32 bytes into the stack frame)
    4. mov r8d, 63 sets the maximum copy length to 63 bytes

    5. call strncpy copies the password into the local buffer

    6. Character-by-character password check: The code compares each character of the password against the ASCII values for "secret":

    7. cmp BYTE PTR 32[rsp], 115 checks if byte 0 equals 's' (ASCII 115)
    8. cmp BYTE PTR 33[rsp], 101 checks if byte 1 equals 'e' (ASCII 101)
    9. And so on for 'c' (99), 'r' (114), 'e' (101), 't' (116)

    10. If any character doesn't match, it jumps to .L2 to print "Access denied"

    11. Output: If all characters match, it loads the address of "Access granted" into rcx and calls puts. Otherwise, it calls puts with "Access denied".

    12. Missing security step: Notice there's no memset() call before the function returns. The password data remains on the stack.

    test_password Function¶

    This function demonstrates the vulnerability by calling process_password("secret"):

    1. Stack setup: sub rsp, 72 allocates 72 bytes for the stack frame.

    2. Embedding the password string: Instead of passing a pointer to a string, the compiler inlines the literal "secret" directly onto the stack:

    3. movabs rax, 127979059635571 loads the bytes 's', 'e', 'c', 'r', 'e', 't' packed into a 64-bit value
    4. Multiple mov instructions write zeros to the rest of the buffer (padding and null terminator)

    5. mov QWORD PTR [rsp], rax writes the packed password bytes to the stack

    6. Password check: The same character-by-character comparisons happen, checking against the inlined password.

    7. Stack cleanup and return: add rsp, 72 deallocates the stack space and returns.

    Why the Optimizer Removed memset()¶

    The -O3 optimization level is aggressive. The compiler analyzed the code and determined: - The local_copy buffer is only read during the password check - After the check completes, the buffer is never read again - Therefore, the memset() call that writes zeros to it has no observable effect - The optimizer eliminated it as dead code

    This is technically correct from a functional perspectivethe program produces the same output. But it's a security disaster because sensitive data (the password) remains in memory where it could potentially be recovered by an attacker with access to the stack.

    Key Observations¶

    • Inlining optimization: The compiler inlined the string literal "secret" directly into test_password's stack frame rather than passing a pointer, which is more efficient but also demonstrates how the optimizer aggressively optimizes this code.
    • No stack frame pointer: Neither function uses rbp (the frame pointer register), which is typical for optimized code where the compiler can track stack offsets directly.
    • The vulnerability is real: This demonstrates why security-critical code should use compiler-specific mechanisms (like explicit_bzero() on some systems or volatile qualifiers) to prevent memset() from being optimized away.
