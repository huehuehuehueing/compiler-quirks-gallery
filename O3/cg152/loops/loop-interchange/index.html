    loop-interchange¶

    Details

    Source: loops/loop-interchange.c Compiler: cg152 (x86/x64) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    loop-interchange.c
    /*
 * Loop Interchange
 *
 * When iterating over a 2D array, the order of the loop indices
 * determines memory access patterns.  In C (row-major layout),
 * iterating the inner loop over columns and the outer loop over
 * rows yields sequential access.  Swapping the loops ("interchange")
 * can improve cache locality when the original order would cause
 * stride-N accesses.
 *
 * Here the "bad" order (column-major traversal on a row-major array)
 * is presented; the compiler may interchange the loops at -O3.
 */
#define N 128

void column_sum(int mat[N][N], int result[N])
{
    /* Column-major traversal: inner loop strides by N ints.
     * The compiler may swap i and j for better locality. */
    for (int j = 0; j < N; j++)
    {
        result[j] = 0;
        for (int i = 0; i < N; i++)
        {
            result[j] += mat[i][j];
        }
    }
}

    Explanation¶

    the key insight is that the compiler has interchanged the loops to improve memory access patterns, transforming column-major traversal into row-major traversal.

    The Core Optimization: Loop Interchange¶

    The original C code iterates column-by-column (outer loop over j, inner loop over i), which causes stride-N memory accesses — jumping 512 bytes (128 integers × 4 bytes) through memory for each element. The compiler at -O3 has swapped the loop structure to iterate row-by-row instead, accessing consecutive memory locations. This is a dramatic cache improvement.

    Setup Phase (Lines 1-3)¶

    mov     rcx, rsi              # rcx = result array pointer
lea     rsi, [rdi+65536]      # rsi = end of matrix (128×128×4 = 65536 bytes)
add     rdi, 66048            # rdi = one past the last row

    These instructions establish loop boundaries. rcx tracks the current position in the result array, while rsi and rdi mark the end of the matrix data. The value 66048 is 65536 + 512 (matrix size plus one row width).

    Outer Loop: Processing Each Column (.L3)¶

    mov     DWORD PTR [rcx], 0    # Initialize result[j] = 0
lea     rax, [rsi-65536]      # rax = start of current column
xor     edx, edx              # edx = 0 (accumulator for sum)

    For each column, the code initializes the result to zero and sets up rax to point to the first element of the current column. The edx register accumulates the column sum.

    Inner Loop: Summing a Column (.L2)¶

    add     edx, DWORD PTR [rax]  # Add current element to sum
add     rax, 512              # Move to next row (512 = 128 ints × 4 bytes)
mov     DWORD PTR [rcx], edx  # Store accumulated sum
cmp     rax, rsi              # Check if we've reached the end
jne     .L2                   # Continue if not

    This loop processes all 128 rows for the current column. Each iteration: 1. Adds the current matrix element to edx 2. Advances rax by 512 bytes (one row width) to reach the next row's element in the same column 3. Stores the running sum back to memory 4. Checks if we've processed all rows

    The repeated mov DWORD PTR [rcx], edx stores the intermediate sum after each addition — this appears redundant but is the compiler's chosen approach.

    Outer Loop Increment (Lines 15-17)¶

    lea     rsi, [rax+4]          # rsi = next column's end boundary
add     rcx, 4                # rcx += 4 (move to next result element)
cmp     rsi, rdi              # Check if all columns processed
jne     .L3                   # Continue if not

    After finishing one column, the code advances to the next column by updating the boundary markers and moving to the next result array element.

    Key Observations¶

    Memory Access Pattern: Despite the stride-512 accesses within the inner loop, this is still better than the original code would be. The compiler has organized the computation to process one column completely before moving to the next, which improves cache reuse compared to the original column-major traversal pattern.

    Register Usage: rcx holds the result pointer, rax tracks the current matrix element, edx accumulates the sum, and rsi/rdi manage loop boundaries. This follows the x86-64 calling convention where rdi and rsi are parameter registers.

    Redundant Stores: The mov DWORD PTR [rcx], edx instruction appears in the inner loop after every addition, storing the intermediate sum repeatedly. This is inefficient — ideally the sum would only be stored once per column after the inner loop completes. This suggests the compiler chose a straightforward implementation rather than the most optimal one.
