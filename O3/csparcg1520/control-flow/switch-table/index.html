    switch-table¶

    Details

    Source: control-flow/switch-table.c Compiler: csparcg1520 (SPARC) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    switch-table.c
    /*
 * Switch optimization: jump tables vs. binary search vs. if-else.
 *
 * Compiler chooses implementation based on case distribution:
 * - Dense cases: jump table (O(1))
 * - Sparse cases: binary search (O(log n)) or if-else chain
 */

/* Dense cases - likely becomes a jump table */
int day_of_week(int day)
{
    switch (day) {
        case 0: return 'S';  /* Sunday */
        case 1: return 'M';
        case 2: return 'T';
        case 3: return 'W';
        case 4: return 'T';
        case 5: return 'F';
        case 6: return 'S';  /* Saturday */
        default: return '?';
    }
}

/* Sparse cases - likely if-else or binary search */
int sparse_switch(int x)
{
    switch (x) {
        case 1:     return 10;
        case 100:   return 20;
        case 10000: return 30;
        case 999999: return 40;
        default:    return 0;
    }
}

/* Two cases - becomes simple comparison */
int binary_choice(int x)
{
    switch (x) {
        case 0: return -1;
        case 1: return 1;
        default: return 0;
    }
}

    Explanation¶

    the key insight is that the compiler uses three different switch implementation strategies depending on how densely packed the case values are.

    day_of_week: Jump Table (Dense Cases)¶

    This function demonstrates the most efficient switch strategy for consecutive values:

    mov     %o0, %g1              # Copy input parameter to %g1
cmp     %g1, 6                # Compare with 6 (highest case value)
bgu,pn  %icc, .L5             # If greater than 6, branch to default
 mov    63, %o0               # (delay slot) Set default return '?' (ASCII 63)
sll     %g1, 2, %g1           # Multiply index by 4 (shift left 2 bits)
sethi   %hi(CSWTCH.1), %g2    # Load high bits of jump table address
or      %g2, %lo(CSWTCH.1), %g2  # Load low bits of jump table address
ld      [%g2+%g1], %o0        # Load return value from table

    Why a jump table? The cases (0-6) are densely packed with no gaps. The compiler creates a lookup table (CSWTCH.1) containing the ASCII values (83='S', 77='M', 84='T', etc.). The input is used as an index: multiply by 4 (since each table entry is 4 bytes), then load the corresponding value. This is O(1) constant time no comparisons needed for valid inputs.

    The bounds check (cmp %g1, 6) ensures the input is in range; out-of-range values get the default return value '?' (ASCII 63).

    sparse_switch: Binary Search (Sparse Cases)¶

    This function has widely scattered case values (1, 100, 10000, 999999), so a jump table would waste memory:

    sethi   %hi(9216), %g2        # Load 10000 (high bits)
or      %g2, 784, %g2         # Complete 10000 (9216*1024 + 784 = 10000)
cmp     %o0, %g2              # Compare input with 10000
be,pn   %icc, .L6             # If equal, branch to return 30
 mov    30, %g1               # (delay slot) Set return value 30
bg,pn   %icc, .L8             # If input > 10000, branch to check 999999
 cmp    %o0, 1                # (delay slot) Compare with 1
be,pn   %icc, .L6             # If equal to 1, branch to return
 mov    10, %g1               # (delay slot) Set return value 10
mov     20, %g1               # Set return value 20 (for case 100)
cmp     %o0, 100              # Compare with 100
movne   %icc, 0, %g1          # If not equal, set return 0 (default)

    The compiler uses a binary search tree approach: it first checks the middle value (10000), then branches based on whether the input is larger or smaller. This is O(log n) fewer comparisons than checking every case sequentially.

    Notice the use of conditional move instructions (movne, be) these avoid branches by conditionally updating registers based on the previous comparison result. This is an optimization that reduces branch mispredictions.

    binary_choice: Arithmetic Encoding (Two Cases)¶

    With only two cases, the compiler uses a clever arithmetic trick:

    xor     %o0, 1, %g1           # XOR input with 1
cmp     %g0, %g1              # Compare result with zero
subx    %g0, -1, %g1          # Subtract with carry: if zero, result is -1; else 0
cmp     %o0, 0                # Compare input with 0
mov     -1, %o0               # Set default return -1
jmp     %o7+8                 # Return
 movne  %icc, %g1, %o0        # (delay slot) If input != 0, use computed value
