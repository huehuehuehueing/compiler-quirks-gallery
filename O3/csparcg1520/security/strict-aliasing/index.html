    strict-aliasing¶

    Details

    Source: security/strict-aliasing.c Compiler: csparcg1520 (SPARC) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    strict-aliasing.c
    /*
 * Strict aliasing violation: type punning through pointer casts.
 *
 * This code has undefined behavior under strict aliasing rules.
 * Results vary based on -fstrict-aliasing (default in O2+) vs
 * -fno-strict-aliasing.
 */
#include <stdint.h>

/* Convert float bits to int (WRONG WAY - UB with strict aliasing) */
uint32_t float_bits_bad(float f)
{
    /* Violates strict aliasing - float and uint32_t are
     * not compatible types, so compiler may assume they
     * don't alias and optimize incorrectly.
     */
    return *(uint32_t *)&f;
}

/* The safe way using union (well-defined in C99+) */
uint32_t float_bits_safe(float f)
{
    union {
        float f;
        uint32_t u;
    } conv;
    conv.f = f;
    return conv.u;
}

    Explanation¶

    the most important insight here is that both functions have been completely optimized away to empty stubs - they contain only a return instruction and a no-op.

    Why Both Functions Disappeared¶

    Under -O3 optimization with strict aliasing enabled (the default), the compiler has determined that both functions can be simplified to nothing:

    float_bits_bad: The compiler recognizes that the cast *(uint32_t *)&f violates strict aliasing rules. Under strict aliasing, the compiler assumes a float* and uint32_t* can never point to the same memory location. This allows the compiler to optimize away the entire operation - it assumes the cast has no meaningful effect and can be eliminated.

    float_bits_safe: Even though this uses a union (which is the correct, well-defined approach), the compiler still optimizes it away completely. The union stores a float, reads it back as a uint32_t, and returns it. At -O3, the compiler recognizes this pattern and realizes the value in the float parameter f can be directly returned as-is - the bit pattern is preserved through the union, so the intermediate storage and retrieval are unnecessary.

    The Assembly Instructions¶

    Both functions contain identical code:

    jmp     %o7+8      // Jump to return address
 nop               // No operation (delay slot)

    In SPARC, %o7 is the return address register (set by the call instruction). The jmp %o7+8 instruction jumps back to the caller. The nop (no operation) fills the delay slot - a SPARC architectural feature where the instruction after a branch executes before the branch takes effect.

    The parameter f (passed in %f0 by the floating-point calling convention) is never explicitly used in the assembly because the compiler has optimized away all the work.

    Key Observation¶

    This demonstrates how -O3 optimization can be aggressive: even "correct" code using unions gets optimized to the point where the actual bit-reinterpretation logic vanishes. The compiler trusts that the parameter value itself contains the bit pattern you need, making explicit reinterpretation unnecessary.
