    fast-math¶

    Details

    Source: arithmetic/fast-math.c Compiler: csparcg1520 (SPARC) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    fast-math.c
    /*
 * Fast-math optimizations: -ffast-math / -Ofast effects.
 *
 * These flags allow the compiler to reorder floating-point
 * operations, assuming associativity (which floats don't have).
 * Can change results or break special value handling.
 */

/* With -ffast-math, this may be reordered and give different results */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++)
        sum += arr[i];
    return sum;
}

/* NaN check may be optimized away with -ffast-math */
int is_nan(float x)
{
    /* With -ffast-math, compiler assumes no NaN values exist,
     * so this comparison may be optimized to always return 0.
     */
    return x != x;
}

/* Division by variable may become multiplication by reciprocal */
float divide_by_y(float x, float y)
{
    /* With -ffast-math: may become x * (1.0f / y)
     * Less accurate but faster on some architectures.
     */
    return x / y;
}

/* Algebraic simplification (inexact with floats!) */
float algebraic(float a, float b, float c)
{
    /* With -ffast-math: (a + b) + c may be reordered to a + (b + c)
     * These can give different results with floating point!
     */
    return (a + b) + c;
}

    Explanation¶

    the key insight is that this code demonstrates how floating-point operations are handled on SPARC, with particular attention to parameter passing and the unusual stack usage patterns even with -O3 optimization.

    Parameter Passing on SPARC¶

    SPARC uses a specific calling convention: integer arguments are passed in registers %o0, %o1, %o2, etc. (output registers), and floating-point values are passed as integers in these same registers, then converted to floating-point registers when needed. This is important for understanding why you see store-then-load patterns.

    sum_array Function¶

    This function demonstrates the most optimized code here:

    cmp     %o1, 0              // Compare n with 0
ble,pn  %icc, .L4          // If n <= 0, branch to .L4 (early exit)
 sll    %o1, 2, %o1        // Shift n left by 2 (multiply by 4 for byte offset)

    The sll (shift left logical) instruction multiplies n by 4 because each float is 4 bytes. This happens in the delay slot (the instruction after the branch that executes regardless of the branch outcome).

    sethi   %hi(.LC0), %g1     // Load high bits of address for 0.0f constant
add     %o0, %o1, %o1      // Calculate end address: arr + (n*4)
ld      [%g1+%lo(.LC0)], %f0  // Load 0.0f into %f0 (the sum accumulator)

    The loop itself is tight and efficient:

    .L3:
ld      [%o0], %f8         // Load next array element
add     %o0, 4, %o0        // Increment pointer by 4 bytes
cmp     %o1, %o0           // Compare end address with current position
bne,pt  %icc, .L3          // Branch if not equal (continue loop)
 fadds  %f0, %f8, %f0      // Add to sum (in delay slot)

    The fadds instruction (floating-point add single-precision) executes in the branch delay slot, meaning it runs before the branch decision is made. This is efficient pipelining. The loop continues until %o0 reaches the end address stored in %o1.

    is_nan Function¶

    This function checks if a float equals itself (NaN != NaN):

    add     %sp, -80, %sp      // Allocate 80 bytes on stack
st      %o0, [%sp+76]      // Store the float parameter to stack
mov     0, %o0             // Pre-load return value as 0
ld      [%sp+76], %f8      // Load it back into floating-point register
fcmps   %fcc0, %f8, %f8    // Compare %f8 with itself

    The fcmps instruction sets the floating-point condition code register %fcc0. If the value is NaN, the comparison sets the "not equal" condition.

    movne   %fcc0, 1, %o0      // If not equal (NaN detected), move 1 into %o0

    This conditional move executes only if the "not equal" condition is true, returning 1 for NaN, otherwise 0.

    Why the stack operations? Even with -O3, the compiler uses the stack to convert the integer register parameter to a floating-point register. This is because SPARC's calling convention passes floats as integers, requiring this intermediate storage.

    divide_by_y Function¶

    st      %o0, [%sp+76]      // Store x to stack
ld      [%sp+76], %f8      // Load x into %f8
st      %o1, [%sp+76]      // Store y to stack
ld      [%sp+76], %f9      // Load y into %f9
fdivs   %f8, %f9, %f0      // Divide: %f0 = %f8 / %f9

    The fdivs instruction performs single-precision floating-point division. The result goes directly into %f0, which is the return value register
