    static-memcpy¶

    Details

    Source: string-literals/static-memcpy.c Compiler: csparcg1520 (SPARC) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    static-memcpy.c
    #include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[5];

    memcpy(buf, "AAAA", 4);
    buf[sizeof(buf) - 1] = '\0';

    /* make sure the above is not treated as dead code */
    puts(buf);
}

    Explanation¶

    the key insight here is that the compiler has eliminated the memcpy call entirely and replaced it with direct memory writes, which is a significant optimization.

    Stack Frame Setup¶

    save    %sp, -104, %sp

    This instruction creates a new stack frame for the function. The save instruction is SPARC's way of setting up a function prologue. It allocates 104 bytes of stack space (the negative value indicates space allocation) and saves the caller's registers. This creates room for local variables like the buf[5] array.

    The Optimization: Constant Folding¶

    The most important thing to understand here is what the compiler didn't do: it didn't call memcpy. Instead, it recognized that: - The source string "AAAA" is a compile-time constant - The destination buffer is a local variable - The copy size is fixed at 4 bytes

    So the compiler inlined the copy operation as direct memory writes:

    sethi   %hi(1094795264), %g1      // Load upper 22 bits of "AAAA"
or      %g1, 321, %g1             // OR in lower 10 bits to complete "AAAA"
st      %g1, [%fp-8]              // Store the 4-byte constant to buf

    The values 1094795264 and 321 combine to form the ASCII representation of "AAAA" (0x41414141 in hex). This is much faster than calling a function—the compiler has essentially replaced a function call with a single store instruction.

    Null Terminator¶

    stb     %g0, [%fp-4]

    This stores a byte of zero (%g0 is the zero register in SPARC) at offset -4 from the frame pointer. This is the null terminator ('\0') being written to buf[4] (the last position in the 5-byte array).

    Preparing for puts Call¶

    add     %fp, -8, %o0               // Calculate address of buf
call    puts, 0                    // Call puts function
 st     %g1, [%fp-8]               // (delay slot) Store "AAAA" to buf

    The add instruction calculates the address of the buffer (frame pointer minus 8 bytes) and places it in %o0. By SPARC calling convention, the first argument to a function goes in %o0, so this passes the buffer address to puts.

    Notice the st instruction appears after the call instruction—this is a delay slot. In SPARC, the instruction immediately following a branch or call executes before the branch takes effect. The compiler uses this to store the "AAAA" constant to the buffer right before calling puts, saving an instruction.

    Function Return¶

    return  %i7+8
 nop

    The return instruction (which is a synthetic instruction for jmpl %i7+8, %g0) jumps back to the caller. The nop (no operation) fills the delay slot—it does nothing but is required by SPARC's pipeline architecture.

    Summary¶

    This is highly optimized code (-O3). The compiler has eliminated the memcpy function call by recognizing the constant values and inlining them as direct memory operations. This transforms what could have been an expensive function call into just a couple of store instructions, demonstrating how modern compilers optimize away unnecessary overhead when they can prove the operation is safe.
