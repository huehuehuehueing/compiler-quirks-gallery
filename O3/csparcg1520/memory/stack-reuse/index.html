    stack-reuse¶

    Details

    Source: memory/stack-reuse.c Compiler: csparcg1520 (SPARC) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    stack-reuse.c
    /*
 * Stack slot reuse: non-overlapping locals share stack space.
 *
 * The compiler analyzes variable lifetimes and reuses
 * stack slots for variables that are never live at the same time.
 */
#include <stdio.h>

void example(int flag)
{
    if (flag) {
        /* This buffer is only live in this branch */
        char buffer_a[100];
        snprintf(buffer_a, sizeof(buffer_a), "Path A: %d", flag);
        printf("%s\n", buffer_a);
    } else {
        /* This buffer may reuse the same stack slot as buffer_a
         * since they're never both alive at the same time.
         */
        char buffer_b[100];
        snprintf(buffer_b, sizeof(buffer_b), "Path B: %d", flag);
        printf("%s\n", buffer_b);
    }

    /* Total stack usage may be ~100 bytes, not 200 */
}

    Explanation¶

    the key insight here is stack slot reuse - the compiler allocates only 200 bytes of stack space total, even though the source code declares two 100-byte buffers. This is possible because the buffers are never alive at the same time.

    Stack Frame Setup¶

    save    %sp, -200, %sp

    This instruction creates a new stack frame and allocates 200 bytes of local storage. In SPARC, save is the function prologue instruction that: - Creates a new register window (SPARC uses overlapping register windows for efficient function calls) - Adjusts the stack pointer (%sp) downward by 200 bytes to reserve space for local variables - The negative value -200 means "allocate 200 bytes"

    The 200-byte allocation is the critical detail: even though the source code has two separate 100-byte buffers (buffer_a and buffer_b), the compiler only reserves 200 bytes total, not 400. This demonstrates stack slot reuse - both buffers will use the same memory location because they're never both needed simultaneously.

    Branch Condition Check¶

    cmp     %i0, 0
be,pt   %icc, .L2
 mov    0, %o3

    These instructions check the flag parameter (passed in %i0 by convention): - cmp compares %i0 with 0 - be,pt branches to .L2 if equal (the "else" path). The pt hint tells the processor this branch is predicted taken - mov 0, %o3 executes in the delay slot (a SPARC quirk where the instruction after a branch executes before the branch takes effect)

    Path A: True Branch (flag != 0)¶

    mov     %i0, %o3
mov     100, %o1
add     %fp, -104, %o0
sethi   %hi(.LC0), %o2
call    snprintf, 0
 or     %o2, %lo(.LC0), %o2

    This prepares arguments for snprintf: - %o0 = buffer address: %fp - 104 (the frame pointer minus 104 bytes gives the buffer location) - %o1 = size: 100 - %o2 = format string address for "Path A: %d" (loaded using sethi/or to construct the full address) - %o3 = the flag value (the %d argument)

    The buffer is placed at %fp - 104, which is 104 bytes below the frame pointer. This leaves room for the 100-byte buffer plus 4 bytes of padding/alignment.

    After snprintf returns, puts is called with the same buffer address to print it.

    Path B: False Branch (flag == 0)¶

    .L2:
mov     100, %o1
add     %fp, -104, %o0
sethi   %hi(.LC1), %o2
call    snprintf, 0
 or     %o2, %lo(.LC1), %o2

    This is nearly identical to Path A, except: - The format string is "Path B: %d" (.LC1 instead of .LC0) - The buffer address is still %fp - 104 - the exact same location as buffer_a

    This is the stack slot reuse in action: buffer_b reuses the same stack memory as buffer_a because the two branches are mutually exclusive.

    Return Sequence¶

    return  %i7+8
 nop

    Both paths end with return %i7+8 (return to the caller, with the +8 accounting for the delay slot). The nop (no operation) fills the delay slot harmlessly.

    Key Observations¶

    Stack efficiency: The compiler allocated 200 bytes instead of 400 because it analyzed variable lifetimes and determined the two buffers never coexist. This is a form of liveness analysis - the compiler tracks which variables are "live" (potentially used) at each point in the code.

    Identical buffer placement: Both branches use %fp - 104 for their buffer, confirming they share the same stack slot.
