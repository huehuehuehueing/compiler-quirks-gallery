    fast-math¶

    Details

    Source: arithmetic/fast-math.c Compiler: armv7-clang2110 (ARM) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    fast-math.c
    /*
 * Fast-math optimizations: -ffast-math / -Ofast effects.
 *
 * These flags allow the compiler to reorder floating-point
 * operations, assuming associativity (which floats don't have).
 * Can change results or break special value handling.
 */

/* With -ffast-math, this may be reordered and give different results */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++)
        sum += arr[i];
    return sum;
}

/* NaN check may be optimized away with -ffast-math */
int is_nan(float x)
{
    /* With -ffast-math, compiler assumes no NaN values exist,
     * so this comparison may be optimized to always return 0.
     */
    return x != x;
}

/* Division by variable may become multiplication by reciprocal */
float divide_by_y(float x, float y)
{
    /* With -ffast-math: may become x * (1.0f / y)
     * Less accurate but faster on some architectures.
     */
    return x / y;
}

/* Algebraic simplification (inexact with floats!) */
float algebraic(float a, float b, float c)
{
    /* With -ffast-math: (a + b) + c may be reordered to a + (b + c)
     * These can give different results with floating point!
     */
    return (a + b) + c;
}

    Explanation¶

    the key insight is that this code uses ARM's floating-point registers and instructions to perform mathematical operations efficiently, with the compiler making specific choices about how to handle floating-point arithmetic.

    Floating-Point Registers and Calling Convention¶

    ARM32 uses special floating-point registers (s0, s1, s2, etc.) for floating-point values, separate from general-purpose registers like r0, r1. By convention, function parameters and return values use s0, s1, s2 for floats. This is why you see floating-point values being passed and returned in these registers rather than in r0/r1.

    sum_array() - Loop with Accumulation¶

    vldr    s0, .LCPI0_0          // Load 0.0f into s0 (the accumulator)
cmp     r1, #1                // Compare n with 1
bxlt    lr                     // If n < 1, return immediately
.LBB0_1:
    vldmia  r0!, {s2}         // Load next array element into s2, increment r0
    vadd.f32 s0, s0, s2       // Add: s0 = s0 + s2
    subs    r1, r1, #1        // Decrement counter, update flags
    bne     .LBB0_1            // Loop if counter != 0
bx      lr                     // Return

    What's happening: The function initializes s0 to 0.0f, then loops through the array. Each iteration loads one float element (vldmia with the ! means "load and auto-increment"), adds it to the running sum, and decrements the counter. The bne (branch if not equal) jumps back to the loop label as long as the counter is non-zero.

    The early exit check (cmp r1, #1 / bxlt lr) handles the case where n < 1, avoiding unnecessary loop setup.

    is_nan() - NaN Detection¶

    vcmp.f32 s0, s0               // Compare s0 with itself
vmrs    APSR_nzcv, fpscr      // Move floating-point flags to CPU flags
mov     r0, #0                // Set r0 = 0 (default return value)
movvs   r0, #1                // If overflow flag set, set r0 = 1
bx      lr                     // Return

    What's happening: A NaN value has the special property that it's not equal to itself. The vcmp.f32 s0, s0 compares the value with itself and sets floating-point condition flags. The vmrs instruction copies those flags into the CPU's condition register (APSR_nzcv). If the value is NaN, the overflow flag (vs condition) is set, so movvs conditionally sets r0 to 1. Otherwise, r0 remains 0.

    divide_by_y() - Simple Division¶

    vdiv.f32 s0, s0, s1           // s0 = s0 / s1
bx      lr                     // Return

    What's happening: This is straightforwardone instruction performs the division. The vdiv.f32 instruction divides s0 by s1 and stores the result back in s0. With -O3 optimization, the compiler doesn't attempt to convert this to a reciprocal multiplication (which would be less accurate anyway).

    algebraic() - Sequential Additions¶

    vadd.f32 s0, s0, s1           // s0 = s0 + s1
vadd.f32 s0, s0, s2           // s0 = s0 + s2
bx      lr                     // Return

    What's happening: The function performs two additions in sequence: first a + b, then the result plus c. This matches the source code's parenthesization (a + b) + c. Even with -O3, the compiler respects the explicit grouping in the source code and doesn't reorder these operations.
