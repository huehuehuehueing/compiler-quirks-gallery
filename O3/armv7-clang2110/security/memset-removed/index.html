    memset-removed¶

    Details

    Source: security/memset-removed.c Compiler: armv7-clang2110 (ARM) Scenario: O3 - Aggressive Optimization (-O3)

    Source Code¶

    memset-removed.c
    /*
 * Security issue: memset() to clear sensitive data is removed
 * as dead code when the buffer is not used afterward.
 *
 * This is a real security vulnerability pattern - passwords
 * and keys may remain in memory after "clearing".
 */
#include <string.h>
#include <stdio.h>

void process_password(const char *password)
{
    char local_copy[64];

    /* Copy password for processing */
    strncpy(local_copy, password, sizeof(local_copy) - 1);
    local_copy[sizeof(local_copy) - 1] = '\0';

    /* Do something observable with the password (simulate auth check) */
    if (local_copy[0] == 's' && local_copy[1] == 'e' &&
        local_copy[2] == 'c' && local_copy[3] == 'r' &&
        local_copy[4] == 'e' && local_copy[5] == 't') {
        puts("Access granted");
    } else {
        puts("Access denied");
    }

    /* Security: clear the password from stack
     * BUG: This memset is often REMOVED by optimizer
     * because local_copy is not used afterward!
     */
    memset(local_copy, 0, sizeof(local_copy));
}

void test_password(void)
{
    process_password("secret");
}

    Explanation¶

    the key insight here is that the memset() call to clear the password has been completely removed by the optimizer, which is the security vulnerability the code comments warn about.

    Stack Setup and Function Prologue¶

    push    {r4, r5, r11, lr}
sub     sp, sp, #64

    The function starts by saving registers and allocating 64 bytes on the stack. In ARM32, r11 is the frame pointer and lr (link register) holds the return address. The sub sp, sp, #64 creates space for the local_copy buffer (which is 64 bytes). This is standard function prologuethe compiler is preparing the stack frame.

    Copying the Password¶

    mov     r1, r0          ; r1 = password pointer (parameter)
mov     r0, sp          ; r0 = address of local_copy buffer
mov     r2, #63         ; r2 = 63 (max bytes to copy)
bl      strncpy         ; call strncpy

    The function sets up parameters for strncpy(): the destination buffer (sp), the source password (r0), and the count (63). By ARM calling convention, the first three parameters go in r0, r1, and r2. After strncpy returns, the password has been copied into the stack buffer.

    Null Terminator¶

    mov     r2, #0
strb    r2, [sp, #63]   ; local_copy[63] = '\0'

    This manually sets the last byte to null terminator (since strncpy might not if the string is exactly 63 characters).

    Password Validation Check¶

    The next section loads individual bytes from the buffer and compares them:

    ldrb    r3, [sp, #5]    ; load byte at offset 5 ('e' from "secret")
ldrb    r2, [sp, #4]    ; load byte at offset 4 ('e')
ldrb    lr, [sp, #1]    ; load byte at offset 1 ('e')
ldrb    r5, [sp, #3]    ; load byte at offset 3 ('r')
ldrb    r1, [sp]        ; load byte at offset 0 ('s')
ldrb    r4, [sp, #2]    ; load byte at offset 2 ('c')

    These loads fetch the characters to check if they spell "secret". The compiler then performs comparisons using cmp and eor (exclusive OR) instructions to check each character. The eor instructions are clever: eor r1, r1, #115 XORs with 115 (ASCII 's'), which results in 0 only if the byte was 's'. If any character doesn't match, r0 is set to point to the "Access denied" string via movne (move if not equal).

    The Critical Optimization: Missing memset()¶

    Here's the crucial part: the memset(local_copy, 0, sizeof(local_copy)) call is completely absent from the assembly.

    The source code explicitly calls memset() to clear the password from memory, but the optimizer removed it because: - The local_copy buffer is never read after the password check - From the compiler's perspective, clearing memory that won't be used again is "dead code" - The optimizer aggressively removes this as unnecessary

    This is the security vulnerability: the password remains in the stack memory even though the code intended to clear it.

    Function Epilogue and puts() Call¶

    bl      puts            ; call puts with r0 pointing to the message
add     sp, sp, #64     ; deallocate stack space
pop     {r4, r5, r11, pc}  ; restore registers and return

    The function calls puts() with the appropriate message string pointer in r0, then cleans up the stack and returns.
