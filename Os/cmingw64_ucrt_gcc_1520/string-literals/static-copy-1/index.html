    static-copy-1¶

    Details

    Source: string-literals/static-copy-1.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    static-copy-1.c
    /*
 * String copy with strcpy - compiler may inline as immediate stores.
 *
 * For small, known-length strings, strcpy becomes direct
 * memory stores with the string data embedded in instructions.
 */
#include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[100];
    strcpy(buf, "ABCDXXXXCCCC");

    /* Make observable to prevent dead code elimination */
    puts(buf);
}

    Explanation¶

    the key insight here is that despite the -Os optimization flag (optimize for size), the compiler is not inlining the string copy as the comment suggests it might. Instead, it's calling the actual strcpy function. Let me walk through what's happening:

    Stack Setup and Buffer Allocation¶

    sub     rsp, 152

    The function starts by subtracting 152 from rsp (the stack pointer register). This allocates space on the stack for local variables. Since buf is declared as char buf[100], you might expect 100 bytes, but 152 bytes are allocated. The extra space accounts for stack alignment requirements on x86-64 (the stack must be 16-byte aligned for function calls) and any other compiler-managed overhead.

    Setting Up Function Arguments¶

    lea     rdx, .LC0[rip]
lea     rcx, 44[rsp]

    These two lea (Load Effective Address) instructions set up the arguments for strcpy. The lea instruction calculates an address without actually accessing memory—it's just arithmetic:

    • lea rdx, .LC0[rip]: Loads the address of the string literal "ABCDXXXXCCCC" into rdx. The [rip] means the address is calculated relative to the instruction pointer (position-independent code).
    • lea rcx, 44[rsp]: Loads the address of the buffer into rcx. This is rsp + 44, which is where the 100-byte buffer buf is located on the stack.

    By the x86-64 calling convention (specifically the Microsoft x64 calling convention used by MinGW), the first two arguments are passed in rcx and rdx respectively. So strcpy(buf, "ABCDXXXXCCCC") becomes strcpy(rcx, rdx).

    Calling strcpy¶

    call    strcpy

    This calls the strcpy function, which copies the string from the address in rdx to the address in rcx. The strcpy function returns a pointer to the destination string in rax (by convention, return values go in rax).

    Preparing for puts()¶

    mov     rcx, rax
call    puts

    The return value from strcpy (now in rax) is moved to rcx, which is the first argument register for the next function call. Then puts is called to print the buffer. This makes the side effect of the string copy observable, preventing the compiler from eliminating it as dead code.

    Cleanup and Return¶

    nop
add     rsp, 152
ret

    The nop (no operation) is a padding instruction, likely added for alignment. Then rsp is restored by adding back the 152 bytes we subtracted at the start, and ret returns from the function.

    Key Observation¶

    Despite -Os optimization, the compiler chose not to inline the string copy as immediate memory stores. Instead, it calls the actual strcpy function. This is likely because the string is 12 bytes long (including the null terminator), which is reasonably sized, and calling strcpy keeps the code compact—important when optimizing for size. An inlined version would embed the string data directly as a series of mov instructions, which could actually be larger in this case.
