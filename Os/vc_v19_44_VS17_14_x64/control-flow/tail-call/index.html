    tail-call¶

    Details

    Source: control-flow/tail-call.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    tail-call.c
    /*
 * Tail call optimization: recursive calls become jumps.
 *
 * When a function's last action is calling another function
 * (or itself), the call can become a jump, saving stack space.
 */

/* Tail-recursive factorial - can be optimized to a loop */
int factorial_tail(int n, int acc)
{
    if (n <= 1)
        return acc;
    /* Last action is the recursive call - tail position */
    return factorial_tail(n - 1, n * acc);
}

int factorial(int n)
{
    return factorial_tail(n, 1);
}

/* NOT tail recursive - multiplication after recursive call */
int factorial_not_tail(int n)
{
    if (n <= 1)
        return 1;
    /* Multiplication happens AFTER the call returns */
    return n * factorial_not_tail(n - 1);
}

/* Mutual tail recursion */
int is_even(unsigned n);
int is_odd(unsigned n);

int is_even(unsigned n)
{
    if (n == 0) return 1;
    return is_odd(n - 1);  /* Tail call */
}

int is_odd(unsigned n)
{
    if (n == 0) return 0;
    return is_even(n - 1);  /* Tail call */
}

    Explanation¶

    the key insight is that this compiler did NOT apply tail call optimization despite the -Os flag, even though the code is structured perfectly for it.

    The Core Issue: Missed Tail Call Optimization¶

    Look at factorial_tail it's a textbook tail-recursive function where the last action is calling itself. In an optimized world, this would become a loop. Instead, you see an actual call factorial_tail instruction, which means the function calls itself recursively. This will consume stack space with each recursion, defeating the purpose of tail recursion.

    How the Functions Work¶

    factorial_tail (lines with factorial_tail PROC):

    The function receives two parameters: - n in register ecx (by x64 calling convention, first integer parameter goes in ecx) - acc in register edx (second parameter goes in edx)

    The assembly stores these on the stack at offsets 48 and 56 from rsp, then:

    1. Base case check: cmp DWORD PTR n$[rsp], 1 compares n to 1
    2. If n 1: jumps to $LN1@factorial_ and returns acc (the accumulated result)
    3. If n > 1:
    4. Multiplies n * acc using imul eax, DWORD PTR acc$[rsp]
    5. Decrements n with dec ecx
    6. Calls itself with the new values
    7. The result comes back in eax

    factorial (wrapper function):

    This simply calls factorial_tail with initial accumulator value of 1. It sets edx = 1 (the accumulator) and passes n in ecx, then delegates to factorial_tail.

    factorial_not_tail (the non-tail-recursive version):

    This one actually demonstrates why tail recursion matters. After the recursive call factorial_not_tail returns, it performs multiplication: imul ecx, eax. This means the function must wait for the recursive call to complete before it can finish its work it cannot simply jump to the next iteration.

    is_even and is_odd (mutual recursion):

    These functions call each other back and forth. is_even checks if n == 0 (returns 1), otherwise calls is_odd(n-1). is_odd does the opposite. Again, these are structured as tail calls but compiled as actual function calls.

    Stack Frame Setup Pattern¶

    Every function follows this pattern: 

    mov DWORD PTR [rsp+8], ecx    ; Save first parameter
sub rsp, 40                    ; Allocate 40 bytes of stack space
...
add rsp, 40                    ; Deallocate
ret 0

    The sub rsp, 40 allocates space for local variables and maintains stack alignment (x64 requires 16-byte alignment). This stack allocation happens even though these functions don't really need local variables it's just how the compiler structured the code.

    Why This Matters¶

    With tail call optimization, factorial_tail(1000, 1) would run in constant stack space (like a loop). Without it, each recursive call adds a stack frame, so you'd run out of stack memory around n 100,000 or so. The -Os flag (optimize for size) apparently didn't enable tail call optimization in this MSVC compiler version, which is a limitation of this particular compiler configuration.
