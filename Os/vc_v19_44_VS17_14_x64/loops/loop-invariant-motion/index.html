    loop-invariant-motion¶

    Details

    Source: loops/loop-invariant-motion.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    loop-invariant-motion.c
    /*
 * Loop-Invariant Code Motion (LICM)
 *
 * The optimizer recognises computations inside a loop whose result
 * does not change across iterations and moves ("hoists") them before
 * the loop.  This avoids redundant work on every iteration.
 *
 * In the example below, `base * scale` is invariant with respect to
 * the loop index i, so the compiler should compute it once and reuse
 * the result inside the loop body.
 */

void fill_scaled(int *arr, int n, int base, int scale)
{
    for (int i = 0; i < n; i++)
    {
        /* base * scale is loop-invariant: it does not depend on i. */
        arr[i] = base * scale + i;
    }
}

    Explanation¶

    the key insight is that the compiler is NOT performing loop-invariant code motion (LICM) here â€” it's recalculating base * scale on every loop iteration instead of hoisting it outside the loop.

    The Core Problem¶

    Look at these instructions inside the loop body: 

    mov     eax, DWORD PTR base$[rsp]
imul    eax, DWORD PTR scale$[rsp]
add     eax, DWORD PTR i$1[rsp]

    The imul (integer multiply) instruction executes every iteration, even though base * scale produces the same result each time. An optimized version would compute this once before the loop and reuse it.

    Stack Frame Setup¶

    The function begins by saving parameters to the stack: 

    mov     DWORD PTR [rsp+32], r9d      # scale parameter
mov     DWORD PTR [rsp+24], r8d      # base parameter
mov     DWORD PTR [rsp+16], edx      # n parameter
mov     QWORD PTR [rsp+8], rcx       # arr parameter (pointer)
sub     rsp, 24                       # allocate space for local variable i

    This follows the x64 calling convention where the first four integer parameters arrive in registers (rcx, rdx, r8, r9). The compiler saves them to the stack for later access. The sub rsp, 24 allocates space for the loop counter variable i.

    Loop Structure¶

    The loop uses a classic three-part pattern:

    Initialization (before the loop): 

    mov     DWORD PTR i$1[rsp], 0        # i = 0
jmp     SHORT $LN4@fill_scale        # jump to condition check

    Loop body ($LN2@fill_scale): 

    mov     eax, DWORD PTR i$1[rsp]      # load i
inc     eax                           # increment it
mov     DWORD PTR i$1[rsp], eax      # store back to stack

    Condition check ($LN4@fill_scale): 

    mov     eax, DWORD PTR n$[rsp]       # load n
cmp     DWORD PTR i$1[rsp], eax      # compare i with n
jge     SHORT $LN3@fill_scale        # exit if i >= n

    Array Storage Operation¶

    Inside the loop, after computing the value: 

    movsxd  rcx, DWORD PTR i$1[rsp]      # sign-extend i to 64-bit (rcx)
mov     rdx, QWORD PTR arr$[rsp]     # load array pointer
mov     DWORD PTR [rdx+rcx*4], eax   # store result at arr[i]

    The movsxd instruction converts the 32-bit loop counter to a 64-bit value (required for address calculation). The final mov uses indexed addressing: [rdx+rcx*4] means "the address of arr plus i times 4 bytes" (since each int is 4 bytes).

    Why This Isn't Optimized¶

    Even with -Os (optimize for size), the compiler chose not to hoist the multiplication. An optimized version would look like: 

    mov     eax, base * scale            # compute once before loop
loop:
    add     eax, i                    # add loop counter
    store to array
    increment i
    check condition

    This would eliminate the expensive imul from the loop body, reducing work per iteration significantly.
