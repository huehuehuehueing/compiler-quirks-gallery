    msvc-guard-cf¶

    Details

    Source: hardening/msvc-guard-cf.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    msvc-guard-cf.c
    /* @gallery-hints
 *   replace-flags: /O2 /guard:cf
 *   compiler-only: vc_v19_44_VS17_14_x64, vc_v19_44_VS17_14_x86
 */

/*
 * MSVC Control Flow Guard (/guard:cf)
 *
 * Control Flow Guard (CFG) is a Windows platform security feature
 * that validates every indirect call at runtime.  The compiler
 * replaces indirect CALL instructions with a dispatch through
 * __guard_dispatch_icall_fptr, which checks the target address
 * against a bitmap of valid call targets before transferring
 * control.
 *
 * Compare with cf-protection.c (GCC/Clang):
 *   - GCC/Clang use Intel CET hardware: ENDBR64 marks valid
 *     landing pads, and the CPU faults on violations.
 *   - MSVC CFG is a software bitmap check performed in user-mode
 *     before every indirect call; it works on all x86/x64 CPUs.
 *
 * Look for: rex_jmp __guard_dispatch_icall_fptr (or
 * __guard_check_icall_fptr) where a plain indirect call would
 * normally appear.
 */

typedef int (*op_fn)(int, int);

int add(int a, int b) { return a + b; }
int sub(int a, int b) { return a - b; }

int apply(op_fn f, int x, int y)
{
    return f(x, y);
}

int demo(void)
{
    return apply(add, 10, 3) + apply(sub, 10, 3);
}

    Explanation¶

    the key insight here is that Control Flow Guard (CFG) replaces normal indirect function calls with a security check, and the compiler has optimized the demo function to a constant.

    The Security Feature: Control Flow Guard¶

    The most important pattern in this code is the rex_jmp __guard_dispatch_icall_fptr instruction in the apply function. This is MSVC's Control Flow Guard—a Windows security feature that validates indirect function calls at runtime.

    Why this matters: When you call a function through a pointer (an "indirect call"), the CPU normally just jumps to whatever address is in that pointer. A malicious attacker could try to redirect that pointer to execute arbitrary code. CFG prevents this by checking the target address against a bitmap of valid call targets before allowing the jump.

    Breaking Down Each Function¶

    add and sub Functions¶

    These are straightforward:

    add:    lea eax, [rcx+rdx]    ; Load Effective Address: eax = rcx + rdx
        ret

    The lea (Load Effective Address) instruction performs addition without actually accessing memory—it just calculates rcx + rdx and stores the result in eax. By convention, the first two integer parameters are passed in rcx and rdx, so this adds the two input values and returns the result in eax (the return value register).

    sub:    sub ecx, edx          ; ecx = ecx - edx
        mov eax, ecx          ; eax = ecx (move result to return register)
        ret

    The sub function subtracts edx from ecx, then moves the result to eax for return.

    apply Function: The CFG Security Check¶

    This is where the security feature appears:

    apply:  mov eax, edx          ; eax = x (second parameter)
        mov r9, rcx           ; r9 = f (function pointer)
        mov ecx, eax          ; ecx = x (prepare first call parameter)
        mov edx, r8d          ; edx = y (prepare second call parameter)
        mov rax, r9           ; rax = f (move function pointer to rax)
        rex_jmp QWORD PTR __guard_dispatch_icall_fptr

    The function is setting up parameters for the indirect call: ecx and edx hold the two arguments (x and y), and rax holds the function pointer. Then instead of a normal call rax, it uses rex_jmp __guard_dispatch_icall_fptr.

    What rex_jmp does: This is a special jump instruction (the rex prefix is a size override). It jumps to the CFG dispatcher function, which: 1. Checks if the address in rax is in the valid call targets bitmap 2. If valid, calls the function and returns the result 3. If invalid, crashes the program (preventing the attack)

    demo Function: Compile-Time Optimization¶

    demo:   mov eax, 20           ; eax = 20
        ret

    This is the most interesting optimization. The compiler completely eliminated the function calls! Here's why:

    • apply(add, 10, 3) returns 10 + 3 = 13
    • apply(sub, 10, 3) returns 10 - 3 = 7
    • 13 + 7 = 20

    With /O2 optimization enabled, the compiler evaluated all of this at compile time and replaced the entire function with a single constant. This is called constant folding—when the compiler can determine that an expression always produces the same value, it computes that value once and hardcodes it.

    Key Observations¶

    • CFG adds security overhead: The rex_jmp __guard_dispatch_icall_fptr replaces what would normally be a direct call rax instruction. This dispatcher function performs a bitmap lookup before allowing the call, adding a small runtime cost for security.
    • Optimization still applies: Even with CFG enabled, the compiler aggressively optimizes—notice how demo becomes just a constant return value.
    • Parameter passing convention: The x64 calling convention uses rcx, rdx, r8, and r9 for the first four integer parameters,
