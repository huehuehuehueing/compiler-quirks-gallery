    wrapv-vs-ubsan¶

    Details

    Source: security/wrapv-vs-ubsan.c Compiler: cg152 (x86/x64) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    wrapv-vs-ubsan.c
    /* @gallery-hints
 *   extra-flags: -fwrapv
 */

/*
 * -fwrapv: well-defined signed integer overflow
 *
 * By default, signed integer overflow is undefined behaviour in C,
 * and compilers exploit this for optimisation (e.g. assuming i+1 > i).
 * With -fwrapv, signed overflow wraps around in two's complement,
 * which is well-defined.
 *
 * Compare the assembly with and without -fwrapv: the optimizer may
 * remove overflow checks or fold comparisons differently when
 * overflow is undefined.
 */

int will_overflow(int x)
{
    /* Without -fwrapv the compiler may assume this never wraps,
     * optimising the comparison to always-true. */
    if (x + 1 > x)
        return 1;
    return 0;
}

int abs_safe(int x)
{
    /* With well-defined wrapping, the compiler must handle INT_MIN. */
    if (x < 0)
        x = -x;
    return x;
}

    Explanation¶

    the key insight is that the -fwrapv flag changes how the compiler handles signed integer overflow, making it well-defined instead of undefined behavior. This affects both functions differently.

    will_overflow Function¶

    The compiler has optimized this function to check a single condition:

    xor     eax, eax           # Clear eax to 0
cmp     edi, 2147483647    # Compare input with INT_MAX (2147483647)
setne   al                 # Set al to 1 if not equal, 0 if equal
ret

    What's happening: The function returns 1 if the input is not equal to INT_MAX (the largest 32-bit signed integer), and 0 otherwise.

    Why this works: With -fwrapv, the compiler knows that x + 1 > x is only false when x is INT_MAX. When you add 1 to INT_MAX, it wraps around to INT_MIN (the most negative number), which is definitely not greater than INT_MAX. For any other value, x + 1 will be greater than x.

    The setne instruction (set if not equal) is a conditional instruction that sets the low byte of eax to 1 or 0 based on the comparison result. Since we cleared eax first with xor, the upper 32 bits are zeroed, giving us a clean return value.

    abs_safe Function¶

    This function implements absolute value with proper handling of the edge case:

    mov     eax, edi           # Copy input to eax (return register)
neg     eax                # Negate eax (compute -x)
cmovs   eax, edi           # If sign flag set, move original input back to eax
ret

    What's happening: The function negates the input, then conditionally moves the original value back if the negation produced a negative result.

    Why this matters: The neg instruction sets the CPU's sign flag if the result is negative. The cmovs (conditional move if sign) instruction checks this flag—if the negation was negative, it means the original input was INT_MIN (the only value whose negation is still negative in two's complement). In that case, we keep the original value rather than returning the undefined result of negating INT_MIN.

    Register conventions: By convention in the amd64 calling convention, the first integer argument is passed in edi (the 32-bit portion of rdi), and the return value goes in eax.

    The -Os optimization flag (optimize for size) keeps both functions compact—no unnecessary stack operations or branches, just the essential comparisons and conditional moves.
