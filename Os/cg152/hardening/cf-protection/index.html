    cf-protection¶

    Details

    Source: hardening/cf-protection.c Compiler: cg152 (x86/x64) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    cf-protection.c
    /* @gallery-hints
 *   extra-flags: -fcf-protection=full
 *   compiler-only: cg152, clang1910
 */

/*
 * Control-Flow Enforcement (CET / IBT + Shadow Stack)
 *
 * -fcf-protection=full enables Intel CET instrumentation:
 *   - IBT (Indirect Branch Tracking): indirect calls/jumps must land
 *     on an ENDBR64 instruction, otherwise a #CP fault occurs.
 *   - Shadow Stack: CALL pushes return addresses to a hardware shadow
 *     stack; RET checks both stacks match, preventing ROP attacks.
 *
 * Look for: ENDBR64 at function entry points and at indirect branch
 * targets.
 */

typedef int (*op_fn)(int, int);

int add(int a, int b) { return a + b; }
int sub(int a, int b) { return a - b; }

int apply(op_fn f, int x, int y)
{
    return f(x, y);
}

int demo(void)
{
    return apply(add, 10, 3) + apply(sub, 10, 3);
}

    Explanation¶

    the key insight here is that every function starts with endbr64, which is a security instruction that prevents attackers from jumping into the middle of functions. This is the most important pattern to understand.

    The Security Foundation: endbr64¶

    The -fcf-protection=full flag enables Intel Control-Flow Enforcement Technology (CET), which adds security checks to prevent certain types of attacks. The endbr64 instruction (End Branch 64-bit) acts as a "landing pad" marker. When code makes an indirect call or jump (like calling through a function pointer), the CPU verifies that execution lands on an endbr64 instruction. If it doesn't, the CPU raises a fault, stopping the attack.

    Think of it like a security checkpoint: indirect branches must land on these marked locations, or the program halts.

    The Simple Functions: add and sub¶

    Both functions follow the same pattern:

    add function: 

    endbr64                    # Security checkpoint
lea     eax, [rdi+rsi]     # Load Effective Address: eax = rdi + rsi
ret                        # Return

    The lea (Load Effective Address) instruction performs address calculation without memory access. Here it computes rdi + rsi and stores the result in eax. By convention, the first two integer parameters arrive in rdi and rsi, and the return value goes in eax. So this adds the two parameters and returns the result in one instruction.

    sub function: 

    endbr64                    # Security checkpoint
mov     eax, edi           # eax = first parameter (edi)
sub     eax, esi           # eax = eax - second parameter (esi)
ret                        # Return

    This copies the first parameter into eax, subtracts the second parameter, and returns. It's slightly less efficient than add (which used lea to do both operations at once), but the compiler chose this approach under -Os (optimize for size).

    The Function Pointer Handler: apply¶

    This is where the security mechanism becomes visible:

    endbr64                    # Security checkpoint
mov     rax, rdi           # rax = function pointer (first parameter)
mov     edi, esi           # edi = first argument (move from esi)
mov     esi, edx           # esi = second argument (move from edx)
jmp     rax                # Jump to the function pointer

    The apply function receives a function pointer in rdi (first parameter), and two integers in esi and edx (second and third parameters). It: 1. Saves the function pointer to rax 2. Rearranges the arguments into the correct registers for the target function (edi and esi) 3. Uses jmp rax to jump to the function pointer

    The critical detail: jmp rax is an indirect jump (jumping to an address stored in a register). The CPU will verify that rax points to an endbr64 instruction. Since both add and sub start with endbr64, the jump succeeds. If an attacker tried to jump to a different location, the CPU would fault.

    The Orchestrator: demo¶

    endbr64                    # Security checkpoint
push    rbx                # Save rbx (callee-saved register)
mov     edx, 3             # Third parameter: 3
mov     esi, 10            # Second parameter: 10
mov     edi, OFFSET FLAT:add    # First parameter: address of add function
call    apply              # Call apply(add, 10, 3)
mov     edx, 3             # Third parameter: 3
mov     esi, 10            # Second parameter: 10
mov     edi, OFFSET FLAT:sub    # First parameter: address of sub function
mov     ebx, eax           # Save first result in rbx
call    apply              # Call apply(sub, 10, 3)
add     eax, ebx           # eax = second result + first result
pop     rbx                # Restore rbx
ret                        # Return

    The demo function: 1. Saves rbx (a callee-saved register, meaning demo must restore it before returning)
