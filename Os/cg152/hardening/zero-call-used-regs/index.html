    zero-call-used-regs¶

    Details

    Source: hardening/zero-call-used-regs.c Compiler: cg152 (x86/x64) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    zero-call-used-regs.c
    /* @gallery-hints
 *   extra-flags: -fzero-call-used-regs=used
 *   compiler-only: cg152, clang1910
 */

/*
 * Zero Call-Used Registers
 *
 * -fzero-call-used-regs=used clears every register that the function
 * actually used before returning. This mitigates information leakage
 * through registers and makes ROP gadgets less useful because
 * attackers cannot rely on register contents at a ret instruction.
 *
 * Look for: xor or zero instructions on registers (eax, ecx, edx,
 * xmm0, etc.) immediately before the ret instruction.
 */

int compute(int a, int b, int c)
{
    int result = a * b + c;
    return result;
}

long long widen(int x)
{
    return (long long)x * x;
}

    Explanation¶

    the key insight here is that this code demonstrates a security feature that clears registers before returning from functions.

    The Security Feature: -fzero-call-used-regs=used¶

    The most important pattern to notice is the xor instructions appearing right before each ret instruction. These aren't part of the actual computation—they're clearing registers to prevent information leakage. When a function returns, attackers could potentially read sensitive data left in registers. This flag forces the compiler to zero out any registers the function actually used.

    compute() Function Breakdown¶

    imul    edi, esi          # Multiply: edi = a * b
lea     eax, [rdi+rdx]    # Add: eax = (a*b) + c
xor     edx, edx          # Clear edx (security)
xor     esi, esi          # Clear esi (security)
xor     edi, edi          # Clear edi (security)
ret

    The computation: - imul edi, esi: Multiplies the first parameter (a in edi) by the second parameter (b in esi), storing the result back in edi. By convention in the amd64 calling convention, integer parameters are passed in rdi, rsi, rdx, etc. - lea eax, [rdi+rdx]: Uses lea (Load Effective Address) to add the third parameter (c in rdx) to our multiplication result. Despite its name, lea doesn't access memory—it just calculates an address and stores it. Here it's being used as a fast addition: eax = edi + edx. The result goes into eax because by convention, integer return values are passed back in rax/eax.

    The security cleanup: - The three xor instructions zero out edx, esi, and edi because the function used these registers to hold the input parameters. The xor instruction (XOR a register with itself) is the fastest way to zero a register.

    widen() Function Breakdown¶

    movsx   rax, edi          # Sign-extend: rax = (long long)x
imul    rax, rax          # Multiply: rax = x * x
xor     edi, edi          # Clear edi (security)
ret

    The computation: - movsx rax, edi: Sign-extends the 32-bit input parameter (x in edi) to a full 64-bit value in rax. This is necessary because we're casting int to long long—we need to preserve the sign bit and fill the upper 32 bits appropriately. - imul rax, rax: Multiplies the 64-bit value by itself, storing the result back in rax (the return value register).

    The security cleanup: - xor edi, edi clears the input parameter register before returning.

    Key Observations¶

    The -Os flag (optimize for size) combined with -fzero-call-used-regs=used creates compact code that still includes security hardening. Notice that lea is used for addition instead of separate add instructions—this is a size optimization. The register clearing adds a small security cost (a few extra instructions) but prevents attackers from reading sensitive values that might remain in registers after the function returns.
