    tail-call¶

    Details

    Source: control-flow/tail-call.c Compiler: mipsg1520 (MIPS) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    tail-call.c
    /*
 * Tail call optimization: recursive calls become jumps.
 *
 * When a function's last action is calling another function
 * (or itself), the call can become a jump, saving stack space.
 */

/* Tail-recursive factorial - can be optimized to a loop */
int factorial_tail(int n, int acc)
{
    if (n <= 1)
        return acc;
    /* Last action is the recursive call - tail position */
    return factorial_tail(n - 1, n * acc);
}

int factorial(int n)
{
    return factorial_tail(n, 1);
}

/* NOT tail recursive - multiplication after recursive call */
int factorial_not_tail(int n)
{
    if (n <= 1)
        return 1;
    /* Multiplication happens AFTER the call returns */
    return n * factorial_not_tail(n - 1);
}

/* Mutual tail recursion */
int is_even(unsigned n);
int is_odd(unsigned n);

int is_even(unsigned n)
{
    if (n == 0) return 1;
    return is_odd(n - 1);  /* Tail call */
}

int is_odd(unsigned n)
{
    if (n == 0) return 0;
    return is_even(n - 1);  /* Tail call */
}

    Explanation¶

    the key insight is that the compiler has transformed recursive function calls into loops and jumps, eliminating the stack overhead that normally comes with recursion.

    The Core Optimization: Tail Call Elimination¶

    The most important pattern here is that tail calls (recursive calls at the end of a function) have been converted to jumps and loops instead of actual function calls. This saves stack space and avoids the overhead of setting up new stack frames.

    factorial_tail and factorial: Converted to Loops¶

    Let's trace through factorial_tail:

    move    $2,$5          # $2 = acc (second parameter, in $5)
$L3:
slt     $3,$4,2        # $3 = (n < 2)? Set $3 to 1 if true
bne     $3,$0,$L5      # If n >= 2, skip to $L5 (exit)
mult    $2,$4          # Multiply: acc * n
mflo    $2             # Move result from multiply to $2
b       $L3            # Jump back to loop start
addiu   $4,$4,-1       # Decrement n (delay slot)
$L5:
jr      $31            # Return

    What's happening: Instead of making recursive calls, the compiler created a loop: - $2 holds the accumulator (the result being built up) - $4 holds n (the countdown value) - The loop multiplies $2 * $4, decrements n, and jumps back - When n < 2, it exits and returns $2

    This is loop unrolling in reversethe compiler recognized that the tail-recursive pattern is mathematically equivalent to a simple loop and generated that instead.

    The factorial function does the same thing but initializes the accumulator to 1 first (li $2,1), then enters the same loop.

    factorial_not_tail: Also Optimized (Surprisingly!)¶

    Even though the source code says this function is "NOT tail recursive" (because multiplication happens after the recursive call), the compiler still optimized it to a loop:

    li      $2,1           # Start with result = 1
$L13:
slt     $3,$4,2        # Check if n < 2
bne     $3,$0,$L15     # Exit if true
mult    $2,$4          # Multiply result * n
mflo    $2             # Store result
b       $L13           # Loop back
addiu   $4,$4,-1       # Decrement n

    This works because mathematically, n * factorial(n-1) produces the same result as accumulating multiplications in a loop. The compiler is smart enough to recognize this pattern and optimize it anyway.

    Mutual Tail Recursion: Converted to Jumps¶

    The is_even and is_odd functions show a different patternmutual tail recursion converted to jumps:

    is_odd(unsigned int):
beq     $4,$0,$L17     # If n == 0, jump to exit
nop                    # Delay slot (no operation)
j       is_even(unsigned int)  # Jump to is_even
addiu   $4,$4,-1       # Decrement n (delay slot)
$L17:
jr      $31            # Return
move    $2,$0          # Set return value to 0

    Key insight: Instead of calling is_even, the code uses j (jump) to transfer control directly. The parameter n is decremented in the delay slot, and control jumps to is_even without setting up a new stack frame.

    The is_even function does the same thing in reverseit jumps to is_odd when n != 0.

    Why This Matters¶

    Without tail call optimization, each recursive call would: - Push a return address onto the stack - Create a new stack frame - Eventually overflow the stack for large inputs

    With tail call optimization, the code: - Reuses the same stack frame - Uses jumps instead of calls - Runs in constant stack space, no matter how deep the recursion

    Calling Convention Details¶

    In MIPS, by convention: - $4 holds the first parameter (n) - $5 holds the second parameter
