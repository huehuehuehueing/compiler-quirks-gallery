    loop-interchange¶

    Details

    Source: loops/loop-interchange.c Compiler: mipsg1520 (MIPS) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    loop-interchange.c
    /*
 * Loop Interchange
 *
 * When iterating over a 2D array, the order of the loop indices
 * determines memory access patterns.  In C (row-major layout),
 * iterating the inner loop over columns and the outer loop over
 * rows yields sequential access.  Swapping the loops ("interchange")
 * can improve cache locality when the original order would cause
 * stride-N accesses.
 *
 * Here the "bad" order (column-major traversal on a row-major array)
 * is presented; the compiler may interchange the loops at -O3.
 */
#define N 128

void column_sum(int mat[N][N], int result[N])
{
    /* Column-major traversal: inner loop strides by N ints.
     * The compiler may swap i and j for better locality. */
    for (int j = 0; j < N; j++)
    {
        result[j] = 0;
        for (int i = 0; i < N; i++)
        {
            result[j] += mat[i][j];
        }
    }
}

    Explanation¶

    the key insight is that this code implements a 2D array column summation using nested loops, and the assembly reveals how the compiler manages memory access patterns and loop control.

    Function Setup and Register Allocation¶

    The function receives two parameters: - $4: pointer to the 2D array mat[128][128] - $5: pointer to the result array

    The compiler uses these registers throughout: - $2: outer loop counter (column index j), starts at 0 - $10: constant 128 (loop limit) - $9: constant 512 (128 × 4 bytes, used to check when j reaches 128) - $8: address of current result element - $6: address of current matrix element being read - $7: accumulator for the column sum - $3: inner loop counter (row index i) - $11: temporary register for loading matrix values

    Outer Loop Structure ($L6 label)¶

    addu    $8,$5,$2          # Calculate address: result[j]

    This calculates where to store the current column's sum. Since $2 holds j*4 (the byte offset), adding it to $5 (result array base) gives the address of result[j].

    sw      $0,0($8)          # Initialize result[j] = 0

    Each column sum starts at zero.

    Inner Loop Structure ($L2 label)¶

    The inner loop accumulates all values in column j:

    addu    $6,$4,$2          # Start of column j in matrix
lw      $11,0($6)         # Load mat[i][j]
addu    $7,$7,$11         # Add to running sum
sw      $7,0($8)          # Store updated sum back
addiu   $6,$6,512         # Move to next row (stride by 512 bytes)

    The critical detail here is addiu $6,$6,512: this adds 512 bytes (128 integers × 4 bytes per integer) to move from mat[i][j] to mat[i+1][j]. This is a stride-512 memory access pattern, which means the processor jumps far through memory for each element—potentially causing cache misses.

    The inner loop counter $3 increments from 0 to 128, and the loop branches back at $L2 until all 128 rows are processed.

    Outer Loop Continuation¶

    addiu   $2,$2,4           # j++  (increment by 4 bytes)
bne     $2,$9,$L6         # Continue if j < 128
addu    $8,$5,$2          # Prepare next result address (delay slot)

    The outer loop increments $2 by 4 (moving to the next column) and repeats until $2 equals 512 (128 columns × 4 bytes).

    Key Observation: Loop Interchange Not Applied¶

    The comment in the source code mentions that the compiler may interchange the loops at -O3 for better cache locality. However, with -Os (optimize for size), the compiler kept the original column-major traversal. This means the code performs stride-512 memory accesses—jumping across the entire matrix width for each element—which is inefficient for cache performance. An interchanged version would iterate rows in the outer loop and columns in the inner loop, creating sequential memory access instead.
