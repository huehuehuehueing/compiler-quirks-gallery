    auto-vectorize¶

    Details

    Source: simd/auto-vectorize.c Compiler: armv8-clang2110 (ARM) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    auto-vectorize.c
    /*
 * Auto-vectorization: loops converted to SIMD operations.
 *
 * Compiler transforms scalar loops into vector operations
 * (SSE, AVX, NEON, etc.) processing multiple elements at once.
 */

/* Simple loop - prime candidate for vectorization */
void add_arrays(float *dst, const float *a, const float *b, int n)
{
    for (int i = 0; i < n; i++) {
        dst[i] = a[i] + b[i];
    }
    /* With -O3: becomes SIMD adds processing 4-8 floats at once */
}

/* Reduction - trickier to vectorize but often done */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++) {
        sum += arr[i];
    }
    return sum;
    /* May use horizontal add instructions or partial sums */
}

/* Loop with dependency - CANNOT be vectorized */
void fibonacci_array(int *arr, int n)
{
    arr[0] = 0;
    arr[1] = 1;
    for (int i = 2; i < n; i++) {
        /* Each iteration depends on previous two - serial */
        arr[i] = arr[i-1] + arr[i-2];
    }
}

/* Conditional in loop - may use masked operations */
void clamp_array(float *arr, int n, float min, float max)
{
    for (int i = 0; i < n; i++) {
        if (arr[i] < min) arr[i] = min;
        else if (arr[i] > max) arr[i] = max;
    }
}

    Explanation¶

    the key insight is that none of these loops are vectorized despite being compiled with -Os (optimize for size). Let me walk through why and what each function does.

    Why No Vectorization?¶

    The -Os flag prioritizes code size over speed. SIMD (Single Instruction Multiple Data) vectorizationprocessing multiple elements in one instructionwould require larger code with special setup. The compiler chose scalar operations (processing one element at a time) to keep the code compact.

    add_arrays Function¶

    This function adds two arrays element-by-element.

    The loop structure: 

    cmp     w3, #1          // Compare n with 1
b.lt    .LBB0_3         // If n < 1, skip loop and return
mov     w8, w3          // Copy n into w8 (loop counter)
.LBB0_2:                // Loop start
    ldr     s0, [x1], #4    // Load float from array a, advance pointer by 4 bytes
    subs    x8, x8, #1      // Decrement counter and set flags
    ldr     s1, [x2], #4    // Load float from array b, advance pointer by 4 bytes
    fadd    s0, s0, s1      // Add the two floats
    str     s0, [x0], #4    // Store result to dst, advance pointer by 4 bytes
    b.ne    .LBB0_2         // If counter != 0, loop again

    Key points: - s0 and s1 are 32-bit floating-point registers (the 's' means single-precision) - The [x1], #4 syntax means "load from address in x1, then increment x1 by 4 bytes" (post-indexed addressing) - Each iteration processes exactly one float (4 bytes) - A vectorized version would load 4 floats at once using NEON instructions and process them together

    sum_array Function¶

    This function sums all elements in an array.

    The setup and loop: 

    movi    d0, #0000000000000000  // Initialize d0 (64-bit register) to 0.0
cmp     w1, #1                 // Compare n with 1
b.lt    .LBB1_3                // If n < 1, skip loop
mov     w8, w1                 // Copy n into loop counter
.LBB1_2:                        // Loop start
    ldr     s1, [x0], #4       // Load next float from array
    subs    x8, x8, #1         // Decrement counter
    fadd    s0, s0, s1         // Add to running sum
    b.ne    .LBB1_2            // Loop if more elements

    Key points: - d0 holds the accumulating sum (initialized to 0.0) - Each iteration adds one float to the sum - The fadd instruction creates a data dependency: each iteration must wait for the previous addition to complete before starting the next one - This is why reductions are hard to vectorizeyou can't easily parallelize when each step depends on the previous result

    fibonacci_array Function¶

    This function fills an array with Fibonacci numbersa classic example of non-vectorizable code.

    The setup: 

    adrp    x8, .LCPI2_0           // Load address of constant data (0, 1)
ldr     d0, [x8, :lo12:.LCPI2_0]  // Load 64-bit value containing [0, 1]
str     d0, [x0]               // Store to arr[0] and arr[1]
cmp     w1, #3                 // If n < 3, we're done
