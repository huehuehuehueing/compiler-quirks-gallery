    stack-clash¶

    Details

    Source: hardening/stack-clash.c Compiler: armv8-clang2110 (ARM) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    stack-clash.c
    /* @gallery-hints
 *   extra-flags: -fstack-clash-protection
 *   compiler-exclude: vc_v19_44_VS17_14_x64, vc_v19_44_VS17_14_x86, avrg1520
 */

/*
 * Stack Clash Protection
 *
 * -fstack-clash-protection inserts probes into large stack
 * allocations so that every page of the stack is touched in order.
 * This prevents "stack clash" attacks where a large allocation
 * jumps the guard page and overlaps the heap.
 *
 * Look for: loop-based or sequential page-sized probes (usually
 * stores to rsp at 4096-byte intervals) when the function has a
 * large local frame.
 */

void large_frame(int n)
{
    volatile char buf[16384];  /* larger than one page */
    /* Touch a few spots so the compiler keeps the buffer. */
    buf[0] = 'A';
    buf[4096] = 'B';
    buf[8192] = 'C';
    buf[16383] = 'D';
    (void)n;
}

    Explanation¶

    the key insight here is that the compiler has completely optimized away the large 16KB buffer allocation despite the -fstack-clash-protection flag being enabled.

    What You'd Normally Expect¶

    With a 16,384-byte local buffer and stack clash protection enabled, you'd typically see: 1. A large sub sp, sp, #16384 instruction to allocate the buffer 2. A loop or series of str instructions that "probe" (touch) each 4096-byte page of the stack to prevent stack clash attacks

    What Actually Happens Here¶

    The compiler recognizes that the buffer is only used to store four individual characters at specific offsets (buf[0], buf[4096], buf[8192], buf[16383]). Since the buffer is volatile, the compiler must preserve the stores, but it doesn't need to allocate the full 16KB on the stack.

    The actual stack frame is only 16 bytes (see sub sp, sp, #16), not 16,384 bytes. Here's what the function does:

    sub     sp, sp, #16          // Allocate 16 bytes on the stack
mov     w8, #65              // Load 'A' (ASCII 65) into w8
mov     w9, #66              // Load 'B' (ASCII 66) into w9
strb    w8, [sp, #12]        // Store 'A' at sp+12 (buf[0])
mov     w8, #67              // Load 'C' (ASCII 67) into w8
strb    w9, [sp, #8]         // Store 'B' at sp+8 (buf[4096])
strb    w8, [sp, #4]         // Store 'C' at sp+4 (buf[8192])
mov     w8, #68              // Load 'D' (ASCII 68) into w8
strb    w8, [sp], #16        // Store 'D' at sp, then restore sp
ret                          // Return

    Key Details¶

    • Register w8 and w9: These are 32-bit registers (the lower 32 bits of 64-bit registers x8 and x9). The compiler uses them to hold the character values before storing.
    • strb instruction: This stores a single byte (the 'b' suffix means "byte"). The compiler only needs to store the actual bytes being written, not allocate space for the entire buffer.
    • [sp, #offset] addressing: This notation means "the address at sp plus an offset". The compiler maps the four buffer accesses to different offsets within the small 16-byte stack frame.
    • strb w8, [sp], #16: This is a post-indexed store—it stores the byte, then automatically increments sp by 16, effectively deallocating the frame in one instruction.

    Why Stack Clash Protection Doesn't Appear¶

    Stack clash protection probes are only needed when allocating large contiguous stack space (typically more than one page, which is 4096 bytes). Since the compiler optimized this down to a 16-byte allocation, no probing is necessary—the allocation is small enough that it can't "clash" with the heap.
