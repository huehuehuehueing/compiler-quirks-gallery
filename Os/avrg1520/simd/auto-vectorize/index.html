    auto-vectorize¶

    Details

    Source: simd/auto-vectorize.c Compiler: avrg1520 (AVR) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    auto-vectorize.c
    /*
 * Auto-vectorization: loops converted to SIMD operations.
 *
 * Compiler transforms scalar loops into vector operations
 * (SSE, AVX, NEON, etc.) processing multiple elements at once.
 */

/* Simple loop - prime candidate for vectorization */
void add_arrays(float *dst, const float *a, const float *b, int n)
{
    for (int i = 0; i < n; i++) {
        dst[i] = a[i] + b[i];
    }
    /* With -O3: becomes SIMD adds processing 4-8 floats at once */
}

/* Reduction - trickier to vectorize but often done */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++) {
        sum += arr[i];
    }
    return sum;
    /* May use horizontal add instructions or partial sums */
}

/* Loop with dependency - CANNOT be vectorized */
void fibonacci_array(int *arr, int n)
{
    arr[0] = 0;
    arr[1] = 1;
    for (int i = 2; i < n; i++) {
        /* Each iteration depends on previous two - serial */
        arr[i] = arr[i-1] + arr[i-2];
    }
}

/* Conditional in loop - may use masked operations */
void clamp_array(float *arr, int n, float min, float max)
{
    for (int i = 0; i < n; i++) {
        if (arr[i] < min) arr[i] = min;
        else if (arr[i] > max) arr[i] = max;
    }
}

    Explanation¶

    Key Insight: This is unoptimized code compiled with -Os (size optimization) for AVR microcontrollers. The AVR architecture has no SIMD (vector) instructions, so despite the source code comments about vectorization, all operations are scalar (single-element) and processed one at a time.

    Understanding AVR Basics¶

    AVR is an 8-bit microcontroller architecture. Unlike x86 or ARM processors, AVR has: - 8-bit registers (r0-r31), but operations on larger values require multiple registers - No floating-point hardware — floating-point math uses library functions like __addsf3 (add single-precision float) - No SIMD instructions — vectorization is impossible on this architecture - Limited registers — the compiler must carefully manage which values stay in registers vs. stack memory

    add_arrays(float*, float const*, float const*, int):
    push r12
    push r13
    ...
    mov r12,__zero_reg__    // r12:r13 = loop counter i (initialized to 0)

.L3:

    ldd r30,Y+3
    ldd r31,Y+4
    ld r22,Z+
    ld r23,Z+
    ld r24,Z+
    ld r25,Z+

    // Load b[i] (4 bytes for float)
    mov r30,r16
    mov r31,r17
    ld r18,Z+
    ld r19,Z+
    ld r20,Z+
    ld r21,Z+

    rcall __addsf3          // Call floating-point add library function

    // Store result to dst[i]
    mov r30,r14
    mov r31,r15
    st Z+,r22
    st Z+,r23
    st Z+,r24
    st Z+,r25

    // Increment loop counter
    ldi r31,-1
    sub r12,r31             // i++ (subtract -1 = add 1)
    sbc r13,r31
    rjmp .L2

    What's happening: - Loop counter i lives in r12:r13 (16-bit value using two 8-bit registers) - Each iteration loads 4 bytes from array a, 4 bytes from array b, calls the floating-point library function __addsf3, then stores 4 bytes to dst - No vectorization — processes one float (4 bytes) per loop iteration, not multiple floats at once - Stack usage — pointers and the array size are stored on the stack and reloaded each iteration (inefficient, but typical for -Os)
