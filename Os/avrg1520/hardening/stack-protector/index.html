    stack-protector¶

    Details

    Source: hardening/stack-protector.c Compiler: avrg1520 (AVR) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    stack-protector.c
    /* @gallery-hints
 *   extra-flags: -fstack-protector-all
 *   compiler-exclude: vc_v19_44_VS17_14_x64, vc_v19_44_VS17_14_x86
 */

/*
 * Stack Protector (Stack Canary)
 *
 * With -fstack-protector-all the compiler inserts a canary value
 * between local variables and the saved return address. Before the
 * function returns, the canary is checked; if it was overwritten
 * (e.g. by a buffer overflow), the program aborts.
 *
 * Look for: canary load from %fs:0x28 (or %gs:0x14 on 32-bit),
 * comparison before the function epilogue, and a call to
 * __stack_chk_fail on mismatch.
 */
#include <string.h>

void copy_input(const char *input)
{
    char buf[64];
    strcpy(buf, input);   /* intentionally unsafe for demonstration */
    (void)buf;
}

    Explanation¶

    the key insight here is that this code demonstrates stack frame setup and teardown on AVR, with special attention to how the compiler manages the stack pointer safely.

    Stack Frame Setup (Lines 1-13)¶

    The function starts by creating space for local variables:

    push r28
push r29

    These instructions save the frame pointer registers (r28 and r29). Think of the frame pointer as a bookmark that marks where the current function's local data begins. By saving the old values, the function can restore them when it returns.

    in r28,__SP_L__
in r29,__SP_H__

    These read the current stack pointer (the address of the top of the stack) into r28 and r29. The stack pointer is split across two 8-bit I/O ports because AVR is an 8-bit architecture.

    subi r28,64
sbc r29,__zero_reg__

    These subtract 64 from the stack pointer, creating 64 bytes of space for the buf[64] array. The sbc (subtract with carry) handles the upper byte of the 16-bit subtraction.

    The next three instructions update the actual stack pointer in the I/O ports:

    in __tmp_reg__,__SREG__
cli
out __SP_H__,r29
out __SREG__,__tmp_reg__
out __SP_L__,r28

    This pattern is important: it disables interrupts (cli), updates the stack pointer, then restores the interrupt state. This prevents an interrupt from occurring between updating the high and low bytes of the stack pointer, which could corrupt the stack.

    Function Body (Lines 14-18)¶

    mov r22,r24
mov r23,r25
mov r24,r28
mov r25,r29
adiw r24,1
rcall strcpy

    By AVR calling convention, the input parameter (input pointer) arrives in r24:r25 (a 16-bit register pair). The code: 1. Copies it to r22:r23 (the second parameter for strcpy) 2. Loads the buffer address into r24:r25 by copying the frame pointer and adding 1 3. Calls strcpy to copy the input string into the buffer

    The adiw r24,1 adds 1 to skip past the canary value that the stack protector inserts.

    Stack Frame Teardown (Lines 19-27)¶

    subi r28,-64
sbci r29,-1

    These restore the stack pointer by adding 64 back (subtracting -64). This mirrors the setup phase.

    in __tmp_reg__,__SREG__
cli
out __SP_H__,r29
out __SREG__,__tmp_reg__
out __SP_L__,r28

    Again, the interrupt-safe stack pointer update pattern.

    pop r29
pop r28
ret

    Finally, restore the saved frame pointer registers and return to the caller.

    Key Observations¶

    • Stack protector overhead: The -fstack-protector-all flag causes the compiler to allocate 66 bytes instead of 64 (note .L__stack_usage = 66), with the extra 2 bytes being the canary value. However, the actual canary checking code isn't visible here—it would typically appear before the function returns.
    • Interrupt safety: The repeated pattern of saving/restoring the status register and disabling interrupts during stack pointer updates is crucial on AVR, where the stack pointer is a 16-bit value split across two 8-bit I/O ports.
    • Frame pointer usage: r28:r29 serve as the frame pointer pair, allowing the function to reference local variables at known offsets from this pointer.
