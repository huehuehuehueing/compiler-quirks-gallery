    inline-expansion¶

    Details

    Source: control-flow/inline-expansion.c Compiler: avrg1520 (AVR) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    inline-expansion.c
    /*
 * Function inlining: small functions are expanded at call site.
 *
 * Eliminates call overhead and enables further optimizations
 * by exposing the function body to the caller's context.
 */

/* Tiny function - almost certainly inlined */
static inline int square(int x)
{
    return x * x;
}

/* Using static helps inlining (internal linkage) */
static int cube(int x)
{
    return x * x * x;
}

/* Wrapper that uses inlined functions */
int compute(int x)
{
    /* After inlining, this becomes:
     * return (x * x) + (x * x * x);
     * Which may further simplify to:
     * return x * x * (1 + x);
     */
    return square(x) + cube(x);
}

/* Recursive functions are NOT inlined (infinite expansion) */
int fibonacci(int n)
{
    if (n <= 1) return n;
    return fibonacci(n-1) + fibonacci(n-2);
}

/* Large functions may not be inlined (code size cost) */
int large_function(int x)
{
    int result = 0;
    for (int i = 0; i < 100; i++) {
        result += x * i;
        result ^= (result >> 3);
        result += (result << 5);
    }
    return result;
}

    Explanation¶

    the key insight is that this code demonstrates function inlining - where small functions get expanded directly into their callers, eliminating function call overhead.

    The compute Function: Inlining in Action¶

    The most important pattern here is that square() and cube() are completely inlined - they don't exist as separate functions. Instead, their code is expanded directly into compute():

    compute(int):
        mov r18,r24          # Save input x to r18
        mov r19,r25          # (r24:r25 holds the 16-bit input)
        mov r22,r24          # Set up for first multiplication
        mov r23,r25
        rcall __mulhi3       # Call multiply: x * x
        mov r30,r24          # Save result (x²) to r30:r31
        mov r31,r25
        mov r24,r18          # Restore original x
        mov r25,r19
        mov r22,r30          # Set up: x * (x²)
        mov r23,r31
        rcall __mulhi3       # Call multiply: x * x²
        add r24,r30          # Add the two results
        adc r25,r31          # (x² + x³)
        ret

    Why this matters: Instead of calling square() and cube() as separate functions (which would require pushing/popping registers and jumping), the compiler directly computes x * x and x * x * x inline. The -Os flag (optimize for size) still allows this because these functions are tiny.

    Register Usage Pattern¶

    The AVR calling convention uses r24:r25 (a 16-bit pair) for both input and output: - Input: x arrives in r24:r25 - Output: Result leaves in r24:r25 - Temporary registers: r18-r23 are caller-saved, so the function can use them freely

    The code saves the original x value in r18:r19 because it needs x multiple times but the multiply function (__mulhi3) overwrites r24:r25.

    The fibonacci Function: Why Recursion Isn't Inlined¶

    This function cannot be inlined because it's recursive (calls itself):

    fibonacci(int):
        push r16             # Save caller-saved registers
        push r17
        push r28
        push r29
        mov r28,r24          # Store n in r28:r29
        mov r29,r25
        ldi r16,0            # Initialize accumulator to 0
        ldi r17,0
.L4:
        mov r24,r28          # Load n into r24:r25
        cpi r28,2            # Compare n with 2
        cpc r29,__zero_reg__
        brlt .L6             # If n < 2, jump to base case
        sbiw r24,1           # Decrement n by 1
        rcall fibonacci(int) # Recursive call: fib(n-1)
        sbiw r28,2           # Decrement n by 2
        add r16,r24          # Accumulate result
        adc r17,r25
        rjmp .L4             # Loop back
.L6:
        add r24,r16          # Add base case result
        adc r25,r17
        pop r29              # Restore registers
        pop r28
        pop r17
        pop r16
        ret

    Key observations: - The function saves registers (push) because it needs to preserve values across recursive calls - It uses r28:r29 to hold n and r16:r17 as an accumulator - The base case (n <= 1) returns n directly - This is a very inefficient implementation - it recalculates the same values repeatedly (exponential time complexity)
