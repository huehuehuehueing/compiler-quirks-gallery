    unrollme-2¶

    Details

    Source: loops/unrollme-2.c Compiler: avrg1520 (AVR) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    unrollme-2.c
    #include <stdio.h>

void foo(void)
{
    int i;
    char buf[8];

    for (i = 0; i < sizeof(buf); i++)
    {
        buf[i] = 'A';
    }

    /*
     * by dereferencing buf this should force the compile to NOT
     * eliminate the above as dead code. loop will likely still
     * be unrolled
     */
    printf("%s", buf);
}

    Explanation¶

    the key insight is that the compiler has replaced the loop with a single memset call - a significant optimization that fills the buffer efficiently in one operation rather than looping 8 times.

    Stack Frame Setup¶

    The function begins by saving registers and allocating space for the local buffer:

    push r16
push r17
push r28
push r29
in r28,__SP_L__
in r29,__SP_H__
sbiw r28,8

    The push instructions save four registers (r16, r17, r28, r29) that the function will use. Then r28:r29 (a 16-bit register pair called the frame pointer on AVR) is loaded with the current stack pointer position. The sbiw r28,8 instruction subtracts 8 from this pair, allocating 8 bytes on the stack for the buf array. The subsequent in and out instructions safely update the actual stack pointer with interrupts disabled to prevent corruption.

    Loop Optimization: Replaced with memset¶

    Rather than executing the loop 8 times, the compiler has recognized that filling a buffer with a constant value is a perfect use case for the standard library memset function:

    ldi r20,lo8(8)      // Load 8 (the count)
ldi r21,0           // High byte of count
ldi r22,lo8(65)     // Load 65 (ASCII 'A')
ldi r23,0           // High byte of character
mov r16,r28         // Copy frame pointer to r16
mov r17,r29         // Copy frame pointer high byte to r17
subi r16,-1         // Add 1 to get buffer address
sbci r17,-1         // Add carry to high byte
mov r24,r16         // Move buffer address to r24:r25
mov r25,r17         // (parameter passing convention)
rcall memset        // Call memset

    This is a loop unrolling optimization at the library level. Instead of 8 individual store operations, memset handles the entire operation. The parameters are set up following AVR's calling convention: r24:r25 holds the buffer address, r22:r23 holds the character value (65 = 'A'), and r20:r21 holds the count (8).

    printf Call Setup¶

    push r17
push r16
ldi r24,lo8(.LC0)   // Load format string address
ldi r25,hi8(.LC0)
push r25
push r24
rcall printf

    The buffer address (in r16:r17) is pushed onto the stack as the second argument. The format string address ("%s" stored at label .LC0) is loaded into r24:r25 and also pushed. The rcall printf calls the function.

    Stack Cleanup and Return¶

    pop __tmp_reg__     // Pop 4 times to discard
pop __tmp_reg__     // the arguments that were
pop __tmp_reg__     // pushed for printf
pop __tmp_reg__
adiw r28,8          // Deallocate the 8-byte buffer
in __tmp_reg__,__SREG__
cli
out __SP_H__,r29
out __SREG__,__tmp_reg__
out __SP_L__,r28
pop r29
pop r28
pop r17
pop r16
ret

    The function pops the pushed arguments, restores the stack pointer by adding 8 back (undoing the earlier subtraction), and then restores all saved registers before returning.

    Key Observations¶

    The -Os optimization flag (optimize for size) has caused the compiler to replace the explicit loop with a memset callthis is smaller in code size than 8 individual store instructions, even though it involves a function call. On AVR microcontrollers where code space is limited, this trade-off makes sense.

    The frame pointer manipulation with interrupts disabled ensures the stack pointer update is atomiccritical on AVR since the stack pointer is 16 bits but can only be written 8 bits at a time.
