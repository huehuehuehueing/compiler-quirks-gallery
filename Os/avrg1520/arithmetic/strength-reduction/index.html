    strength-reduction¶

    Details

    Source: arithmetic/strength-reduction.c Compiler: avrg1520 (AVR) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    strength-reduction.c
    /*
 * Strength reduction: expensive operations replaced with cheaper ones.
 *
 * Multiplication by powers of 2 becomes shifts.
 * Division by constants becomes multiplication + shift.
 */

int multiply_by_8(int x)
{
    /* Becomes: x << 3 */
    return x * 8;
}

int multiply_by_15(int x)
{
    /* May become: (x << 4) - x */
    return x * 15;
}

unsigned divide_by_3(unsigned x)
{
    /* Division is expensive. Compiler uses "magic number" multiplication:
     * Becomes something like: (x * 0xAAAAAAAB) >> 33
     */
    return x / 3;
}

unsigned modulo_power_of_2(unsigned x)
{
    /* Becomes: x & 7 */
    return x % 8;
}

    Explanation¶

    the key insight is that the compiler is performing strength reduction — replacing expensive arithmetic operations with cheaper alternatives.

    Function 1: multiply_by_8(int)¶

    ldi r18,3          ; Load 3 into r18 (loop counter)
1:
lsl r24            ; Logical shift left r24
rol r25            ; Rotate left r25 (with carry)
dec r18            ; Decrement counter
brne 1b            ; Branch if not equal (loop back)
ret

    Key insight: Instead of a single multiply instruction, the compiler uses a loop that shifts left 3 times. Each left shift doubles the value, so shifting 3 times multiplies by 2³ = 8.

    Here's how it works: - r24 and r25 hold the 16-bit input value (AVR uses two 8-bit registers for 16-bit integers) - lsl r24 shifts the lower byte left by 1 bit, moving the leftmost bit into the carry flag - rol r25 rotates the upper byte left, using that carry bit as input - This repeats 3 times total, multiplying the value by 8 - The loop uses dec and brne (branch if not equal) to count down

    Function 2: multiply_by_15(int)¶

    ldi r22,lo8(15)    ; Load 15 into r22
ldi r23,0          ; Clear r23 (upper byte)
rcall __mulhi3     ; Call multiplication library function
ret

    Key insight: The compiler couldn't optimize this to shifts alone, so it calls a library multiplication function (__mulhi3). The parameters are loaded into r22:r23 (the 16-bit divisor/multiplier), and the result is returned in r24:r25.

    Function 3: divide_by_3(unsigned int)¶

    ldi r22,lo8(3)     ; Load 3 into r22
ldi r23,0          ; Clear r23
rcall __udivmodhi4 ; Call unsigned division/modulo library
mov r24,r22        ; Copy result from r22 to r24
mov r25,r23        ; Copy result from r23 to r25
ret

    Key insight: Division is expensive, so the compiler calls a library function (__udivmodhi4) that handles the division. The divisor (3) goes into r22:r23, and the result comes back in the same registers. The mov instructions then copy the result to r24:r25 (the standard return registers).

    Function 4: modulo_power_of_2(unsigned int)¶

    andi r24,lo8(7)    ; AND r24 with 7 (binary: 0000 0111)
ldi r25,0          ; Clear r25
ret

    Key insight: This is the most efficient optimization. Modulo by a power of 2 becomes a simple bitwise AND operation. Since 8 = 2³, the remainder when dividing by 8 is just the lowest 3 bits of the number. The andi instruction masks everything except those 3 bits.

    Overall Pattern¶

    The compiler is using the -Os flag (optimize for size), which explains why: - Multiply by 8 uses a loop of shifts (smaller code than a multiply instruction) - Multiply by 15 and divide by 3 call library functions (reusable code saves space) - Modulo by 8 uses a single bitwise AND (the absolute smallest solution)

    This demonstrates how modern compilers intelligently replace expensive operations with cheaper alternatives — a technique called strength reduction.
