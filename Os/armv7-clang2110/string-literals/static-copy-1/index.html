    static-copy-1¶

    Details

    Source: string-literals/static-copy-1.c Compiler: armv7-clang2110 (ARM) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    static-copy-1.c
    /*
 * String copy with strcpy - compiler may inline as immediate stores.
 *
 * For small, known-length strings, strcpy becomes direct
 * memory stores with the string data embedded in instructions.
 */
#include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[100];
    strcpy(buf, "ABCDXXXXCCCC");

    /* Make observable to prevent dead code elimination */
    puts(buf);
}

    Explanation¶

    the key insight is that the compiler is manually copying the string data into the buffer using load-and-store instructions, rather than calling a strcpy function. This is a compiler optimization called inlining.

    Function Setup and Stack Frame¶

    push    {r11, lr}
sub     sp, sp, #104

    The function starts by saving two registers to the stack: r11 (frame pointer) and lr (link register, which holds the return address). Then it allocates 104 bytes of stack space for the local char buf[100] array plus some padding.

    Think of the stack like a scratch pad: the function needs temporary space to work, so it reserves 104 bytes. The sp register (stack pointer) marks where this space begins.

    Setting Up the String Copy¶

    ldr     r12, .LCPI0_0
mov     r0, sp
mov     lr, r0

    Here the compiler: 1. Loads an offset value into r12 (this points to where the string data is stored in memory) 2. Moves sp (the buffer's address) into r0, then copies it to lr

    Now lr holds the address of buf where we'll write the string data.

    Position-Independent Code Setup¶

    .LPC0_0:
add     r12, pc, r12

    This is a position-independent code technique. The compiler adds the current program counter (pc) to the offset in r12 to calculate the actual address of the string data. This allows the code to work correctly regardless of where it's loaded in memory.

    The Actual String Copy¶

    ldm     r12!, {r1, r2, r3}
stm     lr!, {r1, r2, r3}
ldrb    r1, [r12]
strb    r1, [lr]

    This is the core of the optimization. Instead of calling strcpy, the compiler directly copies the string bytes:

    • ldm (load multiple) reads 3 registers' worth of data (12 bytes) from the string in memory into r1, r2, and r3. The ! means the address in r12 is incremented afterward.
    • stm (store multiple) writes those same 3 registers into the buffer. Again, lr is incremented.
    • ldrb (load byte) reads the final single byte (the null terminator \0)
    • strb (store byte) writes that null terminator to complete the string

    This bulk-copy approach is much faster than calling a function, especially for small strings.

    Function Cleanup and Return¶

    bl      puts
add     sp, sp, #104
pop     {r11, pc}

    After the string is copied, bl (branch with link) calls puts to print the buffer. Then the function cleans up by deallocating the 104 bytes of stack space and restoring the saved registers. The pop {r11, pc} instruction restores the frame pointer and jumps back to the caller by loading the return address into pc.

    Why This Optimization Matters¶

    The -Os flag (optimize for size) tells the compiler to prioritize small code. For a known, small string like "ABCDXXXXCCCC", inlining the copy as direct memory operations is both smaller and faster than calling strcpy, which would need to check each byte for the null terminator in a loop.
