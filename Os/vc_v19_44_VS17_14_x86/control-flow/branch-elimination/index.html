    branch-elimination¶

    Details

    Source: control-flow/branch-elimination.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    branch-elimination.c
    /*
 * Branch elimination: compile-time constant conditions.
 *
 * When a branch condition is known at compile time,
 * the dead branch is eliminated entirely.
 */
#include <limits.h>

/* Debug flag - in release builds, debug code is eliminated */
#define DEBUG 0

void maybe_log(const char *msg)
{
    if (DEBUG) {
        /* This entire block is removed when DEBUG is 0 */
        extern void log_message(const char *);
        log_message(msg);
    }
}

/* Platform-specific code */
int get_pointer_size(void)
{
    if (sizeof(void *) == 8) {
        /* Only this branch exists on 64-bit platforms */
        return 64;
    } else {
        /* Only this branch exists on 32-bit platforms */
        return 32;
    }
}

/* Impossible conditions eliminated */
int check_range(int x)
{
    /* If x is already validated as positive elsewhere,
     * and optimizer knows it, this check may be eliminated.
     */
    if (x < 0)
        return -1;

    /* This is always false for signed int */
    if (x > INT_MAX)
        return -2;

    return x;
}

    Explanation¶

    the key insight is that the compiler eliminates branches whose conditions are known at compile time, leaving only the code paths that can actually execute.

    maybe_log Function¶

    xor     eax, eax
je      SHORT $LN2@maybe_log

    The compiler evaluates DEBUG (which is 0) at compile time. It sets eax to 0 with xor eax, eax, then immediately jumps over the log_message call with je (jump if equal). The entire logging block is unreachable dead codethe compiler keeps the jump instruction but removes the actual function call. This is dead code elimination: when a condition is provably false, the compiler removes the unreachable branch.

    get_pointer_size Function¶

    xor     eax, eax
je      SHORT $LN2@get_pointe
push    64
pop     eax
jmp     SHORT $LN3@get_pointe
...
$LN2@get_pointe:
push    32
pop     eax

    Here's where it gets interesting: the compiler knows this is a 64-bit platform (amd64), so sizeof(void *) == 8 is always true. However, the compiler still generates both branches in the assembly. The xor eax, eax followed by je creates a jump that always takes the false path, making the 32-bit branch unreachable at runtime. The 64-bit return value (64) is loaded via push 64 / pop eax. This is a quirk of this particular compilerit generates both code paths but makes one unreachable through the conditional jump.

    check_range Function¶

    cmp     DWORD PTR _x$[ebp], 0
jge     SHORT $LN2@check_rang
or      eax, -1
jmp     SHORT $LN1@check_rang

    This function performs actual runtime checks because the parameter x is unknown at compile time. The compiler:

    1. Compares x to 0: cmp sets CPU flags based on whether x is negative
    2. Jumps if greater-or-equal: jge skips the -1 return if x >= 0
    3. Returns -1 for negative values: or eax, -1 sets eax to -1 (all bits set)

    Then it checks the second condition:

    cmp     DWORD PTR _x$[ebp], 2147483647
jle     SHORT $LN3@check_rang
push    -2
pop     eax

    This compares x against INT_MAX (2147483647). However, this check is logically impossible to fail for a signed 32-bit integerno valid signed int can exceed INT_MAX. The compiler still generates the comparison and branch, but in practice, the condition never triggers. A more aggressive optimizer might eliminate this dead code entirely.

    Key Observations¶

    • Compile-time constants (DEBUG = 0, sizeof(void *) on a known platform) are evaluated during compilation, and unreachable branches are eliminated or made unreachable via unconditional jumps
    • Runtime values (function parameters) require actual comparison instructions and conditional branches
    • Impossible conditions (like a signed int exceeding INT_MAX) still generate code, though they're unreachablethis compiler doesn't eliminate them, but a higher optimization level might
