    auto-vectorize¶

    Details

    Source: simd/auto-vectorize.c Compiler: clang1910 (x86/x64) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    auto-vectorize.c
    /*
 * Auto-vectorization: loops converted to SIMD operations.
 *
 * Compiler transforms scalar loops into vector operations
 * (SSE, AVX, NEON, etc.) processing multiple elements at once.
 */

/* Simple loop - prime candidate for vectorization */
void add_arrays(float *dst, const float *a, const float *b, int n)
{
    for (int i = 0; i < n; i++) {
        dst[i] = a[i] + b[i];
    }
    /* With -O3: becomes SIMD adds processing 4-8 floats at once */
}

/* Reduction - trickier to vectorize but often done */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++) {
        sum += arr[i];
    }
    return sum;
    /* May use horizontal add instructions or partial sums */
}

/* Loop with dependency - CANNOT be vectorized */
void fibonacci_array(int *arr, int n)
{
    arr[0] = 0;
    arr[1] = 1;
    for (int i = 2; i < n; i++) {
        /* Each iteration depends on previous two - serial */
        arr[i] = arr[i-1] + arr[i-2];
    }
}

/* Conditional in loop - may use masked operations */
void clamp_array(float *arr, int n, float min, float max)
{
    for (int i = 0; i < n; i++) {
        if (arr[i] < min) arr[i] = min;
        else if (arr[i] > max) arr[i] = max;
    }
}

    Explanation¶

    what's happening, why, and what it reveals about how the compiler handles these different loop patterns.

    Key Insight: No Vectorization Despite -Os¶

    The most important observation is that none of these loops are vectorized. Even though the code comments suggest vectorization might happen with -O3, this is compiled with -Os (optimize for size), which prioritizes smaller code over speed. The compiler chose scalar (single-element) operations instead of SIMD (Single Instruction Multiple Data) operations that process multiple elements at once.

    add_arrays - Element-by-Element Addition¶

    test    ecx, ecx              // Check if n <= 0
jle     .LBB0_3               // If so, skip loop and return
mov     eax, ecx              // Save n for loop counter
xor     ecx, ecx              // Clear ecx (loop index i = 0)
.LBB0_2:                       // Loop start
    movss   xmm0, [rsi + 4*rcx]   // Load a[i] into xmm0 (scalar float)
    addss   xmm0, [rdx + 4*rcx]   // Add b[i] to xmm0
    movss   [rdi + 4*rcx], xmm0   // Store result in dst[i]
    inc     rcx                    // i++
    cmp     rax, rcx               // Compare i with n
    jne     .LBB0_2                // Loop if i != n
.LBB0_3:
    ret

    What's happening: This processes one float at a time in a loop. The xmm0 register (a 128-bit SIMD register) is being used, but only for its scalar (single-value) capabilityit holds just one 32-bit float. The 4*rcx addressing calculates the byte offset (each float is 4 bytes).

    Why scalar instead of vector? With -Os, the compiler prioritizes code size. A vectorized version would need setup code to handle the main loop in chunks of 4-8 floats, plus a cleanup loop for remaining elements. That's more code bytes, so -Os avoids it.

    sum_array - Accumulating a Sum¶

    test    esi, esi              // Check if n <= 0
jle     .LBB1_1               // If so, jump to return 0.0
mov     eax, esi              // Save n
xorps   xmm0, xmm0            // Initialize sum = 0.0 (xmm0 = 0.0)
xor     ecx, ecx              // Clear ecx (loop index i = 0)
.LBB1_4:                       // Loop start
    addss   xmm0, [rdi + 4*rcx]   // sum += arr[i]
    inc     rcx                    // i++
    cmp     rax, rcx               // Compare i with n
    jne     .LBB1_4                // Loop if i != n
ret
.LBB1_1:
    xorps   xmm0, xmm0            // Return 0.0
    ret

    What's happening: This accumulates a running sum in xmm0. Each iteration adds one array element to the accumulator. The xorps instruction clears xmm0 to 0.0 (XOR-ing a register with itself produces zero).

    The dependency chain: Each iteration depends on the previous sum value, so this cannot be vectorizedthe CPU must wait for each addition to complete before starting the next one. This is a fundamental limitation, not a compiler choice.
