    loop-invariant-motion¶

    Details

    Source: loops/loop-invariant-motion.c Compiler: clang1910 (x86/x64) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    loop-invariant-motion.c
    /*
 * Loop-Invariant Code Motion (LICM)
 *
 * The optimizer recognises computations inside a loop whose result
 * does not change across iterations and moves ("hoists") them before
 * the loop.  This avoids redundant work on every iteration.
 *
 * In the example below, `base * scale` is invariant with respect to
 * the loop index i, so the compiler should compute it once and reuse
 * the result inside the loop body.
 */

void fill_scaled(int *arr, int n, int base, int scale)
{
    for (int i = 0; i < n; i++)
    {
        /* base * scale is loop-invariant: it does not depend on i. */
        arr[i] = base * scale + i;
    }
}

    Explanation¶

    the key insight is that the compiler has applied Loop-Invariant Code Motion (LICM) and SIMD vectorization to optimize this loop.

    The Core Optimization: LICM¶

    The most important thing to understand is what happens at the very beginning:

    imul    ecx, edx

    This single instruction computes base * scale once, before the loop starts. In the C code, this multiplication happens inside the loop on every iteration, but the compiler recognizes that base and scale never changethey're loop-invariant. So it hoists this computation outside the loop and reuses the result (ecx) throughout.

    This is the entire point of LICM: avoid redundant work by computing constant values once instead of repeatedly.

    Vectorization: Processing Multiple Elements at Once¶

    The compiler goes further and uses SIMD instructions (Single Instruction, Multiple Data) to process multiple array elements in parallel. Think of SIMD like processing a batch of items at once instead of one at a time.

    The key registers here are XMM registers (xmm0 through xmm7), which are 128-bit vector registers that can hold multiple integers simultaneously. The compiler processes 2 integers per loop iteration using these vector operations.

    Setup Phase (Before the Loop)¶

    test    esi, esi          ; Check if n <= 0
jle     .LBB0_7           ; If so, skip the loop entirely
imul    ecx, edx          ; ecx = base * scale (computed once!)
mov     eax, esi          ; eax = n
inc     esi               ; esi = n + 1
dec     rax               ; rax = n - 1
movq    xmm0, rax         ; Load n-1 into vector register
pshufd  xmm0, xmm0, 68    ; Duplicate it across vector lanes
shr     esi               ; esi = (n+1) / 2 (number of pairs)
shl     rsi, 3            ; rsi = pairs * 8 (byte offset for loop end)

    The compiler prepares several things: - Early exit: If n <= 0, jump to the end (no iterations needed) - The invariant: ecx holds base * scale - Vector setup: Prepares index values in xmm0 for the vectorized loop - Loop bounds: Calculates how many pairs of elements to process

    The Vectorized Loop Body¶

    .LBB0_2:
    movdqa  xmm5, xmm1        ; Load current indices
    pxor    xmm5, xmm2        ; Bit manipulation for signed comparison
    movdqa  xmm6, xmm5        ; Copy for comparison
    pcmpgtd xmm6, xmm0        ; Compare: is index < n?
    pcmpeqd xmm5, xmm0        ; Compare: is index == n?
    ; ... more comparison logic ...
    movd    edx, xmm5         ; Extract comparison result
    not     edx               ; Invert it
    test    dl, 1             ; Check if we should write
    je      .LBB0_4           ; Skip if not

    paddq   xmm1, xmm4        ; Increment indices by 2

    add     ecx, 2            ; Increment base*scale by 2 for next iteration
    cmp     rsi, rax          ; Check loop condition
    jne     .LBB0_2           ; Continue if more pairs to process
