    fortify-source¶

    Details

    Source: hardening/fortify-source.c Compiler: clang1910 (x86/x64) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    fortify-source.c
    /* @gallery-hints
 *   extra-flags: -D_FORTIFY_SOURCE=2
 *   compiler-only: cg152, clang1910
 *   scenario-exclude: O0
 */

/*
 * FORTIFY_SOURCE - compile-time and runtime buffer overflow checks.
 *
 * When _FORTIFY_SOURCE is defined (requires optimization >= -O1),
 * glibc replaces common functions like memcpy, strcpy, sprintf with
 * checked variants (__memcpy_chk, etc.) that verify the destination
 * buffer size at compile time or runtime.
 *
 * Look for: calls to __memcpy_chk or __strcpy_chk instead of plain
 * memcpy / strcpy.  If the compiler can prove the copy is safe at
 * compile time, the check is elided entirely.
 */
#include <string.h>

void safe_copy(char *dst, unsigned long dstsize, const char *src)
{
    /* With FORTIFY_SOURCE, this becomes __memcpy_chk which will
     * abort at runtime if srclen > dstsize. */
    size_t srclen = strlen(src) + 1;
    if (srclen <= dstsize)
        memcpy(dst, src, srclen);
}

void known_size_copy(void)
{
    char buf[32];
    /* Compiler knows buf is 32 bytes, so it can check at compile time. */
    strcpy(buf, "hello");
    (void)buf;
}

    Explanation¶

    the key insight here is how FORTIFY_SOURCE enables compile-time and runtime buffer overflow protection, and how the compiler optimizes based on what it can verify.

    safe_copy Function¶

    The core pattern: This function validates that the source string fits in the destination buffer before copying. The assembly implements this check and conditionally calls memcpy.

    Setup and Parameter Preservation¶

    push    r15
push    r14
push    rbx
mov     rbx, rdx        ; rbx = src (3rd parameter)
mov     r15, rsi        ; r15 = dstsize (2nd parameter)
mov     r14, rdi        ; r14 = dst (1st parameter)

    The function saves three registers (r15, r14, rbx) because it needs to preserve the original parameters across the strlen call. By convention in the amd64 calling convention, these are caller-saved registers — the function must restore them before returning. The parameters are copied into these registers because strlen will clobber rdi (the first parameter register).

    Computing Source Length¶

    mov     rdi, rdx        ; rdi = src (prepare for strlen call)
call    strlen@PLT      ; rax = strlen(src)
inc     rax             ; rax += 1 (add 1 for null terminator)

    strlen returns the string length in rax, and we add 1 to account for the null terminator that memcpy must also copy.

    The Safety Check¶

    cmp     rax, r15        ; compare srclen with dstsize
jbe     .LBB0_2         ; if srclen <= dstsize, jump to memcpy

    This compares the required length (rax) against the available buffer size (r15). The jbe instruction means "jump if below or equal" — if the source fits, we proceed to the copy. If not, we skip the copy and return safely.

    Conditional Copy Path¶

    .LBB0_2:
mov     rdi, r14        ; rdi = dst (1st parameter for memcpy)
mov     rsi, rbx        ; rsi = src (2nd parameter for memcpy)
mov     rdx, rax        ; rdx = srclen (3rd parameter for memcpy)
pop     rbx
pop     r14
pop     r15
jmp     memcpy@PLT      ; tail call to memcpy

    If the check passes, the function restores the saved registers and tail-calls memcpy (using jmp instead of call). This is an optimization: instead of calling memcpy and then returning, we jump directly to it, letting memcpy's return go straight back to the original caller. This saves one return instruction.

    known_size_copy Function¶

    known_size_copy():
        ret

    This is the most striking optimization: The entire function body vanishes. Here's why:

    The source code calls strcpy(buf, "hello") where: - buf is a 32-byte local buffer - "hello" is a 6-byte string (including null terminator)

    The compiler can prove at compile time that this copy is safe — the string is much smaller than the buffer. With -Os (optimize for size) and _FORTIFY_SOURCE=2, the compiler:

    1. Verifies the copy is safe
    2. Eliminates the entire strcpy call (dead code elimination)
    3. Leaves only the function prologue/epilogue

    The (void)buf; statement prevents the compiler from optimizing away the buffer allocation itself, but since nothing uses the buffer, the copy becomes unnecessary.

    Key Observations¶

    • FORTIFY_SOURCE with -Os: The compiler generates runtime checks (strlen + comparison) for dynamic sizes, but completely eliminates checks for compile-time-provable cases
    • Tail-call optimization: The jmp memcpy@PLT pattern avoids unnecessary return instructions
    • Register preservation: Caller-saved registers are used to hold parameters across function calls that would otherwise clobber them
