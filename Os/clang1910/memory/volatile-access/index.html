    volatile-access¶

    Details

    Source: memory/volatile-access.c Compiler: clang1910 (x86/x64) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    volatile-access.c
    /*
 * Volatile prevents optimization of memory accesses.
 *
 * Without volatile, repeated reads/writes may be cached in
 * registers or eliminated. Volatile forces actual memory access.
 */

/* Hardware register simulation */
volatile int *const HARDWARE_REG = (volatile int *)0x40000000;

void write_hardware(int value)
{
    /* Each write MUST go to memory - cannot be cached or reordered */
    *HARDWARE_REG = value;
    *HARDWARE_REG = value;  /* Not eliminated - both writes happen */
}

int read_hardware(void)
{
    /* Each read MUST come from memory - cannot use cached value */
    int a = *HARDWARE_REG;
    int b = *HARDWARE_REG;
    return a + b;  /* Two actual memory reads */
}

/* Compare: non-volatile version */
int *non_volatile_ptr;

int read_non_volatile(void)
{
    /* Optimizer may read once and reuse the value */
    int a = *non_volatile_ptr;
    int b = *non_volatile_ptr;  /* May be optimized to: b = a */
    return a + b;
}

    Explanation¶

    the key insight is how the volatile keyword forces the compiler to perform actual memory accesses, while non-volatile pointers allow optimization.

    The Core Difference: Volatile vs. Non-Volatile Memory Access¶

    The most important pattern here is how volatile prevents the compiler from being clever. When you mark a pointer as volatile, you're telling the compiler "I know this memory location might change unexpectedly (like a hardware register), so read/write it every single time—don't try to optimize by caching values in registers."

    write_hardware(int) - Two Writes That Both Happen¶

    mov     dword ptr [1073741824], edi
mov     dword ptr [1073741824], edi
ret

    The address 1073741824 is 0x40000000 in decimal—the hardware register location. Notice both mov instructions are identical and both present.

    In normal code, a compiler would see "write the same value twice" and eliminate the redundant write. But because HARDWARE_REG is volatile, the compiler must execute both writes. This is essential for hardware: the first write might trigger an action, and the second write might be needed for a different reason.

    The parameter value arrives in edi (the lower 32 bits of rdi, following the x86-64 calling convention where the first integer argument goes in rdi).

    read_hardware() - Two Reads That Both Happen¶

    mov     eax, dword ptr [1073741824]
add     eax, dword ptr [1073741824]
ret

    Here's the elegant part: the compiler reads from the hardware register twice, but it does so efficiently:

    1. First read: mov eax, dword ptr [1073741824] — reads the value into eax
    2. Second read: add eax, dword ptr [1073741824] — reads again and adds it to eax in one instruction

    The add instruction both reads from memory and performs the addition. This is more efficient than doing a separate mov followed by add, because the add instruction can read directly from memory. The result (the sum) stays in eax, which is where the return value goes by convention.

    Both reads are forced to happen because of volatile, even though they're from the same address.

    read_non_volatile() - Optimization in Action¶

    mov     rax, qword ptr [rip + non_volatile_ptr]
mov     eax, dword ptr [rax]
add     eax, eax
ret

    This is the dramatic difference. Notice the final instruction: add eax, eax. This is doubling the value (adding it to itself), not reading from memory twice.

    Here's what happened:

    1. Load the pointer: mov rax, qword ptr [rip + non_volatile_ptr] — gets the address stored in non_volatile_ptr
    2. Read once: mov eax, dword ptr [rax] — reads the value from that address into eax
    3. Reuse the value: add eax, eax — adds the value to itself (equivalent to multiplying by 2)

    The compiler optimized away the second memory read because non_volatile_ptr is not marked volatile. The compiler reasoned: "Since this isn't volatile, the value at that address can't change between my two reads, so I'll just read once and reuse the value."

    This is a massive performance difference on real hardware—memory reads are slow (typically 100+ CPU cycles), while register operations are fast (1 cycle). The non-volatile version is orders of magnitude faster.

    Why This Matters¶

    • Volatile: Forces the compiler to trust that memory might change unexpectedly. Essential for hardware registers, memory-mapped I/O, and shared memory between threads.
    • Non-volatile: Allows the compiler to assume memory is stable and optimize aggressively. Faster, but unsafe if the memory actually does change.

    The -Os optimization flag (optimize for size) still applies these optimizations—it just means the compiler also tries to keep code small, but it doesn't disable the fundamental optimization of eliminating redundant reads from non-volatile memory.
