    fast-math¶

    Details

    Source: arithmetic/fast-math.c Compiler: clang1910 (x86/x64) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    fast-math.c
    /*
 * Fast-math optimizations: -ffast-math / -Ofast effects.
 *
 * These flags allow the compiler to reorder floating-point
 * operations, assuming associativity (which floats don't have).
 * Can change results or break special value handling.
 */

/* With -ffast-math, this may be reordered and give different results */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++)
        sum += arr[i];
    return sum;
}

/* NaN check may be optimized away with -ffast-math */
int is_nan(float x)
{
    /* With -ffast-math, compiler assumes no NaN values exist,
     * so this comparison may be optimized to always return 0.
     */
    return x != x;
}

/* Division by variable may become multiplication by reciprocal */
float divide_by_y(float x, float y)
{
    /* With -ffast-math: may become x * (1.0f / y)
     * Less accurate but faster on some architectures.
     */
    return x / y;
}

/* Algebraic simplification (inexact with floats!) */
float algebraic(float a, float b, float c)
{
    /* With -ffast-math: (a + b) + c may be reordered to a + (b + c)
     * These can give different results with floating point!
     */
    return (a + b) + c;
}

    Explanation¶

    the key insight is that this code is compiled with -Os (optimize for size), not -ffast-math, so the floating-point operations remain conservative and correct. The comments in the source code describe what would happen with -ffast-math, but that's not what we're seeing here.

    sum_array function¶

    This function demonstrates a loop that accumulates floating-point values:

    test    esi, esi              // Check if n (in esi) is <= 0
jle     .LBB0_1               // If so, jump to return 0.0

    The first two instructions handle the edge case: if n is zero or negative, skip the loop and return 0.0.

    mov     eax, esi              // Copy n into eax (loop counter limit)
xorps   xmm0, xmm0            // Zero out xmm0 (this will hold the sum)
xor     ecx, ecx              // Zero out ecx (loop counter i)

    Here, the compiler sets up the loop. The xorps instruction clears the XMM0 register—this is a floating-point register that will accumulate the sum. By XORing a register with itself, you get zero (a common trick that's faster than loading a constant).

    .LBB0_4:                      // Loop label
    addss   xmm0, dword ptr [rdi + 4*rcx]  // Add arr[i] to sum
    inc     rcx               // i++
    cmp     rax, rcx          // Compare loop counter to limit
    jne     .LBB0_4           // Loop if not equal
ret

    The loop body is straightforward: addss (add single-precision scalar) loads one float from the array at address rdi + 4*rcx (where rdi is the array pointer and rcx is the index, multiplied by 4 since floats are 4 bytes) and adds it to xmm0. The loop continues until rcx equals rax (the count n).

    Key observation: Even though the source code comments mention that -ffast-math could reorder these additions, the actual assembly shows a straightforward sequential accumulation. With -Os, the compiler prioritizes correctness over aggressive optimizations.

    is_nan function¶

    xor     eax, eax              // Zero out eax (will hold return value)
ucomiss xmm0, xmm0            // Compare xmm0 (the float) with itself
setp    al                    // Set al to 1 if parity flag is set
ret

    This is elegant: a NaN is the only floating-point value that is not equal to itself. The ucomiss instruction compares the float in xmm0 with itself. If it's NaN, the comparison sets the parity flag (a CPU flag that tracks certain conditions). The setp instruction copies that parity flag into the low byte of eax (the return register), giving 1 for NaN and 0 otherwise.

    This code is NOT optimized away—despite the source comment suggesting -ffast-math might eliminate it, the actual compiled code correctly detects NaN values.

    divide_by_y function¶

    divss   xmm0, xmm1            // Divide xmm0 (x) by xmm1 (y)
ret

    This is a single instruction: divss performs single-precision scalar division. The first operand (xmm0) is both the dividend and the destination for the result. By convention, floating-point parameters are passed in XMM registers, so x arrives in xmm0 and y in xmm1.

    No reciprocal optimization: The code uses direct division, not the multiplication-by-reciprocal that -ffast-math might enable. This is more accurate.
