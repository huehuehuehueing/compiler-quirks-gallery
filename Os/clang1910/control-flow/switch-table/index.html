    switch-table¶

    Details

    Source: control-flow/switch-table.c Compiler: clang1910 (x86/x64) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    switch-table.c
    /*
 * Switch optimization: jump tables vs. binary search vs. if-else.
 *
 * Compiler chooses implementation based on case distribution:
 * - Dense cases: jump table (O(1))
 * - Sparse cases: binary search (O(log n)) or if-else chain
 */

/* Dense cases - likely becomes a jump table */
int day_of_week(int day)
{
    switch (day) {
        case 0: return 'S';  /* Sunday */
        case 1: return 'M';
        case 2: return 'T';
        case 3: return 'W';
        case 4: return 'T';
        case 5: return 'F';
        case 6: return 'S';  /* Saturday */
        default: return '?';
    }
}

/* Sparse cases - likely if-else or binary search */
int sparse_switch(int x)
{
    switch (x) {
        case 1:     return 10;
        case 100:   return 20;
        case 10000: return 30;
        case 999999: return 40;
        default:    return 0;
    }
}

/* Two cases - becomes simple comparison */
int binary_choice(int x)
{
    switch (x) {
        case 0: return -1;
        case 1: return 1;
        default: return 0;
    }
}

    Key Insight: Compiler Chooses Different Strategies Based on Case Distribution¶

    The compiler uses three different approaches depending on how the switch cases are distributed:

    1. Dense cases (consecutive values) jump table (fastest)
    2. Sparse cases (scattered values) binary search with comparisons
    3. Simple cases (just 2-3 options) arithmetic tricks

    day_of_week(int) - Jump Table Strategy¶

    This function has cases 0-6 (dense and consecutive), so the compiler uses a jump tablea lookup array that provides O(1) constant-time access:

    mov     eax, 63              // eax = '?' (ASCII 63, the default return)
cmp     edi, 6               // Is input > 6?
ja      .LBB0_2              // If yes, jump to return (skip table lookup)
mov     eax, edi             // eax = input value (0-6)
lea     rcx, [rip + .Lswitch.table.day_of_week(int)]  // Load table address
mov     eax, dword ptr [rcx + 4*rax]  // Load result from table[input]
.LBB0_2:
ret

    How it works: - First, eax is pre-loaded with 63 (the ASCII code for '?', the default case) - The input (edi) is checked: if it's greater than 6, we skip the table lookup and return the default - For valid inputs (0-6), the code loads the address of the lookup table using lea (Load Effective Address) - [rcx + 4*rax] calculates the memory address: table base + (input 4). The 4 is because each entry is a 4-byte integer - The value at that address is loaded into eax and returned

    The lookup table (at the bottom): 

    .Lswitch.table.day_of_week(int):
    .long   83   // case 0: 'S' (Sunday)
    .long   77   // case 1: 'M'
    .long   84   // case 2: 'T'
    .long   87   // case 3: 'W'
    .long   84   // case 4: 'T'
    .long   70   // case 5: 'F'
    .long   83   // case 6: 'S' (Saturday)

    This is extremely fast because it's just one bounds check, one address calculation, and one memory loadno branching needed for the common case.

    sparse_switch(int) - Binary Search Strategy¶

    This function has cases at 1, 100, 10000, and 999999 (very sparse), so a jump table would waste memory. Instead, the compiler uses binary search with comparisons:

    cmp     edi, 9999            // Compare input to 9999 (midpoint)
jg      .LBB1_3              // If greater, jump to upper half
cmp     edi, 1               // Compare to 1
je      .LBB1_2              // If equal, return 10
cmp     edi, 100             // Compare to 100
jne     .LBB1_9              // If not equal, return default (0)
mov     eax, 20              // Return 20
ret

    The strategy: - First comparison (cmp edi, 9999) divides the cases into two groups: 9999 and >9999 - For the lower group, it checks 1 and 100 - For the upper group (not shown in detail), it checks 10000 and 999999 - Each path leads to the appropriate return value or the default case

    This approach uses fewer comparisons than a linear if-else chain, and doesn't waste memory on a sparse table.
