    if-conversion¶

    Details

    Source: control-flow/if-conversion.c Compiler: clang1910 (x86/x64) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    if-conversion.c
    /*
 * If-Conversion (branch to conditional move)
 *
 * Short if-else chains that select between two values can be
 * replaced with conditional-move instructions (cmov on x86).
 * This eliminates a branch, avoiding potential misprediction
 * penalties on modern super-scalar CPUs.
 *
 * Look for: cmovl / cmovge / cmovne instead of jl / jge / jne
 * with a branch over a mov.
 */

int min(int a, int b)
{
    if (a < b)
        return a;
    return b;
}

int clamp(int x, int lo, int hi)
{
    if (x < lo)
        x = lo;
    if (x > hi)
        x = hi;
    return x;
}

int abs_val(int x)
{
    return x < 0 ? -x : x;
}

    Explanation¶

    the key insight is that the compiler is using conditional move instructions (cmov*) instead of branches, which is a powerful optimization technique.

    The Core Pattern: Conditional Moves Instead of Branches¶

    The most important thing to understand here is that these three functions avoid traditional branching (jmp instructions). Instead, they use cmov instructions—these conditionally move a value into a register based on CPU flags, without jumping. This is faster on modern processors because it avoids branch misprediction penalties.

    Think of it like this: instead of saying "if this condition is true, jump to a different part of the code," the CPU says "compute both possible results, then pick the right one based on the condition." This keeps the CPU pipeline full and running smoothly.

    Function-by-Function Breakdown¶

    min(int, int) - The Simplest Case¶

    mov     eax, esi          // eax = b (the second parameter)
cmp     edi, esi          // compare a (edi) with b (esi)
cmovl   eax, edi          // if a < b (less), move a into eax
ret

    Parameter passing: By convention, the first integer parameter goes in edi (that's a), and the second goes in esi (that's b). The return value goes in eax.

    What happens: 1. Start by assuming the result is b (move it into eax) 2. Compare a with b using cmp—this sets CPU flags based on the comparison 3. cmovl (conditional move if less) checks those flags: if a < b, it overwrites eax with a 4. Return whatever is in eax

    The beauty here: no branch, no jump. The CPU executes both the mov and the cmovl, but only one actually changes the result.

    clamp(int, int, int) - Two Conditional Moves in Sequence¶

    mov     eax, esi          // eax = lo (second parameter)
cmp     edi, esi          // compare x (edi) with lo (esi)
cmovg   eax, edi          // if x > lo (greater), move x into eax
cmp     eax, edx          // compare result with hi (edx)
cmovge  eax, edx          // if result >= hi, move hi into eax
ret

    Parameters: x in edi, lo in esi, hi in edx.

    What happens: 1. Start with lo as the result 2. First comparison: if x > lo, replace the result with x (this handles the lower bound) 3. Second comparison: if the current result is >= hi, replace it with hi (this handles the upper bound) 4. Return the clamped value

    This chains two conditional moves together to implement the two-condition logic without any branches.

    abs_val(int) - A Clever Trick with neg¶

    mov     eax, edi          // eax = x
neg     eax               // eax = -x (negate it)
cmovs   eax, edi          // if sign flag set (was negative), keep original x
ret

    What happens: 1. Copy x into eax 2. neg negates the value (computes -x) and sets the sign flag based on the result 3. cmovs (conditional move if sign) checks: if the negated value is negative (meaning the original was positive), it restores the original x 4. Return the absolute value

    This is elegant: negate first, then conditionally undo it. The neg instruction sets flags that tell us whether the original value was negative.

    Why This Matters¶

    Compiled with -Os (optimize for size), the compiler chose cmov instructions because they're: - Smaller in code size than branch-based alternatives (no jump instructions needed) - Faster on modern CPUs because they avoid branch misprediction—the CPU doesn't have to guess which way the code will go
