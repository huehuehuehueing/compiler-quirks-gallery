    loop-interchange¶

    Details

    Source: loops/loop-interchange.c Compiler: csparc64g1520 (SPARC64) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    loop-interchange.c
    /*
 * Loop Interchange
 *
 * When iterating over a 2D array, the order of the loop indices
 * determines memory access patterns.  In C (row-major layout),
 * iterating the inner loop over columns and the outer loop over
 * rows yields sequential access.  Swapping the loops ("interchange")
 * can improve cache locality when the original order would cause
 * stride-N accesses.
 *
 * Here the "bad" order (column-major traversal on a row-major array)
 * is presented; the compiler may interchange the loops at -O3.
 */
#define N 128

void column_sum(int mat[N][N], int result[N])
{
    /* Column-major traversal: inner loop strides by N ints.
     * The compiler may swap i and j for better locality. */
    for (int j = 0; j < N; j++)
    {
        result[j] = 0;
        for (int i = 0; i < N; i++)
        {
            result[j] += mat[i][j];
        }
    }
}

    Explanation¶

    the key insight is that this code implements a nested loop that sums columns of a 2D array, and the assembly reveals how the compiler handles memory access patterns.

    The Core Structure¶

    The assembly has two nested loops: - Outer loop (.L6): iterates through each column (128 times) - Inner loop (.L2): accumulates values down each column (128 times)

    Register Usage and Setup¶

    Before diving into the loops, the compiler initializes registers:

    mov     0, %g1              # %g1 = 0 (column index, increments by 4 bytes)
st      %g0, [%o1+%g1]      # result[0] = 0 (initialize first result)

    Here, %g1 tracks the current column offset in bytes (multiplied by 4 since integers are 4 bytes). The %o1 register holds the address of the result array (by SPARC calling convention, the second parameter goes in %o1).

    The Inner Loop: Accumulating Column Values¶

    The inner loop (.L2) is where the actual work happens:

    .L2:
    lduw    [%g3], %g5          # Load unsigned word from mat[i][j]
    add     %g3, 512, %g3       # Move to next row (512 = 128 ints × 4 bytes)
    addcc   %g2, -1, %g2        # Decrement row counter, set condition codes
    add     %g4, %g5, %g4       # Accumulate: result += mat[i][j]
    bne,pt  %icc, .L2           # Branch if not done (rows remaining)
     st     %g4, [%o1+%g1]      # [delay slot] Store accumulated sum

    Key observations:

    • %g3 holds the current memory address within the matrix. It starts at %o0 + %g1 (the address of mat[0][j])
    • %g2 counts down from 128 (the number of rows)
    • %g4 accumulates the column sum
    • The add %g3, 512, %g3 instruction moves to the next row by adding 512 bytes (since each row has 128 integers × 4 bytes/integer)
    • The st instruction in the delay slot stores the accumulated sum back to result[j] on each iteration—this happens after the branch decision is made (a SPARC quirk called the delay slot)

    The Outer Loop: Moving to the Next Column¶

    After the inner loop completes, the outer loop advances to the next column:

    add     %g1, 4, %g1             # Move to next column (4 bytes = 1 integer)
cmp     %g1, 512                # Compare: have we processed all 128 columns?
bne,a,pt %xcc, .L6              # Branch if not done
 st     %g0, [%o1+%g1]          # [delay slot] Initialize next result to 0

    The 512 represents 128 columns × 4 bytes per integer.

    Memory Access Pattern¶

    This code exhibits stride-N access: each lduw loads from addresses that are 512 bytes apart (one row apart in the matrix). This is inefficient for cache performance because: - Modern CPUs load entire cache lines (typically 64 bytes) - Loading from mat[0][j], then mat[1][j], then mat[2][j] means most of each cache line is wasted - An optimized version would iterate rows in the outer loop and columns in the inner loop, achieving sequential access

    The -Os flag (optimize for size) doesn't perform loop interchange, which would require more complex analysis and might increase code size.

    Function Return¶

    jmp     %o7+8                   # Jump to return address
 nop                            # [delay slot] No operation

    This returns control to the caller using the return address stored in %o7 (by SPARC convention).
