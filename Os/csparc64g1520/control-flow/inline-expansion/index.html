    inline-expansion¶

    Details

    Source: control-flow/inline-expansion.c Compiler: csparc64g1520 (SPARC64) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    inline-expansion.c
    /*
 * Function inlining: small functions are expanded at call site.
 *
 * Eliminates call overhead and enables further optimizations
 * by exposing the function body to the caller's context.
 */

/* Tiny function - almost certainly inlined */
static inline int square(int x)
{
    return x * x;
}

/* Using static helps inlining (internal linkage) */
static int cube(int x)
{
    return x * x * x;
}

/* Wrapper that uses inlined functions */
int compute(int x)
{
    /* After inlining, this becomes:
     * return (x * x) + (x * x * x);
     * Which may further simplify to:
     * return x * x * (1 + x);
     */
    return square(x) + cube(x);
}

/* Recursive functions are NOT inlined (infinite expansion) */
int fibonacci(int n)
{
    if (n <= 1) return n;
    return fibonacci(n-1) + fibonacci(n-2);
}

/* Large functions may not be inlined (code size cost) */
int large_function(int x)
{
    int result = 0;
    for (int i = 0; i < 100; i++) {
        result += x * i;
        result ^= (result >> 3);
        result += (result << 5);
    }
    return result;
}

    the key patterns, register usage, and how the compiler handled different function types.

    The Core Pattern: Function Inlining¶

    The most important insight here is that the compiler inlined the small functions (square and cube) directly into compute, eliminating function call overhead. Meanwhile, larger and recursive functions were not inlined. Let me walk through each function to show you why.

    compute: Inlined Functions in Action¶

    compute:
        smul    %o0, %o0, %g1      # x * x  g1 (square)
        smul    %g1, %o0, %o0      # (x*x) * x  o0 (cube)
        add     %o0, %g1, %o0      # cube + square  o0
        jmp     %o7+8              # return
         sra    %o0, 0, %o0        # sign-extend result (delay slot)

    What's happening: Instead of calling square(x) and cube(x) as separate functions, the compiler expanded them inline: - Line 1: smul (signed multiply) computes x * x and stores it in %g1 - Line 2: smul multiplies that result by x again to get x * x * x - Line 3: add combines both results - Line 4-5: Return to caller

    Why this matters: By inlining, the compiler avoided two function call instructions (which would save/restore registers and jump around). The parameter arrives in %o0 (by SPARC calling convention, the first integer argument goes in %o0), and the result leaves in %o0 as well.

    The sra %o0, 0, %o0 instruction (shift right arithmetic by 0) is a sign-extension operationit ensures the 32-bit result is properly formatted for return.

    fibonacci: No Inlining (Recursive Function)¶

    fibonacci:
        save    %sp, -176, %sp     # create new stack frame
        mov     0, %i5             # accumulator = 0
.L4:
        cmp     %i0, 1             # compare n with 1
        ble,pn  %icc, .L6          # if n <= 1, jump to base case
         add    %i0, -1, %o0       # prepare n-1 (delay slot)
        add     %i0, -2, %i0       # n = n-2
        call    fibonacci, 0       # recursive call
         sra    %o0, 0, %o0        # sign-extend n-1 (delay slot)
        sra     %i0, 0, %i0        # sign-extend n-2
        ba,pt   %xcc, .L4          # loop back
         add    %i5, %o0, %i5      # accumulate result (delay slot)
.L6:
        add     %i5, %i0, %i0      # final sum
        return  %i7+8              # return
         sra    %o0, 0, %o0        # sign-extend (delay slot)

    Why no inlining: Recursive functions cannot be inlinedthe compiler would need to expand them infinitely. Instead, the compiler creates a proper stack frame with save (which allocates 176 bytes of local storage) and makes actual function calls.

    Key register roles here: - %i0: input parameter (n), preserved across calls via the register window - %i5: accumulator for summing results - %o0: used to pass arguments to recursive calls - %i7: return address (saved by save instruction)

    The ble,pn instruction (branch if less-or-equal, predict not taken) handles the base case. The pn hint tells the CPU this branch is unlikely, so it optimizes for the recursive path.
