    auto-vectorize¶

    Details

    Source: simd/auto-vectorize.c Compiler: csparc64g1520 (SPARC64) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    auto-vectorize.c
    /*
 * Auto-vectorization: loops converted to SIMD operations.
 *
 * Compiler transforms scalar loops into vector operations
 * (SSE, AVX, NEON, etc.) processing multiple elements at once.
 */

/* Simple loop - prime candidate for vectorization */
void add_arrays(float *dst, const float *a, const float *b, int n)
{
    for (int i = 0; i < n; i++) {
        dst[i] = a[i] + b[i];
    }
    /* With -O3: becomes SIMD adds processing 4-8 floats at once */
}

/* Reduction - trickier to vectorize but often done */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++) {
        sum += arr[i];
    }
    return sum;
    /* May use horizontal add instructions or partial sums */
}

/* Loop with dependency - CANNOT be vectorized */
void fibonacci_array(int *arr, int n)
{
    arr[0] = 0;
    arr[1] = 1;
    for (int i = 2; i < n; i++) {
        /* Each iteration depends on previous two - serial */
        arr[i] = arr[i-1] + arr[i-2];
    }
}

/* Conditional in loop - may use masked operations */
void clamp_array(float *arr, int n, float min, float max)
{
    for (int i = 0; i < n; i++) {
        if (arr[i] < min) arr[i] = min;
        else if (arr[i] > max) arr[i] = max;
    }
}

    Explanation¶

    the key insight is that this is unoptimized code compiled with -Os (optimize for size), and there is no vectorization happening. SPARC is a scalar architecture without SIMD instructions, so the compiler generates simple scalar loops instead.

    Overall Structure¶

    The code contains four functions, each implementing a loop. Let me walk through the most important patterns:

    Register Conventions on SPARC¶

    Before diving into each function, understand SPARC's calling convention: - %o0-%o7: Output registers (used for parameters and return values) - %g0-%g7: Global registers (general purpose) - %f0-%f31: Floating-point registers - %o7: Return address register

    add_arrays: Element-by-Element Addition¶

    add_arrays:
    mov     0, %g1          # %g1 = 0 (byte offset into arrays)
    mov     0, %g2          # %g2 = 0 (loop counter)
.L2:
    cmp     %g2, %o3        # Compare counter with n
    bl,a,pt %icc, .L3      # If counter < n, branch to .L3
     ld     [%o1+%g1], %f8  # Load a[i] into %f8 (delay slot)
    jmp     %o7+8           # Return to caller
     nop
.L3:
    add     %g2, 1, %g2     # Increment counter
    ld      [%o2+%g1], %f9  # Load b[i] into %f9
    fadds   %f8, %f9, %f8   # Add: %f8 = a[i] + b[i]
    st      %f8, [%o0+%g1]  # Store result to dst[i]
    ba,pt   %xcc, .L2       # Branch back to loop start
     add    %g1, 4, %g1     # Increment byte offset by 4 (delay slot)

    Key pattern: This is a simple scalar loop processing one float (4 bytes) per iteration. The compiler uses: - %g1 as a byte offset (incremented by 4 each iteration) - %g2 as the loop counter - %f8 and %f9 for the two input floats - fadds (scalar floating-point add) to add them

    Notice the delay slots (instructions after branch instructions that execute before the branch takes effect): - The ld instruction after bl,a,pt executes before the branch decision - The add %g1, 4, %g1 after ba,pt executes before jumping back

    This is typical SPARC behaviorthe processor executes the next instruction regardless of the branch outcome.

    Why No Vectorization?¶

    SPARC doesn't have SIMD instructions like SSE or AVX. Even with -O3, you wouldn't see vectorization here. The compiler generates the most efficient scalar code it can.
