    stack-protector¶

    Details

    Source: hardening/stack-protector.c Compiler: csparcg1520 (SPARC) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    stack-protector.c
    /* @gallery-hints
 *   extra-flags: -fstack-protector-all
 *   compiler-exclude: vc_v19_44_VS17_14_x64, vc_v19_44_VS17_14_x86
 */

/*
 * Stack Protector (Stack Canary)
 *
 * With -fstack-protector-all the compiler inserts a canary value
 * between local variables and the saved return address. Before the
 * function returns, the canary is checked; if it was overwritten
 * (e.g. by a buffer overflow), the program aborts.
 *
 * Look for: canary load from %fs:0x28 (or %gs:0x14 on 32-bit),
 * comparison before the function epilogue, and a call to
 * __stack_chk_fail on mismatch.
 */
#include <string.h>

void copy_input(const char *input)
{
    char buf[64];
    strcpy(buf, input);   /* intentionally unsafe for demonstration */
    (void)buf;
}

    Explanation¶

    the key insight here is that this code demonstrates stack canary protection - a security mechanism that detects buffer overflows by placing a special "guard value" on the stack.

    The Core Security Mechanism¶

    The compiler has inserted canary checks at the beginning and end of the function. Think of a canary like a tripwire: if a buffer overflow corrupts the stack, it will overwrite this canary value, and the program will detect the tampering before returning.

    Step-by-Step Breakdown¶

    Function prologue (setup): 

    save    %sp, -168, %sp
    This creates a new stack frame with 168 bytes of space. In SPARC, the save instruction is like opening a new "window" of registers for this function. The negative value indicates how much stack space to allocate.

    Loading the canary (lines 2-3): 

    ld      [%g7+20], %g1      ; Load canary from secure location
st      %g1, [%fp-4]       ; Store it on the stack
    The canary value is loaded from %g7+20 (a thread-local storage location managed by the OS) and stored at [%fp-4] - just 4 bytes below the frame pointer. This places it right next to where the return address will be saved, so any buffer overflow that corrupts the return address will also corrupt the canary.

    Preparing the function call (lines 4-7): 

    mov     0, %g1             ; Clear %g1 (likely for alignment/convention)
mov     %i0, %o1           ; Copy input parameter to %o1 (second argument)
call    strcpy, 0          ; Call strcpy
 add    %fp, -72, %o0      ; Calculate buffer address: %fp - 72 → %o0 (first argument)
    The add instruction (executed in the delay slot before the call) calculates the address of the 64-byte buffer at %fp-72. The two arguments to strcpy are now set up: %o0 = buffer address, %o1 = input string pointer.

    Verifying the canary (lines 8-12): 

    ld      [%fp-4], %g2       ; Load the stored canary
ld      [%g7+20], %g1      ; Load the current canary from secure storage
xorcc   %g2, %g1, %g2      ; XOR them together and set condition codes
    The xorcc instruction compares the two canary values: if they're identical, XOR produces zero and sets the zero flag. If they differ (meaning overflow occurred), the result is non-zero.

    Conditional branch (lines 11-12): 

    be,pt   %icc, .L3          ; Branch if equal (canary intact) to .L3
 nop                        ; Delay slot
    be means "branch if equal" - if the canaries match, jump to .L3 (the normal return). The ,pt suffix is a branch prediction hint meaning "predict taken" (the compiler expects the canary to be valid most of the time).

    Failure path (lines 13-14): 

    call    __stack_chk_fail, 0
 nop
    If the canaries don't match, this calls the runtime function __stack_chk_fail, which terminates the program to prevent the corrupted return address from being used.

    Normal return (lines 15-17): 

    .L3:
return  %i7+8              ; Return to caller
 nop
    If the canary check passes, the function returns normally using %i7+8 (the return address plus 8 bytes, accounting for SPARC's delay slot).

    Key Observations¶

    • Canary placement: Stored at [%fp-4], positioned to catch overflows of the 64-byte buffer at [%fp-72]
    • Thread-local storage: The canary comes from %g7+20, which is OS-managed per-thread storage - this prevents attackers from predicting the canary value
    • XOR comparison: Using xorcc is an efficient way to compare values while setting condition codes in a single instruction
    • Minimal overhead: With `-
