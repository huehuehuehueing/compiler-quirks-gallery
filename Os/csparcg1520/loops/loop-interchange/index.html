    loop-interchange¶

    Details

    Source: loops/loop-interchange.c Compiler: csparcg1520 (SPARC) Scenario: Os - Size Optimization (-Os)

    Source Code¶

    loop-interchange.c
    /*
 * Loop Interchange
 *
 * When iterating over a 2D array, the order of the loop indices
 * determines memory access patterns.  In C (row-major layout),
 * iterating the inner loop over columns and the outer loop over
 * rows yields sequential access.  Swapping the loops ("interchange")
 * can improve cache locality when the original order would cause
 * stride-N accesses.
 *
 * Here the "bad" order (column-major traversal on a row-major array)
 * is presented; the compiler may interchange the loops at -O3.
 */
#define N 128

void column_sum(int mat[N][N], int result[N])
{
    /* Column-major traversal: inner loop strides by N ints.
     * The compiler may swap i and j for better locality. */
    for (int j = 0; j < N; j++)
    {
        result[j] = 0;
        for (int i = 0; i < N; i++)
        {
            result[j] += mat[i][j];
        }
    }
}

    Explanation¶

    the key insight is that this code implements a column-major traversal of a row-major array, which creates inefficient memory access patterns. Let me walk through what's happening.

    The Core Pattern: Nested Loops in Assembly¶

    This function has two nested loops from the C code: - Outer loop: iterates over columns (j from 0 to 128) - Inner loop: iterates over rows (i from 0 to 128)

    The assembly implements these loops using two key sections:

    Outer loop (.L6 label): - Manages the column index, incrementing by 4 bytes at a time (add %g1, 4, %g1) - Compares against 512 (which is 128 columns × 4 bytes per integer)

    Inner loop (.L2 label): - Accumulates values from the matrix - Increments by 1 each iteration (add %g2, 1, %g2) - Compares against 128

    Register Usage and Calling Convention¶

    SPARC uses a specific calling convention for parameter passing: - %o0: first parameter (the matrix mat) - %o1: second parameter (the result array) - %o7: return address (used at the end with jmp %o7+8)

    Working registers used: - %g1: outer loop counter (column index, in bytes) - %g2: inner loop counter (row index) - %g3: current address being read from the matrix - %g4: accumulator for the column sum - %o5: temporary for loading matrix values

    Step-by-Step Execution¶

    Initialization (before .L6): 

    mov     0, %g1              // g1 = 0 (column index starts at 0)
st      %g0, [%o1+%g1]      // result[0] = 0 (initialize first column sum)

    Outer loop body (.L6): 

    add     %o0, %g1, %g3       // g3 = address of mat[0][current_column]
mov     0, %g4              // g4 = 0 (reset accumulator for this column)
mov     0, %g2              // g2 = 0 (reset inner loop counter)

    Inner loop (.L2): 

    ld      [%g3], %o5          // Load mat[i][j] into o5
add     %g4, %o5, %g4       // Add to accumulator
st      %g4, [%o1+%g1]      // Store running sum back to result[j]
add     %g2, 1, %g2         // Increment row counter
cmp     %g2, 128            // Check if done with this column
bne,pt  %icc, .L2           // Branch back if not done
 add    %g3, 512, %g3       // (delay slot) Move to next row: add 512 bytes

    The add %g3, 512, %g3 in the delay slot is crucial—it adds 512 bytes (the width of one row in a 128×128 integer array) to move down to the next row in the same column.

    Outer loop continuation: 

    add     %g1, 4, %g1         // Move to next column (4 bytes = 1 integer)
cmp     %g1, 512            // Check if all 128 columns done
bne,a,pt %icc, .L6          // Branch back if not done
 st     %g0, [%o1+%g1]      // (delay slot, annulled) Initialize next result

    Why This Pattern Is Inefficient¶

    The inner loop increments %g3 by 512 bytes each iteration. This means: - First iteration reads from column 0, row 0 - Second iteration reads from column 0, row 1 (512 bytes away) - Third iteration reads from column 0, row 2 (another 512 bytes away)
