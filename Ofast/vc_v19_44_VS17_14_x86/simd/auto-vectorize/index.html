    auto-vectorize¶

    Details

    Source: simd/auto-vectorize.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    auto-vectorize.c
    /*
 * Auto-vectorization: loops converted to SIMD operations.
 *
 * Compiler transforms scalar loops into vector operations
 * (SSE, AVX, NEON, etc.) processing multiple elements at once.
 */

/* Simple loop - prime candidate for vectorization */
void add_arrays(float *dst, const float *a, const float *b, int n)
{
    for (int i = 0; i < n; i++) {
        dst[i] = a[i] + b[i];
    }
    /* With -O3: becomes SIMD adds processing 4-8 floats at once */
}

/* Reduction - trickier to vectorize but often done */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++) {
        sum += arr[i];
    }
    return sum;
    /* May use horizontal add instructions or partial sums */
}

/* Loop with dependency - CANNOT be vectorized */
void fibonacci_array(int *arr, int n)
{
    arr[0] = 0;
    arr[1] = 1;
    for (int i = 2; i < n; i++) {
        /* Each iteration depends on previous two - serial */
        arr[i] = arr[i-1] + arr[i-2];
    }
}

/* Conditional in loop - may use masked operations */
void clamp_array(float *arr, int n, float min, float max)
{
    for (int i = 0; i < n; i++) {
        if (arr[i] < min) arr[i] = min;
        else if (arr[i] > max) arr[i] = max;
    }
}

    Explanation¶

    the key insight is that despite the -Ofast optimization flag, this compiler (MSVC 32-bit) did NOT apply vectorization to any of these loops. Each function processes data element-by-element using scalar (single-value) operations instead of SIMD (vectorized) operations.

    add_arrays Function¶

    This function should be the easiest to vectorize, yet it processes one float at a time:

    movss   xmm0, DWORD PTR [eax+edx*4]    ; Load one float from array a
addss   xmm0, DWORD PTR [esi+ecx*4]    ; Add one float from array b
movss   DWORD PTR [eax+edx*4], xmm0    ; Store result to dst

    The movss instruction (move scalar single-precision) loads a single 32-bit float value into the xmm0 register. The addss instruction (add scalar single-precision) adds just one float. This repeats for every element in the loop. A vectorized version would use movaps (move aligned packed) and addps (add packed single-precision) to process 4 floats simultaneously, cutting the loop iterations by 75%.

    The loop structure itself is straightforward: initialize i to 0, jump to the condition check, increment i by 1 each iteration, and branch back until i >= n.

    sum_array Function¶

    This reduction loop also stays scalar:

    movss   xmm0, DWORD PTR _sum$[ebp]     ; Load current sum
addss   xmm0, DWORD PTR [eax+edx*4]    ; Add one array element
movss   DWORD PTR _sum$[ebp], xmm0     ; Store sum back to stack

    Each iteration loads the accumulated sum from the stack, adds one array element, and stores it back. A vectorized reduction would maintain multiple partial sums in parallel (one in each lane of a vector register), then combine them at the end with horizontal add operations.

    Notice the return path uses fld (floating-point load) to move the result into the x87 floating-point stack for returning to the callerthis is the 32-bit calling convention for returning floats.

    fibonacci_array Function¶

    This function correctly remains scalar because the loop has a data dependency that prevents vectorization: each element depends on the two previous elements:

    mov     eax, DWORD PTR [edx+ecx*4-4]   ; Load arr[i-1]
add     eax, DWORD PTR [edx+ecx*4-8]   ; Add arr[i-2]
mov     DWORD PTR [edx+ecx*4], eax     ; Store to arr[i]

    You cannot process multiple Fibonacci values in parallel because computing arr[5] requires arr[4] and arr[3], which haven't been computed yet. The compiler correctly recognizes this and doesn't attempt vectorization.

    clamp_array Function¶

    This function has conditional branches inside the loop, which complicates vectorization:

    comiss  xmm0, DWORD PTR [eax+edx*4]    ; Compare min with arr[i]
jbe     SHORT $LN5@clamp_arra          ; Jump if arr[i] >= min
movss   DWORD PTR [edx+ecx*4], xmm0    ; Store min value

    The comiss instruction (compare ordered scalar single-precision) compares two floats and sets CPU flags. The jbe (jump if below or equal) branches based on the comparison result. A vectorized version would use masked operations (like blendps) to conditionally update elements without branching, but this compiler didn't apply that optimization.
