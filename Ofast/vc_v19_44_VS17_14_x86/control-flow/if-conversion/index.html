    if-conversion¶

    Details

    Source: control-flow/if-conversion.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    if-conversion.c
    /*
 * If-Conversion (branch to conditional move)
 *
 * Short if-else chains that select between two values can be
 * replaced with conditional-move instructions (cmov on x86).
 * This eliminates a branch, avoiding potential misprediction
 * penalties on modern super-scalar CPUs.
 *
 * Look for: cmovl / cmovge / cmovne instead of jl / jge / jne
 * with a branch over a mov.
 */

int min(int a, int b)
{
    if (a < b)
        return a;
    return b;
}

int clamp(int x, int lo, int hi)
{
    if (x < lo)
        x = lo;
    if (x > hi)
        x = hi;
    return x;
}

int abs_val(int x)
{
    return x < 0 ? -x : x;
}

    Explanation¶

    the key insight is that despite the -Ofast optimization flag, this compiler is NOT using conditional-move instructions (cmov) as the source code comments suggest it should. Instead, it's using traditional branch instructions (jge, jle, jmp), which is surprising for an optimized build.

    Understanding the Core Pattern¶

    All three functions follow the same structure: 1. Set up a stack frame with push ebp and mov ebp, esp (this creates a reference point for accessing function parameters) 2. Compare values using cmp (which sets CPU flags based on the comparison) 3. Branch conditionally using jge (jump if greater-or-equal) or jle (jump if less-or-equal) 4. Execute different code paths depending on the branch 5. Clean up and return with pop ebp and ret

    This is the traditional "branch-based" approach to if-else logic.

    The min Function¶

    mov     eax, DWORD PTR _a$[ebp]      ; Load parameter 'a' into eax (return register)
cmp     eax, DWORD PTR _b$[ebp]      ; Compare a with b
jge     SHORT $LN2@min               ; If a >= b, jump to $LN2@min
mov     eax, DWORD PTR _a$[ebp]      ; Otherwise, keep a in eax (already there)
jmp     SHORT $LN1@min               ; Jump to end
$LN2@min:
mov     eax, DWORD PTR _b$[ebp]      ; Load b into eax
$LN1@min:
pop     ebp
ret     0

    What's happening: The function loads a into eax (the return register by convention). If a >= b, it jumps to load b instead. Otherwise, a stays in eax. Either way, the correct minimum value ends up in eax for return.

    The clamp Function¶

    This function has two if-statements, so it uses two separate branch sequences:

    mov     eax, DWORD PTR _x$[ebp]      ; Load x
cmp     eax, DWORD PTR _lo$[ebp]     ; Compare x with lo
jge     SHORT $LN2@clamp             ; If x >= lo, skip the first adjustment
mov     ecx, DWORD PTR _lo$[ebp]     ; Load lo into ecx
mov     DWORD PTR _x$[ebp], ecx      ; Store lo back to x's stack location
$LN2@clamp:
mov     edx, DWORD PTR _x$[ebp]      ; Load (possibly updated) x into edx
cmp     edx, DWORD PTR _hi$[ebp]     ; Compare x with hi
jle     SHORT $LN3@clamp             ; If x <= hi, skip the second adjustment
mov     eax, DWORD PTR _hi$[ebp]     ; Load hi into eax
mov     DWORD PTR _x$[ebp], eax      ; Store hi back to x
$LN3@clamp:
mov     eax, DWORD PTR _x$[ebp]      ; Load final x into eax for return

    Key observation: The compiler stores x on the stack and repeatedly loads/stores it. This is inefficientan optimized version would keep the value in a register throughout.
