    overflow-check-removed¶

    Details

    Source: security/overflow-check-removed.c Compiler: vc_v19_44_VS17_14_x86 (x86/x64 (Windows)) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    overflow-check-removed.c
    /*
 * Security issue: integer overflow check removed.
 *
 * Signed integer overflow is undefined behavior in C.
 * Compiler may assume it never happens and remove checks.
 */
#include <limits.h>

int safe_add(int a, int b)
{
    /* Attempt to check for overflow before it happens.
     * With optimization, this check may be REMOVED because
     * the compiler assumes signed overflow is impossible.
     */
    if (a > 0 && b > 0 && a > INT_MAX - b)
        return -1;  /* overflow would occur */

    if (a < 0 && b < 0 && a < INT_MIN - b)
        return -1;  /* underflow would occur */

    return a + b;
}

    Explanation¶

    the key insight is that the compiler has kept all the overflow checks intact despite using -Ofast optimization. This is important because it shows the compiler respects the explicit logic you wrote, even though signed integer overflow is technically undefined behavior in C.

    Function Setup and Parameter Access¶

    The function starts with standard stack frame setup: - push ebp and mov ebp, esp establish a stack frame so the function can access its parameters - Parameters a and b are accessed via [ebp+8] and [ebp+12] respectively (this is the x86 calling convention for 32-bit integers)

    First Overflow Check: Positive Numbers¶

    The first section checks if adding two positive numbers would overflow:

    cmp DWORD PTR _a$[ebp], 0
jle SHORT $LN2@safe_add          ; if a <= 0, skip this check
cmp DWORD PTR _b$[ebp], 0
jle SHORT $LN2@safe_add          ; if b <= 0, skip this check

    If both a and b are positive, the code continues: 

    mov eax, 2147483647              ; load INT_MAX (0x7fffffff)
sub eax, DWORD PTR _b$[ebp]      ; compute INT_MAX - b
cmp DWORD PTR _a$[ebp], eax      ; check if a > INT_MAX - b
jle SHORT $LN2@safe_add          ; if not, skip overflow return
or eax, -1                        ; set eax to -1 (overflow detected)
jmp SHORT $LN1@safe_add          ; return -1

    This implements the condition a > INT_MAX - b by computing the threshold and comparing against it.

    Second Underflow Check: Negative Numbers¶

    The second section (starting at $LN2@safe_add) checks if adding two negative numbers would underflow:

    cmp DWORD PTR _a$[ebp], 0
jge SHORT $LN3@safe_add          ; if a >= 0, skip this check
cmp DWORD PTR _b$[ebp], 0
jge SHORT $LN3@safe_add          ; if b >= 0, skip this check

    If both are negative: 

    mov ecx, -2147483648             ; load INT_MIN (0x80000000)
sub ecx, DWORD PTR _b$[ebp]      ; compute INT_MIN - b
cmp DWORD PTR _a$[ebp], ecx      ; check if a < INT_MIN - b
jge SHORT $LN3@safe_add          ; if not, skip underflow return
or eax, -1                        ; set eax to -1 (underflow detected)
jmp SHORT $LN1@safe_add          ; return -1

    This mirrors the first check but for negative numbers.

    Safe Addition Path¶

    If neither overflow nor underflow is detected, execution reaches $LN3@safe_add: 

    mov eax, DWORD PTR _a$[ebp]
add eax, DWORD PTR _b$[ebp]

    The result is computed and placed in eax (the return value register by convention).

    Return¶

    Both paths converge at $LN1@safe_add: 

    pop ebp
ret 0

    The function restores the stack frame and returns, with the result in eax.

    Key Observations¶

    • All checks are preserved: Despite -Ofast, the compiler kept your overflow detection logic. The compiler doesn't assume overflow is impossible here because the checks happen before the addition that could overflow.
    • Register usage: eax and ecx are used for temporary calculations; ebp manages the stack frame to access parameters.
    • Control flow: The code uses conditional jumps (jle, jge) to skip sections, creating three possible paths through the function depending on the input values.
