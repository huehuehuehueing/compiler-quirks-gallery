    inline-expansion¶

    Details

    Source: control-flow/inline-expansion.c Compiler: armv8-clang2110 (ARM) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    inline-expansion.c
    /*
 * Function inlining: small functions are expanded at call site.
 *
 * Eliminates call overhead and enables further optimizations
 * by exposing the function body to the caller's context.
 */

/* Tiny function - almost certainly inlined */
static inline int square(int x)
{
    return x * x;
}

/* Using static helps inlining (internal linkage) */
static int cube(int x)
{
    return x * x * x;
}

/* Wrapper that uses inlined functions */
int compute(int x)
{
    /* After inlining, this becomes:
     * return (x * x) + (x * x * x);
     * Which may further simplify to:
     * return x * x * (1 + x);
     */
    return square(x) + cube(x);
}

/* Recursive functions are NOT inlined (infinite expansion) */
int fibonacci(int n)
{
    if (n <= 1) return n;
    return fibonacci(n-1) + fibonacci(n-2);
}

/* Large functions may not be inlined (code size cost) */
int large_function(int x)
{
    int result = 0;
    for (int i = 0; i < 100; i++) {
        result += x * i;
        result ^= (result >> 3);
        result += (result << 5);
    }
    return result;
}

    Explanation¶

    the key insight here is function inlining - how the compiler decides which functions to expand inline versus keep as separate function calls.

    compute(int) - Aggressive Inlining¶

    mul     w8, w0, w0          // w8 = x * x (square)
madd    w0, w8, w0, w8      // w0 = w8*w0 + w8 = (x*x)*x + (x*x)
ret

    The key pattern: Both square() and cube() functions have been completely inlined and fused into a single expression. The compiler recognized that: - square(x) returns x * x - cube(x) returns x * x * x - Their sum is (x*x) + (x*x*x) = x*x * (1 + x)

    The madd instruction (multiply-add) is a single CPU operation that computes w8*w0 + w8 efficiently. This eliminates all function call overhead and produces just 2 instructions total.

    fibonacci(int) - Recursive Calls NOT Inlined¶

    stp     x29, x30, [sp, #-32]!    // Save frame pointer and return address
stp     x20, x19, [sp, #16]      // Save caller-saved registers
mov     x29, sp                  // Set up new frame pointer
cmp     w0, #2                   // Compare input with 2
b.ge    .LBB1_2                  // If >= 2, go to recursive case

    Why no inlining here: Recursive functions cannot be inlined because the compiler would need infinite copies of the function body. Instead, the compiler keeps this as a proper function with: - Stack frame setup using stp (store pair) to save registers that will be modified - Calling convention compliance - registers x19, x20 are caller-saved, so they're preserved across recursive calls - Base case optimization - the cmp w0, #2 and branch check if n < 2 (the base case), allowing early return without recursion

    The recursive calls at .LBB1_3 use bl fibonacci(int) (branch with link), which is a proper function call that saves the return address.

    large_function(int) - Loop Stays in Place¶

    mov     w9, wzr              // w9 = 0 (accumulator for x*i)
mov     w8, wzr              // w8 = 0 (result)
mov     w10, #100            // w10 = 100 (loop counter)
.LBB2_1:
    add     w8, w8, w9       // result += x*i
    subs    w10, w10, #1     // Decrement counter, set flags
    add     w9, w9, w0       // x*i += x (increment by x each iteration)
    eor     w8, w8, w8, asr #3    // result ^= (result >> 3)
    add     w8, w8, w8, lsl #5    // result += (result << 5)
    b.ne    .LBB2_1          // Loop while counter != 0

    Why not inlined: This function contains a 100-iteration loop with complex operations. Inlining would expand the code 100 times, bloating the caller's code size dramatically. The compiler keeps it as a loop because: - The loop body repeats, so keeping it compact saves code space - The function is "large" - inlining would violate the compiler's heuristics for code size vs. performance trade-offs

    The loop uses subs (subtract and set flags) to decrement the counter and check for zero in one instruction, which is efficient for tight loops.

    Key Observations¶

    Inlining decisions depend on: 1. Function size - square() and cube() are tiny (1-2 operations), so they're inlined 2. Call frequency - compute() calls them once each, making inlining worthwhile 3. Recursion - fibonacci() cannot be inlined (infinite expansion problem)
