    switch-table¶

    Details

    Source: control-flow/switch-table.c Compiler: armv8-clang2110 (ARM) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    switch-table.c
    /*
 * Switch optimization: jump tables vs. binary search vs. if-else.
 *
 * Compiler chooses implementation based on case distribution:
 * - Dense cases: jump table (O(1))
 * - Sparse cases: binary search (O(log n)) or if-else chain
 */

/* Dense cases - likely becomes a jump table */
int day_of_week(int day)
{
    switch (day) {
        case 0: return 'S';  /* Sunday */
        case 1: return 'M';
        case 2: return 'T';
        case 3: return 'W';
        case 4: return 'T';
        case 5: return 'F';
        case 6: return 'S';  /* Saturday */
        default: return '?';
    }
}

/* Sparse cases - likely if-else or binary search */
int sparse_switch(int x)
{
    switch (x) {
        case 1:     return 10;
        case 100:   return 20;
        case 10000: return 30;
        case 999999: return 40;
        default:    return 0;
    }
}

/* Two cases - becomes simple comparison */
int binary_choice(int x)
{
    switch (x) {
        case 0: return -1;
        case 1: return 1;
        default: return 0;
    }
}

    Explanation¶

    the key insight is that the compiler uses three different strategies for switch statements depending on how the cases are distributed.

    day_of_week(int) - Jump Table Strategy¶

    This function uses a jump table because the cases are dense (0-6 with no gaps):

    cmp     w0, #6              // Compare input with 6
b.hi    .LBB0_2             // If input > 6, branch to default case
adrp    x8, .Lswitch.table.day_of_week(int)  // Load table address (high bits)
add     x8, x8, :lo12:.Lswitch.table.day_of_week(int)  // Add low bits
ldr     w0, [x8, w0, uxtw #2]  // Load result from table at offset
ret

    How it works: Instead of comparing the input against each case value, the compiler creates a lookup table containing all the return values (83='S', 77='M', 84='T', etc.). The input becomes an index into this table. The [x8, w0, uxtw #2] means "load from address x8 plus (input 4 bytes)" the #2 shift is because each entry is 4 bytes wide.

    Why this is fast: One comparison, one memory load, done. This is O(1) constant time regardless of how many cases exist.

    sparse_switch(int) - Binary Search Strategy¶

    This function uses a series of comparisons because the cases are sparse (1, 100, 10000, 999999):

    mov     w8, #9999          // Load 9999 into w8
cmp     w0, w8              // Compare input with 9999
b.gt    .LBB1_4             // If input > 9999, jump to check larger values
cmp     w0, #1              // Check if input == 1
b.eq    .LBB1_8             // If yes, return 10
cmp     w0, #100            // Check if input == 100
b.ne    .LBB1_7             // If no, go to default
mov     w0, #20             // Return 20
ret

    The compiler splits the cases into two groups using a pivot value (9999). If the input is 9999, it checks cases 1 and 100. If the input is >9999, it checks 10000 and 999999. This is a binary search tree approach it eliminates half the remaining cases with each comparison.

    Why not a jump table here? A jump table would need 999,999 entries (one for each possible index), wasting enormous amounts of memory. The binary search approach uses only a few comparisons instead.

    binary_choice(int) - Conditional Move Strategy¶

    This function uses conditional instructions instead of branches:

    cmp     w0, #1              // Compare input with 1
cset    w8, eq              // Set w8 to 1 if equal, 0 otherwise
cmp     w0, #0              // Compare input with 0
csinv   w0, w8, wzr, ne     // Conditional select: if not equal, use w8; else use inverted wzr (-1)
ret

    How it works: - cset w8, eq sets w8 to 1 if the previous comparison was equal, 0 otherwise - csinv w0, w8, wzr, ne is a conditional select: "if the condition is not-equal, select w8; otherwise select the bitwise inverse of wzr (which is -1)"

    Why this approach? With only two cases, the compiler avoids branch instructions entirely. Branches can be slow because they disrupt the CPU's instruction pipeline. Using conditional moves (cset, csinv) keeps the pipeline flowing smoothly the CPU executes both paths' setup work, then selects the correct result at the end.

    Key Observations¶

    1. Jump tables (dense cases): O(1) lookup, minimal code, but wastes memory if cases are sparse
    2. Binary search (sparse cases): O(log n) comparisons, memory-efficient for scattered values
    3. **Conditional
