    loop-invariant-motion¶

    Details

    Source: loops/loop-invariant-motion.c Compiler: armv8-clang2110 (ARM) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    loop-invariant-motion.c
    /*
 * Loop-Invariant Code Motion (LICM)
 *
 * The optimizer recognises computations inside a loop whose result
 * does not change across iterations and moves ("hoists") them before
 * the loop.  This avoids redundant work on every iteration.
 *
 * In the example below, `base * scale` is invariant with respect to
 * the loop index i, so the compiler should compute it once and reuse
 * the result inside the loop body.
 */

void fill_scaled(int *arr, int n, int base, int scale)
{
    for (int i = 0; i < n; i++)
    {
        /* base * scale is loop-invariant: it does not depend on i. */
        arr[i] = base * scale + i;
    }
}

    Explanation¶

    the key insight is that the compiler has successfully applied Loop-Invariant Code Motion (LICM) and vectorization to optimize this loop.

    The Core Optimization: LICM in Action¶

    The most important thing to understand is what happens at the very beginning:

    mul     w8, w3, w2

    This single multiplication of base * scale happens once before the loop starts, not repeatedly inside the loop. This is LICMâ€”the compiler recognized that base * scale produces the same result every iteration, so it computed it once and stored the result in register w8 for reuse. Without this optimization, the multiplication would happen 8+ times unnecessarily.

    Register Usage and Calling Convention¶

    The function receives four parameters following the ARM64 calling convention: - w0 = pointer to array (arr) - w1 = loop count (n) - w2 = base - w3 = scale

    The w prefix indicates 32-bit registers (the lower 32 bits of 64-bit x registers). After the multiplication, w8 holds the invariant value base * scale.

    Two-Path Strategy: Vectorized and Scalar Loops¶

    The code splits into two execution paths based on whether there are enough elements to vectorize:

    Path 1: Vectorized Loop (8+ elements) 

    cmp     w1, #8
b.hs    .LBB0_3

    If n >= 8, the code jumps to .LBB0_3 and uses SIMD instructions (Single Instruction, Multiple Data). These vector registers (v0, v1, etc.) process 4 integers at a time, so 8 elements get processed in 2 iterations instead of 8.

    Key vectorization setup: - dup v0.4s, w8 â€” duplicates the invariant value across 4 lanes of a vector register - ldr q3, [x11, :lo12:.LCPI0_0] â€” loads the constant vector [0, 1, 2, 3] from memory - The loop body computes base*scale + [0,1,2,3], then base*scale + [4,5,6,7], storing 8 values per iteration

    Path 2: Scalar Loop (remaining elements) 

    .LBB0_7:
    subs    x9, x9, #1
    str     w8, [x11], #4
    add     w8, w8, #1
    b.ne    .LBB0_7

    After the vectorized loop finishes, any remaining elements (0-7) are handled with a simple scalar loop that stores one value per iteration.

    Why This Matters¶

    Without LICM, the assembly would recompute base * scale inside the loop thousands of times. With LICM, it's computed once. Combined with vectorization (processing 4 elements per instruction), this code is dramatically faster than a naive implementation.

    The -Ofast flag enabled these aggressive optimizations, including the use of SIMD instructions that a lower optimization level wouldn't generate.
