    overflow-check-removed¶

    Details

    Source: security/overflow-check-removed.c Compiler: armv8-clang2110 (ARM) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    overflow-check-removed.c
    /*
 * Security issue: integer overflow check removed.
 *
 * Signed integer overflow is undefined behavior in C.
 * Compiler may assume it never happens and remove checks.
 */
#include <limits.h>

int safe_add(int a, int b)
{
    /* Attempt to check for overflow before it happens.
     * With optimization, this check may be REMOVED because
     * the compiler assumes signed overflow is impossible.
     */
    if (a > 0 && b > 0 && a > INT_MAX - b)
        return -1;  /* overflow would occur */

    if (a < 0 && b < 0 && a < INT_MIN - b)
        return -1;  /* underflow would occur */

    return a + b;
}

    Explanation¶

    the key insight is that the compiler has aggressively optimized away most of the overflow checks, leaving only partial validation logic.

    The Core Issue: Undefined Behavior Optimization¶

    The C code attempts to check for signed integer overflow before it happens. However, signed integer overflow is undefined behavior in C—the language standard says it's not allowed to happen. The compiler, compiled with -Ofast (aggressive optimization), assumes overflow will never occur and removes most of the safety checks accordingly. What remains is a heavily transformed version of the original logic.

    Register and Calling Convention Setup¶

    Before diving into the logic, understand the calling convention: - w0 and w1 are the first two 32-bit integer parameters (the a and b arguments) - w8 and w9 are temporary registers the function can freely use - The function returns its result in w0

    The Positive Overflow Check (Lines 1-8)¶

    cmp     w0, #1              // Compare a with 1
b.lt    .LBB0_4             // If a < 1, skip to underflow check
cmp     w1, #1              // Compare b with 1
b.lt    .LBB0_4             // If b < 1, skip to underflow check
eor     w8, w1, #0x7fffffff // w8 = b XOR 0x7fffffff (INT_MAX)
cmp     w0, w8              // Compare a with (b XOR INT_MAX)
b.ls    .LBB0_4             // If a <= (b XOR INT_MAX), skip overflow return
mov     w0, #-1             // Return -1 (overflow detected)
ret

    This section checks if both a and b are positive and would overflow. The eor (exclusive OR) instruction with 0x7fffffff (INT_MAX) is a clever bit manipulation: it computes INT_MAX - b without using subtraction. If a > INT_MAX - b, overflow would occur, so the function returns -1.

    The Negative Underflow Check (Lines 10-17)¶

    .LBB0_4:
mov     w8, #-2147483648   // w8 = INT_MIN (-2147483648)
and     w9, w1, w0         // w9 = a AND b (both negative check)
sub     w8, w8, w1         // w8 = INT_MIN - b
cmp     w0, w8             // Compare a with (INT_MIN - b)
add     w8, w1, w0         // w8 = a + b (the actual sum)
ccmp    w9, #0, #0, lt     // Conditional compare: if a < INT_MIN - b, check if w9 < 0
csinv   w0, w8, wzr, ge    // If both negative and underflow: return -1, else return sum
ret

    This section handles negative underflow. It checks if both a and b are negative (w9 = a AND b will be negative if both are) and if a < INT_MIN - b (underflow condition). The ccmp (conditional compare) instruction only performs the comparison if the previous condition was true. Finally, csinv (conditional select inverted) returns either the sum (w8) or -1 depending on whether underflow occurred.

    Key Observations¶

    The compiler kept some checks but optimized them heavily. The positive overflow path uses bit manipulation instead of subtraction, and the negative underflow path uses conditional instructions (ccmp, csinv) to avoid branching. These are sophisticated optimizations that reduce the number of branches—branches are expensive on modern processors because they can disrupt instruction execution pipelines.

    However, the compiler's assumption about undefined behavior is risky. While the assembly technically implements the overflow logic, the compiler's aggressive optimizations elsewhere in the program (based on the assumption that overflow never happens) could still cause problems in real-world usage.
