    stack-protector¶

    Details

    Source: hardening/stack-protector.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    stack-protector.c
    /* @gallery-hints
 *   extra-flags: -fstack-protector-all
 *   compiler-exclude: vc_v19_44_VS17_14_x64, vc_v19_44_VS17_14_x86
 */

/*
 * Stack Protector (Stack Canary)
 *
 * With -fstack-protector-all the compiler inserts a canary value
 * between local variables and the saved return address. Before the
 * function returns, the canary is checked; if it was overwritten
 * (e.g. by a buffer overflow), the program aborts.
 *
 * Look for: canary load from %fs:0x28 (or %gs:0x14 on 32-bit),
 * comparison before the function epilogue, and a call to
 * __stack_chk_fail on mismatch.
 */
#include <string.h>

void copy_input(const char *input)
{
    char buf[64];
    strcpy(buf, input);   /* intentionally unsafe for demonstration */
    (void)buf;
}

    Explanation¶

    the key insight is that this code demonstrates stack canary protection - a security mechanism that detects buffer overflows by placing a sentinel value on the stack.

    Stack Frame Setup¶

    The function starts by allocating 120 bytes of stack space: 

    sub     rsp, 120

    This creates room for: - A 64-byte buffer (buf[64]) - Additional space for alignment and the canary value - The canary itself is stored at offset 104 from the stack pointer

    Loading the Canary¶

    mov     rcx, QWORD PTR .refptr.__stack_chk_guard[rip]
mov     rax, QWORD PTR [rcx]
mov     QWORD PTR 104[rsp], rax

    These three instructions load the canary value (a random 8-byte number) from a protected memory location and store it on the stack:

    1. mov rcx, ... - Load the address of the canary guard variable
    2. mov rax, [rcx] - Dereference that address to get the actual canary value
    3. mov [rsp+104], rax - Store the canary on the stack at a fixed offset

    Think of the canary as a "tripwire" - if a buffer overflow corrupts the stack, it will likely overwrite this value.

    Function Call Setup and Execution¶

    xor     eax, eax
lea     rcx, 32[rsp]
call    strcpy

    Before calling strcpy: - xor eax, eax clears the eax register (sets it to 0) - lea rcx, 32[rsp] calculates the address of the buffer (buf) - it's 32 bytes into the stack frame - call strcpy invokes the string copy function with the buffer address in rcx (following the calling convention where the first parameter goes in rcx)

    Canary Verification (The Security Check)¶

    mov     rdx, QWORD PTR .refptr.__stack_chk_guard[rip]
mov     rax, QWORD PTR 104[rsp]
sub     rax, QWORD PTR [rdx]
jne     .L4

    Before returning, the code verifies the canary wasn't corrupted:

    1. Load the current canary value from the protected location into rax
    2. Load the stored canary from the stack into rax (wait - this overwrites it, so actually: load the stored canary, then subtract the current canary)
    3. sub rax, [rdx] - Subtract the current canary from the stored one. If they match, the result is 0
    4. jne .L4 - Jump to .L4 (the error handler) if they're not equal (if the result is non-zero)

    Normal Return vs. Security Failure¶

    If the canary check passes (values match): 

    add     rsp, 120
ret
    The stack is cleaned up and the function returns normally.

    If the canary was overwritten: 

    .L4:
    call    __stack_chk_fail
    The program calls __stack_chk_fail, which terminates the program - preventing the attacker's code from executing.

    Key Observations¶

    • Security overhead: The -fstack-protector-all flag adds canary checks to every function, even simple ones. This has a small performance cost (a few extra memory operations per function call/return).
    • Placement strategy: The canary is positioned between local variables and the return address, so any buffer overflow that corrupts the return address will almost certainly corrupt the canary first.
    • Why this matters: Without this protection, an attacker could overflow buf[64] and overwrite the return address to redirect execution. The canary makes this attack detectable.
