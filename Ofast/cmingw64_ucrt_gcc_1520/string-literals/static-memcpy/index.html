    static-memcpy¶

    Details

    Source: string-literals/static-memcpy.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    static-memcpy.c
    #include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[5];

    memcpy(buf, "AAAA", 4);
    buf[sizeof(buf) - 1] = '\0';

    /* make sure the above is not treated as dead code */
    puts(buf);
}

    Explanation¶

    the key insight here is that the compiler has completely eliminated the memcpy call and replaced it with direct memory writes - this is an aggressive optimization that's possible because the compiler knows exactly what data is being copied at compile time.

    Stack Setup and Buffer Allocation¶

    sub     rsp, 56

    The function starts by subtracting 56 from rsp (the stack pointer register). This allocates 56 bytes of stack space for local variables. Your buf array only needs 5 bytes, but the compiler reserves extra space for alignment and safety - this is standard practice on 64-bit systems where stack alignment matters for performance and compatibility.

    The Optimization: Compile-Time Data Substitution¶

    Instead of calling memcpy to copy "AAAA" into the buffer, the compiler has done something clever:

    mov     DWORD PTR 43[rsp], 1094795585

    This single instruction writes 4 bytes directly to memory at offset 43 from rsp. The value 1094795585 is the hexadecimal representation of the ASCII string "AAAA" stored as a 32-bit integer (in little-endian format: 0x41414141). The compiler recognized that copying a constant string is unnecessary overhead - it can just write the bytes directly.

    lea     rcx, 43[rsp]

    This loads the address of the buffer into rcx. The lea instruction (Load Effective Address) calculates an address without accessing memory - it's just arithmetic. By convention, rcx is the first parameter register on Windows 64-bit calling conventions, so this sets up the pointer argument for puts.

    Null Termination¶

    mov     BYTE PTR 47[rsp], 0

    This writes a single zero byte (null terminator) at offset 47, which is exactly where buf[4] would be (offset 43 + 4 bytes). This satisfies the requirement that buf[sizeof(buf) - 1] = '\0'.

    Function Call and Cleanup¶

    call    puts
nop
add     rsp, 56
ret

    The call instruction invokes puts with the buffer pointer in rcx. The nop (no operation) is padding - likely added for alignment or code size reasons. Finally, add rsp, 56 deallocates the stack space, and ret returns to the caller.

    Why This Optimization Matters¶

    The compiler eliminated an entire function call (memcpy) by recognizing that: 1. The source data is a compile-time constant 2. The destination size is known 3. The operation can be replaced with direct memory writes

    This is the -Ofast optimization level at work - it's aggressive about replacing function calls with inline operations when it's safe to do so. An unoptimized version would actually call memcpy, which would be significantly slower.
