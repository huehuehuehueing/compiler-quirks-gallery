    overflow-check-removed¶

    Details

    Source: security/overflow-check-removed.c Compiler: cmingw64_ucrt_gcc_1520 (x86/x64 (Windows)) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    overflow-check-removed.c
    /*
 * Security issue: integer overflow check removed.
 *
 * Signed integer overflow is undefined behavior in C.
 * Compiler may assume it never happens and remove checks.
 */
#include <limits.h>

int safe_add(int a, int b)
{
    /* Attempt to check for overflow before it happens.
     * With optimization, this check may be REMOVED because
     * the compiler assumes signed overflow is impossible.
     */
    if (a > 0 && b > 0 && a > INT_MAX - b)
        return -1;  /* overflow would occur */

    if (a < 0 && b < 0 && a < INT_MIN - b)
        return -1;  /* underflow would occur */

    return a + b;
}

    Explanation¶

    the key insight is that the compiler kept the overflow checks intact despite using -Ofast optimization. This is important because signed integer overflow is undefined behavior in C, and the compiler could theoretically remove these checks entirely.

    Register Usage and Calling Convention¶

    The function uses the x86-64 calling convention where: - ecx holds the first parameter (a) - edx holds the second parameter (b) - eax is used for the return value

    Note: ecx and edx are 32-bit registers (the e prefix means "extended 32-bit"). When you use a 32-bit register, the upper 32 bits of the 64-bit register are automatically zeroed.

    Control Flow: Three Main Paths¶

    The function branches into three different paths based on the signs of a and b:

    Path 1: Both positive (lines 1-9) 

    test    ecx, ecx        // Check if a > 0
jle     .L2             // If a ≤ 0, jump to Path 2
test    edx, edx        // Check if b > 0
jle     .L2             // If b ≤ 0, jump to Path 2

    If both are positive, the code checks: a > INT_MAX - b (which prevents overflow when adding two positive numbers). It loads INT_MAX (2147483647), subtracts b, then compares with a. If the check fails (jl = jump if less), it returns -1 at .L6.

    Path 2: At least one is non-positive (lines 11-18) 

    test    ecx, edx        // Bitwise AND of a and b
jns     .L4             // Jump if result is non-negative (both positive)

    This clever instruction checks if both numbers are negative: test ecx, edx performs a bitwise AND. If the sign bit is set in both (meaning both are negative), the result is negative, so jns (jump if non-negative) doesn't execute. The code then checks: a < INT_MIN - b (which prevents underflow when adding two negative numbers).

    Path 3: Safe to add (lines 8-9 and 19-20) 

    lea     eax, [rcx+rdx]  // Load effective address: eax = a + b
ret

    The lea instruction (Load Effective Address) computes a + b without actually accessing memory—it's just arithmetic. This is the normal return path when no overflow/underflow would occur.

    Why the Checks Remain¶

    Even with -Ofast (aggressive optimization), the compiler kept these checks because: 1. The checks are explicit in the source code and affect the function's behavior 2. Removing them would change the function's observable behavior (returning -1 vs. overflowing) 3. The compiler respects the programmer's intent to handle edge cases

    The optimization here is in the instruction selection: using lea for addition and test with bitwise AND for the sign check are efficient choices that the optimizer made.
