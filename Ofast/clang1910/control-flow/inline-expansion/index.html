    inline-expansion¶

    Details

    Source: control-flow/inline-expansion.c Compiler: clang1910 (x86/x64) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    inline-expansion.c
    /*
 * Function inlining: small functions are expanded at call site.
 *
 * Eliminates call overhead and enables further optimizations
 * by exposing the function body to the caller's context.
 */

/* Tiny function - almost certainly inlined */
static inline int square(int x)
{
    return x * x;
}

/* Using static helps inlining (internal linkage) */
static int cube(int x)
{
    return x * x * x;
}

/* Wrapper that uses inlined functions */
int compute(int x)
{
    /* After inlining, this becomes:
     * return (x * x) + (x * x * x);
     * Which may further simplify to:
     * return x * x * (1 + x);
     */
    return square(x) + cube(x);
}

/* Recursive functions are NOT inlined (infinite expansion) */
int fibonacci(int n)
{
    if (n <= 1) return n;
    return fibonacci(n-1) + fibonacci(n-2);
}

/* Large functions may not be inlined (code size cost) */
int large_function(int x)
{
    int result = 0;
    for (int i = 0; i < 100; i++) {
        result += x * i;
        result ^= (result >> 3);
        result += (result << 5);
    }
    return result;
}

    Explanation¶

    the key insight is that function inlining eliminates function call overhead by expanding function bodies directly at the call site, and you can see three different outcomes here depending on function characteristics.

    compute(int) - Aggressive Inlining with Algebraic Simplification¶

    compute(int):
        lea     eax, [rdi + 1]
        imul    edi, edi
        imul    eax, edi
        ret

    This is the most dramatic optimization. The source code calls two functions: square(x) and cube(x). Here's what happened:

    1. Both functions were inlined - the compiler expanded square(x) (which computes x * x) and cube(x) (which computes x * x * x) directly into compute's body
    2. Algebraic simplification occurred - instead of computing (x * x) + (x * x * x) separately, the compiler recognized this equals x * x * (1 + x) and reorganized the computation

    Let me trace through the registers (by convention, rdi/edi holds the first integer parameter): - lea eax, [rdi + 1] - Load Effective Address computes x + 1 into eax (this is just arithmetic, not a memory access) - imul edi, edi - Multiply x by itself, storing x back in edi - imul eax, edi - Multiply (x + 1) by x, storing the result in eax - ret - Return with result in eax (by convention, return values go in eax)

    Why this works: Small functions like square and cube are perfect candidates for inlining because the function call overhead (pushing return address, jumping) costs more than the actual computation.

    fibonacci(int) - No Inlining (Recursive Function)¶

    fibonacci(int):
        push    r14
        push    rbx
        push    rax
        mov     r14d, edi
        xor     ebx, ebx
        cmp     edi, 2
        jge     .LBB1_2
        ...
        call    fibonacci(int)
        ...

    This function cannot be inlined because it's recursive (calls itself). If the compiler tried to inline it, the expansion would never terminateyou'd get infinite code. Instead, the compiler keeps it as a regular function with explicit call instructions.

    Notice the stack setup: - push r14, push rbx, push rax - Save registers that this function will use (these are "callee-saved" registers, meaning the function must preserve their original values for the caller) - The function maintains state across recursive calls using these preserved registers

    The function implements a tail-call optimization pattern where it unrolls the recursion slightly to reduce call overhead, but it still makes recursive calls for larger values.

    large_function(int) - Loop Unrolling (Not Inlined)¶

    large_function(int):
        lea     ecx, [rdi + 2*rdi]      // Precompute x*3
        lea     edx, [4*rdi]            // Precompute x*4
        lea     esi, [rdi + rdi]        // Precompute x*2
        xor     r8d, r8d                // Initialize loop counter
        mov     r9d, 100                // Loop count
        xor     eax, eax                // Initialize result
.LBB2_1:
        add     eax, r8d
        mov     r10d, eax
        sar     r10d, 3                 // Shift right by 3 (divide by 8)
        xor     r10d, eax               // XOR operation
        ...
        add     r8d, edx                // Increment by x*4
        add     r9d, -4                 // Decrement loop counter by 4
        jne     .LBB2_1
        ret

    This function was not inlined (it's too large), but the compiler applied loop unrolling: the original loop runs 100 times
