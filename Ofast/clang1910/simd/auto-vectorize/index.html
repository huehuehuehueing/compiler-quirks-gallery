    auto-vectorize¶

    Details

    Source: simd/auto-vectorize.c Compiler: clang1910 (x86/x64) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    auto-vectorize.c
    /*
 * Auto-vectorization: loops converted to SIMD operations.
 *
 * Compiler transforms scalar loops into vector operations
 * (SSE, AVX, NEON, etc.) processing multiple elements at once.
 */

/* Simple loop - prime candidate for vectorization */
void add_arrays(float *dst, const float *a, const float *b, int n)
{
    for (int i = 0; i < n; i++) {
        dst[i] = a[i] + b[i];
    }
    /* With -O3: becomes SIMD adds processing 4-8 floats at once */
}

/* Reduction - trickier to vectorize but often done */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++) {
        sum += arr[i];
    }
    return sum;
    /* May use horizontal add instructions or partial sums */
}

/* Loop with dependency - CANNOT be vectorized */
void fibonacci_array(int *arr, int n)
{
    arr[0] = 0;
    arr[1] = 1;
    for (int i = 2; i < n; i++) {
        /* Each iteration depends on previous two - serial */
        arr[i] = arr[i-1] + arr[i-2];
    }
}

/* Conditional in loop - may use masked operations */
void clamp_array(float *arr, int n, float min, float max)
{
    for (int i = 0; i < n; i++) {
        if (arr[i] < min) arr[i] = min;
        else if (arr[i] > max) arr[i] = max;
    }
}

    Explanation¶

    the key insight is that the compiler applied different optimization strategies to each function based on whether vectorization was possible.

    The Core Pattern: Vectorization Opportunities¶

    The -Ofast flag tells the compiler to prioritize speed, including aggressive auto-vectorization (converting scalar loops into SIMD operations that process multiple elements simultaneously). However, not all loops can be vectorizedit depends on data dependencies.

    Function-by-Function Analysis¶

    add_arrays: Vectorized (SIMD operations)¶

    The visible assembly shows the function uses vector instructions (the omitted 71 lines contain the SIMD loop). This makes sense because: - Each dst[i] = a[i] + b[i] is independentcomputing element 0 doesn't depend on element 1 - The compiler can load 4-8 floats at once into vector registers (like xmm0), add them in parallel, and store results back - This is the ideal vectorization candidate

    The control flow at the end (cmp r8, r9, jne .LBB0_10) handles the scalar cleanup loopafter processing elements in chunks of 4 or 8, any remaining elements (if n isn't evenly divisible) are handled one at a time.

    sum_array: Partially Vectorized with Scalar Fallback¶

    The visible code shows: 

    xorps   xmm0, xmm0          # Initialize vector register to 0.0
...
addss   xmm0, dword ptr [rdi + 4*rcx]  # Add single float
inc     rcx
cmp     rax, rcx
jne     .LBB1_7             # Loop back

    This is scalar code (single-element operations), not vectorized. Why? Reductions (summing all elements) are tricky to vectorize because: - You need to add element 0 to element 1, then add that result to element 2, etc. - This creates a dependency chaineach step depends on the previous result - The omitted 26 lines likely contain a vectorized version that processes multiple partial sums in parallel, then combines them at the end

    The addss instruction adds a single scalar float to xmm0. The loop counter increments by 1 each iteration, processing one element per loop cycle.

    fibonacci_array: Not Vectorized (Serial Loop)¶

    The visible code shows: 

    mov     dword ptr [rdx + 4*rsi], eax  # Store result
inc     rsi                            # Next index
cmp     rcx, rsi
jne     .LBB2_6                        # Loop back

    This is pure scalar codeone element per iteration. The Fibonacci sequence cannot be vectorized because: - arr[i] = arr[i-1] + arr[i-2] means each element depends on the two previous elements - You can't compute arr[5] until you've computed arr[4] and arr[3] - This is a fundamental data dependency that prevents parallel processing

    The compiler has no choice but to execute this serially.

    clamp_array: Vectorized with Masking¶

    The omitted 128 lines contain the complex logic. The visible code shows conditional branches (jne .LBB3_22), which suggests the compiler used masked vector operations: - Load multiple floats into a vector register - Compare all of them against min and max simultaneously - Use a mask to selectively update only the elements that need clamping - This avoids the branch misprediction penalty of checking each element individually

    Key Observations¶

    Vectorization requires independence: The compiler can only vectorize loops where iterations don't depend on each other. add_arrays , sum_array (partially) , fibonacci_array , clamp_array .

    Scalar cleanup is necessary: When array size isn't a multiple of the vector width (4 or 8 floats), the remaining elements are processed one at a time in a scalar loop.

    Register usage: xmm0, xmm1, etc. are vector registers (128-bit, holding 4 floats). The addss instruction operates on a single scalar float within a vector register.
