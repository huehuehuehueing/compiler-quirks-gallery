    stack-reuse¶

    Details

    Source: memory/stack-reuse.c Compiler: csparc64g1520 (SPARC64) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    stack-reuse.c
    /*
 * Stack slot reuse: non-overlapping locals share stack space.
 *
 * The compiler analyzes variable lifetimes and reuses
 * stack slots for variables that are never live at the same time.
 */
#include <stdio.h>

void example(int flag)
{
    if (flag) {
        /* This buffer is only live in this branch */
        char buffer_a[100];
        snprintf(buffer_a, sizeof(buffer_a), "Path A: %d", flag);
        printf("%s\n", buffer_a);
    } else {
        /* This buffer may reuse the same stack slot as buffer_a
         * since they're never both alive at the same time.
         */
        char buffer_b[100];
        snprintf(buffer_b, sizeof(buffer_b), "Path B: %d", flag);
        printf("%s\n", buffer_b);
    }

    /* Total stack usage may be ~100 bytes, not 200 */
}

    Explanation¶

    the key insight here is stack slot reuse - both buffers use the same memory location because they never exist at the same time.

    The Core Pattern: Single Stack Allocation for Two Buffers¶

    The function allocates only one 100-byte buffer on the stack, not two. Both buffer_a and buffer_b use the same stack location at offset 1943 from the frame pointer (%fp). This is possible because: - buffer_a only exists in the if branch - buffer_b only exists in the else branch - They're never both alive simultaneously

    This saves 100 bytes of stack space compared to allocating separate slots.

    Stack Frame Setup¶

    save    %sp, -288, %sp

    This instruction creates a new stack frame by: - Saving the caller's registers - Allocating 288 bytes of local stack space (which includes the 100-byte buffer plus space for other temporary values)

    The %sp register holds the stack pointer, and -288 is the negative offset (SPARC convention uses negative values for stack allocation).

    Control Flow: The Branch¶

    brz,pt  %i0, .L2
 sethi  %hi(.LC1), %o2

    This checks the flag parameter (passed in %i0 by calling convention): - brz = "branch if zero" - pt = "predict taken" (a hint that this branch is likely) - If flag is zero, jump to .L2 (the else path) - The sethi instruction in the delay slot (executed before the branch takes effect) begins loading the address of string .LC1 ("Path B: %d")

    Path A: The if Branch¶

    sethi   %hi(.LC0), %o2          // Load high bits of "Path A: %d" address
mov     %i0, %o3                // Copy flag to %o3 (format argument)
or      %o2, %lo(.LC0), %o2     // Complete the address with low bits
mov     100, %o1                // Buffer size = 100
call    snprintf, 0
 add    %fp, 1943, %o0          // Buffer address = fp + 1943

    This prepares arguments for snprintf: - %o0 = buffer address (the shared stack slot) - %o1 = size (100) - %o2 = format string address (".LC0") - %o3 = the flag value to format

    Then calls snprintf to write the formatted string into the buffer.

    call    puts, 0
 add    %fp, 1943, %o0          // Pass buffer address to puts

    Calls puts to print the buffer. The add instruction in the delay slot prepares the argument.

    Path B: The else Branch (.L2)¶

    mov     0, %o3                  // Set format argument to 0 (flag is 0)
or      %o2, %lo(.LC1), %o2     // Complete address of "Path B: %d"
mov     100, %o1                // Buffer size = 100
call    snprintf, 0
 add    %fp, 1943, %o0          // Same buffer address!

    Notice the identical buffer address (%fp + 1943). Both branches write to the same 100-byte stack slot. The only differences are: - Different format string (.LC1 instead of .LC0) - Different format argument (0 instead of flag)

    Return and Cleanup¶

    return  %i7+8
 nop

    This returns to the caller. The %i7 register holds the return address (by SPARC calling convention), and +8 accounts for the delay slot. The nop (no operation) fills the delay slot.

    Key Observations¶

    Stack efficiency: The compiler recognized that the two 100-byte buffers have non-overlapping lifetimes and allocated only one stack slot, reducing total stack usage from 200 bytes to 100 bytes.
