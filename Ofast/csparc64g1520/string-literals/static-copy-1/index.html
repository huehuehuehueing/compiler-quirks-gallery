    static-copy-1¶

    Details

    Source: string-literals/static-copy-1.c Compiler: csparc64g1520 (SPARC64) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    static-copy-1.c
    /*
 * String copy with strcpy - compiler may inline as immediate stores.
 *
 * For small, known-length strings, strcpy becomes direct
 * memory stores with the string data embedded in instructions.
 */
#include <string.h>
#include <stdio.h>

void foo(void)
{
    char buf[100];
    strcpy(buf, "ABCDXXXXCCCC");

    /* Make observable to prevent dead code elimination */
    puts(buf);
}

    Explanation¶

    the key insight here is that the compiler has inlined the string data directly into memory stores rather than calling strcpy. This is a powerful optimization for small, known-length strings.

    The Core Strategy: String Inlining¶

    Instead of calling the strcpy function to copy "ABCDXXXXCCCC" byte-by-byte, the compiler has embedded the string data as immediate values in instructions and stored them directly into the buffer. This eliminates the function call overhead entirely.

    Stack Frame Setup¶

    save    %sp, -288, %sp

    This instruction creates a new stack frame (a reserved area of memory for local variables). The -288 means the compiler allocates 288 bytes on the stack—enough space for the 100-byte buf array plus some extra space for alignment and other needs. The save instruction also saves the return address in register %i7 so the function can return later.

    Building and Storing the String Data¶

    The compiler constructs the string "ABCDXXXXCCCC" in pieces using registers %g1 and %g2:

    sethi   %hi(1094860800), %g1      // Load upper 32 bits of first part
or      %g1, 836, %g1             // Fill in lower bits
sllx    %g1, 32, %g1              // Shift left 32 bits (for 64-bit value)
add     %fp, 1943, %o0            // Calculate buffer address
add     %g1, %g2, %g1             // Combine the two parts
stx     %g1, [%fp+1943]           // Store 8 bytes to buffer

    This sequence builds a 64-bit value containing multiple characters and stores it as a single 8-byte write (stx = store extended/64-bit). This is much faster than writing one byte at a time.

    sethi   %hi(1128480768), %g1      // Load upper bits of remaining string
or      %g1, 835, %g1             // Fill in lower bits
st      %g1, [%fp+1951]           // Store 4 bytes to buffer

    The remaining characters are stored as a 4-byte value (st = store word).

    stb     %g0, [%fp+1955]           // Store null terminator (0 byte)

    Finally, a null terminator (%g0 is always zero in SPARC) is written to mark the end of the string.

    Function Call and Return¶

    add     %fp, 1943, %o0            // Put buffer address in %o0 (first parameter)
call    puts, 0                   // Call puts() function
 st     %g1, [%fp+1951]           // (delay slot - executes before call)

    By convention in SPARC, the first function parameter goes in register %o0. The buffer address is placed there, then puts() is called to print it. The st instruction in the delay slot (the line after call) executes before the actual function call—this is a SPARC quirk where one instruction always executes after a branch/call.

    return  %i7+8                      // Return to caller
 nop                               // (delay slot - no operation)

    The return instruction jumps back to the caller (address stored in %i7), with +8 accounting for the instruction size. The nop (no operation) in the delay slot does nothing.

    Why This Optimization Matters¶

    For a small string like "ABCDXXXXCCCC" (12 bytes), calling strcpy would involve: - Function call overhead (jumping to another function) - Loop logic to copy bytes one at a time - Checking for the null terminator

    Instead, the compiler directly writes the string in just 2-3 memory operations, making it significantly faster. This is called string inlining and is a common optimization for constant strings.
