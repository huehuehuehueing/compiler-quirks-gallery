    static-init-1¶

    Details

    Source: string-literals/static-init-1.c Compiler: csparc64g1520 (SPARC64) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    static-init-1.c
    /*
 * Static string initialization with concatenation.
 *
 * Adjacent string literals are concatenated at compile time.
 * The initializer may become a single memcpy or inline stores.
 */
#include <string.h>
#include <stdio.h>

void foo(void)
{
    /* Adjacent strings concatenated at compile time: "AAAABBBB" */
    char buf[100] = { "AAAA" "BBBB" };

    /* Make observable to prevent dead code elimination */
    puts(buf);
}

    Explanation¶

    the key insight here is how the compiler handles string initialization and buffer setup.

    The Core Pattern: String Initialization via Direct Memory Writes¶

    The most important thing to understand is that the compiler has pre-computed the concatenated string "AAAABBBB" at compile time and is writing it directly into memory as binary data, rather than doing any string operations at runtime.

    Stack Frame Setup¶

    save    %sp, -288, %sp

    This instruction creates a new stack frame for the function. The save instruction is SPARC's way of setting up a function's local workspace. It allocates 288 bytes of stack space (the negative value indicates space allocation). This large allocation is needed because buf[100] requires 100 bytes, plus additional space for alignment and other local variables.

    Building the String Data¶

    The next section constructs the 8-byte string "AAAABBBB" in registers before writing it to memory:

    sethi   %hi(1094795264), %g1      // Load upper bits of "AAAA"
sethi   %hi(1111638016), %g2      // Load upper bits of "BBBB"
or      %g1, 321, %g1             // Complete "AAAA" in g1
or      %g2, 578, %g2             // Complete "BBBB" in g2
sllx    %g1, 32, %g1              // Shift "AAAA" left 32 bits
add     %g1, %g2, %g1             // Combine: "AAAABBBB" now in g1

    The compiler uses a two-step process: sethi (set high bits) loads the upper 22 bits of a value, then or fills in the lower 10 bits. The sllx (shift left extended) shifts the "AAAA" portion 32 bits to the left, positioning it in the upper half of a 64-bit register. Finally, add combines both parts into a single 64-bit value containing the full concatenated string.

    Clearing the Rest of the Buffer¶

    stx     %g0, [%fp+1951]    // Write 8 zero bytes
stx     %g0, [%fp+1959]    // Write 8 zero bytes
... (multiple more zero writes)

    The stx instruction stores an 8-byte (64-bit) value. %g0 is the zero register—it always contains 0. The compiler clears the remaining 92 bytes of the 100-byte buffer by writing zeros at 8-byte intervals. This initializes the rest of the buffer to null bytes, which is necessary for the string to be properly null-terminated.

    Passing the Buffer to puts()¶

    add     %fp, 1943, %o0     // Calculate buffer address
call    puts, 0            // Call puts function

    The add instruction computes the address of the buffer by adding an offset (1943) to the frame pointer (%fp). By SPARC calling convention, the first argument to a function goes in register %o0, so the buffer address is placed there. Then call invokes the puts() function.

    Function Return¶

    return  %i7+8              // Return to caller
 nop                       // Delay slot (no operation)

    The return instruction jumps back to the caller. The nop (no operation) in the delay slot is a SPARC quirk—branch instructions execute one more instruction before the jump takes effect, so a nop is placed there as a placeholder.

    Key Observations¶

    • String concatenation happens at compile time: The compiler merged "AAAA" and "BBBB" into a single 8-byte constant before any code runs.
    • Direct memory initialization: Rather than using memcpy or character-by-character assignment, the compiler writes the pre-computed string directly to the stack.
    • Efficient 64-bit writes: The compiler uses 8-byte stores (stx) to clear the buffer quickly, rather than byte-by-byte operations.
    • The -Ofast optimization level enables this aggressive approach—the compiler assumes it can safely initialize the entire buffer this way.
