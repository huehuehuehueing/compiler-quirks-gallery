    fast-math¶

    Details

    Source: arithmetic/fast-math.c Compiler: csparc64g1520 (SPARC64) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    fast-math.c
    /*
 * Fast-math optimizations: -ffast-math / -Ofast effects.
 *
 * These flags allow the compiler to reorder floating-point
 * operations, assuming associativity (which floats don't have).
 * Can change results or break special value handling.
 */

/* With -ffast-math, this may be reordered and give different results */
float sum_array(const float *arr, int n)
{
    float sum = 0.0f;
    for (int i = 0; i < n; i++)
        sum += arr[i];
    return sum;
}

/* NaN check may be optimized away with -ffast-math */
int is_nan(float x)
{
    /* With -ffast-math, compiler assumes no NaN values exist,
     * so this comparison may be optimized to always return 0.
     */
    return x != x;
}

/* Division by variable may become multiplication by reciprocal */
float divide_by_y(float x, float y)
{
    /* With -ffast-math: may become x * (1.0f / y)
     * Less accurate but faster on some architectures.
     */
    return x / y;
}

/* Algebraic simplification (inexact with floats!) */
float algebraic(float a, float b, float c)
{
    /* With -ffast-math: (a + b) + c may be reordered to a + (b + c)
     * These can give different results with floating point!
     */
    return (a + b) + c;
}

    Explanation¶

    the key insight is that -Ofast enables aggressive floating-point optimizations that can change program behavior, and you can see this directly in the generated code.

    The Most Important Pattern: Aggressive Optimizations¶

    With -Ofast, the compiler assumes floating-point operations follow ideal mathematical rules (they don't), allowing it to make simplifications that can produce different results than standard code. Let's see this in action:

    is_nan Function: Optimization Away a Safety Check¶

    is_nan:
        jmp     %o7+8
         mov    0, %o0

    This is striking: the function immediately returns 0 without any computation. The source code checks x != x (a NaN detection trick—NaN is the only value not equal to itself), but -ffast-math tells the compiler "assume no NaN values exist," so the entire check is eliminated. The compiler simply returns the constant 0.

    Why this matters: If your code actually receives a NaN, this function will incorrectly report it as "not NaN." This is the trade-off of -ffast-math—speed over correctness.

    divide_by_y Function: Direct Division (Not Reciprocal Optimization)¶

    divide_by_y:
        jmp     %o7+8
         fdivs  %f1, %f3, %f0

    Interestingly, this still uses fdivs (floating-point divide) rather than converting to multiplication by reciprocal. The parameters arrive in %f1 (x) and %f3 (y) by SPARC calling convention, and the result goes to %f0 for return. The jmp %o7+8 returns to the caller (in SPARC, %o7 holds the return address).

    algebraic Function: Reordered Operations¶

    algebraic:
        fadds   %f3, %f5, %f5
        jmp     %o7+8
         fadds  %f5, %f1, %f0

    This computes (b + c) + a instead of (a + b) + c. The parameters arrive as: - %f1 = a - %f3 = b
    - %f5 = c

    The code first adds b + c (storing in %f5), then adds that result to a. With -ffast-math, the compiler assumes this reordering doesn't matter, but floating-point addition is not associative—different orderings can produce different results due to rounding.

    sum_array Function: Loop with Accumulation¶

    This is the most complex function:

            cmp     %o1, 0              // Compare n with 0
        ble,pn  %icc, .L4           // If n <= 0, jump to .L4 (return 0)
         srl    %o1, 0, %o1         // Zero-extend n to 64-bit (delay slot)
        fzeros  %f0                 // Initialize sum to 0.0
        sllx    %o1, 2, %o1         // Multiply n by 4 (convert count to byte offset)
        add     %o0, %o1, %g1       // Calculate end address: arr + n*4
.L3:
        ld      [%o0], %f8          // Load arr[i] into %f8
        add     %o0, 4, %o0         // Move to next element (4 bytes)
        cmp     %g1, %o0            // Compare end address with current position
        bne,pt  %xcc, .L3           // If not equal, loop back
         fadds  %f0, %f8, %f0       // Add to sum (delay slot)
.L4:
        jmp     %o7+8               // Return
         fzeros %f0                 // Return 0.0 if n <= 0

    Key observations:

    1. Early exit for empty arrays: If n <= 0, it jumps to .L4 and returns 0.0 via fzeros %f0.

    2. Loop structure: The loop uses a pointer-based approach
