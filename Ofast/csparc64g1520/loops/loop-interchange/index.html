    loop-interchange¶

    Details

    Source: loops/loop-interchange.c Compiler: csparc64g1520 (SPARC64) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    loop-interchange.c
    /*
 * Loop Interchange
 *
 * When iterating over a 2D array, the order of the loop indices
 * determines memory access patterns.  In C (row-major layout),
 * iterating the inner loop over columns and the outer loop over
 * rows yields sequential access.  Swapping the loops ("interchange")
 * can improve cache locality when the original order would cause
 * stride-N accesses.
 *
 * Here the "bad" order (column-major traversal on a row-major array)
 * is presented; the compiler may interchange the loops at -O3.
 */
#define N 128

void column_sum(int mat[N][N], int result[N])
{
    /* Column-major traversal: inner loop strides by N ints.
     * The compiler may swap i and j for better locality. */
    for (int j = 0; j < N; j++)
    {
        result[j] = 0;
        for (int i = 0; i < N; i++)
        {
            result[j] += mat[i][j];
        }
    }
}

    Explanation¶

    the key insight is that the compiler has not interchanged the loops despite -Ofast optimization. The code still performs column-major traversal (striding through memory by 512 bytes at a time), which is inefficient for cache locality.

    Setup and Constants¶

    sethi   %hi(65536), %g5
mov     0, %g4
add     %o0, %g5, %g5

    The function receives two parameters: mat (the 2D array) in %o0 and result (the output array) in %o1. These are standard SPARC calling convention registers for the first two arguments.

    The code calculates 65536 (which is 0x10000 in hex) and adds it to %o0. Since the matrix is 128×128 integers (512 bytes per row), 65536 bytes represents exactly 128 rows × 512 bytes. This value becomes the loop boundary stored in %g5 — it marks where the inner loop should stop.

    %g4 is initialized to 0 and will track the column index (multiplied by 4 for byte offset, since integers are 4 bytes).

    Outer Loop: Processing Each Column¶

    .L3:
    st      %g0, [%o1+%g4]      // result[j] = 0
    add     %o0, %g4, %g1       // g1 = &mat[0][j]
    mov     0, %g2              // g2 = sum accumulator

    The outer loop (.L3) processes each column. It: 1. Initializes result[j] to zero by storing %g0 (the zero register) 2. Calculates the address of the first element in column j by adding the column offset %g4 to the base address %o0 3. Resets the accumulator %g2 to zero for summing this column

    Inner Loop: Summing a Column¶

    .L2:
    lduw    [%g1], %g3          // Load mat[i][j]
    add     %g1, 512, %g1       // Move to next row (512 bytes = 128 ints)
    cmp     %g1, %g5            // Check if we've reached the end
    add     %g2, %g3, %g2       // Accumulate: sum += mat[i][j]
    bne,pt  %xcc, .L2           // Branch if not equal (continue loop)
     st     %g2, [%o1+%g4]      // Delay slot: store updated sum

    The inner loop (.L2) is where the inefficiency becomes visible:

    • lduw [%g1], %g3: Load an unsigned word (32-bit integer) from memory at address %g1
    • add %g1, 512, %g1: Increment the address by 512 bytes — this jumps to the next row in the same column (stride-N access pattern)
    • cmp %g1, %g5: Compare the current address against the boundary to check if we've processed all 128 rows
    • add %g2, %g3, %g2: Add the loaded value to the running sum
    • bne,pt %xcc, .L2: Branch back if not done (the ,pt hint tells the CPU this branch is likely taken)

    The delay slot instruction (st %g2, [%o1+%g4]) executes after the branch decision is made — it stores the accumulated sum back to memory on each iteration. This is inefficient because the sum is written repeatedly; ideally it would only be written once after the loop completes.
