    inline-expansion¶

    Details

    Source: control-flow/inline-expansion.c Compiler: csparc64g1520 (SPARC64) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    inline-expansion.c
    /*
 * Function inlining: small functions are expanded at call site.
 *
 * Eliminates call overhead and enables further optimizations
 * by exposing the function body to the caller's context.
 */

/* Tiny function - almost certainly inlined */
static inline int square(int x)
{
    return x * x;
}

/* Using static helps inlining (internal linkage) */
static int cube(int x)
{
    return x * x * x;
}

/* Wrapper that uses inlined functions */
int compute(int x)
{
    /* After inlining, this becomes:
     * return (x * x) + (x * x * x);
     * Which may further simplify to:
     * return x * x * (1 + x);
     */
    return square(x) + cube(x);
}

/* Recursive functions are NOT inlined (infinite expansion) */
int fibonacci(int n)
{
    if (n <= 1) return n;
    return fibonacci(n-1) + fibonacci(n-2);
}

/* Large functions may not be inlined (code size cost) */
int large_function(int x)
{
    int result = 0;
    for (int i = 0; i < 100; i++) {
        result += x * i;
        result ^= (result >> 3);
        result += (result << 5);
    }
    return result;
}

    The Big Picture: Function Inlining vs. Recursion¶

    The most important insight here is contrast: the compute function is tiny and gets completely inlined (no function call overhead), while fibonacci is recursive and cannot be inlined, resulting in massive, complex code. The large_function is also not inlined due to its size.

    compute Function: Inlining in Action¶

    compute:
        smul    %o0, %o0, %g1      # x * x  g1 (square)
        smul    %g1, %o0, %o0      # (x*x) * x  o0 (cube)
        add     %o0, %g1, %o0      # cube + square  o0
        jmp     %o7+8              # return
         sra    %o0, 0, %o0        # sign-extend result (delay slot)

    What's happening:

    The compiler inlined both square(x) and cube(x) directly into compute. Instead of making two function calls, it:

    1. Line 1: Computes x * x using smul (signed multiply), storing in %g1
    2. Line 2: Multiplies that result by x again to get x, storing in %o0
    3. Line 3: Adds the square and cube together
    4. Line 4-5: Returns via jmp %o7+8 (jump to return address)

    Register conventions in SPARC: - %o0 is the input parameter (by calling convention) and also the return value register - %g1 is a temporary register (global, caller-saved) - %o7 holds the return address - The sra %o0, 0, %o0 in the delay slot sign-extends the result (a common SPARC pattern)

    Why inlining matters: Without inlining, this would require two call instructions plus stack setup. Inlining eliminates all that overhead and lets the compiler see the full computation (x*x + x*x*x), which could theoretically be further optimized to x*x*(1+x).

    fibonacci Function: Recursive Complexity¶

    This is where things get dramatically different. The function is recursive (calls itself), so it cannot be inlined. The compiler instead uses aggressive loop unrolling and optimization techniques to handle the exponential recursion more efficiently.

    Key observations:

    1. Stack frame setup: save %sp, -256, %sp allocates a 256-byte stack frame for local variables and spill space
    2. Massive code size: The function spans hundreds of instructions with multiple nested loops (.L5, .L11, .L15, .L19, .L23, .L27, .L35, .L37)
    3. Heavy register usage: The compiler uses almost every available register (%i0-%i5, %g1-%g5, %l0-%l7, %o0-%o7) to avoid memory access
    4. Stack spilling: Multiple st (store) and lduw (load unsigned word) instructions show the compiler is saving/restoring values to the stack frame because it ran out of registers

    The core recursion pattern (simplified): 

    cmp     %i0, 1          # if (n <= 1)
ble,pn  %icc, .L6       #   return n
...
call    fibonacci, 0    # fibonacci(n-1)
...
call    fibonacci, 0    # fibonacci(n-2)
...
add     %i5, %o0, %i5   # sum the results

    The pn suffix on branches means "predict not taken" the compiler expects the base case to be rare, optimizing for the recursive path.

    Why this is so complex: Recursive fibonacci has exponential time complexity. The compiler tries to optimize by unrolling loops and reusing computations, but there's a fundamental limit: the algorithm itself is inefficient. No amount of assembly optimization can fix an O(2^n) algorithm.
