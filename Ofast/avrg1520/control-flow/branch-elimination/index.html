    branch-elimination¶

    Details

    Source: control-flow/branch-elimination.c Compiler: avrg1520 (AVR) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    branch-elimination.c
    /*
 * Branch elimination: compile-time constant conditions.
 *
 * When a branch condition is known at compile time,
 * the dead branch is eliminated entirely.
 */
#include <limits.h>

/* Debug flag - in release builds, debug code is eliminated */
#define DEBUG 0

void maybe_log(const char *msg)
{
    if (DEBUG) {
        /* This entire block is removed when DEBUG is 0 */
        extern void log_message(const char *);
        log_message(msg);
    }
}

/* Platform-specific code */
int get_pointer_size(void)
{
    if (sizeof(void *) == 8) {
        /* Only this branch exists on 64-bit platforms */
        return 64;
    } else {
        /* Only this branch exists on 32-bit platforms */
        return 32;
    }
}

/* Impossible conditions eliminated */
int check_range(int x)
{
    /* If x is already validated as positive elsewhere,
     * and optimizer knows it, this check may be eliminated.
     */
    if (x < 0)
        return -1;

    /* This is always false for signed int */
    if (x > INT_MAX)
        return -2;

    return x;
}

    Explanation¶

    the key insight here is dead code elimination - the compiler is removing entire branches that can never execute based on compile-time constants.

    The Core Pattern: Dead Code Elimination¶

    When the compiler can determine at compile time that a condition is always true or always false, it removes the unreachable code entirely. This is one of the most powerful optimizations because it reduces code size and improves performance by eliminating unnecessary branches.

    Function-by-Function Breakdown¶

    maybe_log() - Complete Elimination¶

    maybe_log(char const*):
    ret

    This function compiles to a single ret instruction. Here's why: the source code has if (DEBUG) where DEBUG is defined as 0. Since the compiler knows this condition is always false, it eliminates the entire if block (the call to log_message()). The function body becomes empty, leaving only the return instruction.

    Key insight: This is how debug code gets completely removed in release builds without runtime overhead.

    get_pointer_size() - Single Path Execution¶

    get_pointer_size():
    ldi r24,lo8(32)      ; Load 32 into r24 (low byte of return value)
    ldi r25,0            ; Load 0 into r25 (high byte of return value)
    ret

    The function returns a 16-bit integer (on AVR, int is 16 bits). By convention, return values use registers r24 and r25 (the lower 16 bits of the return register pair).

    The source code checks if (sizeof(void *) == 8), but on this AVR platform, pointers are 2 bytes (16 bits), not 8 bytes. The compiler knows this at compile time, so it: - Eliminates the if branch entirely - Keeps only the else branch (returning 32) - Loads the constant 32 directly into the return registers

    Why two instructions? AVR's ldi (Load Immediate) instruction can only load 8-bit values. To load a 16-bit value, the compiler loads the low byte into r24 and the high byte into r25.

    check_range() - Partial Elimination¶

    check_range(int):
    sbrs r25,7           ; Skip next instruction if bit 7 of r25 is set
    ret                  ; Return immediately if positive
    ldi r24,lo8(-1)      ; Load -1 into r24
    ldi r25,lo8(-1)      ; Load -1 into r25
    ret

    This function receives a 16-bit signed integer in r24:r25 (by calling convention). Here's what happens:

    1. sbrs r25,7 - "Skip if Bit in Register is Set". This checks bit 7 of r25 (the high byte), which is the sign bit for a signed 16-bit integer. If the sign bit is 0 (positive number), the next instruction is skipped.

    2. First ret - If the number is positive (sign bit = 0), return immediately with the original value still in r24:r25.

    3. ldi r24,lo8(-1) and ldi r25,lo8(-1) - If the number is negative (sign bit = 1), load -1 into the return registers and return.

    What happened to the other checks? The compiler eliminated them: - The if (x > INT_MAX) check is gone because a signed 16-bit integer can never exceed INT_MAX (by definition, INT_MAX is the largest value that fits in a signed 16-bit integer). This condition is always false. - The compiler kept only the negative check because that's the only condition that can actually occur.

    Why This Matters¶

    With -Ofast optimization, the compiler performs constant folding (evaluating constant expressions at compile time) and dead code elimination (removing unreachable branches). This results in: - Smaller code size - no unused branches - Faster execution - no unnecessary branches to predict or execute - Zero runtime overhead - debug code and platform-specific branches vanish entirely
