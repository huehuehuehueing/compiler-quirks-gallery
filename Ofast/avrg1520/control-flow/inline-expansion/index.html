    inline-expansion¶

    Details

    Source: control-flow/inline-expansion.c Compiler: avrg1520 (AVR) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    inline-expansion.c
    /*
 * Function inlining: small functions are expanded at call site.
 *
 * Eliminates call overhead and enables further optimizations
 * by exposing the function body to the caller's context.
 */

/* Tiny function - almost certainly inlined */
static inline int square(int x)
{
    return x * x;
}

/* Using static helps inlining (internal linkage) */
static int cube(int x)
{
    return x * x * x;
}

/* Wrapper that uses inlined functions */
int compute(int x)
{
    /* After inlining, this becomes:
     * return (x * x) + (x * x * x);
     * Which may further simplify to:
     * return x * x * (1 + x);
     */
    return square(x) + cube(x);
}

/* Recursive functions are NOT inlined (infinite expansion) */
int fibonacci(int n)
{
    if (n <= 1) return n;
    return fibonacci(n-1) + fibonacci(n-2);
}

/* Large functions may not be inlined (code size cost) */
int large_function(int x)
{
    int result = 0;
    for (int i = 0; i < 100; i++) {
        result += x * i;
        result ^= (result >> 3);
        result += (result << 5);
    }
    return result;
}

    The Core Pattern: Function Inlining in Action¶

    The most important insight here is that the compute() function shows successful inlining the small square() and cube() functions have been expanded directly into compute()'s code rather than being called separately. This eliminates the overhead of function calls.

    Understanding the compute() Function¶

    Looking at the visible portion of compute():

    mov r22,r30
mov r23,r31
rcall __mulhi3
add r24,r30
adc r25,r31
ret

    Here's what's happening step-by-step:

    1. mov r22,r30 and mov r23,r31: These copy values between registers. In AVR, 16-bit values (like int) span two 8-bit registers. The registers r30:r31 hold one value, and r22:r23 hold another.

    2. rcall __mulhi3: This calls a library function that multiplies two 16-bit integers. The __mulhi3 is a built-in helper function for multiplication. By convention, AVR passes parameters in specific registers and returns results in r24:r25.

    3. add r24,r30 and adc r25,r31: These add two 16-bit values together:

    4. add adds the low bytes (r24 + r30)
    5. adc (add with carry) adds the high bytes, accounting for any overflow from the low byte addition

    6. The result stays in r24:r25 (the return value register)

    7. ret: Returns from the function with the result in r24:r25.

    The inlining worked: instead of calling square() and cube() as separate functions, the compiler expanded their multiplication operations inline and combined them into a single addition.

    Why Inlining Matters¶

    Without inlining, each function call would require: - Pushing return addresses onto the stack - Jumping to the function - Jumping back - Cleaning up

    With inlining, the compiler eliminates all that overhead and can optimize the combined operations together.

    The fibonacci() Function: Why It's NOT Inlined¶

    The assembly shows fibonacci() is a full function with: - push r2 through push r6: Saving registers that the function will use - 623 lines of code (shown as omitted) - Corresponding pop instructions to restore those registers

    Recursive functions like fibonacci() cannot be inlined because inlining would require infinite expansion each recursive call would need another copy of the function body, which is impossible. The compiler must keep it as a callable function.

    The large_function(): Stack Frame Setup¶

    The visible code shows ldd r24,Y+9 this is loading a value from the stack frame using the Y register (which points to the stack). Large functions like large_function() create stack frames to store local variables (like the result and i variables in the loop). The compiler doesn't inline these because the code size cost would be too high.

    Key Observations¶

    • Inlining succeeded for small functions: square() and cube() were expanded into compute(), eliminating call overhead
    • AVR uses register pairs for 16-bit integers: Notice how operations use pairs like r24:r25 and r30:r31
    • Recursive functions stay as functions: fibonacci() must remain callable because recursion prevents inlining
    • Large functions use stack frames: The ldd instruction shows local variables are stored on the stack rather than in registers
