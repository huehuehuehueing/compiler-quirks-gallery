    overflow-check-removed¶

    Details

    Source: security/overflow-check-removed.c Compiler: mipsg1520 (MIPS) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    overflow-check-removed.c
    /*
 * Security issue: integer overflow check removed.
 *
 * Signed integer overflow is undefined behavior in C.
 * Compiler may assume it never happens and remove checks.
 */
#include <limits.h>

int safe_add(int a, int b)
{
    /* Attempt to check for overflow before it happens.
     * With optimization, this check may be REMOVED because
     * the compiler assumes signed overflow is impossible.
     */
    if (a > 0 && b > 0 && a > INT_MAX - b)
        return -1;  /* overflow would occur */

    if (a < 0 && b < 0 && a < INT_MIN - b)
        return -1;  /* underflow would occur */

    return a + b;
}

    Explanation¶

    the key insight is that the compiler kept the overflow checks intact despite using -Ofast optimization. This is important because signed integer overflow is undefined behavior in C, and the compiler could theoretically remove these checks—but it didn't.

    Register Conventions¶

    Before diving into the code, understand that MIPS uses specific registers by convention: - $4 and $5: parameter registers (hold function arguments a and b) - $2: return value register (where the result goes) - $31: return address register (used by jr $31 to return from the function)

    Control Flow Overview¶

    The function checks two conditions: 1. Positive overflow: both a and b are positive, and adding them would exceed INT_MAX 2. Negative underflow: both a and b are negative, and adding them would go below INT_MIN

    If either condition is true, return -1. Otherwise, return a + b.

    First Check: Positive Overflow (lines 1-11)¶

    blez    $4,$L2          # if a <= 0, skip to $L2
nop                     # (delay slot - no operation)

blez    $5,$L3          # if b <= 0, skip to $L3 (return a+b)
nop

    These branches skip the positive overflow check if either a or b is not positive. The nop (no operation) instructions fill the delay slot—a MIPS quirk where the instruction after a branch executes before the branch takes effect.

    li      $2,2147418112   # load INT_MAX into $2 (0x7fff0000)
ori     $2,$2,0xffff    # complete the 32-bit value (0x7fffffff = INT_MAX)
subu    $2,$2,$5        # $2 = INT_MAX - b
slt     $2,$2,$4        # $2 = 1 if (INT_MAX - b) < a, else 0
bne     $2,$0,$L11      # if overflow detected, jump to $L11 (return -1)

    This calculates whether a > INT_MAX - b (the overflow condition). If true, it jumps to $L11 which returns -1.

    Second Check: Negative Underflow (lines 18-28)¶

    $L2:
beq     $4,$0,$L3       # if a == 0, skip underflow check
nop

bgez    $5,$L3          # if b >= 0, skip underflow check
nop

    These skip the negative underflow check if a is zero or b is non-negative.

    li      $2,-2147483648  # load INT_MIN into $2 (0x80000000)
subu    $2,$2,$5        # $2 = INT_MIN - b
slt     $2,$4,$2        # $2 = 1 if a < (INT_MIN - b), else 0
beq     $2,$0,$L3       # if no underflow, jump to $L3 (return a+b)

    This checks whether a < INT_MIN - b (the underflow condition). If the condition is false (no underflow), it returns the sum.

    Return Paths¶

    $L3:
jr      $31             # return from function
addu    $2,$4,$5        # $2 = a + b (executes in delay slot)

    This is the normal return path: add a and b, then return.

    $L11:
jr      $31             # return from function
li      $2,-1           # $2 = -1 (executes in delay slot)

    This returns -1 when overflow is detected.

    Key Observation¶

    The compiler preserved all the overflow checks even with -Ofast. This is actually correct behavior—while signed overflow is undefined in C, the compiler still needs to execute the code as written. The checks remain because they're part of the control flow logic, not dead code that can be eliminated.
