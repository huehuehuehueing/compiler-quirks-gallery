    memcpy-inline¶

    Details

    Source: memory/memcpy-inline.c Compiler: mipsg1520 (MIPS) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    memcpy-inline.c
    /*
 * memcpy inlining: small copies become inline loads/stores.
 *
 * For small, known-size copies, the compiler generates
 * inline code instead of calling memcpy.
 */
#include <string.h>

struct small {
    int a, b;
};

void copy_8_bytes(void *dst, const void *src)
{
    /* Small copy - likely inlined as single 64-bit load/store
     * on 64-bit architectures, or two 32-bit ops on 32-bit.
     */
    memcpy(dst, src, 8);
}

void copy_struct(struct small *dst, const struct small *src)
{
    /* Structure copy - same as memcpy but type-aware.
     * Compiler knows alignment and size.
     */
    *dst = *src;
}

void copy_varying(void *dst, const void *src, int use_large)
{
    /* Compile-time unknown size - must call memcpy or
     * generate size-checking code.
     */
    if (use_large)
        memcpy(dst, src, 1024);
    else
        memcpy(dst, src, 8);
}

    Explanation¶

    the key insight is that the compiler is inlining small memory copies as direct load/store instructions instead of calling the memcpy function.

    Understanding the Core Pattern¶

    MIPS uses a special pair of instructions for unaligned memory access: lwl/lwr (Load Word Left/Right) and swl/swr (Store Word Left/Right). These work together to load or store a 32-bit word from potentially unaligned addresses:

    • lwl loads the left (most significant) bytes
    • lwr loads the right (least significant) bytes
    • Together they reconstruct a complete 32-bit value

    This is necessary on MIPS because the architecture doesn't support unaligned loads directlythe compiler must use this two-instruction pattern to safely handle any address alignment.

    Function 1: copy_8_bytes (8-byte copy)¶

    lwl     $3,0($5)      # Load left part of first word from src
lwl     $2,4($5)      # Load left part of second word from src
lwr     $3,3($5)      # Load right part of first word, complete $3
lwr     $2,7($5)      # Load right part of second word, complete $2
swl     $3,0($4)      # Store left part of first word to dst
swr     $3,3($4)      # Store right part of first word
swl     $2,4($4)      # Store left part of second word to dst
jr      $31           # Return to caller
swr     $2,7($4)      # Store right part of second word (delay slot)

    What's happening: The compiler inlined the 8-byte copy as four pairs of load/store operations. Register $5 holds the source pointer, $4 holds the destination pointer. Two 32-bit words are loaded into $3 and $2, then stored to the destination. The jr $31 (jump to return address) executes after the final swr due to MIPS's branch delay slotthe instruction after a branch executes before the branch takes effect.

    Function 2: copy_struct (8-byte struct copy)¶

    lw      $3,0($5)      # Load first 4 bytes from src
lw      $2,4($5)      # Load second 4 bytes from src
sw      $3,0($4)      # Store first 4 bytes to dst
jr      $31           # Return
sw      $2,4($4)      # Store second 4 bytes (delay slot)

    Key difference: This uses simple lw/sw (Load/Store Word) instead of the lwl/lwr pairs. The compiler knows the struct is properly aligned (both pointers are 4-byte aligned), so it can use the faster, simpler aligned load/store instructions. This is more efficient than the unaligned version.

    Function 3: copy_varying (conditional 8 or 1024-byte copy)¶

    This is more complex because the size is determined at runtime:

    Branch decision: 

    beq     $6,$0,$L5     # If use_large == 0, jump to $L5 (8-byte path)
addiu   $8,$5,1024    # Calculate end address (src + 1024)
