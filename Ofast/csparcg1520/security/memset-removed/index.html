    memset-removed¶

    Details

    Source: security/memset-removed.c Compiler: csparcg1520 (SPARC) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    memset-removed.c
    /*
 * Security issue: memset() to clear sensitive data is removed
 * as dead code when the buffer is not used afterward.
 *
 * This is a real security vulnerability pattern - passwords
 * and keys may remain in memory after "clearing".
 */
#include <string.h>
#include <stdio.h>

void process_password(const char *password)
{
    char local_copy[64];

    /* Copy password for processing */
    strncpy(local_copy, password, sizeof(local_copy) - 1);
    local_copy[sizeof(local_copy) - 1] = '\0';

    /* Do something observable with the password (simulate auth check) */
    if (local_copy[0] == 's' && local_copy[1] == 'e' &&
        local_copy[2] == 'c' && local_copy[3] == 'r' &&
        local_copy[4] == 'e' && local_copy[5] == 't') {
        puts("Access granted");
    } else {
        puts("Access denied");
    }

    /* Security: clear the password from stack
     * BUG: This memset is often REMOVED by optimizer
     * because local_copy is not used afterward!
     */
    memset(local_copy, 0, sizeof(local_copy));
}

void test_password(void)
{
    process_password("secret");
}

    Explanation¶

    the key insight here is that the memset() call to clear the password has been completely removed by the optimizer, which is the security vulnerability the code comments warn about.

    The Critical Security Issue¶

    Notice that the source code calls memset(local_copy, 0, sizeof(local_copy)) to clear the password from memory, but there are zero instructions in the assembly that perform this clearing. The -Ofast optimization level aggressively removed this call because the compiler determined that local_copy is never read after the memset(). This is a real security vulnerability: the password remains in the stack memory even though the code intended to erase it.

    Stack Frame Setup¶

    Both functions start with: 

    save    %sp, -160, %sp

    This is SPARC's function prologue. The save instruction creates a new stack frame by allocating 160 bytes of space. The -64 offsets you see later (like [%fp-64]) refer to the 64-byte buffer local_copy within this frame. The extra space accounts for register windows and alignment requirements on SPARC.

    Password Checking Logic¶

    The core logic checks if the password matches "secret" by examining individual characters:

    1. First character check ([%fp-64]): Load the first byte and compare to 115 (ASCII 's') 

      ldsb    [%fp-64], %g1      # Load signed byte from buffer
cmp     %g1, 115           # Compare to 's'
bne,pn  %icc, .L2          # Branch if not equal (predict not taken)

    2. Subsequent characters: Each character is loaded and compared in sequence:

    3. [%fp-63] 101 ('e')
    4. [%fp-62] 99 ('c')
    5. [%fp-61] 114 ('r')
    6. [%fp-60] 101 ('e')
    7. [%fp-59] 116 ('t')

    The ldsb instruction loads a signed byte (the 's' prefix means sign-extend), which is appropriate for character data. The cmp instruction sets condition flags, and branch instructions like bne (branch if not equal) and be (branch if equal) use those flags to control flow.

    Branch Prediction Hints¶

    Notice the branch annotations: - bne,pn "predict not taken" (expects the password to NOT match) - be,pn "predict not taken" (expects early exit) - bne,pt "predict taken" (expects to continue checking)

    These hints help the CPU's branch predictor optimize execution. The compiler expects most passwords won't match, so it predicts early exits.

    Function Call Convention¶

    When calling strncpy() and puts(): - %o0, %o1, %o2 are used for the first three arguments (output registers by SPARC convention) - The call instruction jumps to the function - The instruction after call (in the delay slot) executes before the jump takes effect this is why you see add %fp, -64, %o0 after the call instruction; it sets up the destination buffer address

    The restore instruction at the end of the function both returns from the function and restores the previous stack frame.

    The Missing memset()¶

    The entire 64-byte clearing operation is absent. In optimized code, this would normally be a loop or series of store instructions writing zeros to the buffer. The compiler eliminated it because it performed dead code elimination recognizing that nothing reads from local_copy after the memset() call, it removed the call entirely.
