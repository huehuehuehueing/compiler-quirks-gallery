    msvc-gs-canary¶

    Details

    Source: hardening/msvc-gs-canary.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    msvc-gs-canary.c
    /* @gallery-hints
 *   replace-flags: /O2 /GS
 *   compiler-only: vc_v19_44_VS17_14_x64, vc_v19_44_VS17_14_x86
 */

/*
 * MSVC Buffer Security Check (/GS)
 *
 * /GS is MSVC's stack canary, analogous to GCC's -fstack-protector.
 * The compiler places a copy of __security_cookie (XORed with RSP)
 * between local buffers and the saved return address.  Before the
 * function returns it calls __security_check_cookie; if the canary
 * was corrupted the process terminates.
 *
 * Compare with stack-protector.c (GCC/Clang):
 *   - GCC reads the canary from a thread-local slot (%fs:0x28).
 *   - MSVC reads a global __security_cookie and XORs it with RSP,
 *     making each frame's canary value unique.
 *
 * Look for: mov rax, __security_cookie / xor rax, rsp at the
 * prologue, and __security_check_cookie before the epilogue.
 */
#include <string.h>

void copy_input(const char *input)
{
    char buf[64];
    strcpy(buf, input);   /* intentionally unsafe for demonstration */
    (void)buf;
}

    Explanation¶

    the key insight here is that this code demonstrates MSVC's stack canary protection (/GS flag), which guards against buffer overflow attacks by placing a security token on the stack.

    The Core Protection Mechanism¶

    When you compile with /GS, the compiler automatically inserts a "canary" value (a security token) into the stack frame. If a buffer overflow corrupts this canary, the program detects it and terminates before the attacker can hijack the return address.

    Step-by-Step Breakdown¶

    Function prologue (setup): 

    sub     rsp, 120                      ; Allocate 120 bytes on stack
mov     rax, QWORD PTR __security_cookie  ; Load global security cookie
xor     rax, rsp                      ; XOR it with current stack pointer
mov     QWORD PTR __$ArrayPad$[rsp], rax  ; Store result as canary

    The prologue does three things: 1. Allocates stack space (sub rsp, 120) - this creates room for the 64-byte buffer plus padding and the canary itself 2. Loads the security cookie - __security_cookie is a global value that MSVC provides 3. Creates a unique canary - by XORing the cookie with rsp (the stack pointer), each function call gets a different canary value, even if the same function is called multiple times

    The canary is stored at offset __$ArrayPad$ on the stack, positioned after the local buffer so any overflow will hit it first.

    Function body: 

    mov     rdx, rcx                      ; Move input parameter to rdx
lea     rcx, QWORD PTR buf$[rsp]      ; Load address of buf into rcx
call    strcpy                        ; Call strcpy(buf, input)

    This sets up the strcpy call. By convention on x64 Windows, the first parameter goes in rcx and the second in rdx, so the compiler moves the input pointer to rdx and loads the buffer address into rcx.

    Function epilogue (verification): 

    mov     rcx, QWORD PTR __$ArrayPad$[rsp]  ; Reload the stored canary
xor     rcx, rsp                      ; XOR with current stack pointer again
call    __security_check_cookie       ; Verify canary is intact
add     rsp, 120                      ; Deallocate stack space
ret     0                             ; Return to caller

    Before returning, the code: 1. Reloads the canary from the stack 2. Recomputes the XOR with the current stack pointer (which should match the original if nothing was corrupted) 3. Calls the security check function - if the canary doesn't match, this function terminates the process 4. Cleans up and returns

    Key Observations¶

    • The XOR trick: By XORing with rsp, MSVC makes each frame's canary unique. This is different from GCC's approach, which uses a fixed thread-local value. The advantage is that even if an attacker knows the global cookie, they can't predict the canary without knowing the stack pointer.

    • Canary placement: The canary (__$ArrayPad$) is positioned after the buffer (buf$), so a buffer overflow will corrupt the canary before reaching the return address.

    • Optimization applied: Despite /O2 optimization, the compiler still generates all the canary code—security checks are never optimized away, even at high optimization levels.

    • Stack layout: The 120-byte allocation includes the 64-byte buffer, padding, the 8-byte canary, and space for the strcpy call's stack frame.
