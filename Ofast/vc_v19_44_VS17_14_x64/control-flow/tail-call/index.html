    tail-call¶

    Details

    Source: control-flow/tail-call.c Compiler: vc_v19_44_VS17_14_x64 (x86/x64 (Windows)) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    tail-call.c
    /*
 * Tail call optimization: recursive calls become jumps.
 *
 * When a function's last action is calling another function
 * (or itself), the call can become a jump, saving stack space.
 */

/* Tail-recursive factorial - can be optimized to a loop */
int factorial_tail(int n, int acc)
{
    if (n <= 1)
        return acc;
    /* Last action is the recursive call - tail position */
    return factorial_tail(n - 1, n * acc);
}

int factorial(int n)
{
    return factorial_tail(n, 1);
}

/* NOT tail recursive - multiplication after recursive call */
int factorial_not_tail(int n)
{
    if (n <= 1)
        return 1;
    /* Multiplication happens AFTER the call returns */
    return n * factorial_not_tail(n - 1);
}

/* Mutual tail recursion */
int is_even(unsigned n);
int is_odd(unsigned n);

int is_even(unsigned n)
{
    if (n == 0) return 1;
    return is_odd(n - 1);  /* Tail call */
}

int is_odd(unsigned n)
{
    if (n == 0) return 0;
    return is_even(n - 1);  /* Tail call */
}

    Explanation¶

    the key insight is that the compiler did NOT apply tail call optimization (TCO) here, even though the code is written to enable it. Let me walk through what's happening.

    The Core Issue: Tail Calls Remain as Calls¶

    In the factorial_tail function, the recursive call is in tail position (the last action before returning). With aggressive optimization (-Ofast), you might expect the compiler to convert this into a jmp instruction instead of a call. However, the assembly shows a regular call factorial_tail instruction, which means the stack frame is preserved and a return address is pushed.

    This is a limitation of the MSVC compiler on x64 it doesn't perform tail call optimization as aggressively as some other compilers (like GCC with -O2 or higher). The x64 calling convention and MSVC's implementation make TCO less automatic than on other architectures.

    How Each Function Works¶

    factorial_tail (lines with $LN4: label):

    The function receives two parameters: - ecx = n (the countdown value) - edx = acc (the accumulator)

    These are stored on the stack at offsets 48 and 56 respectively (following the x64 calling convention where the caller reserves 32 bytes of "shadow space" plus additional stack space).

    mov DWORD PTR [rsp+16], edx    ; Store acc parameter
mov DWORD PTR [rsp+8], ecx     ; Store n parameter
sub rsp, 40                     ; Allocate 40 bytes of stack space

    The function then checks if n <= 1: 

    cmp DWORD PTR n$[rsp], 1
jg SHORT $LN2@factorial_       ; Jump if n > 1

    If the base case is reached (n <= 1), it loads the accumulator and jumps to the exit: 

    mov eax, DWORD PTR acc$[rsp]   ; Return value = acc
jmp SHORT $LN1@factorial_      ; Jump to cleanup

    Otherwise, it computes the next recursive call: 

    mov eax, DWORD PTR n$[rsp]
imul eax, DWORD PTR acc$[rsp]  ; eax = n * acc (new accumulator)
mov ecx, DWORD PTR n$[rsp]
dec ecx                         ; ecx = n - 1
mov edx, eax                    ; edx = new accumulator (for next call)
call factorial_tail             ; Recursive call (NOT a tail jump!)

    After the call returns, it cleans up and returns: 

    add rsp, 40                     ; Deallocate stack space
ret 0

    factorial_not_tail (the non-tail-recursive version):

    This function demonstrates why TCO matters. After the recursive call returns, it performs multiplication: 

    call factorial_not_tail         ; Recursive call
mov ecx, DWORD PTR n$[rsp]     ; Load n again
imul ecx, eax                   ; Multiply n * result
mov eax, ecx                    ; Move result to return register

    This multiplication after the call means the stack frame must be preserved during the recursive call you can't convert this to a jump because you need to remember n for the multiplication.

    is_even and is_odd (mutual recursion):

    These functions demonstrate tail calls between different functions: 

    is_even: ... call is_odd        ; Tail call to is_odd
is_odd:  ... call is_even       ; Tail call to is_even

    Again, these are regular call instructions, not jumps. Each call preserves the stack frame.

    Key Observations¶

    • Stack frame overhead: Every function allocates 40 bytes of stack space (sub rsp, 40) and deallocates it on exit (add rsp, 40). With true TCO, recursive calls wouldn't need this cleanup.
    • Parameter passing: The x64 calling convention uses ecx and edx for the first two integer parameters, but MSVC stores them on the stack immediately, which adds overhead.
    • Why TCO didn't happen: MSVC's x64 implementation doesn
