    wrapv-vs-ubsan¶

    Details

    Source: security/wrapv-vs-ubsan.c Compiler: armv7-clang2110 (ARM) Scenario: Ofast - Fast Math Optimization (-Ofast)

    Source Code¶

    wrapv-vs-ubsan.c
    /* @gallery-hints
 *   extra-flags: -fwrapv
 */

/*
 * -fwrapv: well-defined signed integer overflow
 *
 * By default, signed integer overflow is undefined behaviour in C,
 * and compilers exploit this for optimisation (e.g. assuming i+1 > i).
 * With -fwrapv, signed overflow wraps around in two's complement,
 * which is well-defined.
 *
 * Compare the assembly with and without -fwrapv: the optimizer may
 * remove overflow checks or fold comparisons differently when
 * overflow is undefined.
 */

int will_overflow(int x)
{
    /* Without -fwrapv the compiler may assume this never wraps,
     * optimising the comparison to always-true. */
    if (x + 1 > x)
        return 1;
    return 0;
}

int abs_safe(int x)
{
    /* With well-defined wrapping, the compiler must handle INT_MIN. */
    if (x < 0)
        x = -x;
    return x;
}

    Explanation¶

    the key insight is that -fwrapv forces the compiler to handle signed integer overflow as well-defined behavior, which changes how these functions compile.

    will_overflow(int) Function¶

    The most important thing to understand here: the compiler is checking whether x + 1 > x by testing if adding 1 to x would overflow (wrap around in two's complement arithmetic).

    Here's how it works step-by-step:

    mvn     r1, #-2147483648
    This loads a special value into r1. The mvn instruction (Move Not) loads the bitwise NOT of the immediate value. Since -2147483648 is the minimum 32-bit signed integer (INT_MIN), this effectively loads INT_MAX (the maximum signed integer). Think of it as: "what's the largest value we can add to without overflowing?"

    subs    r0, r0, r1
    This subtracts r1 (which is INT_MAX) from r0 (the input x), and the s suffix means it sets the condition flags based on the result. This is a clever trick: if x > INT_MAX, the subtraction would underflow (wrap around), setting the "not equal" flag.

    movne   r0, #1
    If the "not equal" flag is set (meaning overflow occurred), move 1 into r0. Otherwise, r0 remains 0 from the subtraction.

    bx      lr
    Return to the caller (branch to link register).

    Why this approach? With -fwrapv, the compiler must actually check for overflow rather than assuming x + 1 > x is always true. This implementation avoids computing x + 1 directly and instead uses a mathematical equivalence to detect overflow.

    abs_safe(int) Function¶

    This function computes the absolute value while handling the edge case where x = INT_MIN (which cannot be negated safely).

    cmp     r0, #0
    Compare the input r0 with zero, setting condition flags.

    rsbmi   r0, r0, #0
    This is the key instruction: rsb means "Reverse Subtract" and mi means "execute if minus" (if the previous comparison showed a negative number). So this says: "if r0 is negative, compute 0 - r0" (which is the negation). The mi suffix makes this conditional—it only executes when r0 < 0.

    bx      lr
    Return to the caller.

    Why this works with -fwrapv: When x = INT_MIN, negating it would overflow (since INT_MIN has no positive counterpart in two's complement). With -fwrapv, this overflow wraps around and produces INT_MIN again—which is technically correct behavior under the wrapping semantics, even though it's not the mathematical absolute value. The compiler accepts this because overflow is now well-defined.
